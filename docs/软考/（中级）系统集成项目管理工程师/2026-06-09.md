## 今日核心概念

### 1. 中间件与构件技术

定义：中间件是位于操作系统之上、应用程序之下的一层软件，屏蔽底层异构性，为上层应用提供统一的标准接口和服务。构件是可复用的、独立部署的软件单元，具有明确的接口和契约。

关键要点：
- 中间件分类：消息中间件（MQ）、交易中间件（OTM）、对象中间件（CORBA/EJB）、数据库中间件（ODBC/JDBC）
- 构件特征：可复用性、独立性、封装性、可组装性
- 构件与中间件关系：中间件为构件提供运行环境和通信机制，构件在中间件之上组装
- 企业应用集成（EAI）：表示集成、数据集成、控制集成、业务流程集成、功能集成，层次由浅到深
- 常见考点：中间件分类对应典型产品（WebLogic/JBoss→对象中间件，RabbitMQ→消息中间件），EAI集成层次的深度判断

### 2. 项目管理办公室（PMO）

定义：PMO是对与项目相关的治理过程进行标准化，并促进资源、方法论、工具和技术共享的一种组织结构。PMO是项目管理知识体系的组织保障。

关键要点：
- PMO三种类型：支持型（咨询顾问角色，低控制度）、控制型（强制遵守流程和模板，中度控制）、指令型（直接管理项目，高控制度）
- PMO核心职能：制定项目管理标准和方法论、提供项目模板和表单、监控项目绩效、培训项目经理、多项目资源协调
- PMO与项目经理的区别：项目经理关注单个项目目标，PMO关注企业级项目组合的战略一致性
- PMO不直接管理项目日常执行，而是提供统一的管理框架和决策支持
- 常见考点：三种类型控制力度排序，PMO职能判断正误

### 3. 成本基准与项目预算

定义：成本基准是经过批准的、按时间段分配的项目预算，不包括管理储备，是测量、监督和控制成本绩效的依据。项目预算等于成本基准加上管理储备。

关键要点：
- 成本基准组成：各工作包估算成本之和 + 应急储备（已知-未知风险），按时间段分配形成S曲线
- 项目预算 = 成本基准 + 管理储备（未知-未知风险），管理储备由管理层审批
- 应急储备 vs 管理储备：应急储备纳入成本基准，项目经理可动用；管理储备不纳入成本基准，需管理层批准才能动用
- 储备分析贯穿估算过程：估算活动成本时确定应急储备，制定预算时汇总并追加管理储备
- 常见考点：成本基准包含什么不包含什么、两种储备的区别与使用权限、项目预算的构成公式

### 4. 质量审计与过程改进

定义：质量审计是对质量管理活动及其结果进行独立的、结构化的审查，确定质量活动和标准是否符合既定政策和程序，识别可改进的最佳实践。过程改进是持续优化项目管理流程以提高效率和质量。

关键要点：
- 质量审计目的：识别全部违规和差距、分享最佳实践、改进过程、提升团队效率
- 质量审计属于实施质量保证过程的核心工具，是质量管理体系的外部监督机制
- PDCA循环（戴明环）：计划（Plan）→执行（Do）→检查（Check）→行动（Act），是过程改进的基本方法论
- CMMI过程改进路径：初始级→已管理级→已定义级→量化管理级→优化级
- 质量审计 vs 质量控制：审计关注过程合规性（做了没有、做得对不对），控制关注产品符合性（成果合不合格）
- 常见考点：质量审计的目的判断、PDCA各阶段对应活动、审计与控制的区别

### 5. UML核心图与面向对象基础

定义：UML（统一建模语言）是面向对象软件工程的标准建模语言，通过一系列图形化符号描述软件系统的结构和行为。面向对象三大特征是封装、继承、多态。

关键要点：
- UML结构图（静态）：类图（类、属性、方法及关系）、构件图（系统物理组件）、部署图（硬件节点与软件分布）、对象图（实例快照）、包图（组织结构）
- UML行为图（动态）：用例图（参与者与用例交互）、序列图/顺序图（对象间消息时序）、活动图（工作流程与决策）、状态图（对象状态转换）、通信图（对象间消息交互）
- 面向对象三大特征：封装（隐藏实现、暴露接口）、继承（代码复用、单继承/多继承）、多态（同一接口不同实现）
- 设计模式三大分类：创建型（工厂/单例/建造者）、结构型（适配器/代理/装饰器）、行为型（观察者/策略/命令）
- 常见考点：区分结构图与行为图、类图中关系类型（泛化/实现/关联/聚合/组合/依赖）、用例图包含(include)与扩展(extend)关系

## 案例分析题

**题目**：

某市"智慧人社"综合管理系统项目，总投资预算800万元（含管理储备60万元），工期10个月。项目进入第5个月时，项目经理组织了质量审计，发现以下问题：

（1）成本基准中应急储备为50万元。第5月末检查：计划价值（PV）=400万元，挣值（EV）=320万元，实际成本（AC）=380万元。

（2）审计发现开发团队未使用组织规定的编码规范和代码审查流程，测试用例评审也缺少记录。

（3）项目采用面向对象方法开发，架构设计中类之间存在大量依赖关系，系统耦合度高，复用性差。项目经理要求采用设计模式降低耦合。

（4）由于需求变更频繁，项目经理未经评审直接批准了3项变更并使用了管理储备。

**问题1**：计算第5月末的成本偏差（CV）、进度偏差（SV）、成本绩效指数（CPI）和进度绩效指数（SPI），并分析项目当前状态。

**问题2**：针对质量审计发现的问题（2），请指出应采取哪些改进措施？

**问题3**：问题（3）中，请推荐两种合适的设计模式来降低系统耦合度，并说明理由。

**问题4**：问题（4）中，项目经理的做法有哪些不妥之处？正确做法是什么？

**参考答案**：

**问题1**：
- CV = EV - AC = 320 - 380 = -60万元（成本超支60万元）
- SV = EV - PV = 320 - 400 = -80万元（进度落后80万元）
- CPI = EV / AC = 320 / 380 ≈ 0.84（每花1元只产生0.84元价值，成本超支）
- SPI = EV / PV = 320 / 400 = 0.80（进度效率仅为计划的80%）

项目状态分析：CV为负、CPI<1表示成本超支；SV为负、SPI<1表示进度落后。项目处于成本超支且进度滞后的双偏差状态，需采取纠正措施。

**问题2**：
① 严格执行组织编码规范，将规范纳入项目质量计划和质量核对表
② 建立代码审查制度，设置审查周期、审查人员和审查标准，留存审查记录
③ 完善测试用例评审流程，评审后形成签字确认的评审记录
④ 针对审计发现制定纠正措施和预防措施，跟踪整改落实
⑤ 加强质量培训，提升团队质量意识
⑥ 建立过程资产库，将质量审计发现纳入组织过程资产

**问题3**：
① 工厂方法模式：将对象创建过程抽象为工厂接口，客户端通过工厂获取对象实例，无需直接依赖具体类，降低创建层面的耦合
② 适配器模式：在不修改已有类代码的前提下，通过适配器将不兼容的接口转换为客户端期望的接口，实现已有组件的复用，降低调用层面的耦合
③ 也可答：代理模式（通过代理对象控制对实际对象的访问，隔离客户端与真实对象）、观察者模式（松耦合的事件通知机制）

**问题4**：
不妥之处：
① 变更未经评审直接批准——违反整体变更控制流程
② 项目经理无权直接动用管理储备——管理储备由管理层审批
③ 未建立变更控制委员会（CCB）或未走正式变更流程

正确做法：
① 任何变更必须提出变更请求，提交CCB评审
② CCB评审通过后由项目经理更新项目管理计划和项目文件
③ 管理储备的使用需管理层（项目发起人或高层管理者）审批
④ 变更记录需更新到变更日志，做好配置管理

**踩分要点**：
- 挣值四个公式每个正确得2分（CV/SV/CPI/SPI=8分）
- 状态分析：双偏差（成本超支+进度落后）得2分
- 质量改进措施：编码规范执行、代码审查、测试评审记录、纠正预防措施各1-2分（6-8分）
- 设计模式：模式名称+理由各2分（4分）
- 变更控制问题：未经评审（2分）、管理储备审批权限错误（2分）、正确流程（CCB评审+管理层审批管理储备）（4分）

## 记忆口诀

**中间件分类诀**
消息中间件传信息，交易中间件保事务
对象中间件管组件，数据库中间件连数据

**PMO三型诀**
支持型像顾问，出主意不插手
控制型像教练，定规矩盯着走
指令型像主帅，亲自上阵管到底

**成本基准储备诀**
应急储备入基准，项目经理能动用
管理储备在基准外，高层批准才能花
项目预算加一起，基准加上管理储备

**质量审计诀**
审计看过程合不合规，控制看产品合不合格
PDCA循环转不停，计划执行检查再行动
质量审计找差距，最佳实践来分享

**UML图分类诀**
结构图是静态的：类图构件部署对象包
行为图是动态的：用例序列活动状态通信
五图结构五图行为，十种图要分得清
类图关系六种多：泛化实现关联聚合组合依赖

**设计模式三分类诀**
创建型造对象（工厂单例建造者）
结构型搭框架（适配器代理装饰器）
行为型管交互（观察者策略命令者）
