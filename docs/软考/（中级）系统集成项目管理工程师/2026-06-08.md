## 今日核心概念

### 1. 软件测试方法（V模型映射）

定义：测试是验证软件是否满足需求、发现缺陷的过程。测试应尽早介入，贯穿整个生命周期。

V模型测试阶段对应关系：
- 需求分析 → 验收测试（用户角度验证业务需求）
- 概要设计 → 系统测试（整体功能、性能、安全测试）
- 详细设计 → 集成测试（模块间接口、交互测试）
- 编码实现 → 单元测试（最小可测单元，函数/方法级）

测试层次口诀：从内到外，单元→集成→系统→验收。

关键要点：
- 单元测试由开发人员完成，白盒为主
- 集成测试策略：自顶向下（需桩模块）、自底向上（需驱动模块）、三明治（两者结合）
- 系统测试关注非功能性需求（性能、压力、安全、兼容性）
- 验收测试由用户/客户主导，α测试（开发环境）和β测试（用户现场）
- 白盒测试：关注程序内部逻辑结构（语句覆盖、判定覆盖、条件覆盖、路径覆盖）
- 黑盒测试：关注输入输出功能（等价类划分、边界值分析、因果图、错误推测）
- 灰盒测试：介于两者之间，既关注功能也关注内部结构

### 2. 需求工程

定义：需求工程是系统化获取、分析、记录、验证和管理需求的过程。需求分两类——功能性需求（系统做什么）和非功能性需求（系统做得怎么样：性能、可靠性、安全性、可用性）。

需求获取方法（7种）：
- 访谈：一对一或一对多，结构化或半结构化
- 问卷调查：覆盖面广，适合统计分析
- 头脑风暴：发散思维，收集创意
- 联合需求计划（JRP）：利益相关方集中讨论，快速达成共识
- 原型法：抛弃型原型（验证需求）或演化型原型（迭代开发）
- 文档分析：研究现有系统文档、业务规程
- 观察法：实地观察用户工作流程

需求规格说明书（SRS）必须满足八个特性：完整性、正确性、可行性、必要性、优先级、无歧义、可验证、可修改。

关键要点：
- 需求跟踪矩阵（RTM）：建立需求从来源→设计→实现→测试的双向追溯链
- 需求变更控制：所有需求变更必须走正式变更控制流程
- 需求确认：通过评审、原型演示确保干系人对需求达成一致

### 3. OSI七层模型与TCP/IP协议簇

定义：OSI（开放系统互连）模型是ISO制定的网络通信参考模型，将网络通信分为七层。TCP/IP协议簇是互联网实际使用的协议体系，分四层。

OSI七层与TCP/IP四层对应：

| OSI七层 | TCP/IP四层 | 主要协议/设备 |
|---------|-----------|-------------|
| 应用层 | 应用层 | HTTP/HTTPS、FTP、SMTP、DNS、SNMP |
| 表示层 | 应用层 | 数据加密/解密、压缩、格式转换 |
| 会话层 | 应用层 | 会话建立/维护/终止 |
| 传输层 | 传输层 | TCP（可靠）、UDP（不可靠但快） |
| 网络层 | 网络层 | IP、ICMP、ARP、路由器 |
| 数据链路层 | 网络接口层 | MAC地址、交换机、网桥、VLAN |
| 物理层 | 网络接口层 | 网线、光纤、集线器、中继器 |

关键要点：
- TCP三次握手：SYN → SYN-ACK → ACK，建立可靠连接
- TCP四次挥手：FIN → ACK → FIN → ACK，断开连接
- UDP无连接，速度快，适合实时应用（视频通话、直播）
- 路由器工作在网络层（IP地址转发），交换机工作在数据链路层（MAC地址转发）
- DNS将域名解析为IP地址，默认端口53
- SNMP网络管理协议，默认端口161/162

### 4. 数据库范式（1NF/2NF/3NF）

定义：范式是关系数据库设计中规范数据结构的规则，目的是消除数据冗余、避免更新异常。

**第一范式（1NF）**：每个字段不可再分，即"原子性"。
- 反例：字段"联系方式"包含"电话,邮箱"
- 正例：拆为"电话"和"邮箱"两个独立字段

**第二范式（2NF）**：在满足1NF基础上，每个非主键字段必须完全依赖于主键（消除部分依赖）。
- 反例：表(学号, 课程号, 成绩, 学生姓名)，学生姓名只依赖于学号（部分依赖）
- 正例：拆为学生表(学号, 学生姓名)和选课表(学号, 课程号, 成绩)

**第三范式（3NF）**：在满足2NF基础上，非主键字段不能依赖于其他非主键字段（消除传递依赖）。
- 反例：表(学号, 班级号, 班主任)，班主任依赖于班级号，班级号依赖于学号（传递依赖）
- 正例：拆为学生表(学号, 班级号)和班级表(班级号, 班主任)

关键要点：
- 范式越高冗余越低，但查询时需要更多表连接，性能可能下降
- 实务中通常到3NF即可，有时为性能有意保留部分冗余（反规范化）
- E-R模型三要素：实体（矩形）、属性（椭圆）、联系（菱形），联系类型：1:1、1:N、M:N

### 5. 大数据与云计算关键技术

定义：
- 大数据：无法用传统工具采集、存储、管理、分析的超大规模数据集合，特征为5V（Volume大量、Velocity高速、Variety多样、Value低价值密度、Veracity真实性）
- 云计算：通过网络按需提供可配置计算资源（网络、服务器、存储、应用、服务），特征为按需自助、宽带接入、资源池化、快速弹性、可计量服务

云计算的三种服务模式：
- IaaS（基础设施即服务）：提供虚拟化计算资源，用户部署和管理自己的应用——如阿里云ECS、AWS EC2
- PaaS（平台即服务）：提供应用开发和部署平台，用户只管代码——如阿里云ACE、Google App Engine
- SaaS（软件即服务）：提供完整的应用软件，用户只管使用——如钉钉、腾讯文档、Salesforce

云计算四种部署模式：公有云、私有云、混合云、社区云。

大数据关键技术：
- 分布式存储：HDFS（Hadoop分布式文件系统）
- 分布式计算：MapReduce（并行计算框架）
- NoSQL数据库：HBase、MongoDB（处理非结构化数据）
- 数据仓库技术：ETL（抽取-转换-加载）、OLAP（联机分析处理）

关键要点：
- 大数据与云计算的关系：云计算是大数据的计算平台，大数据是云计算的重要应用
- 物联网（IoT）三层架构：感知层（传感器/RFID）→ 网络层（WiFi/5G/NB-IoT）→ 应用层（智能家居/智慧城市）
- 人工智能三要素：数据、算法、算力

## 案例分析题

**题目**：

某市统计局"智慧统计数据分析平台"项目建设过程中发生了以下情况：

**【背景】**
项目预算200万元，工期8个月。采用结构化方法开发。项目经理王某带领12人团队。

**【事件一】**
需求分析阶段，王某组织开发人员直接根据招标文件中的功能清单编写了需求规格说明书（SRS），未组织用户参与需求评审。开发进入详细设计阶段后，统计局业务处室提出大量需求变更，包括新增8个统计报表、调整数据采集频率等。开发方称需求已冻结，要求走正式变更流程，用户方认为"招标文件写得不详细，你们应该早就考虑到"，双方产生争执。

**【事件二】**
系统测试阶段发现，数据传输模块在并发用户超过100时响应时间从2秒骤升至30秒。经排查，传输层选择了UDP协议以追求速度，但大量数据包在公网传输中丢包率达15%，导致频繁重传。而系统规格说明书中明确要求"并发200用户时响应时间不超过5秒"。

**【事件三】**
数据库设计中，原始方案将"统计指标"表设计为：指标表(指标编码, 指标名称, 统计周期, 填报单位编码, 填报单位名称, 单位负责人, 负责人电话)。数据库管理员指出该设计违反第三范式。

**【事件四】**
王某决定采用云计算方案，将系统部署到公有云。但统计局领导担忧数据安全问题，要求全部部署在局内机房。王某评估后发现如果改部署模式，需要追加预算80万元购买服务器和存储设备。

**【子问题】**

（1）请分析事件一中，需求工程存在哪些问题？从需求获取、需求分析、需求确认三个维度回答。（6分）

（2）事件二中，从OSI模型角度分析传输层协议选择的问题；如果改为TCP协议，对系统有什么影响？应该采取什么整体解决方案？（6分）

（3）事件三中，请指出原始数据库设计的范式问题，并给出符合3NF的修正方案。（4分）

（4）事件四中，王某本来想用什么部署模式，用户要求什么模式？请说明两种部署模式的优缺点对比，并指出追加预算的来源应纳入什么管理过程。（4分）

---

**参考答案**：

（1）需求工程问题分析（6分）：

需求获取方面（2分）：
- 仅依据招标文件编写SRS，未采用访谈、JRP等需求获取方法与实际用户深入沟通
- 未安排现场观察、文档分析等方式了解统计局实际业务流程
- 缺少需求获取的多种方法组合，信息源单一

需求分析方面（2分）：
- 未对需求进行分类（功能需求/非功能需求）、优先级排序
- 未建立需求跟踪矩阵（RTM），缺少需求与设计、测试的双向追溯
- 招标文件作为业务需求不等于系统需求，缺少需求细化和分析过程

需求确认方面（2分）：
- 未组织用户参与需求评审，需求规格说明书未经用户签字确认
- 缺少原型演示等需求验证手段
- 需求基线未正式确立即进入设计阶段，导致后续大量变更

（2）传输层协议分析（6分）：

协议选择问题（2分）：
- UDP是无连接协议，不保证可靠传输，无确认和重传机制
- 数据传输场景需要数据完整性，公网环境丢包率高，不应使用UDP
- 正确选择应为TCP（面向连接、可靠传输、流量控制、拥塞控制）

改为TCP的影响（2分）：
- 优点：建立连接确保可靠传输，自动重传丢包，保障数据完整性
- 缺点：三次握手增加连接建立开销；拥塞控制可能降低吞吐量，实时性略降

整体解决方案（2分）：
- 传输层改用TCP保证可靠传输
- 应用层可引入数据压缩减少传输量
- 同时做网络层面的优化：CDN加速、专线/VPN、断点续传机制
- 可考虑混合方案：控制指令用TCP，非关键实时展示用UDP+应用层重传

（3）数据库范式问题（4分）：

范式问题分析（2分）：
- 填报单位编码→填报单位名称、单位负责人（部分依赖）
- 填报单位名称→单位负责人、负责人电话（传递依赖）
- 该设计违反2NF和3NF，存在严重数据冗余和更新异常

3NF修正方案（2分）：
- 统计指标表(指标编码, 指标名称, 统计周期, 填报单位编码)
- 填报单位表(填报单位编码, 填报单位名称, 单位负责人编码)
- 负责人表(负责人编码, 负责人姓名, 负责人电话)
- 指标编码→填报单位编码，填报单位编码→负责人编码，消除传递依赖

（4）部署模式分析（4分）：

王某方案：公有云；用户要求：私有云/本地部署（2分）

两种模式对比（1分）：

| 对比维度 | 公有云 | 私有云/本地部署 |
|---------|-------|--------------|
| 数据安全 | 数据存储在第三方，存在合规风险 | 数据在内部，安全可控 |
| 建设成本 | 按需付费，前期成本低 | 一次性投入大，需购买硬件 |
| 运维成本 | 云服务商运维 | 需自行配备运维团队 |
| 扩展性 | 弹性伸缩，快速扩展 | 扩容需采购硬件，周期长 |

追加预算管理（1分）：追加的80万元预算变更应纳入**整体变更控制过程**——提交变更请求→CCB审批→更新成本基准和项目管理计划→通知相关干系人。

---

**踩分要点**：

- 事件一：三个维度各2分，必须逐个维度分析缺陷，不能笼统说"需求管理不好"
- 事件二：UDP"不保证可靠传输"必须写明，TCP影响需优缺点都要说，整体方案必须涉及传输层+应用层+网络层
- 事件三：必须指明"违反2NF和3NF"，只说"违反范式"扣1分；修正方案须拆出两个独立表（填报单位表+负责人表）
- 事件四：公有云/私有云各写至少两条优缺点，预算追加必须关联到"整体变更控制"

## 记忆口诀

**V模型测试口诀**
需→验、概→系、详→集、编→单。
需求对验收，概要对系统，详细对集成，编码对单元。左边出右边测，步步对应不要错。

**需求获取七法口诀**
访谈问卷头脑风，原型文档联合观。
一对一访谈、调查问卷、头脑风暴、原型法、文档分析、JRP联合计划、观察法——七法各有所长，组合使用效果佳。

**OSI七层口诀**
应用表示会话传，网络链路物理连。
从上到下记忆：Application应用层、Presentation表示层、Session会话层、Transport传输层、Network网络层、Data Link数据链路层、Physical物理层。

**TCP与UDP区别口诀**
TCP最可靠（三次握手、确认重传），UDP就是快（无连接、无确认）。
数据传输用TCP，实时直播用UDP。考场上判断传输需求——数据丢了不行选TCP，丢一点无所谓选UDP。

**三范式口诀**
一消可分、二消部分、三消传递。
1NF字段不可分，2NF完全依赖主键，3NF非主键之间不依赖。层层递进减冗余，实务三范就够用。

**云服务模式口诀**
I-P-S，基础平台软件三层递进。
IaaS给基础设施（服务器/存储/网络），PaaS给开发平台（中间件/数据库/运行时），SaaS给完整应用（开箱即用）。用户管的东西越来越少，供应商管的东西越来越多。
