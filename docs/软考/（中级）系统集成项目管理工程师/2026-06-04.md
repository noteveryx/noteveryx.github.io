# 软考（中级）系统集成项目管理工程师 - 2026-06-04

## 今日核心概念

### 1. 信息系统开发方法

**定义**：信息系统开发方法是指导信息系统开发过程的一套系统化的方法体系，主要包括结构化方法、面向对象方法、原型法和敏捷开发方法。

**关键要点**：
- 结构化方法：自顶向下、逐步求精、模块化设计，强调数据流和数据存储，核心工具为数据流图（DFD）和E-R图。优点：逻辑性强、文档规范；缺点：开发周期长、难以适应需求变化
- 面向对象方法：以对象为核心，将数据和操作封装在一起，核心概念包括类、对象、继承、多态。使用UML建模。优点：复用性强、易于维护；缺点：前期分析设计投入大
- 原型法：快速构建一个可运行的原型系统，让用户试用后提出改进意见，逐步完善。适用于需求不明确的场景。优点：用户参与度高、风险降低；缺点：可能导致文档不够规范、系统结构不够理想
- 敏捷开发：以人为核心、迭代、循序渐进的方法，强调快速交付可工作的软件，响应变化胜过遵循计划。代表方法：Scrum、XP。适用于需求变化频繁的中小型项目

### 2. RACI矩阵（责任分配矩阵）

**定义**：RACI矩阵是一种用于明确项目团队成员角色与职责的工具，通过四种角色定义确保每项工作任务都有人负责、有人参与、有人知晓。

**关键要点**：
- R（Responsible，执行者）：实际完成工作任务的人，每项任务至少有一个R
- A（Accountable，负责人）：对任务结果承担最终责任的人，每项任务有且仅有一个A
- C（Consulted，被咨询者）：在决策或行动前需要征询其意见的人，通常是领域专家，双向沟通
- I（Informed，被通知者）：需要被告知决策或行动结果的人，单向沟通
- 使用原则：每个任务必须有且仅有一个A，可以有多个R但避免过多，A和R最好不是同一个人以便形成制衡
- 与RAM（责任分配矩阵）的关系：RAM是通用概念，RACI是RAM的一种具体表现形式

### 3. 配置管理核心要素

**定义**：配置管理是通过技术和管理手段，对项目产品及其组成部分的识别、控制、状态记录和审核的过程，确保项目成果的一致性和可追溯性。

**关键要点**：
- 配置项（CI）：被纳入配置管理的项目组成部分，如需求文档、设计文档、源代码、测试用例等。每个配置项有唯一的标识符
- 基线（Baseline）：经过正式评审和批准、作为后续工作基础的配置项集合。常见的三大基线：功能基线（需求确认后）、分配基线（设计确认后）、产品基线（测试通过后）
- 配置库三库结构：开发库（动态库，程序员工作空间，可自由修改）→ 受控库（主库，存放已通过评审的配置项，变更需审批）→ 产品库（静态库，存放已发布的产品版本，只读）
- 配置状态报告：记录和报告配置项的当前状态，包括变更申请状态、实施状态等
- 配置审计：分为功能配置审计（检查是否满足功能需求）和物理配置审计（检查配置项是否与文档一致）

### 4. 项目管理计划与监控项目工作

**定义**：项目管理计划是整合所有子计划的综合性文档，定义了项目如何执行、监控和收尾。监控项目工作是跟踪、审查项目进展，确保项目按计划执行的过程。

**关键要点**：
- 项目管理计划的内容包括：选定的项目生命周期、范围基准、进度基准、成本基准、子管理计划（范围、进度、成本、质量、人力资源、沟通、风险、采购、干系人）和配置管理计划、过程改进计划
- 计划不是一成不变的，需要在项目生命周期中持续维护和更新（渐进明细）
- 监控项目工作的输入：项目管理计划、工作绩效数据、进度预测、成本预测等
- 监控项目工作的输出：工作绩效信息（对比分析后的结果）、变更请求、项目管理计划更新、项目文件更新
- 区分：工作绩效数据 → 各控制过程分析处理 → 工作绩效信息 → 监控项目工作汇总 → 工作绩效报告
- 纠正措施：使实际绩效重新符合计划；预防措施：降低潜在偏离的风险；缺陷补救：修复不合格的产品组件

### 5. 政府采购方式

**定义**：政府采购是指各级国家机关、事业单位和团体组织使用财政性资金采购依法制定的集中采购目录以内或者采购限额标准以上的货物、工程和服务的行为。熟练掌握采购方式对系统集成项目管理中的采购环节至关重要。

**关键要点**：
- 公开招标：采购人依法以招标公告的方式邀请不特定的供应商参加投标。适用：金额较大的项目，是政府采购的主要方式
- 邀请招标：以投标邀请书的方式邀请三家以上特定供应商投标。适用：具有特殊性只能从有限范围采购、公开招标费用占采购金额比例过大
- 竞争性谈判：邀请三家以上供应商就采购事宜进行谈判。适用：没有供应商投标或没有合格标的、技术复杂不能确定详细规格、招标时间不能满足紧急需要、不能事先计算价格总额
- 询价：向三家以上供应商发出询价单，邀请报价后比较确定成交供应商。适用：规格标准统一、现货货源充足且价格变化幅度小的项目
- 单一来源采购：向唯一供应商直接采购。适用：只能从唯一供应商处采购、发生了不可预见紧急情况、必须保证原有项目一致性或服务配套要求需从原供应商处添购
- 公开招标数额标准：货物和服务类一般200万元以上，工程类一般400万元以上（各地标准可能不同）

## 案例分析题

**题目**：

某市人力资源和社会保障局委托A公司开发"智慧人社综合服务平台"，项目预算500万元，建设周期10个月。项目经理小张采用敏捷开发方法进行项目开发。

【场景一】项目启动阶段，小张组织团队编写了项目管理计划，包括范围基准、进度基准和成本基准，但由于开发方法为敏捷模式，小张认为详细需求会随迭代逐步明确，因此未制定详细的范围管理计划和需求跟踪矩阵，仅定义了一个粗粒度的产品待办事项列表（Product Backlog）。

【场景二】第3个迭代结束时，客户提出大量需求变更，称前期需求理解有偏差。小张逐个接受了变更，并直接安排团队开始实施。第5个月时，项目进度严重滞后，PV=320万，EV=220万，AC=260万。此时团队已有12人，其中3人为新手。

【场景三】项目后期发现代码版本混乱，部分已修复的Bug在新版本中再次出现。经查，团队使用一个共享文件夹进行代码管理，多个开发人员可随意覆盖文件，没有明确的版本控制和基线管理制度。

【问题1】（8分）请指出项目经理小张在项目管理中存在哪些问题。

【问题2】（6分）请计算项目第5个月的CV、SV、CPI、SPI，并分析当前项目的成本和进度执行情况。

【问题3】（6分）针对场景三中的配置管理混乱问题，请给出具体的改进措施，说明如何使用配置库三库结构进行管理。

【问题4】（5分）假设项目剩余工作的绩效指数按当前CPI执行（典型偏差），请计算ETC和EAC。如果项目剩余工作必须按原计划效率完成（非典型偏差），ETC和EAC又是多少？

**参考答案**：

【问题1】小张存在的问题：
1. 未制定范围管理计划，缺乏需求变更的规范化流程
2. 未建立需求跟踪矩阵，无法追溯需求来源和变更影响
3. 变更控制流程缺失：对客户提出的变更没有进行影响分析（对范围、进度、成本、质量的影响），没有提交CCB审批，直接接受并实施
4. 项目监控不力：到第5个月才发现进度严重滞后，缺乏定期的绩效测量和分析
5. 团队配置管理缺失：没有建立配置管理体系，没有使用版本控制工具，没有建立基线管理
6. 资源管理不当：3名新手未经充分培训即投入开发，可能影响开发效率和质量
7. 进度和成本没有进行有效的偏差分析和纠正措施制定

【问题2】
CV = EV - AC = 220 - 260 = -40（万元），成本超支40万元
SV = EV - PV = 220 - 320 = -100（万元），进度落后100万元
CPI = EV ÷ AC = 220 ÷ 260 = 0.846，小于1，成本效率低，每花1元只完成0.846元的活
SPI = EV ÷ PV = 220 ÷ 320 = 0.688，小于1，进度效率低，仅完成了计划的68.8%

结论：项目成本超支、进度严重滞后，需要进行纠偏。

【问题3】改进措施：
1. 建立开发库（动态库）：每个开发人员的个人工作空间，可自由修改代码，但须按时签到签出
2. 建立受控库（主库）：存放经过评审和测试通过的代码版本。从开发库签入受控库需要经过代码评审和审批流程
3. 建立产品库（静态库）：存放已发布的产品版本，只读访问。将经过集成测试和验收测试的正式版本存入产品库
4. 配合配置管理流程：版本号规则 → 变更申请 → 变更评审 → 实施变更 → 验证 → 更新基线 → 通知相关人员
5. 引入版本控制工具（如Git/SVN），禁止使用共享文件夹方式管理代码
6. 由配置管理员（CMO）统一管理配置库，严格权限控制

【问题4】
BAC（完工预算）= 项目总预算 = 500万元

典型偏差（按当前CPI执行）：
ETC = (BAC - EV) ÷ CPI = (500 - 220) ÷ 0.846 = 280 ÷ 0.846 = 330.97万元
EAC = AC + ETC = 260 + 330.97 = 590.97万元

非典型偏差（按原计划效率）：
ETC = BAC - EV = 500 - 220 = 280万元
EAC = AC + ETC = 260 + 280 = 540万元

结论：如果按当前效率继续，项目最终将超支约90.97万元；如果后续工作能按原计划效率完成，最终将超支40万元。

**踩分要点**：
- 问题1：须答出"无范围管理计划""未进行变更影响分析""未提交CCB审批"等关键词，答出配置管理和资源管理问题可加分
- 问题2：公式和计算过程须写出，仅写结果可能被扣步骤分；结论须明确说明"超支"和"滞后"
- 问题3：三库结构的名称和用途必须说清，"共享文件夹不可行""需配置管理员"是关键词
- 问题4：典型vs非典型的公式不同，必须区分；ETC和EAC的单位（万元）不能漏

## 记忆口诀

**开发方法选择诀**：
需求稳定结构化，需求多变敏捷佳；
复用优先面向对象，需求模糊原型探。

**RACI矩阵诀**：
R是干活的，A是担责的；C来出主意，I只被告知。
每项任务一个A，R多要精简，AC可分离，制衡才合理。

**配置三库诀**：
开发库中随便改，受控库里需审批；
产品库中只读看，三库升级有规矩。
基线就是标准线，变更必须走流程；
审计分为功和物，功能物理不混淆。

**采购方式选择诀**：
公开招标是主路，邀请招标限三家；
竞争谈判技术杂，规格统一询价发；
单一来源唯一供，紧急配套有说法。
两百万货物服务起，四百万工程门槛立。

**挣值公式速记诀**：
CV = EV减AC，正数省钱负超支；
SV = EV减PV，正数提前负落后。
CPI看花钱效，大于1就划算；
SPI看干活速，大于1就领先。
典型偏差用CPI除，非典型直接减法算。
