## 今日核心概念

### 1. 项目范围管理六过程
定义：范围管理确保项目做且只做成功完成项目所需的全部工作，包含规划范围管理、收集需求、定义范围、创建WBS、确认范围、控制范围六个过程。
关键要点：
- 规划范围管理→输出范围管理计划与需求管理计划
- 收集需求→输出需求文件与需求跟踪矩阵（RTM）
- 定义范围→输出项目范围说明书（详细版）
- 创建WBS→输出范围基准（WBS+WBS词典+范围说明书）
- 确认范围→正式验收已完成可交付成果，与质量控制区别：确认范围关注"是否完成"，质量控制关注"是否正确"
- 控制范围→防止范围蔓延（未经控制的范围扩大），区别于范围镀金（主动增加的功能）

### 2. 需求跟踪矩阵（RTM）与需求管理
定义：RTM是一张表格，将每个需求从来源到实现到测试全链条追踪，确保需求不遗漏、不偏离。
关键要点：
- RTM核心列：需求编号、需求描述、业务需求来源、项目目标对应、WBS对应、设计组件、测试用例
- 需求获取七法：访谈、焦点小组、引导式研讨会、群体创新技术（头脑风暴/名义群体/德尔菲/思维导图/亲和图）、群体决策技术、问卷调查、观察
- 需求文件八大特性：唯一标识、完整、可跟踪、可测试、可验证、优先级排序、业务价值、来源可溯
- 需求变更必须走整体变更控制流程，任何未经批准的需求变更都是范围蔓延

### 3. 项目收尾管理（行政收尾与合同收尾）
定义：项目收尾是项目或阶段的正式终结，包括行政收尾和合同收尾两种类型。
关键要点：
- 行政收尾流程：确认工作完成→获得客户验收→移交产品→更新组织过程资产→归档项目文件→释放资源→正式宣布项目结束
- 合同收尾流程：核实合同条款完成→采购审计→解决遗留问题→正式关闭合同→更新组织过程资产
- 行政收尾vs合同收尾区别：行政收尾针对项目整体，合同收尾针对每个采购合同；项目只需一次行政收尾，可能多次合同收尾
- 收尾必须产出：最终产品/服务移交、经验教训登记册更新、组织过程资产更新、项目档案归档

### 4. 项目立项管理与可行性研究
定义：项目立项是项目正式启动前的决策过程，通过可行性研究论证项目必要性、可行性和合理性。
关键要点：
- 可行性研究四维度：技术可行性（能否实现）、经济可行性（是否值得）、操作可行性（能否运行）、社会可行性（是否符合法规与社会影响）
- 财务评价指标：净现值NPV（≥0可行）、投资回收期（越短越好）、内部收益率IRR（≥基准收益率可行）、投资回报率ROI（越高越好）
- 立项流程：项目建议书→可行性研究→项目论证→项目评估→项目审批→项目启动
- 项目章程由发起人发布，项目经理不编写但可参与建议；项目章程授权项目经理动用组织资源

### 5. 系统集成项目验收与移交
定义：系统集成项目验收是项目收尾的核心环节，确认系统满足合同与需求规格要求，正式交付给客户使用。
关键要点：
- 验收三阶段：初验（功能测试+文档审查）→试运行（实际环境运行3-6个月）→终验（全面评估+正式签字）
- 验收依据：合同条款、需求规格说明书、项目范围说明书、测试报告、试运行报告
- 验收文档清单：需求规格说明书、设计文档、测试报告、用户手册、运维手册、培训记录、源代码交付清单（如合同要求）
- 移交内容：系统软件/硬件、全套文档、培训完成证明、运维支持承诺、知识产权交付（如合同约定）

## 案例分析题

**题目**：
某市住建局"智慧住建综合监管平台"项目，合同金额800万元，工期10个月。项目进入第5个月末时，项目经理发现以下问题：

1. 客户业务部门提出3项新需求（增加移动端审批、实时视频监控接入、数据与省厅平台对接），项目经理未走变更流程直接安排开发团队实现。
2. 试运行阶段，用户反馈5项需求与原始需求说明书不一致，经查RTM中3项需求缺失跟踪记录。
3. 挣值分析数据：PV=420万元，EV=360万元，AC=440万元。
4. 项目发起人要求项目按原合同金额和工期收尾。

请回答：
（1）计算CV、SV、CPI、SPI，判断项目绩效。
（2）分析问题1和问题2违反了哪些范围管理原则？给出纠正措施。
（3）针对问题3，若后续按典型偏差发展，计算ETC和EAC（BAC=800万元）。
（4）项目收尾时应完成哪些关键活动？列出至少5项。

**参考答案**：

（1）挣值计算：
- CV = EV - AC = 360 - 440 = -80万元（成本超支）
- SV = EV - PV = 360 - 420 = -60万元（进度落后）
- CPI = EV/AC = 360/440 ≈ 0.82（每花1元只获得0.82元价值）
- SPI = EV/PV = 360/420 ≈ 0.86（进度仅为计划的86%）
项目成本和进度均不达标。

（2）范围管理违规分析：
问题1——直接实现3项新需求未走变更流程，属于**范围蔓延**。违反了"所有范围变更必须经过整体变更控制流程"原则。纠正措施：立即停止未批准需求的开发；补提变更请求，经CCB评审批准后方可实施；若不批准则回退已完成工作。
问题2——RTM缺失跟踪记录，违反了"需求全链条可追踪"原则，导致需求遗漏和偏离。纠正措施：补充完善RTM，将5项不一致需求逐条溯源；对遗漏需求走变更流程确认是否纳入；今后强制要求每个需求必须有RTM记录。

（3）ETC与EAC计算（典型偏差）：
- ETC = (BAC - EV)/CPI = (800 - 360)/0.82 ≈ 536.6万元
- EAC = AC + ETC = 440 + 536.6 ≈ 976.6万元

（4）项目收尾关键活动：
① 获得客户正式验收签字（初验+终验确认）
② 移交全部系统软件、硬件及文档资料
③ 组织项目经验教训总结会议，更新经验教训登记册
④ 归档全部项目文件（合同、计划、报告、变更记录等）至组织过程资产
⑤ 完成采购审计，正式关闭所有合同
⑥ 释放项目团队成员与物理资源
⑦ 组织用户培训，确保运维团队可独立操作
⑧ 发布项目结束公告，通知所有干系人

**踩分要点**：
- CV/SV必须带正负号，CPI/SPI必须保留两位小数，且要有定性判断结论
- 范围蔓延vs范围镀金的区分：未经批准是蔓延，主动超范围是镀金
- RTM的作用是"需求全链条追踪"，必须强调遗漏跟踪→需求偏离
- ETC典型偏差公式是(BAC-EV)/CPI，非典型是(BAC-EV)，必须明确用哪个
- 收尾活动至少写5项，"经验教训总结+归档+验收+移交+释放资源"为最低得分组合

## 记忆口诀

范围管理六步诀：规收定创确控——规划收集定义创建确认控制

RTM追踪诀：源目设测四对齐——需求来源→项目目标→设计组件→测试用例，全链条对齐

范围蔓延vs镀金诀：蔓延被动未批准，镀金主动超范围——蔓延是外界强加未经批准，镀金是自己主动多做

行政收尾vs合同收尾诀：行政收项目，合同收采购——项目一次行政收尾，合同按份分别关闭

验收三步诀：初验试运终验收——初验→试运行→终验，三步走完才算完

立项流程诀：建可论评批启——建议书→可行性→论证→评估→审批→启动

ETC/EAC典型诀：剩余价值除效率——ETC=(BAC-EV)/CPI，AC加ETC得EAC
