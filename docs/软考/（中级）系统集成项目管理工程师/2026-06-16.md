## 今日核心概念

### 1. 国家信息化体系与电子政务
定义：国家信息化体系是由信息技术应用、信息资源、信息网络、信息技术和产业、信息化人才、信息化政策法规和标准规范六个要素组成的有机整体；电子政务是政府机构运用现代信息技术实现政府管理和服务的信息化模式。

关键要点：
- 六要素关系：信息技术应用是龙头（需求牵引），信息资源是核心（核心任务），信息网络是基础设施，信息技术和产业是物质基础，信息化人才是成功之本，政策法规标准是保障
- 电子政务四种模式：G2G（政府对政府，如公文流转）、G2B（政府对企业，如在线审批）、G2C（政府对公民，如社保查询）、G2E（政府对公务员，如内部OA）
- "两网一站四库十二金"：两网=政务内网+政务外网；一站=政府门户网站；四库=人口/法人/空间地理/宏观经济基础信息库；十二金=金税/金关/金财/金审/金盾等12个重点业务系统
- 电子政务发展阶段：起步→单向发布→双向互动→网上办事→一站式服务

### 2. 网络存储与网络设备
定义：网络存储技术是解决数据集中存储、共享和备份的技术方案；网络设备是构建计算机网络的核心硬件设施。

关键要点：
- 三种存储架构：DAS（直连式存储，成本低/扩展差/数据孤岛）、NAS（网络附加存储，文件级共享/即插即用/性能中等）、SAN（存储区域网络，块级存储/高性能/高成本/光纤通道）
- 存储选择口诀：小规模直连DAS，文件共享选NAS，高性能块存储选SAN
- 核心网络设备：路由器（网络层/跨网段路由/ACL访问控制）、交换机（数据链路层/MAC寻址/VLAN划分）、防火墙（网络层/内外网隔离/包过滤+状态检测+应用代理）
- 防火墙三种技术：包过滤（速度快/无法检测应用层攻击）、状态检测（跟踪连接状态/主流技术）、应用代理（应用层深度检测/速度慢）
- 综合布线六大子系统：工作区/水平/管理/垂直干线/设备间/建筑群

### 3. IT服务管理（ITIL）
定义：ITIL（信息技术基础架构库）是一套IT服务管理的最佳实践框架，将IT服务视为资产进行全生命周期管理，从战略到持续改进形成闭环。

关键要点：
- ITIL v3五大核心阶段：服务战略（价值定位/服务组合管理）→服务设计（服务级别SLA/可用性/容量/连续性/安全设计）→服务转移（变更管理/发布部署/知识管理）→服务运营（事件/事故/问题/请求/访问管理）→持续服务改进（PDCA/度量与改进）
- 服务运营五大流程：事件管理（监控检测快速恢复）、事故管理（尽快恢复服务）、问题管理（根因分析永久解决）、请求管理（标准化服务请求）、访问管理（权限控制）
- 服务设计包SDP：定义新服务或变更服务的全部要求，包含SLA/OLA/UC
- SLA vs OLA vs UC：SLA（与客户的服务级别协议）、OLA（内部运营级别协议）、UC（与外部供应商的支撑合同）
- 关键区别：事故管理重快速恢复，问题管理重根因消除

### 4. 软件架构模式
定义：软件架构是系统的高层结构组织方式，定义了组件之间的关系和交互规则，直接影响系统的性能、可扩展性、安全性和可维护性。

关键要点：
- C/S vs B/S：C/S客户端/服务器（性能好/需安装/维护难/跨平台差）、B/S浏览器/服务器（零安装/易维护/跨平台/依赖网络/交互性弱）；趋势：B/S为主，C/S为辅
- MVC架构：Model模型（数据与业务逻辑）/View视图（用户界面）/Controller控制器（请求分发），核心思想：分层解耦、各司其职
- 分层架构：表现层→业务逻辑层→数据访问层→数据存储层，优点是关注点分离，缺点是性能开销
- 微服务架构：将单体应用拆分为独立部署的小服务，每个服务独立数据库、独立部署、独立扩展，适用大型复杂系统，但增加运维复杂度
- 中间件架构：CORBA/DCOM/J2EE/EJB等，解决异构系统集成问题

### 5. 项目风险监控与风险再评估
定义：项目风险监控是在项目全生命周期中持续跟踪已识别风险、监控残余风险、识别新风险，并评估风险应对措施有效性的过程。

关键要点：
- 风险监控七工具：风险再评估（识别新风险/重新评估已有风险）、风险审计（检查风险应对措施有效性）、偏差分析（对比计划与实际偏差）、技术绩效测量（技术指标与风险关联）、储备分析（剩余储备是否充足）、状态审查会（定期风险评审）、风险登记册更新（动态维护）
- 风险再评估时机：里程碑节点、重大变更后、阶段结束、外部环境变化时
- 储备分析要点：应急储备由项目经理支配（已知-未知风险），管理储备由管理层支配（未知-未知风险），随项目推进逐步释放未使用的应急储备
- 风险预警机制：设定风险触发条件（阈值），一旦触及立即启动应急计划或权变措施
- 权变措施vs应急计划：应急计划是事先制定的（已知风险），权变措施是临时应对的（未知风险）

## 案例分析题

**题目**：某市"一网通办"政务服务平台建设项目，预算1200万元，工期14个月。项目进行到第8个月末，项目经理发现：实际成本AC=780万元，已完成工作预算EV=620万元，计划价值PV=700万元。同时出现以下情况：

（1）项目初期未充分识别风险，现在出现新的安全合规要求，需增加等保三级建设内容，预计额外增加成本80万元；
（2）开发团队采用B/S架构，但部分审批功能页面响应缓慢，用户抱怨交互体验差，有人建议改用C/S架构；
（3）项目存储方案原采用NAS，随着数据量增长出现性能瓶颈；
（4）SLA约定系统可用性99.9%，但运维团队反映事故恢复时间经常超标。

请计算CV、SV、CPI、SPI，并分析项目成本和进度状况；针对上述四个问题，分别给出处理方案。

**参考答案**：

一、挣值计算：
- CV = EV - AC = 620 - 780 = -160万元（成本超支）
- SV = EV - PV = 620 - 700 = -80万元（进度滞后）
- CPI = EV/AC = 620/780 ≈ 0.79（每花1元只完成0.79元的工作，成本效率低）
- SPI = EV/PV = 620/700 ≈ 0.89（进度效率低，仅完成计划的89%）

项目状况：成本严重超支、进度滞后，需立即采取纠偏措施。

二、问题处理方案：

问题（1）——新安全合规要求：
- 属于外部环境变化引发的新风险，应启动风险再评估流程
- 通过整体变更控制流程提交变更申请，经CCB审批后执行
- 80万元额外成本：优先使用管理储备（未知-未知风险性质），或调整项目预算基线
- 更新风险登记册，补充安全合规风险及应对措施

问题（2）——B/S架构交互体验差：
- 不建议整体改用C/S架构（改造代价过大，影响进度和成本）
- 采用局部优化方案：对审批等交互密集功能采用富客户端技术（如Vue/React前端框架+WebSocket实时通信）提升体验
- 利用MVC架构优势，仅优化View层表现，不改变整体架构
- 新增性能需求纳入范围变更管理

问题（3）——NAS存储性能瓶颈：
- 分析瓶颈类型：文件级访问瓶颈还是块级I/O瓶颈
- 若为数据库等块级I/O需求，将数据库存储迁移至SAN架构
- 保留NAS用于文件共享，形成SAN+NAS混合存储架构
- 迁移方案纳入变更控制，评估对业务的影响和停机窗口

问题（4）——SLA可用性不达标：
- 区分事故管理与问题管理：事故管理恢复服务（治标），问题管理消除根因（治本）
- 建立问题管理流程，对重复性事故进行根因分析
- 完善OLA（内部运营级别协议），明确各支持团队的响应时限
- 增加事件监控和自动化告警，缩短事故发现和响应时间
- 定期进行服务评审，持续改进

**踩分要点**：
1. 挣值计算必须列出公式和过程，CPI<1和SPI<1要明确说明超支和滞后
2. 新合规要求必须走变更控制流程，不能直接执行
3. 管理储备vs应急储备的适用场景要区分清楚
4. 架构问题不能建议推翻重来，要给出务实的优化方案
5. NAS→SAN的迁移要说明理由（块级vs文件级），不是简单替换
6. 事故管理与问题管理的区别是高频考点，必须点明
7. SLA/OLA/UC三层协议关系要体现

## 记忆口诀

国家信息化六要素诀：
**应资网产人规标，应用龙头资核心**
（应用=龙头，资源=核心，网络=基础，产业=支撑，人才=根本，法规标准=保障）

电子政务模式诀：
**G2G政府间流转，G2B政企在线办，G2C政民查社保，G2E内部办公环**

存储三架构诀：
**DAS直连低扩展，NAS文件共享便，SAN块存高性能，混合架构最常见**

ITIL五阶段诀：
**战设转运改，战略定方向，设计出方案，转移上生产，运营保稳定，改进不停歇**

事故vs问题诀：
**事故快恢复治标，问题找根因治本，事件管理先检测，问题管理后根除**

架构模式诀：
**B/S零装易维护，C/S性能交互强，MVC分层各司职，微服务拆分独立跑**

风险监控诀：
**再评估找新险，审计查效偏差算，储备分析余量看，状态会议定期审**

SLA三层诀：
**SLA对客户承诺，OLA内部运营定，UC供应商支撑，三层协议保服务**
