## 今日核心概念

### 1. 信息安全等级保护2.0（等保2.0）

定义：网络安全等级保护2.0是我国信息安全领域的基础制度，由《信息安全技术 网络安全等级保护基本要求》（GB/T 22239-2019）规定，覆盖云计算/物联网/移动互联网/工业控制系统等新技术场景。

关键要点：
- 五个等级：一级（自主保护/用户系统）、二级（指导保护/一般企业）、三级（监督保护/重要系统，如政务/金融/能源/医疗/教育）、四级（强制保护/关键基础设施）、五级（专控保护/涉密系统）
- 定级要素：受侵害的客体（公民/法人/社会秩序/国家安全）+ 对客体造成侵害的程度（一般损害/严重损害/特别严重损害）二维矩阵
- 二级 vs 三级关键区分：三级要求更严格的安全管理（安全审计/集中管控/边界保护/入侵防范），物理环境要求有机房电磁防护等
- 五个规定动作：定级→备案→安全建设整改→等级测评→监督检查，简称"定备建测监"
- 二级每两年测评一次，三级每年测评一次，四级每半年一次

### 2. 软件架构评估方法（ATAM与SAAM）

定义：软件架构评估是对软件架构设计方案进行分析和评价的方法，识别架构风险、敏感点和权衡点。ATAM（架构权衡分析法）和SAAM（软件架构分析法）是两种主流方法。

关键要点：
- ATAM四阶段：场景与需求收集→架构视图描述与场景映射→属性模型构造与分析→折中点识别。按质量属性（性能/可用性/安全性/可修改性）的效用树对架构进行多目标分析
- SAAM六步骤：场景开发→架构描述→场景分类与分配→单个场景评估→场景交互评估→整体评估（最早提出，关注可修改性）
- 敏感点：只影响一个质量属性的架构决策；权衡点：影响多个质量属性的架构决策（如加密→安全↑但性能↓）
- 风险点：架构决策可能影响多个质量属性，但实际未满足目标
- 考试应用：给定架构方案，识别风险/敏感点/权衡点；ATAM适用于复杂系统多属性评估，SAAM适用于中小型系统

### 3. 大型复杂信息系统项目管理（多项目/项目集/项目组合）

定义：大型复杂信息系统通常涉及多个子项目/项目集/项目组合的管理，需要在组织战略层面进行统筹和协调。

关键要点：
- 项目集（Program）：一组相关且协调管理的项目，目的是获得单独管理无法获得的效益。强调"协同效应"和"组件项目间的依赖管理"
- 项目组合（Portfolio）：为实现战略目标而集中管理的一组项目/项目集/其他工作。强调"投资回报最大化"和"战略对齐"
- 三者关系：项目组合 ⊇ 项目集 ⊇ 项目，项目组合管战略，项目集管协同，项目管交付
- 共同协调：项目集管理通过共同的资源分配、风险管理和利益协调机制，确保各组件项目协同实现预期效益
- 项目集经理 vs 项目经理：项目集经理关注组件间协调和整体效益，项目经理关注单个项目目标的实现
- 收益实现：项目集收尾的核心是收益实现管理（Benefits Realization），而非单纯的交付物验收

### 4. 系统可靠性设计与容错技术

定义：系统可靠性设计是确保系统在规定条件下和规定时间内完成规定功能的能力的设计技术。容错技术是可靠性设计的重要手段，通过冗余实现故障容忍。

关键要点：
- 可靠性指标：MTBF（平均故障间隔时间，Mean Time Between Failures）、MTTR（平均修复时间，Mean Time To Repair）、可用度 A = MTBF / (MTBF + MTTR)
- 串联系统：整体可靠性 = 各子系统可靠性的乘积（R = R1×R2×...×Rn），子系统越多整体可靠性越低
- 并联系统：整体不可靠性 = 各子系统不可靠性的乘积（F = F1×F2×...×Fn），子系统越多整体可靠性越高
- 冗余分类：结构冗余（双机热备/集群/三模冗余TMR-N模冗余NMR）、信息冗余（校验码/RAID/CRC/海明码）、时间冗余（重试/回滚/恢复块/N版本程序设计）
- 考试高频：给定子系统数量和单个可靠性，计算串/并联系统可靠性；RAID级别（0/1/5/6/10）区分与应用场景

### 5. 项目管理过程组与知识领域交叉分析

定义：项目管理五大过程组（启动/规划/执行/监控/收尾）与十大知识领域（整合/范围/进度/成本/质量/资源/沟通/风险/采购/干系人）交叉形成项目管理全景视图。

关键要点：
- 47个过程分布：启动（2）/规划（24）/执行（10）/监控（12）/收尾（1）
- 高频过程归属：项目章程→启动；WBS/进度计划/风险登记册→规划；指导与管理项目工作/管理团队/实施采购→执行；监控项目工作/实施整体变更控制→监控；结束项目或阶段→收尾
- 规划过程组占比最多（24个过程），反映"计划先行"的项目管理哲学
- 监控过程组贯穿项目始终：监控项目工作/实施整体变更控制/控制范围/控制进度/控制成本/控制质量等多个监控过程
- 整合管理是"管理其他知识领域的过程"：项目章程→项目管理计划→指导与管理项目工作→监控项目工作→实施整体变更控制→结束项目或阶段，形成PDCA循环
- 收尾过程组虽然只有"结束项目或阶段"一个过程，但涉及合同收尾、行政收尾、经验教训登记册更新三项关键工作

## 案例分析题

**题目**：

某大型国企启动"集团数字化转型"项目集，包含三个子项目：ERP核心系统建设（预算2000万，工期18个月）、CRM客户管理系统（预算800万，工期12个月）、BI商业智能平台（预算1200万，工期15个月）。项目集经理老李统筹管理。

在ERP子项目执行第8个月时，发生以下情况：

**问题1**：ERP子项目经理小张在第6个月时进行了一次架构评审，仅评估了"性能"一个质量属性。评审中发现数据库采用"集中式单库"方案存在性能问题，建议改为"分布式分库分表"方案。请指出：
（1）小张的评审方法与哪种主流架构评估方法最相似？
（2）该方法（ATAM/SAAM）的标准流程是什么？
（3）采用"分布式分库分表"方案后，预期会带来哪些质量属性的"权衡点"（即哪些属性改善、哪些属性变差）？

**问题2**：ERP系统需要承载集团核心业务，被定为三级等保系统。请回答：
（1）等保2.0的"定级→备案→安全建设整改→等级测评→监督检查"五个动作中，本项目需要在何时完成等级测评？
（2）三级系统相比二级系统，在物理安全、网络安全、主机安全、应用安全、数据安全五个层面，至少各增加一项关键技术要求（至少举3个层面）。

**问题3**：ERP系统采用三台服务器做集群（双机热备+一台冷备），单机MTBF=1000小时，MTTR=10小时。CRM系统采用双机热备，单机MTBF=800小时，MTTR=20小时。请计算：
（1）ERP集群的可用度（按双机并联考虑，不计冷备）
（2）CRM双机的可用度
（3）若将ERP和CRM串联为业务流（ERP失效则CRM不可用，反之亦然），整体可用度是多少？

**问题4**：项目集经理老李应该如何协调三个子项目以实现项目集的整体效益？请从项目集管理核心活动角度给出建议（至少4条）。

**参考答案**：

**问题1**：
（1）小张的评审方法与SAAM（软件架构分析法）最相似，因为SAAM最早提出且最初主要关注可修改性/单一质量属性，但更接近单属性分析，与ATAM多质量属性效用树方法不同。
（2）ATAM标准四阶段：
- 阶段1：场景与需求收集（use case/质量属性场景/约束/环境）
- 阶段2：架构视图描述与场景映射（用多种视图描述架构，将场景映射到架构元素）
- 阶段3：属性模型构造与分析（效用树：质量属性→属性求精→场景）
- 阶段4：折中点识别（分析敏感点/权衡点/风险点）

（3）"分布式分库分表"方案的权衡点分析：
- 改善的属性：性能（并发处理能力提升）、可扩展性（可横向扩展节点）、可用性（单库故障影响范围缩小）
- 变差的属性：数据一致性（分布式事务难以保证强一致，CAP定理制约）、可维护性（运维复杂度显著增加，数据迁移/扩容/数据均衡困难）、可修改性（业务代码需要适配分库分表）、开发成本（开发难度增加，工期延长）、成本（需要更多服务器和数据库License）

**问题2**：
（1）测评时间：三级系统应在安全建设整改完成后、上线运行前完成首次等级测评，且每年开展一次测评。
（2）三级系统相比二级系统的关键技术增强（至少3个层面）：
- 物理安全：机房出入口电子门禁+双人共管、机房电磁防护（电磁屏蔽/电磁干扰防护）、视频监控覆盖关键区域
- 网络安全：网络架构分区分域（生产区/办公区/外联区/测试区）、关键网络设备冗余、入侵防范（IDS/IPS部署）、安全审计（集中审计网络中的安全事件）
- 主机安全：身份鉴别（两种或以上鉴别方式，如口令+数字证书）、强制访问控制、剩余信息保护、安全审计集中管控
- 应用安全：会话超时失效、防范SQL注入/XSS/CSRF、关键操作二次认证、应用安全审计
- 数据安全：数据传输加密（HTTPS/SSL/TLS）、数据存储加密（敏感字段加密）、数据备份与恢复（异地备份）、数据脱敏

**问题3**：
可用度公式：A = MTBF / (MTBF + MTTR)
单机不可靠性：F = MTTR / (MTBF + MTTR)

（1）ERP单机可用度：A1 = 1000/(1000+10) ≈ 0.990
ERP单机不可靠性：F1 = 1 - 0.990 = 0.010
ERP双机并联（热备）：整体不可靠性 = F1 × F1 = 0.01 × 0.01 = 0.0001
ERP双机并联可用度 = 1 - 0.0001 = 0.9999（即99.99%）

（2）CRM单机可用度：A2 = 800/(800+20) ≈ 0.9756
CRM单机不可靠性：F2 = 1 - 0.9756 = 0.0244
CRM双机并联：整体不可靠性 = F2 × F2 = 0.0244² ≈ 0.000595
CRM双机并联可用度 = 1 - 0.000595 ≈ 0.999405（即99.94%）

（3）串联整体可用度：
A整体 = A_ERP × A_CRM = 0.9999 × 0.999405 ≈ 0.999305（约99.93%）

业务可用度评估：单点可用度99.99%的ERP对整体可用度贡献有限，CRM相对ERP可用度较低，成为业务流的薄弱环节，应优先提升CRM单机可靠性（如增加第三台热备或优化MTTR）。

**问题4**：
项目集管理核心活动与建议：
1. 项目集战略一致性管理：确保三个子项目（ERP/CRM/BI）的目标与集团"数字化转型"战略对齐，建立战略地图（Strategy Map），明确每个子项目对战略目标的贡献度
2. 跨项目依赖管理：识别三个子项目之间的依赖关系（如BI需要ERP和CRM提供数据源，CRM需要ERP的主数据），建立依赖矩阵和关键路径监控机制
3. 统一资源协调：在项目集层面建立资源池，统一调度关键资源（架构师/数据库专家/测试经理），避免子项目间资源争夺
4. 统一变更控制：建立项目集级CCB，对涉及多子项目接口的变更进行集中评审，避免局部变更引发整体影响
5. 收益实现管理（核心）：项目集收尾不等于交付物验收，必须持续跟踪业务效益（ROI/客户满意度/库存周转率等），确保"转型"目标真正落地
6. 风险统筹管理：建立项目集级风险登记册，对跨子项目的共性风险（如厂商交付风险/数据治理风险）进行统一应对
7. 治理与沟通：建立项目集指导委员会（Steering Committee），定期向集团高层汇报整体进展

**踩分要点**：
- 问题1：识别SAAM/ATAM（1分）；ATAM四阶段完整（2分）；权衡点至少列出3个改善和3个变差属性（3分）
- 问题2：测评时间正确（每年一次/上线前首次，1分）；至少3个层面的三级增强要求（每层2分，共6分）
- 问题3：可用度公式正确（1分）；ERP双机并联计算（2分）；CRM双机并联计算（2分）；串联整体计算（2分）
- 问题4：每条建议具体可操作（每条2分，共8分）；至少提到"收益实现管理"作为核心（1分）
