## 今日核心概念

### 1. 项目管理过程组与知识领域全景地图

定义：项目管理体系由五大过程组（启动/规划/执行/监控/收尾）和十大知识领域（整合/范围/进度/成本/质量/资源/沟通/风险/采购/干系人）交叉构成，共包含47个过程。

关键要点：
- 五大过程组关系：启动→规划→执行→监控（贯穿始终）→收尾，监控过程组与其他四组并行，覆盖全过程
- 十大知识领域分布：整合管理7个过程、范围管理6个、进度管理7个、成本管理4个、质量管理3个、资源管理6个、沟通管理3个、风险管理7个、采购管理3个、干系人管理4个（合计50个左右，不同版本略有差异）
- 核心记忆：过程组是"时间维度"（做什么先后），知识领域是"专业维度"（管什么内容）
- 47个过程中，监控过程组独占12个（最多），整合管理次之7个——体现"监控无处不在、整合贯穿全程"
- 考试要点：判断过程所属的过程组与知识领域（如"控制范围"属于监控过程组+范围知识领域）

### 2. 项目收尾管理——行政收尾与合同收尾

定义：项目收尾过程组包含"结束项目或阶段"和"采购收尾"两个核心过程，分为行政收尾和合同收尾两类。

关键要点：
- 行政收尾：贯穿项目始终（每个阶段结束都要做），产出经验教训登记册、资源归档、绩效报告
- 合同收尾：仅针对有采购合同的项目，核心是核实合同条款完成情况、签署收尾文件
- 收尾工作清单：范围确认已完成、可交付物已交付、文档已归档、合同已关闭、经验教训已记录、资源已释放
- 行政收尾前置条件：所有过程组的过程均已完成（或被批准提前终止）
- 考试陷阱：经验教训登记册"在项目早期就开始创建，整个项目期间不断更新"，收尾时输出最终版
- 项目提前终止也必须进行收尾，且要明确记录提前终止的原因

### 3. 信息系统服务管理（ITSM）与ITIL

定义：ITSM是一套以服务为中心、对IT系统进行规划、设计、运营和管理的方法论体系；ITIL（Information Technology Infrastructure Library）是ITSM的典型实践框架。

关键要点：
- ITIL v2九流程：服务支持（事件/问题/变更/发布/配置管理）+服务交付（可用性/能力/连续性/IT服务财务管理）
- ITIL v3五生命周期：服务战略、服务设计、服务转换、服务运营、持续服务改进（CSI）
- 服务运营五流程：事件管理、问题管理、变更管理、发布管理、配置管理
- 三大核心角色：服务台（Service Desk）一线响应、技术支持二线解决、研发团队三线根因修复
- 关键协议：SLA（服务级别协议/与客户）、OLA（运营级别协议/内部部门间）、UC（支持合同/与外部供应商）
- 关键概念：事件（影响业务的突发中断）vs 问题（事件根因）——事件解决快，问题解决慢

### 4. 软件需求工程——需求获取与需求规格说明

定义：需求工程是软件开发中获取、分析、规格说明、验证和管理需求的系统性活动，是软件工程的基础环节。

关键要点：
- 需求获取七种方法：访谈、问卷调查、观察、原型法、头脑风暴、焦点小组、文档分析（联合应用规划会议JAD）
- SRS（软件需求规格说明）八大特性：正确性、无歧义、完整性、可验证、一致性、可修改、可追踪、有序组织
- 需求跟踪矩阵RTM：建立需求→设计→编码→测试的双向追溯关系，每条需求至少对应一个测试用例
- 需求分类：业务需求、用户需求、功能需求、非功能需求（性能/安全/可用性/可维护性）、设计约束
- 需求变更控制：所有变更必须经过CCB评审，影响分析后再批准或拒绝
- 考试要点：判断哪些是功能需求（系统做什么）vs 非功能需求（系统做得多好）

### 5. 项目质量管理——质量保证（QA）与质量控制（QC）

定义：QA是过程导向的预防性活动（关注过程是否规范），QC是产品导向的检查性活动（关注产品是否合格）。

关键要点：
- QA核心目标：审计过程合规性、预防缺陷发生、提高团队过程能力
- QC核心目标：通过检查和测试识别缺陷、验证可交付物符合质量标准
- 工具区分：QA用过程审计、过程分析、根因分析；QC用因果图、帕累托图、控制图、测试用例
- 关系模型：QA是"建造桥梁前的图纸审查"，QC是"桥梁建成后的承重测试"
- 考试陷阱：质量保证是"管理过程"，由项目经理/质量经理主导；质量控制是"技术过程"，由测试团队主导
- QA成果：质量审计报告、过程改进建议；QC成果：测试报告、缺陷清单、验收结论

## 案例分析题

**题目**：

某省"政务大数据共享交换平台"项目，预算2500万元，工期24个月，由A公司中标承建。项目经理小李在项目收尾阶段遇到一系列问题：

（1）项目主体功能已上线运行6个月，业务部门反馈整体运行稳定。但项目组发现还有3个次要功能模块未完成（属于非关键业务功能），且合同约定的"数据治理标准制定"工作也未启动。

（2）项目期间发生了多次需求变更，CCB均已审批通过，但小李发现经验教训登记册中只记录了2条内容，且多数变更的应对过程未形成文档。

（3）项目采购了3家分包商的硬件和软件，合同金额合计800万元。3家分包商均已按合同交付货物并通过验收，但分包合同尚未正式关闭（分包合同管理专员说"等明年再做"）。

（4）客户要求尽快完成项目收尾，签署最终验收报告。但项目经理小李认为3个次要功能未完成，不能签署验收报告。

**问题1**：分析本项目收尾管理中存在的主要问题。（6分）

**问题2**：针对行政收尾和合同收尾，分别给出整改建议。（8分）

**问题3**：针对质量保证（QA）与质量控制（QC）的差异，分析小李拒绝签署验收报告的做法是否合理，并说明理由。（6分）

**参考答案**：

**问题1**：
本项目收尾管理存在的主要问题：
1. 范围基线管理失控：合同约定的"数据治理标准制定"工作未启动，说明范围基准未得到有效执行
2. 经验教训沉淀不足：项目经历多次变更，但经验教训登记册仅记录2条，知识资产大量流失
3. 分包合同收尾滞后：3家分包商已交付并通过验收，但分包合同未正式关闭，存在合规风险
4. 收尾决策不合理：项目经理以次要功能未完成为由拒绝签署验收报告，未区分"关键可交付物"与"次要可交付物"
5. 文档管理不规范：变更过程的应对方法未形成文档，组织过程资产沉淀不足

**问题2**：
行政收尾整改建议：
1. 立即组织经验教训总结会，邀请项目核心成员（开发、测试、运维、客户代表）参加，对项目全生命周期的成功经验与失败教训进行系统梳理
2. 更新经验教训登记册，按"问题描述/根本原因/应对措施/预防建议"四要素记录，每条变更都应形成复盘记录
3. 完成项目档案归档：项目计划、需求文档、设计文档、测试报告、变更记录、会议纪要全部归档
4. 对3个次要功能模块与"数据治理标准制定"工作进行分析：评估剩余工作量与影响范围，与客户协商处理方案（放入下一阶段/变更范围/补充合同）
5. 签署最终验收报告：若关键功能已交付并稳定运行，可签署有条件验收报告，明确次要项的处理计划

合同收尾整改建议：
1. 立即启动3家分包合同的收尾流程：核实合同条款完成情况、确认货物验收文件齐全
2. 签署分包合同收尾协议，明确双方权利义务终止
3. 完成合同档案归档，包括合同正本、补充协议、验收文件、付款记录
4. 评估分包商绩效，形成供方绩效评估文档（作为后续采购的参考）
5. 释放履约保证金，办理最终付款结算

**问题3**：
小李拒绝签署验收报告的做法不合理。

QA与QC的差异分析：
- QA（质量保证）是过程导向的预防性活动，关注过程是否规范；QC（质量控制）是产品导向的检查性活动，关注产品是否合格
- 验收报告的签署属于QC范畴——即对已交付产品进行检查和确认
- 项目主体功能已上线运行6个月且稳定，3个未完成的次要功能模块属于非关键业务功能，不应影响整体验收
- 正确做法：识别关键可交付物与次要可交付物，对关键功能进行QC检查（已通过），对次要功能与"数据治理标准制定"列入遗留问题清单，签署有条件验收报告

小李的拒绝反映出：
1. 将QA（过程合规）与QC（产品合格）混淆——QA要求过程合规，不等于所有过程都100%完成才能验收
2. 未对范围基线进行分层管理——关键与次要功能应区别对待
3. 收尾决策缺乏与客户的充分沟通——应通过协商达成共识

合理做法：组织最终验收评审会，对已完成的关键功能进行QC确认，对未完成的次要项达成书面处理协议，签署有条件验收报告。

**踩分要点**：
- 问题1：范围失控（1分）、经验教训不足（1分）、合同收尾滞后（1分）、收尾决策不合理（1分）、文档不规范（1分）、能说出5个主要问题（1分）
- 问题2行政收尾：经验教训总结会（1分）、登记册更新（1分）、档案归档（1分）、遗留项处理（1分）、有条件验收（1分），共5分；合同收尾：分包合同收尾（1分）、收尾协议（1分）、合同归档（1分）、供方评估（1分）共3分
- 问题3：QA vs QC区分正确（2分）、拒绝签署不合理判断（2分）、理由阐述清晰（2分）

整合最多七过程，监控覆盖十二最。

行政收尾贯全程，合同收尾采购停。
经验教训早建档，次要遗留清单清。
关键可交付QC验，有条件验收莫迟疑。

ITIL框架ITSM，v3五生命周期。
事件问题速响应，配置变更控风险。
SLA对客OLA内，UC对供应商言。

需求获取七方法，访谈观察原型佳。
SRS八特性，RTM双向追。

**QA与QC口诀**：
QA看过程QC看品，预防检查两分明。
QA过程审计重在控，QC测试验证在终验。
