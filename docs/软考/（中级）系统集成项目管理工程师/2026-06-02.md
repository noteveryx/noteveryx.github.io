## 今日核心概念

### 1. 成本估算方法

定义：对完成项目活动所需资金进行近似估算的过程，包含四种主要方法。

类比估算：利用历史类似项目的实际成本作为当前项目成本估算的依据，优点是成本低、速度快，缺点是准确性较差，适用于项目早期信息不足时。

参数估算：利用历史数据之间的统计关系和其他变量进行估算，如每平方米建设成本×总平方米数，准确度取决于参数模型的成熟度和基础数据的可靠性。

自下而上估算：对工作包进行详细估算后逐层汇总，先估算最底层活动的成本再向上汇总，准确度最高但耗时最长。

三点估算（PERT）：基于最乐观成本（O）、最可能成本（M）、最悲观成本（P）计算期望成本，公式为期望值=(O+4M+P)/6，标准差=(P-O)/6，用于应对不确定性。

关键要点：估算精确度随项目推进逐步提高（粗略量级估算-50%~+100%、预算估算-10%~+25%、确定性估算-5%~+10%）；三点估算比单点估算更具可靠性；成本估算需考虑风险储备。

### 2. 沟通模型与沟通方法

定义：沟通模型描述了信息从发送方到接收方的传递过程，沟通方法则是沟通信息传递的具体方式。

沟通模型五要素：编码（发送方将思想转化为语言或符号）、信息和反馈（传输媒介）、媒介（传递信息的渠道，如会议、邮件）、噪声（干扰信息传递的因素）、解码（接收方将信息还原为思想）。

沟通方法三类：推式沟通（将信息主动发送给接收方，如邮件、报告、群发消息，优点是不需接收方主动获取，缺点是可能不被阅读）；拉式沟通（信息存储在公共位置供接收方自行获取，如知识库、共享文件夹、企业内网，适用于信息量大的场景）；交互式沟通（双方或多方进行多向信息交换，如会议、电话、即时通讯，效率最高但时间成本也最高）。

沟通公式：沟通渠道数=n×(n-1)/2，其中n为干系人数量。干系人从4人增加到5人，渠道数从6增加到10。

关键要点：沟通方法不存在绝对优劣，需根据干系人需求、信息紧急程度、项目环境灵活选择；沟通管理计划中应明确谁需要什么信息、何时需要、以何种方式传递、由谁发送。

### 3. 马斯洛需求层次理论与赫兹伯格双因素理论

定义：马斯洛需求层次理论认为人的需求是分层次的，低层需求满足后高层需求才成为激励因素。赫兹伯格双因素理论将影响员工满意度的因素分为保健因素和激励因素。

马斯洛五层需求：生理需求（工资、工作环境）→ 安全需求（工作保障、医疗保险）→ 社交需求（团队归属感、人际关系）→ 尊重需求（认可、职位、成就感）→ 自我实现（发挥潜能、创造性工作）。项目经理需要识别团队成员处于哪个需求层次，采用对应的激励手段。

赫兹伯格双因素：保健因素（工作条件、工资福利、公司政策、人际关系）——满足时不会产生满意，不满足时会产生不满意；激励因素（成就感、认可、晋升机会、成长空间、工作挑战性）——满足时产生满意，不满足时不会不满意。实质是"保健因素消除不满，激励因素产生满意"。

关键要点：马斯洛理论适用于分析个体需求层次，赫兹伯格理论适用于工作环境设计；项目经理应保障保健因素、强化激励因素；两个理论在案例分析中经常联合使用。

### 4. 范围确认与范围控制

定义：确认范围（Validate Scope）是正式验收已完成的项目交付成果的过程，由客户或发起人对项目可交付成果进行审查和验收。控制范围（Control Scope）是监督项目和产品的范围状态、管理范围基准变更的过程，贯穿项目全过程。

确认范围核心：对象是已完成的可交付成果，主体是客户或发起人，输出是验收的可交付成果和变更请求（如有不合格），与质量控制（QC）的区别在于QC关注"做得对不对"（正确性），确认范围关注"做的是不是这个"（验收）。

控制范围核心：防止范围蔓延（未获批准的范围扩大，也叫项目范围潜变），管理范围基准变更，只有通过整体变更控制流程批准的变更才能纳入范围基准。范围蔓延的典型案例：客户不断提出"小的额外需求"，项目团队未经变更流程直接接受，导致成本超支、进度延误。

关键要点：确认范围通常在项目或阶段末期进行，也可在中间节点分步验收；确认范围的结果直接影响项目验收和交付；范围控制需与整体变更控制联动，未批先做是大忌。

### 5. 软件生命周期模型

定义：软件生命周期模型描述了软件从概念到退役的全过程阶段划分及各阶段的顺序关系。

瀑布模型：线性顺序执行（需求→设计→编码→测试→运维），每个阶段完成后才能进入下一个阶段，特点是文档驱动、阶段明确、变更成本高。适用于需求明确且稳定的项目。

V模型：瀑布模型的变体，强调各开发阶段与对应测试阶段的关系（需求分析↔验收测试、概要设计↔系统测试、详细设计↔集成测试、编码↔单元测试），强调"左验证右确认"，测试贯穿全过程。

原型模型：先开发一个可运行的原型让用户试用，根据反馈完善需求后再正式开发，适用于需求不明确的项目，缺点是用户可能将原型当作最终产品。

迭代模型：将软件拆分为多个迭代周期，每个迭代都经过完整的开发过程，逐步交付增量。RUP（统一过程）是典型代表，分为初始、细化、构造、移交四个阶段。

敏捷模型：以用户需求演进为核心，通过短周期（冲刺）持续交付有价值的软件，强调个体与交互高于过程与工具、可工作的软件高于详尽的文档、客户合作高于合同谈判、响应变化高于遵循计划。Scrum是典型框架。

关键要点：没有"最好的"模型，只有最合适的模型；考试常见考点是不同模型的适用场景对比。

## 案例分析题

**题目**：

某省政务服务平台升级项目预算1000万元，工期12个月。项目经理李明在项目规划阶段采用自下而上估算法得出成本基准为960万元（含应急储备40万元），其中管理储备80万元。

项目执行到第7个月末时，状态如下：PV=580万元，EV=500万元，AC=600万元。已完成的工作中有3个模块已经过内部质量控制测试通过，但尚未提交甲方验收。

李明发现团队骨干张工近期工作积极性明显下降，经了解：张工认为工资长期未涨（保健因素），且近期连续加班但未获得任何表彰（激励因素）。同时，因张工情绪问题，项目组内部沟通渠道从之前的10条增加到15条。

甲方项目经理王某在第7个月末的非正式沟通中提出希望新增一个移动端适配功能，李明口头答应并表示"小功能，不花钱，我们顺便做了"。该功能实际需要额外投入约15万元成本。

**问题1**（8分）：计算第7个月末的CV、SV、CPI、SPI，并分析项目当前的成本和进度绩效。

**问题2**（6分）：根据赫兹伯格双因素理论，分析张工满意度下降的原因，并给出两个合理的激励措施。

**问题3**（4分）：计算原项目团队人数，以及当前团队人数（依据沟通渠道数变化）。新增了几位成员？

**问题4**（6分）：李明口头答应甲方新增移动端功能的做法是否妥当？请从范围控制、整体变更控制两个角度分析，并说明正确做法。

**问题5**（6分）：已完成内部QC测试但未提交甲方验收的3个模块，在进度计算中EV计入了这些工作，这一处理是否正确？简述确认范围与质量控制的区别，并指出验收应由谁主导。

**参考答案**：

**问题1**：
- CV = EV - AC = 500 - 600 = -100万元，成本超支100万元
- SV = EV - PV = 500 - 580 = -80万元，进度滞后80万元
- CPI = EV/AC = 500/600 ≈ 0.83，小于1，成本超支
- SPI = EV/PV = 500/580 ≈ 0.86，小于1，进度滞后
- 项目整体：成本超支、进度滞后，需分析原因并采取纠正措施

**问题2**：
原因分析：工资未涨属于保健因素缺失，导致不满意度上升；加班无表彰属于激励因素缺失，导致满意度下降。两方面叠加使张工积极性大幅降低。

激励措施：
- 推荐张工为优秀员工，争取公司层面的表彰和奖励（激励因素）
- 安排张工承担技术攻关任务，赋予更多自主权和挑战性工作（激励因素）
- （可选）与人力部门沟通薪资调整的可能性（保健因素）
- 减少不必要的加班或给予调休补偿（保健因素）

**问题3**：
原渠道数10条 → n(n-1)/2 = 10，解得n=5人
当前渠道数15条 → n(n-1)/2 = 15，解得n=6人
新增1位成员。

**问题4**：
做法不妥当，理由：
- 范围控制角度：该功能属于范围蔓延，未走变更控制流程就口头承诺，后续可能导致成本超支（15万）和范围失控
- 整体变更控制角度：新增功能属于重大变更，应提交变更请求→可行性分析→CCB审批→批准后更新范围基准和成本基准→通知干系人→执行变更

正确做法：
1. 记录甲方的变更请求
2. 分析新增功能对范围、成本、进度、质量的影响
3. 提交变更控制委员会（CCB）审批
4. 批准后方可实施，并更新项目管理计划和各基准

**问题5**：
EV计入的处理正确但验收流程尚未完成。已通过QC测试说明交付成果符合质量标准（质量达标），但需甲方正式验收后确认范围才算完成。

确认范围与质量控制的区别：
- 质量控制关注"交付成果是否正确建造"（是否合格），由项目团队内部执行
- 确认范围关注"交付成果是否满足客户需求"（验收通过），由客户或发起人主导
- 通常先QC（内部验证）再确认范围（外部验收），两者可并行但不能替代

验收应由甲方项目经理王某或甲方指定的验收人员主导执行。

**踩分要点**：

| 题号 | 踩分点 | 分值 |
|------|--------|------|
| 问题1 | CV=-100万，SV=-80万，各2分；CPI=0.83，SPI=0.86，各1分；绩效判断2分 | 8分 |
| 问题2 | 保健因素分析2分，激励因素分析2分，激励措施两条各1分 | 6分 |
| 问题3 | 原人数5人1分，当前6人1分，新增1人2分（需推导过程） | 4分 |
| 问题4 | 指出范围蔓延1分，整体变更流程分析2分，正确做法3分 | 6分 |
| 问题5 | EV处理判断1分，QC与确认范围区别3分，验收主导方2分 | 6分 |

## 记忆口诀

### 成本估算四法诀

类比估算法——"拿过去比现在，省时省力不太准，早期项目最好用"
参数估算法——"套公式算总量，模型要靠谱，数据要对路"
自下而上法——"从底往上加，每块都算清，最准也最慢"
三点估算法——"乐最可能悲，加四倍可能，除以六求期望，除以六求方差"

### 双因素理论诀

保健消不满——"工资环境管政策，缺了就闹有也不爽"
激励生满意——"成就认可升成长，有了就嗨缺也无事"
项目经理要记住——"保健要保住，激励要加码，双管齐下才是法"

### 沟通渠道公式诀

干系人增渠道暴——"渠道数=n×(n-1)÷2，一人加入渠道飞"
五人十条六人十五——"五个人十条线，加一个变十五，项目经理沟通忙"

### 范围控制警戒诀

范围蔓延要警惕——"口头答应不能算，变更流程必须走"
确认范围QC分——"QC做对没做对，确认范围验没验收"
"QC在前验在后，客户签字才算够"

### 生命周期模型场景诀

需求明确用瀑布——"稳稳当当线性走，文档驱动不回头"
需求模糊用原型——"先做样子再确认，免得后面白折腾"
快速交付用敏捷——"短周期迭代快，拥抱变化是常态"
复杂项目用迭代——"拆开分步做，每轮都有新收获"
