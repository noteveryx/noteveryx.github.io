## 今日核心概念

### 1. 配置管理与CMDB
定义：配置管理是识别、记录、控制和报告IT基础设施中所有配置项（CI）及其相互关系的过程，其核心产出是配置管理数据库（CMDB），为其他ITSM流程提供准确的配置信息支撑。
关键要点：
- 配置项（CI）：需要纳入配置管理控制的任何IT组件，包括硬件、软件、文档、人员、服务、许可证等，每个CI必须唯一标识
- CMDB与资产库的区别：资产库记录"有什么、值多少钱"，CMDB记录"怎么连接、依赖谁、属于谁"，前者是财务视角，后者是运维视角
- 配置基线（Baseline）：在特定时间点对CI集合的快照，用于变更前后对比和回退验证
- 配置审计：定期验证CMDB中的信息是否与实际环境一致，分为物理审计（现场盘点）和逻辑审计（数据比对）
- 配置管理的价值：提供准确的变更影响评估依据、加快故障定位速度、支持容量规划和安全管理

### 2. 容量管理
定义：容量管理是确保IT资源在当前和未来都能满足业务需求，同时以最优成本提供的服务管理流程，涵盖资源能力、服务能力和业务能力三个层面。
关键要点：
- 三个子流程：业务容量管理（将业务需求翻译为IT资源需求）、服务容量管理（监控和预测现有服务的资源使用趋势）、资源容量管理（管理具体IT资源的使用和性能）
- 容量管理的关键活动：需求预测（根据业务增长预估资源需求）、性能监控（实时采集CPU/内存/存储/网络等指标）、容量缺口分析（当前容量与预测需求的差距）、容量计划制定（包含扩容方案、预算和时间表）
- 阈值管理：设定资源使用率的预警阈值（如CPU利用率>70%告警、>85%严重告警），实现事前预警而非事后救火
- 容量数据库（CDB）：记录历史资源使用数据、业务增长数据、性能测试结果，是容量预测的数据基础
- 与变更管理联动：任何重大业务上线或扩容变更前，必须经过容量评估

### 3. 服务级别管理
定义：服务级别管理是协调IT服务提供方与客户之间关于服务级别协商、监控、评审和改进的全过程管理活动，是连接业务需求与IT服务交付的核心纽带。
关键要点：
- 服务级别管理流程闭环：需求收集→SLA起草→协商与签署→监控与报告→评审与改进，是一个持续循环的过程
- 服务目录（Service Catalog）：向客户正式发布的服务清单，分为业务服务目录（面向客户，描述业务价值）和技术服务目录（面向内部，描述技术细节）
- 服务级别协议三件套：SLA（服务提供方与客户之间）、OLA（内部运营级别协议，约定IT内部各团队之间的支持责任）、UC（支撑合同，与外部供应商签订）
- 服务报告机制：定期（月度/季度）向客户出具服务报告，内容涵盖SLA达成率、未达标事件清单、改进措施及进展
- 服务改进计划（SIP）：当SLA持续未达标时，必须制定结构化的改进计划，明确改进目标、措施、责任人、完成时限

### 4. 信息系统安全管理
定义：信息系统安全管理是为保护信息系统及其数据的机密性、完整性和可用性（CIA三要素），通过制定安全策略、实施安全措施、进行安全监控和审计来降低安全风险的综合管理活动。
关键要点：
- CIA三要素：机密性（Confidentiality，未授权不可访问）、完整性（Integrity，未授权不可篡改）、可用性（Availability，授权用户可随时访问）
- 安全管理核心活动：风险评估（识别资产、威胁、脆弱性）、安全策略制定（组织级安全方针）、安全控制实施（技术+管理+物理三个维度）、安全意识培训、安全事件响应、安全审计与合规检查
- 等保2.0（网络安全等级保护）：五级分类（1-5级），定级对象为信息系统，核心标准为GB/T 22239。定级流程：确定定级对象→初步定级→专家评审→主管部门审批→公安备案
- 安全控制三维度：技术控制（防火墙、加密、入侵检测）、管理控制（安全制度、人员管理、访问控制策略）、物理控制（门禁、监控、机房环境）
- 与ITSM的交叉点：安全事件管理属于事件管理的特殊类别；安全补丁管理属于变更管理的特殊类别

### 5. IT服务外包与供应商管理
定义：IT服务外包是组织将部分或全部IT服务职能委托给外部专业服务提供商执行，供应商管理是对外包关系进行全生命周期管控的过程，确保供应商交付满足合同约定和业务要求。
关键要点：
- 外包类型：基础设施外包（IDC托管、网络运维）、应用外包（开发、测试、维护）、业务流程外包（BPO，如客服中心、数据处理）
- 供应商管理全生命周期：供应商选型（评估能力、资质、口碑）→合同谈判与签署→服务交付监控→绩效考核→关系维护与持续改进→合同续约或终止
- 合同关键条款：服务范围与SLA、定价模式（固定价格/按量计费/混合计费）、数据安全与知识产权、退出机制（Exit Strategy，必须明确数据归还和知识转移的条款）
- 供应商绩效考核：定期评估交付质量、SLA达标率、响应速度、创新能力，建立供应商分级管理（战略供应商/重要供应商/一般供应商）
- 核心风险及应对：锁定风险（多供应商策略）、数据泄露风险（合同约束+审计权）、质量失控风险（明确度量指标+定期评审）

## 案例分析题

**题目**：某省医疗保障局将全省医保信息系统的运维工作外包给A公司，合同期为3年。合同中约定了"7×24小时运维响应，核心交易系统可用率不低于99.9%"。系统上线运行1年后，出现以下问题：

①2026年3月，医保结算系统在月底结算高峰期连续2次宕机，每次恢复时间超过6小时，影响了全省数千家定点医院和药店的正常结算；
②医保局信息中心多次向A公司提出"调取系统资源使用监控数据"的要求，A公司表示"该数据涉及公司技术机密，不便提供"，双方产生纠纷；
③A公司提供的月度运维报告仅包含"当月无重大故障"一句话，没有具体的SLA达标数据、故障处理记录和改进措施；
④合同中未约定数据归还条款，A公司的运维人员拥有生产数据库的直接访问权限，医保局对此缺乏管控；
⑤2026年4月，省医保局新增了异地就医直接结算业务，系统并发量翻倍，但A公司表示"合同中未包含此项需求，扩容需要另行付费并重新评估工期"。

请回答以下问题：

（1）针对上述五种情况，分别指出问题根因和改进措施。

（2）请设计该项目的服务级别管理方案，包含服务目录、SLA指标体系和监控报告机制。

（3）从信息安全角度，指出该外包项目存在哪些安全隐患，应如何整改？

**参考答案**：

（1）五种情况的问题根因与改进措施：

**情况①——结算高峰宕机且恢复时间长**

根因：容量管理缺失（未对月底结算高峰进行容量预测和资源预留）+ 应急响应能力不足（恢复超6小时，远超99.9%可用性要求对应的月度累计停机时间约43.8分钟）。

改进措施：
- 立即开展容量评估，根据月底结算峰值的并发量和交易量，计算所需的CPU、内存、数据库连接池等资源需求，实施扩容
- 建立容量预警机制，设定资源使用率阈值（如>70%预警、>85%严重告警），在月初自动触发扩容准备
- 梳理应急预案，将P0级故障恢复SLA缩短至≤2小时，并定期演练

**情况②——供应商拒绝提供监控数据**

根因：合同条款不完善，未明确约定"运维方有义务向甲方提供运行监控数据和性能报告"，数据透明度缺乏制度保障。

改进措施：
- 立即启动合同补充协议谈判，明确约定：甲方有权随时调取系统运行监控数据、性能报告、日志数据等运维关键信息
- 在合同中约定甲方拥有对系统的审计权，可定期或随机对运维过程进行审计检查
- 建立联合监控平台，甲方可实时查看系统运行状态和关键指标

**情况③——运维报告流于形式**

根因：服务报告机制缺失，合同中未约定报告的内容模板、提交频率和度量指标体系，供应商按照最低标准应付。

改进措施：
- 制定标准化的月度服务报告模板，必须包含：SLA达成率统计、故障清单及处理详情（含MTTR）、资源使用趋势、容量预警、已知问题跟踪、改进计划进展
- 在SLA中明确"月度服务报告在次月5个工作日内提交，甲方10个工作日内完成评审并反馈意见"
- 建立服务评审会议机制，每月召开甲乙双方联合服务评审会

**情况④——数据安全管控缺失**

根因：合同缺少数据安全条款、数据归还条款和退出机制，运维人员权限管理失控，存在严重的数据泄露和业务连续性风险。

改进措施：
- 立即回收A公司运维人员对生产数据库的直接访问权限，改为通过堡垒机/跳板机进行操作，所有操作留痕
- 建立最小权限原则：每个运维人员仅授予完成其职责所需的最小权限，定期审计权限分配
- 签订保密协议（NDA）和数据安全补充条款，明确数据安全责任、违规处罚和数据泄露通知义务
- 补充数据归还和知识转移条款，约定合同终止时A公司必须归还所有数据、文档、知识库，并在交接期间提供技术支持

**情况⑤——新增业务需求超出合同范围**

根因：需求管理机制缺失，合同未建立需求变更和扩容的处理流程，导致业务发展与服务交付脱节。

改进措施：
- 建立"业务需求→容量评估→SLA调整→资源扩容"的需求响应流程，明确需求响应的SLA（如一般需求15个工作日内评估完成，紧急需求5个工作日内）
- 在合同中约定变更管理机制：新增业务需求由双方联合评估影响和费用，签订补充协议后执行
- 每季度进行一次业务需求前瞻会议，甲方提前告知即将上线的业务变化，乙方提前做好容量和资源准备

（2）服务级别管理方案：

**服务目录设计**：

业务服务目录（面向省医保局业务部门）：

| 服务项 | 服务描述 | 服务时间 |
|---|---|---|
| 核心交易服务 | 医保结算、参保登记、费用报销等核心业务 | 7×24 |
| 异地就医服务 | 跨省/跨市异地就医直接结算 | 7×24 |
| 公共查询服务 | 参保信息查询、药品目录查询、定点机构查询 | 7×24 |
| 报表统计服务 | 月度/季度业务报表生成与推送 | 工作日 |

技术服务目录（面向信息中心）：

| 服务项 | 服务描述 | 服务时间 |
|---|---|---|
| 基础设施监控 | 服务器、网络、存储资源实时监控与告警 | 7×24 |
| 数据库管理 | 数据库性能优化、备份恢复、数据归档 | 7×24 |
| 安全管理 | 漏洞扫描、补丁更新、安全事件响应 | 7×24 |
| 变更实施 | 系统升级、配置变更、新功能部署 | 计划维护窗口 |

**SLA指标体系**：

| 指标项 | 目标值 | 计算方式 |
|---|---|---|
| 核心交易系统可用率 | ≥99.9% | （总时间-故障时间）/总时间×100% |
| P0级故障响应时间 | ≤15分钟 | 从故障确认到开始处理的时间 |
| P0级故障解决时间 | ≤2小时 | 从故障确认到服务恢复的时间 |
| P1级故障响应时间 | ≤30分钟 | 同上 |
| P1级故障解决时间 | ≤4小时 | 同上 |
| P2级故障响应时间 | ≤2小时 | 同上 |
| P2级故障解决时间 | ≤8小时 | 同上 |
| 月度服务报告提交 | 次月5个工作日内 | 报告交付及时率 |
| 需求评估响应 | 15个工作日内 | 需求接收至方案输出 |

**监控与报告机制**：

- 实时监控：通过联合监控平台，甲方实时查看系统可用率、资源使用率、告警状态
- 周报：每周一提交上周运行概况，包含告警统计、故障处理汇总、资源趋势
- 月报：次月5个工作日内提交完整月度服务报告，包含SLA达标率、MTTR统计、容量分析、已知问题跟踪、改进计划进展
- 季度评审：每季度召开联合服务评审会，评审SLA达标情况，讨论改进措施，修订下季度SLA目标

（3）信息安全整改措施：

**隐患一：运维人员拥有生产数据库直接访问权限**
整改：建立严格的访问控制体系。生产数据库访问必须通过堡垒机，实施"双人复核"操作机制；采用数据库审计系统记录所有SQL操作；定期回收离职人员权限。

**隐患二：合同缺少保密条款和数据安全约束**
整改：立即签订数据安全补充协议。明确数据分类分级标准（如参保人员个人信息为高敏感数据），约定数据使用范围、禁止数据导出、禁止将数据用于甲方业务以外的目的。

**隐患三：缺少数据归还和销毁条款**
整改：补充退出管理条款。约定合同终止时，乙方必须在30天内归还所有甲方数据（含备份数据），对存储介质上的甲方数据进行彻底销毁（多次覆写或物理销毁），并由第三方出具销毁证明。

**隐患四：甲方对运维过程缺乏审计权**
整改：在合同中明确甲方有权对乙方运维过程进行安全审计，审计内容包括：权限分配合理性、操作日志完整性、安全补丁及时性、数据访问合规性。审计频率不低于每半年一次。

**隐患五：未建立安全事件通报机制**
整改：约定乙方发现安全事件后必须在30分钟内通知甲方，24小时内提交初步调查报告。重大安全事件（数据泄露、系统被入侵等）必须立即启动应急响应，并配合甲方完成安全事件处置和事后分析。

**踩分要点**：
- 第（1）问必须"一一对应"，五种情况分别给出根因和措施，不能合并回答
- 情况①必须同时涉及容量管理和应急响应两个维度，不能只写"加服务器"
- 情况②的关键词是"数据透明度"和"审计权"
- 情况④必须提到"堡垒机""最小权限""数据归还条款"，这是高频考点
- 服务目录要分为"业务服务目录"和"技术服务目录"两个视角
- SLA指标必须可量化、可测量，不能用"尽快""尽量"等模糊表述
- 安全整改必须覆盖人员权限、合同约束、审计机制、应急响应四个层面

## 记忆口诀

**CMDB口诀**："资库管钱，配置管联——资产库记价格，CMDB记关系"

**容量管理三子流程口诀**："业服资——业务定需求，服务看趋势，资源算具体"

**服务级别三件套口诀**："SOU外内供——S对外承诺，O内部协调，U供方约束"

**安全CIA口诀**："CIA三要素——机密不能看，完整不能改，可用不能断"

**供应商管理口诀**："选签监考关——选型签合同，监控做考核，关停有退出"

**等保定级口诀**："定初评审备——定对象、初定级、专家评审、公安备案"
