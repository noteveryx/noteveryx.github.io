## 今日核心概念

### 1. 软件架构评估方法（ATAM/CBAM/SAAM）

定义：架构评估方法是一组用于系统地评价软件架构质量属性的方法论，通过识别风险、权衡点和敏感点来辅助架构决策。

关键要点：
- **ATAM（架构权衡分析法）**：9个步骤，参与者包括评估小组、项目决策者、管理者、架构师和用户代表；输出包括风险主题、风险场景、风险点、权衡点和非风险点
- **CBAM（成本效益分析法）**：在ATAM基础上对架构决策进行经济性分析，构建效用树对质量属性效用进行排序，量化每个决策的ROI
- **SAAM（场景架构分析法）**：早期评估方法，核心是基于场景的评估，通过评估场景实现成本来比较多个架构
- **敏感点vs权衡点vs风险点**：敏感点是影响一个质量属性的参数；权衡点是影响多个质量属性的参数；风险点是潜在问题
- **质量属性效用树**：将质量属性细化为可度量的子属性，构造成树形结构便于评估

### 2. 软件可靠性与可用性工程

定义：软件可靠性是指软件在规定条件下、规定时间内完成规定功能的能力；可用性是系统在任意时刻可正常工作的概率。

关键要点：
- **可靠性指标**：平均失效间隔时间MTBF、平均修复时间MTTR、失效率λ（浴盆曲线分早期/偶然/损耗三阶段）
- **可用性公式**：A = MTBF / (MTBF + MTTR)，常用"9"的个数度量（99.99%=4个9，年停机52分钟）
- **软件可靠性建模**：Musa模型、Jelinski-Moranda模型、Goel-Okumoto NHPP模型
- **可靠性增长测试**：通过不断修复缺陷使可靠性随测试时间增长而提高
- **容错设计**：N版本程序设计（独立开发多版本同时执行投票）、恢复块设计（主备切换）、冗余模式

### 3. 信息安全与等级保护2.0

定义：等级保护2.0是我国网络安全等级保护制度的核心标准体系，覆盖云计算、移动互联网、物联网、工业控制、大数据等新技术领域。

关键要点：
- **五个等级**：用户自主保护级（一级）、系统审计保护级（二级）、安全标记保护级（三级）、结构化保护级（四级）、访问验证保护级（五级）
- **等保2.0标准**：将原等保1.0的"自主/审计/标记/结构化/访问验证"重构为"安全物理环境、安全通信网络、安全区域边界、安全计算环境、安全管理中心"五个层面
- **等保2.0三特性**：被动防护→主动防御；静态防护→动态防护；单点防护→整体防控+感知
- **密码应用安全性评估（密评）**：与等保2.0协同，对密码应用进行专项评估
- **关键信息基础设施（CII）保护**：在等保基础上实施重点保护

### 4. 软件演化与维护策略

定义：软件演化是指软件系统在交付后为适应环境变化、用户新需求和缺陷修复而进行的修改活动；维护是演化的子集。

关键要点：
- **四种维护类型**（按Lehman演化定律）：改正性维护（修正错误，占20%）、适应性维护（环境变化，占25%）、完善性维护（功能增强，占50%）、预防性维护（防止未来问题，占5%）
- **Lehman七条演化定律**：持续变化、复杂度递增、程序组织演化、保守稳定性、增长有限性、激励衰退、内容衰退
- **再工程vs逆向工程**：逆向工程是从代码恢复设计；再工程是在逆向工程基础上重构以改善质量
- **重构技术**：代码重构（提取方法/内联/移动）、数据库重构、架构重构
- **技术债管理**：识别、量化、优先级排序、偿还的闭环管理

### 5. 系统测试策略与方法体系

定义：系统测试是将软件、硬件、外设、网络等元素集成后进行的完整测试，验证系统是否满足规定的需求。

关键要点：
- **测试方法分类**：按执行方式分静态测试（评审）和动态测试（执行）；按是否运行程序分白盒测试（结构测试）和黑盒测试（功能测试）
- **测试阶段**：单元测试→集成测试→系统测试→确认测试→验收测试，遵循"V模型"
- **集成测试策略**：自顶向下（需要桩模块）、自底向上（需要驱动模块）、三明治（混合）、基于调用图、基于路径
- **系统测试类型**：功能测试、性能测试（负载/压力/容量/并发）、安全性测试、兼容性测试、可靠性测试、可用性测试
- **回归测试**：修改后重新执行原有测试用例，可采用选择策略（覆盖修改法、回归测试选择、优先级排序）减少测试量

## 案例分析题

**题目**：某大型银行核心交易系统已运行8年，采用单体架构，承载日均800万笔交易。最近频繁出现可用性下降（MTTR=120分钟/MTBF=720小时）、性能瓶颈（峰值TPS仅2000）和安全合规不达标（未达等保2.0三级）等问题。计划进行架构现代化改造，预算5000万元，改造周期18个月。请回答以下问题：
1. 该系统当前可用性指标为多少？属于几个9级别？若要达到"5个9"指标，需要将MTTR降低到多少？
2. 在ATAM评估中，请识别该系统的敏感点、权衡点和风险点（每类至少2个）。
3. 请用CBAM方法对该项目进行成本效益分析应包含哪些核心步骤？
4. 针对等保2.0三级要求，从"安全计算环境"层面提出至少4项具体改造措施。
5. 该系统包含4个主要子模块（账户/交易/清算/风控），请设计集成测试策略并说明理由。

**参考答案**：

**问题1**：
- 可用性 A = MTBF / (MTBF + MTTR) = 720 / (720 + 2) = 720/722 ≈ 99.72%（年停机约24.5小时）
- 属于3个9级别
- 5个9要求 A=99.999%，即年停机≤5.26分钟。MTTR = MTBF × (1-A)/A = 720 × 0.00001/0.99999 ≈ 0.0072小时 ≈ 0.43分钟（约26秒）

**问题2**：
- 敏感点（影响单一质量属性）：①单点数据库成为性能瓶颈；②网络带宽限制交易并发；③单点登录认证服务器
- 权衡点（影响多个属性）：①交易一致性（高安全vs高可用）；②数据加密强度（安全性vs性能）；③模块化粒度（可维护性vs性能）
- 风险点：①MTBF过短（720小时）随时可能宕机；②MTTR过长（2小时）超出业务容忍度；③不满足等保2.0合规要求面临监管处罚

**问题3**：
CBAM核心步骤：
1. 收集场景并整理为质量属性效用树
2. 对效用树叶子节点进行量化（效用值1-9级）
3. 根据项目目标确定质量属性优先级和权重
4. 评估每个架构决策对效用树的贡献度
5. 计算每个架构决策的"期望值"并排序
6. 使用边际成本/收益分析确定最优决策组合
7. 输出风险和敏感点清单辅助决策

**问题4**：
1. 部署HIDS主机入侵检测系统+日志审计系统（6个月留存）
2. 实施强身份鉴别：双因素认证（MFA）+生物识别+数字证书
3. 数据库透明加密（TDE）+敏感字段应用层加密+密钥分级管理
4. 应用最小权限原则：RBAC细粒度授权+特权账号管理PAM
5. 部署WAF+代码审计+防SQL注入/XSS等应用层防护
6. 主机安全加固：基线检查+补丁管理+端口最小化

**问题5**：
推荐采用**三明治（混合）集成测试策略**：
- 底层：账户模块（被多个上层模块调用）→ 自底向上集成，先测账户基础服务
- 中间：交易模块、清算模块 → 渐增式集成，逐步叠加
- 顶层：风控模块（作为调度协调者）→ 自顶向下集成，使用桩模块模拟下层
- 最后：四模块全联调 + 端到端业务流测试

理由：①账户模块是基础设施，自底向上便于尽早发现基础缺陷；②风控是业务编排层，自顶向下便于验证业务流；③混合策略兼顾测试效率与覆盖度；④配合每日CI自动化集成，降低风险。

**踩分要点**：
- 可用性计算公式务必准确，5个9对应的MTTR计算是高频考点
- 敏感点/权衡点/风险点的区分：单属性vs多属性vs潜在问题是核心
- CBAM在ATAM基础上加入成本效益分析，是"评估→决策"的关键
- 等保2.0五大层面（物理/通信/边界/计算/管理）必须熟记，"安全计算环境"是改造重点
- 集成测试策略选择需结合模块依赖关系和测试目标，不能死记硬背

## 记忆口诀

1. **可用性等级口诀**："三个9一天、四个9一时、五个9一分"——99.9%年停8.76小时、99.99%年停52分钟、99.999%年停5分钟
2. **ATAM风险术语**："敏单权多非无险潜"——敏感点影响单属性、权衡点影响多属性、非风险点无影响、风险点有隐患
3. **等保2.0五层面**："物理通边算管"——物理环境、通信网络、区域边界、计算环境、管理中心（从底到顶）
4. **四种维护比例**："二正二适五完五预"——改正20%、适应25%、完善50%、预防5%
5. **容错三剑客**："多版本、备份件、双机热"——N版本程序设计、恢复块设计、双机热备
6. **集成测试策略**："顶桩底驱三明治"——自顶向下用桩模块、自底向上用驱动模块、三明治混合
