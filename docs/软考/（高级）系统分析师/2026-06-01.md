# 系统分析师每日学习 - 2026-06-01

## 今日核心概念

### 1. 软件体系结构描述语言（ADL）

定义：用于精确描述软件系统体系结构的形式化语言，通过构件（Component）、连接件（Connector）和配置（Configuration）三要素来刻画系统结构。

关键要点：
- **三要素**：构件描述计算单元（接口+实现），连接件描述构件间交互（角色+胶水），配置描述整体拓扑关系
- **代表性ADL**：Rapide（仿真模拟）、Wright（形式化验证连接件）、Acme（架构交换格式）、UniCon（异构集成）
- **与设计模式区别**：ADL面向宏观结构整体描述，设计模式面向局部解决方案复用
- **常见考查点**：ADL能否进行形式化分析验证、Acme作为架构描述交换标准、连接件vs接口的区别

---

### 2. 面向对象设计原则——依赖倒置与接口隔离

定义：依赖倒置原则（DIP）要求高层模块和低层模块都依赖抽象，而非具体实现；接口隔离原则（ISP）要求客户端不应依赖它不需要的接口，应将大接口拆分为小而专一的接口。

关键要点：
- **DIP核心**：抽象不依赖细节，细节依赖抽象；通过依赖注入（DI）实现，常见容器如Spring IoC
- **ISP核心**：胖接口是"接口污染"，应按角色拆分，一个接口只服务一类客户端
- **DIP与OCP关系**：DIP是实现开闭原则的技术手段，二者相辅相成
- **常见考查点**：判断代码或设计图违反了哪条原则，如何重构使其合规；依赖注入三种方式（构造器注入、Setter注入、接口注入）

---

### 3. 软件系统建模——状态图与活动图

定义：状态图（State Diagram）描述对象在其生命周期中响应事件时的动态行为，以状态迁移为核心；活动图（Activity Diagram）描述系统中工作流或算法的控制流，以活动为核心。

关键要点：
- **状态图组成**：初态（实心圆）、终态（同心圆）、状态（圆角矩形）、迁移（带触发事件[条件]/动作的箭头）、历史伪状态（H）
- **活动图组成**：活动节点、控制流边、分支（菱形）、并发（粗横线：分叉/汇合）、泳道（责任划分）
- **适用场景区别**：状态图用于生命周期复杂的对象（订单、工单、协议状态机）；活动图用于业务流程建模、并行流建模
- **常见考查点**：给定业务描述绘制或补全状态图/活动图、识别迁移上的守卫条件与动作、判断顺序图vs活动图的选用场景

---

### 4. 数据仓库建模——维度建模与缓慢变化维

定义：维度建模（Dimensional Modeling）是数据仓库中以事实表为核心、维度表为辐射的建模方法，由Ralph Kimball提出；缓慢变化维（SCD，Slowly Changing Dimension）是处理维度属性随时间变化问题的一组策略。

关键要点：
- **事实表**：存储可度量的业务事件（粒度要明确），包含退化维和外键
- **维度表**：宽表，存储描述性属性；雪花型=维度表被进一步规范化
- **SCD三种类型**：
  - Type 1（覆盖）：直接更新，丢失历史，适合纠错场景
  - Type 2（增行）：新增行+代理键+生效/失效日期，保留完整历史，最常用
  - Type 3（加列）：新增"旧值"列，仅保留一次历史变化
- **常见考查点**：选择合适SCD类型并给出理由、区分星型/雪花型/星座型模式、事实表粒度设计

---

### 5. 系统可维护性设计

定义：可维护性（Maintainability）是指软件在特定条件下，被修改（纠错、改进或适应环境变化）的容易程度，是ISO 25010质量模型中的重要特性。

关键要点：
- **子特性**：模块化（Modularity）、可重用性（Reusability）、可分析性（Analysability）、可修改性（Modifiability）、可测试性（Testability）
- **提高可维护性的设计手段**：高内聚低耦合、信息隐藏与封装、接口抽象、模块化分解、代码注释规范与文档同步、避免重复代码（DRY原则）
- **可维护性度量指标**：圈复杂度（McCabe）、代码行数、注释率、缺陷密度、MTTR（平均修复时间）
- **与可靠性关系**：可靠性强调"不失效"，可维护性强调"失效后可快速修复"，MTBF/MTTR/可用性三者关系：可用性=MTBF/(MTBF+MTTR)
- **常见考查点**：圈复杂度计算（节点-边+2P或区域数法）、给出代码分析耦合内聚级别、SLA对MTTR的要求

## 案例分析题

**题目**：

某省税务局正在建设新一代税收征管信息系统，计划将现有分散的各地市税务系统整合为统一的省级平台。系统需要对纳税人全生命周期状态（注册→正常申报→欠税→注销）进行精确管理，并构建全省纳税人画像数仓，支撑风险分析和稽查决策。

系统分析师小王在需求分析阶段面临以下问题：
1. 各地市系统中"纳税人"对象的状态迁移逻辑不一致，历史数据存在同一纳税人在不同系统中状态矛盾的情况；
2. 数据仓库建模时，"纳税人"维度表中的企业名称、法人代表、注册地址会随时间发生变化，需要保留完整历史变更记录；
3. 新系统对可维护性要求极高，要求任意模块的圈复杂度不超过10，并明确规定模块间耦合方式；
4. 系统中"申报处理"业务逻辑复杂，涉及多个部门并行审核，需要用合适的UML图进行建模。

**请回答以下问题：**

（1）请为"纳税人"对象设计状态图，说明主要状态和关键迁移触发事件（不少于4个状态，不少于5条迁移）。

（2）针对问题②，应选择哪种SCD类型？简述其实现方式及数据表结构设计要点（至少4个字段）。

（3）若某函数的控制流图有10个节点、12条边、1个连通区域，其圈复杂度为多少？若圈复杂度超过阈值，分析师应建议采取什么措施？

（4）"申报处理"涉及多部门并行审核，应选用状态图还是活动图？请说明理由，并描述活动图中表达并行的关键元素。

---

**参考答案**：

**（1）纳税人状态图设计：**

主要状态（5个）：
- **待核实**（初始注册，资料待审）
- **正常**（核实通过，正常申报）
- **逾期**（申报截止日未申报或未缴清）
- **欠税**（正式认定欠税，已产生滞纳金）
- **注销**（终态，已完成注销流程）

关键迁移（≥5条）：
- 待核实 →[提交资料完整且审核通过]→ 正常
- 正常 →[申报截止日未提交申报]→ 逾期
- 逾期 →[在宽限期内补报补缴]→ 正常
- 逾期 →[超过宽限期仍未缴清]→ 欠税
- 欠税 →[缴清欠税及滞纳金]→ 正常
- 正常/欠税 →[提交注销申请并完成清税]→ 注销

**（2）SCD类型选择：**

应选择 **SCD Type 2（增行法）**。

实现方式：当维度属性（企业名称/法人/地址）发生变化时，不修改原记录，而是新增一行，同时更新原记录的失效日期。

数据表关键字段设计（至少4个）：
| 字段名 | 说明 |
|---|---|
| surrogate_key（代理键） | 主键，自增整数，唯一标识每个历史版本 |
| taxpayer_id（自然键） | 纳税人识别号，可重复出现多行 |
| corp_name（企业名称） | 维度属性，随时间变化 |
| legal_person（法人代表） | 维度属性，随时间变化 |
| effective_date（生效日期） | 该版本生效时间 |
| expiry_date（失效日期） | 当前版本为NULL或9999-12-31，历史版本为实际失效时间 |
| is_current（当前标记） | Y/N，快速过滤当前有效版本 |

**（3）圈复杂度计算：**

公式：V(G) = E - N + 2P

- E = 12（边），N = 10（节点），P = 1（连通区域）
- V(G) = 12 - 10 + 2×1 = **4**

圈复杂度为4，未超过阈值10，符合要求。

若超过阈值，分析师应建议：
① **函数拆分**：将复杂函数按单一职责原则拆分为多个子函数；
② **消除嵌套条件**：使用卫语句（Guard Clause）提前返回，减少嵌套层次；
③**策略模式重构**：将复杂的多分支条件判断替换为多态或策略模式；
④ **代码评审与重测**：拆分后重新计算圈复杂度并回归测试。

**（4）选用活动图：**

应选用**活动图**。

理由：申报处理涉及多部门**并行**审核，是业务流程建模，关注的是活动执行顺序和并发关系，而非单个对象的状态机行为。状态图关注对象生命周期中的状态迁移，不适合描述跨部门并行流程。

活动图中表达并行的关键元素：
- **分叉节点（Fork）**：一条控制流进入，多条控制流并发发出（用粗实线横杠表示）
- **汇合节点（Join）**：多条并发控制流到达后，等待全部完成才向下流出（同样用粗实线横杠表示）
- **泳道（Swimlane）**：将不同部门的活动划分到各自泳道，清晰呈现责任边界

---

**踩分要点**：

1. **状态图**：迁移箭头上必须注明触发事件和守卫条件，仅画箭头不写事件名最多得一半分；初态和终态符号画错（初态=实心圆，终态=同心圆）直接扣分。

2. **SCD**：回答"加版本号"而不提代理键+生效失效日期，视为未掌握Type 2核心机制，扣2-3分；仅选Type 1（覆盖）明显不满足"保留完整历史"要求，该问得0分。

3. **圈复杂度**：公式写成V(G)=E-N+2时P漏写，若P=1则数值不变但公式扣1分；注意区分V(G)=区域数（另一种等价算法），两种方法均可。

4. **活动图vs状态图**：若选状态图，理由再充分也得0分（本题考查选型判断能力）；只选对不说明理由得1分，满分需理由+关键元素两部分齐全。

## 记忆口诀

**ADL三要素**：构件连接配置好，Rapide仿真Wright验证，Acme交换格式找。

**SCD三招**：Type 1 覆盖丢历史，Type 2 增行最常用（代理键+生失效），Type 3 加列记一次。

**圈复杂度两公式**：边减点加2P，或者数数封闭区域。阈值10是红线，超了就拆函数。

**状态图活动图选型口诀**：对象生命周期选状态，业务并行流程选活动；并行分叉汇合画横杠，泳道划分责任边界清。

**SOLID五原则速记**：单开里依接，单一/开闭/里氏/依赖倒置/接口隔离，DIP靠注入来实现，ISP胖接口要分家。

**可维护性可用性公式**：可用性=MTBF除以(MTBF+MTTR)，可靠不失效，维护快修复，可用是乘积。
