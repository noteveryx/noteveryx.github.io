# （高级）系统分析师 2026-06-23 学习内容

## 今日核心概念
### 1. 软件需求获取与需求分析
定义：需求获取是从用户、业务、环境等利益相关方中系统化收集、识别和提炼软件系统应满足的需求的过程。需求分析是在获取的基础上对需求进行分类、建模、协商、验证和管理的完整工程活动。

关键要点：
- **需求层次**：业务需求（组织战略层面"为什么做"）、用户需求（用户视角"做什么"）、功能需求（系统具体行为"输入→处理→输出"）、非功能需求（性能/安全/可用性/可维护性等质量属性）、约束条件（技术/法规/预算/时间）。层次间需双向追溯。
- **需求获取方法**：①访谈（结构化/半结构化，针对决策层和管理层）②问卷调查（针对大量分散用户）③观察法（深入现场记录真实工作流）④文档分析（研究现有制度、报表、表单）⑤原型法（快速构建原型引导用户表达）⑥JAD联合应用开发（多方代表集中会议高效达成共识）⑦头脑风暴（激发隐性需求）。不同方法组合使用效果最佳。
- **需求建模**：DFD数据流图（功能建模四要素：加工/数据流/数据存储/外部实体，需保持父图与子图平衡）、ER图（数据建模实体-联系-属性）、UML用例图（需求建模参与者-用例-关系）、状态转换图（行为建模）、判定表/判定树（复杂逻辑建模）。
- **需求验证与确认**：通过需求评审、需求测试、用例驱动验证确认需求完整性（覆盖所有业务）、正确性（不偏离用户意图）、可行性（技术可实现）、无歧义（表述唯一理解）、可测试（每条需求可设计测试用例）、可追溯（双向追溯到源头和实现）。

### 2. 软件架构评估方法体系
定义：软件架构评估是在架构设计完成后、详细设计实施前，对所选架构方案在质量属性、风险点、权衡点等方面进行系统性分析和验证的活动。架构评估是ATAM等方法的核心应用场景。

关键要点：
- **ATAM架构权衡分析法**：分九步执行——①描述ATAM方法（向利益相关者说明）②描述业务动机（业务目标驱动）③描述架构（架构文档讲解）④确定质量属性（效用树生成）⑤分析质量属性效用树（场景优先级排序）⑥分析架构方法（评估架构决策）⑦识别风险点与非风险点（场景是否被架构处理）⑧分析风险点（根因分析）⑨形成评估结论。ATAM核心是效用树（质量属性→场景→优先级）。
- **架构评估核心术语**：敏感点（一个或多个组件对某个质量属性有显著影响）、权衡点（影响多个质量属性且彼此冲突的架构决策，例如安全vs性能、可用性vs成本）、风险点（架构决策可能引发负面结果的隐患）、非风险点（架构已充分处理的点，可作为正面证据）。
- **CBAM成本效益分析法**：在ATAM基础上加入经济性分析。步骤：①收集场景②场景精炼与优先级③效用响应曲线（每个质量属性效用随场景变化的曲线）④确定架构策略对应成本⑤计算每个场景的"期望效用值"⑥按预算约束选择最优架构策略组合。CBAM帮助在有限预算下选择性价比最高的架构策略。
- **其他评估方法**：SAAM（场景替换分析法，最早的方法，适合评估单一质量属性如可修改性）、ARID（基于架构的再工程识别，主动式场景评审）。ATAM适用最广（多质量属性综合评估），CBAM强调成本效益。

### 3. 信息系统安全体系深度
定义：信息系统安全体系是保护信息系统免受威胁、确保机密性、完整性和可用性的完整防护框架，涵盖技术、管理、物理、人员等多层面。

关键要点：
- **安全CIA三要素**：机密性Confidentiality（数据不被未授权访问，加密+访问控制实现）、完整性Integrity（数据不被未授权篡改，数字签名+哈希校验实现）、可用性Availability（授权用户随时可用，冗余+容灾实现）。扩展要素：不可否认性Non-repudiation（数字签名+审计日志）、可控性（访问控制+审计）、真实性（身份认证）。
- **安全体系分层**：物理安全（机房/门禁/监控/防火防水）→网络与系统安全（防火墙/IDS/IPS/VPN/网闸）→应用安全（SQL注入/XSS/CSRF防护/输入校验）→数据安全（加密存储/脱敏/备份恢复）→管理安全（安全策略/培训/审计/应急响应）。分层防御（Defense in Depth）理念：单一防护层失效不影响整体安全。
- **等级保护2.0**：五个等级（自主保护/指导保护/监督保护/强制保护/专控保护，对应等保1-5级）。定级要素：客体（信息系统）+ 受到破坏时所侵害的客体（公民法人/社会秩序/国家安全）+ 对客体造成侵害的程度（一般/严重/特别严重）。保护框架：安全通信网络→安全区域边界→安全计算环境→安全管理中心（技术五层面） + 安全管理（制度/机构/人员/建设/运维五方面）。
- **密码学基础**：对称加密（DES/3DES/AES，加密解密同一密钥，速度快适合大量数据）、非对称加密（RSA/ECC，公钥加密私钥解密，密钥分发安全但速度慢）、哈希算法（MD5/SHA-1/SHA-256，验证完整性）、数字签名（私钥签名+公钥验证，完整性和不可否认性）、PKI公钥基础设施（CA数字证书颁发+CRL吊销列表+RA注册机构）。
- **风险评估方法**：定性评估（专家打分+风险矩阵，定性等级如高/中/低）+定量评估（ALE年预期损失=资产价值×威胁发生概率×脆弱性暴露程度，量化金额）。残余风险 = 原始风险 - 控制措施降低的风险。风险处理四选项：规避/转移/降低/接受。

### 4. 系统性能评价与性能优化
定义：系统性能评价是通过定性和定量方法评估信息系统在响应时间、吞吐量、资源利用率等维度的表现。性能优化是在评价基础上采取针对性改进措施提升系统性能。

关键要点：
- **性能评价指标**：响应时间（用户视角，从发起请求到收到响应的总时间）、吞吐量（单位时间内系统处理请求数TPS/事务数）、并发用户数（同时在线用户数量）、资源利用率（CPU/内存/磁盘/网络使用率）、每秒查询率QPS（数据库视角）、可用性指标（5个9=99.999%年停机5.26分钟）。性能测试类型：负载测试（预期负载）、压力测试（超载找极限）、容量测试（最大承载）、稳定性测试（长时间运行）、并发测试（同时访问冲突）。
- **性能评价方法**：分析评价法（数学模型+理论推导排队论模型M/M/1、M/M/c）、模拟评价法（计算机模拟仿真，灵活但需建模）、测量评价法（实际系统跑基准测试如SPEC、TPC-C，最真实但成本高）。三种方法互为补充：分析快、模拟准、测量真。
- **性能优化策略**：前端优化（CDN加速+静态资源压缩+浏览器缓存+懒加载+SSR/CSR选择）→网络优化（HTTP/2多路复用+WebSocket长连接+减少请求数+域名分片）→应用层优化（缓存三剑客：本地缓存/分布式缓存Redis/CDN+异步消息队列削峰+池化技术线程池/连接池+算法优化降低时间复杂度）→数据库优化（SQL索引+查询重写+读写分离+分库分表+连接池调优）→架构优化（水平扩展+微服务拆分+异步化+服务降级/熔断/限流）。性能优化遵循"二八原则"——80%时间花在20%瓶颈上。
- **性能测试工具**：JMeter（开源负载测试，支持HTTP/数据库/FTP）、LoadRunner（商业功能强大，企业级压测首选）、Gatling（高性能基于Scala）、Locust（Python分布式压测）。监控工具：Prometheus+Grafana（指标采集可视化）、SkyWalking/APM（分布式链路追踪）、ELK（日志分析）。

### 5. 数学建模与运筹学基础
定义：数学建模是用数学语言和方法描述和解决实际问题的过程。运筹学是数学建模的核心方法论，研究在有限资源约束下进行最优决策的学科。

关键要点：
- **线性规划LP**：目标函数+约束条件均为线性，求最优解（最大/最小）。标准形式：Max/Min Z = Σcj·xj，约束Σaij·xj ≤/=/≥ bi，xj ≥ 0。求解方法：单纯形法（迭代找最优顶点）+ 图解法（两变量直观）。对偶理论：每个LP问题有对偶问题，原问题Max对应对偶问题Min，对偶变量即为影子价格（资源边际价值）。软件分析员需理解LP建模思路，工具常用LINDO/LINGO/Python的scipy.optimize.linprog。
- **整数规划IP与动态规划DP**：整数规划变量约束为整数（0-1规划是特例），分支定界法+割平面法求解。0-1规划常用于选址/分配/背包问题。动态规划将多阶段决策问题分解为子问题，核心要素：阶段/状态/决策/策略/状态转移方程+递推关系。经典模型：最短路径、01背包、生产库存、设备更新。DP三大性质：最优子结构+无后效性+子问题重叠。
- **排队论Queueing Theory**：研究排队系统中服务效率与等待成本的平衡。Kendall记号A/B/c/K/N/∞表示到达间隔分布/服务时间分布/服务台数/系统容量/顾客源/排队规则。M/M/1是最基本模型（泊松到达+指数服务+单服务台），关键指标：ρ=λ/μ（服务强度，ρ<1才稳定）、Lq=ρ²/(1-ρ)（队列平均长度）、Wq=ρ/(μ(1-ρ))（平均等待时间）、L=ρ/(1-ρ)（系统平均顾客数）、W=1/(μ-λ)（平均逗留时间）。M/M/c多服务台模型：ρ=λ/(cμ)，Wq公式含C语言爱尔朗公式。
- **图论与网络优化**：最小生成树Prim/Kruskal（连通代价最小）、最短路径Dijkstra（单源正权图）/Floyd（任意两点）、最大流问题（Ford-Fulkerson增广路径算法，最大流最小割定理）、关键路径法CPM（项目进度管理，AOE网求最长路径即关键路径，总时差为零的活动为关键活动）。网络优化广泛应用于通信/物流/项目排期。
- **决策论**：确定型决策（直接选最优方案）、风险型决策（已知概率，期望值法EMV=Σ概率×收益、决策树分析+灵敏度分析）、不确定型决策（不知道概率，乐观/悲观/等可能/最小后悔四种准则）。

## 案例分析题
**题目**：某市市场监管局启动"智慧市场监管一体化平台"建设，整合原工商、质监、食药监三个部门业务系统（已合并为市场监管局），覆盖企业监管、食品安全、特种设备、消费维权、信用监管五大业务域。项目面临以下复杂问题：

（1）三个部门原有6套业务系统，需求文档散落在30+个PDF文件中，部门间业务术语不一致（如"市场主体"vs"企业"vs"商户"实指同一对象），需求获取困难。
（2）平台日均访问量约10万次，预计3年后达到50万次。领导要求响应时间≤2秒，可用性99.99%。架构选型争议大——传统单体架构 vs 微服务架构。
（3）平台涉及200万家企业、50万商户、300万条食品溯源数据，含敏感信息（法人身份证/银行账号），需通过等保三级测评。
（4）建设经费1500万，工期18个月，涉及大量业务规则建模（食品安全检查标准200+项、特种设备检验流程50+类）。

**问题1**（8分）：针对需求获取困难问题，请设计需求工程实施方案，包括需求获取方法组合、需求建模工具选择、需求验证流程，说明如何用"业务术语统一表"解决术语不一致问题。

**问题2**（7分）：针对性能与可用性要求，请用性能评价方法分析该平台的性能瓶颈，说明性能优化策略（至少5个层级），并论证该平台应使用ATAM还是CBAM进行架构评估（说明理由）。

**问题3**（5分）：针对等保三级安全要求，请设计安全防护方案（覆盖技术五层面+管理五方面），并说明密码学技术如何在数据传输和存储环节应用。

**问题4**（5分）：针对业务规则建模复杂问题，请说明如何用线性规划/整数规划方法建立食品检查资源调度模型（设某区有100家食品企业需检查，5个检查组，每组日均检查8家），并说明CPM关键路径法在18个月工期管控中的应用思路。

**参考答案**：

**问题1**：
需求工程实施方案：
- **需求获取方法组合**（多管齐下）：
  - ①文档分析：组建6人需求小组，2周内梳理30+个PDF，输出《现状业务矩阵》和《系统功能清单》
  - ②访谈：分三轮——第一轮各部门领导（业务战略层），第二轮业务骨干（操作流程层），第三轮窗口人员（用户体验层），共访谈60人次
  - ③JAD联合应用开发：组织3场4天集中工作坊，每场40人（5个业务域代表+IT+第三方咨询）
  - ④原型法：快速搭建低保真原型引导用户表达隐性需求，针对争议点迭代3版
  - ⑤问卷调查：向全市2000名企业用户和500名基层监管员发放需求优先级问卷
- **需求建模工具**：DFD数据流图（功能建模）+ ER图（数据建模）+ UML用例图（系统边界建模）+ BPMN流程图（业务流程建模）+ 判定表（复杂规则建模如食品安全检查标准200+项）

业务术语统一表方案：
- ①建立《市场监管业务术语字典》：收集三部门原术语300+条，合并归一为150+条标准术语
- ②核心术语映射示例：
  - 工商"市场主体" ↔ 质监"企业" ↔ 食药监"商户" → 统一为"市场主体"
  - 工商"营业执照" ↔ 食药监"经营许可证" → 统一为"经营许可证"
- ③需求文档强制使用标准术语，定义术语别名表，旧文档建立对照索引
- ④评审环节术语一致性检查作为强制审查项

需求验证流程：需求评审（内部+外部三轮）→需求测试（编写验收测试用例）→需求确认（用户签字）→需求基线（建立版本基线）→需求变更控制（CCB审批+影响分析）

**问题2**：
性能瓶颈分析（测量评价法+排队论分析）：
- 假设日均10万访问集中在8小时工作时段，每秒平均≈3.5次，但峰值可达每秒50次（早9点午2点），平均请求处理时间0.5秒，单服务器TPS=2（处理能力不足！）
- 排队论分析：M/M/1模型，ρ=λ/μ=50×0.5=25（ρ>>1，系统严重过载，必须多服务台分担）
- 关键瓶颈：①单体架构单点性能上限②数据库并发连接池不够③未使用缓存④网络带宽

性能优化策略（6层）：
- ①前端层：CDN加速静态资源+浏览器缓存+图片懒加载+Gzip压缩+首屏SSR渲染
- ②网络层：HTTP/2多路复用+WebSocket长连接（实时监管预警）+DNS预解析+合并请求
- ③应用层：Redis分布式缓存热点数据（如企业信用查询占70%流量）+Kafka消息队列异步处理非实时任务（如检查通知）+线程池+熔断降级（Sentinel/Hystrix）
- ④数据库层：MySQL读写分离（主从3节点）+分库分表（按业务域分库，企业表按区域分表）+SQL索引优化+连接池Druid调优
- ⑤架构层：微服务化拆分（按5个业务域拆5个微服务）+K8s容器化水平扩展（峰值自动扩容50+ Pod）+API网关限流（令牌桶/漏桶）
- ⑥可观测性：Prometheus+Grafana实时监控QPS/响应时间+SkyWalking链路追踪+ELK日志分析

架构评估方法选择：**采用CBAM成本效益分析法**。
- 理由：①该平台有明确预算约束1500万，需在预算内选性价比最高的架构策略组合
- ②仅用ATAM只能识别风险点和权衡点，无法量化经济性
- ③CBAM六步法可计算"缓存vs数据库优化""微服务vs单体"等架构策略的成本效益
- ④具体步骤：列出关键质量场景（响应≤2s/可用99.99%/安全等保三级）→构建效用响应曲线→计算每个架构策略成本（如Redis集群50万+ K8s集群80万）→计算期望效用值→预算约束下选最优策略组合
- ⑤ATAM可作为前置评估（识别风险点），CBAM作为成本决策（最终选型）

**问题3**：
等保三级安全防护方案（技术五层面+管理五方面）：
- **安全通信网络**：①网络分区（业务区/数据区/管理区/DMZ区）②防火墙+IDS/IPS部署③VPN加密通道（移动办公场景）④WAF应用防火墙防SQL注入/XSS

业务规则建模应用：

**整数规划应用——食品检查资源调度模型**：
- 决策变量：xij∈{0,1}，1表示第i检查组检查第j家企业，0表示不查
- 目标函数（最大化覆盖率）：Max Z = Σxij（最大化检查企业数）
- 约束条件：
  - ①每家企业至少被检查1次：Σxij ≥ 1（j=1,2,...,100）
  - ②每检查组工作量约束：Σxij ≤ 8（i=1,2,...,5，每天每组最多8家）
  - ③关键企业（如高风险）必须现场检查：xij=1
  - ④每组工作量均衡：Σxij在合理区间
- 求解：用Python scipy.optimize.milp或LINDO/LINGO求解，0-1规划用分支定界法
- 扩展模型：考虑检查人员专业（食品/特种设备/计量）匹配，加入车辆调度（车辆容量约束），转成带时间窗的车辆路径问题VRPTW

**CPM关键路径法应用——18个月工期管控**：
- 步骤：①WBS工作分解（需求3月+设计3月+开发6月+测试3月+部署3月=18月）
- ②绘制AOE网络图（活动为弧，事件为节点）
- ③估算每项活动工期和依赖关系（前导活动）
- ④计算最早开始ES/最早完成EF/最晚开始LS/最晚完成LF/总时差TF=LS-ES
- ⑤识别关键路径：TF=0的活动串联即为关键路径
- ⑥关键路径上活动延误=项目延误，需重点监控；非关键路径活动有富余时间可调配
- 应用工具：Microsoft Project / Primavera P6 / 国产建文/广联达
