# 软考-（高级）系统分析师 第19次学习

## 今日核心概念

### 1. 系统可靠性工程与冗余设计
定义：通过系统架构与软件设计手段，使系统在规定条件下、规定时间内执行所需功能的能力（GB/T 3187），含可靠性（Reliability）、可用性（Availability）、可维护性（Maintainability）三大维度。
关键要点：
- 可靠性指标：MTTF（平均失效前时间）、MTBF（平均失效间隔时间）、MTTR（平均修复时间）、失效率λ（浴盆曲线：早期损耗期/偶然期/耗损期）
- 串联系统：R=R₁×R₂×...×Rₙ（整体可靠性<最弱环节）；并联系统：R=1-(1-R₁)(1-R₂)...(1-Rₙ)（冗余提升可靠性）
- 冗余技术三类型：①信息冗余（CRC/海明码/奇偶校验检错纠错）；②时间冗余（指令复算/重试机制/卷回恢复）；③结构冗余（静态冗余TMR三模冗余/动态冗余热备份/混合冗余）
- 高可用架构模式：双机热备（Active-Standby）、双机互备（Active-Active）、集群负载均衡（多节点+心跳检测）、异地多活（多机房流量调度）
- 软考关注：可靠性建模、可用性计算公式 A=MTBF/(MTBF+MTTR)，5个9=99.999%即年停机<5.26分钟

### 2. 数据库事务管理与并发控制
定义：事务（Transaction）是数据库执行的基本逻辑单位，满足ACID特性；并发控制通过锁机制和多版本机制保证多事务并发执行的正确性。
关键要点：
- ACID四特性：原子性A（undo日志回滚）、一致性C（约束+触发器）、隔离性I（锁/MVCC）、持久性D（redo日志+checkpoint）
- 四级隔离级别（从低到高）：READ UNCOMMITTED（脏读）→READ COMMITTED（不可重复读，Oracle默认）→REPEATABLE READ（幻读，MySQL InnoDB默认）→SERIALIZABLE（可串行化）
- 锁机制三类型：①共享锁S（读锁，可并发）；②排他锁X（写锁，互斥）；③意向锁IS/IX/SIX（提高加锁效率）
- 两段锁协议2PL：扩展阶段（只加锁不释放）→收缩阶段（只释放不加锁），是可串行化的充分条件
- 死锁四条件（互斥/占有等待/不可剥夺/环路等待）+四处理策略：预防（破坏四条件之一）、避免（银行家算法）、检测（等待图/超时）、恢复（牺牲事务回滚）

### 3. 信息系统项目管理与挣值管理（EVM）
定义：挣值管理（Earned Value Management）是用货币量代替工作量，综合衡量项目范围、进度、成本绩效的集成管理方法。
关键要点：
- 三大基础值：PV（计划值Planned Value，BCWS）、EV（挣值Earned Value，BCWP）、AC（实际成本Actual Cost，ACWP）
- 四大评价指标：①进度偏差SV=EV-PV（>0提前）；②成本偏差CV=EV-AC（>0节约）；③进度绩效SPI=EV/PV（>1提前）；④成本绩效CPI=EV/AC（>1节约）
- 预测指标：EAC=完工估算=AC+(BAC-EV)/CPI；VAC=完工偏差=BAC-EAC；TCPI=剩余工作绩效指数=(BAC-EV)/(BAC-AC)
- 完工预测公式族：典型偏差EAC=AC+(BAC-EV)/CPI；非典型偏差EAC=AC+BAC-EV；当前CPI需保持EAC=AC+(BAC-EV)
- 软考典型计算：已知PV/EV/AC推导进度状态+预测EAC，要求理解指标含义、绘制S曲线、成本/进度双重偏差归因

### 4. 系统架构评估方法（ATAM/CBAM/SAAM）
定义：架构评估是对软件架构进行系统性分析，识别潜在风险、敏感点、权衡点，为架构决策提供量化依据。
关键要点：
- ATAM（架构权衡分析法）九步骤：①描述ATAM方法→②描述业务动机→③描述架构→④确定质量属性效用树→⑤分析架构方法→⑥识别风险点/非风险点/敏感点/权衡点→⑦分析权衡点→⑧形成结论
- 三类术语精解：敏感点（影响一个质量属性的单个属性）、权衡点（影响多个质量属性至少一个）、风险点（架构决策导致的不利后果）、非风险点（决策无关但可分析的点）
- CBAM（成本效益分析法）七步：场景→精化→场景优先级→效用响应曲线→策略效益计算→隐含质量期望值→架构决策
- SAAM（场景式架构分析法）五步：场景开发→架构描述→单个场景评估→场景交互评估→总体评估
- 软考关注：能根据题干场景正确识别四类术语，并基于效用树进行架构方案选型

### 5. 软件质量模型与质量保证（ISO 25010）
定义：ISO 25010是软件产品质量评价的国际标准，定义了八大质量特性及其子特性，是软件质量度量的权威框架。
关键要点：
- 八大质量特性：①功能适合性（完备性/正确性/适合性）；②性能效率（时间特性/资源利用性/容量）；③兼容性（共存性/互操作性）；④可用性（可辨识性/易学性/易操作性/错误防护/用户界面美观性/可访问性）；⑤可靠性（成熟性/可用性/容错性/易恢复性）；⑥安全性（机密性/完整性/不可否认性/可问责性/真实性）；⑦可维护性（模块化/可重用性/可分析性/可修改性/可测试性）；⑧可移植性（适应性/可安装性/可替换性）
- McCall质量模型（早期）：产品修正/产品转移/产品运行三大类共11个因素
- 质量保证QA vs 质量控制QC：QA是过程导向（建立并维护过程），QC是产品导向（验证产品符合性）
- 评审技术：Fagan审查六步（计划→准备→审查会议→返工→跟进→收尾）、走查（Walkthrough）、技术评审（Technical Review）
- 度量指标：缺陷密度=缺陷数/代码规模（KLOC）、平均故障间隔时间MTBF、用户问题报告数

## 案例分析题

**题目**：
某省级人社部门启动"智慧就业服务一体化平台"建设，预算2800万元，工期14个月。系统需覆盖求职招聘、就业见习、失业保险、技能培训四大业务，整合人社、税务、教育、民政4个厅局数据，对接国家人社部、省政务平台、社保系统、银行支付4类外部接口。系统日均访问量预计50万PV，峰值QPS约3000，可用性要求99.99%（年停机<52分钟），敏感数据含个人身份信息与银行账户信息。请回答以下问题：

**问题1**：基于"5个9"可用性要求和上述业务场景，请从可靠性维度设计冗余架构，画出"两地三中心"部署架构图，并推导该架构的可用性计算公式（设单服务器可用性A=99.5%，负载均衡器A=99.9%）。

**问题2**：在项目执行第8个月时，项目经理小李测得以下数据：PV=1200万，EV=1000万，AC=1100万，BAC=2800万。请计算SV/CV/SPI/CPI四项指标，分析项目当前的成本与进度状态，并按"典型偏差"假设计算EAC与VAC，说明项目需要采取的纠偏措施。

**问题3**：项目组计划采用ATAM方法对架构方案进行评估，质量属性包括可用性、可靠性、安全性、可维护性。请列出ATAM的九个步骤，并说明"敏感点""权衡点""风险点""非风险点"四类术语的含义及在本题中的应用示例。

**问题4**：系统数据库采用MySQL InnoDB引擎，在业务高峰期出现"高并发扣减失业保险金"场景下的超卖问题。请从事务隔离级别、锁机制（行锁vs表锁、乐观锁vs悲观锁）、SQL优化三方面分析问题根因，并给出详细解决方案（写出SQL伪代码）。

**参考答案**：

问题1：
- 部署架构：生产中心+同城灾备中心+异地灾备中心三地，核心数据库采用Active-Standby-Standby模式
- 架构组件：负载均衡器（F5/HAProxy双机热备）→ Web应用服务器（4节点Nginx集群）→ 应用服务器（8节点Tomcat）→ 数据库（MySQL主从+半同步复制）
- 同城双活RPO≈0（同步复制）、异地灾备RPO≤30s（异步复制）
- 可用性公式推导：
  - 单链路：Web（4节点99.5%并联）=1-(1-0.995)⁴=1-6.25×10⁻¹¹≈100%
  - 应用层（8节点）≈100%；DB主从切换RTO≤30s视为可用
  - 综合可用性 = A负载均衡×A应用×A数据库 = 0.999×0.99999×0.99999 ≈ 99.997%
  - 满足99.99%要求，但建议DB升级为99.999%双集群

问题2：
- SV=EV-PV=1000-1200=-200万（进度滞后200万）
- CV=EV-AC=1000-1100=-100万（成本超支100万）
- SPI=EV/PV=1000/1200=0.83（进度仅完成83%）
- CPI=EV/AC=1000/1100=0.91（每投入1元仅获得0.91元收益）
- EAC=AC+(BAC-EV)/CPI=1100+(2800-1000)/0.91=1100+1978=3078万
- VAC=BAC-EAC=2800-3078=-278万（预计超支278万）
- 状态分析：项目严重滞后且超支，处于"危机区"（SPI<1, CPI<1）
- 纠偏措施：①赶工（增加资源/并行作业）；②快速跟进（并行后续活动）；③范围管理（砍掉非核心功能MVP）；④加强成本监控；⑤复盘关键路径；⑥与CCB沟通变更基线

问题3：
- ATAM九步骤：①描述ATAM方法→②描述业务动机→③描述架构→④确定质量属性效用树→⑤分析架构方法→⑥识别风险点/非风险点/敏感点/权衡点→⑦分析权衡点→⑧形成评估结论→⑨组织最终讨论
- 术语含义与应用示例：
  - 敏感点：本架构决策对单个质量属性影响大（如"采用MySQL主从复制"对可用性影响大）
  - 权衡点：本架构决策对多个质量属性影响且需取舍（如"增加缓存Redis"提升性能但降低数据一致性）
  - 风险点：本架构决策可能对某些质量属性产生负面影响（如"采用异步消息"提升吞吐量但增加复杂度影响可维护性）
  - 非风险点：与质量属性无关的决策（如"选用Java开发语言"对业务功能无直接影响）

问题4：
- 问题根因分析：
  - 隔离级别：默认REPEATABLE READ下SELECT不加锁导致幻读
  - 锁机制：行锁竞争激烈时可能升级为间隙锁引发死锁；高并发时悲观锁性能低
  - SQL问题：未命中索引导致行锁升级为表锁，热点账户单行I/O瓶颈
- 解决方案（三层防护）：
  ```
  -- 1.乐观锁+条件更新
  UPDATE balance SET amount = amount - #{pay} 
  WHERE user_id = #{userId} AND amount >= #{pay} AND version = #{version};
  -- 返回影响行数=0即扣减失败，前端重试
  
  -- 2.Redis原子扣减+Lua脚本
  local stock = redis.call('GET', KEYS[1])
  if tonumber(stock) >= tonumber(ARGV[1]) then
    return redis.call('DECRBY', KEYS[1], ARGV[1])
  else return -1 end
  
  -- 3.悲观锁（资金场景关键路径）
  BEGIN; 
  SELECT amount FROM balance WHERE user_id = #{userId} FOR UPDATE;
  -- 校验后UPDATE
  COMMIT;
  ```
- 配套优化：①Redis预扣减（库存预热）+消息队列异步落库；②数据库乐观锁兜底；③事务最小化（避免长事务）；④热点账户分库分表（按user_id hash 16库）；⑤K8s+HPA自动扩缩容

**踩分要点**：
- 问题1：必须画出"两地三中心"完整架构图，含负载均衡双机、Web/应用集群、数据库主从、灾备同步链路
- 问题2：四个核心指标必计算正确，EAC公式区分"典型/非典型"两种假设，纠偏措施需结合赶工/快速跟进/范围管理
- 问题3：九步骤顺序正确，四类术语定义准确并能结合本题给出具体架构决策示例
- 问题4：隔离级别+锁机制+SQL优化三方面根因分析完整，解决方案需含具体SQL/Lua代码，分库分表等扩展方案加分

## 记忆口诀
**1. 可靠可用容错性**：MTBF长MTTR短，可用性A=MTBF/(MTBF+MTTR)；五九五二六分钟，双机热备集群稳
**2. 冗余三兄弟**：信息冗余检错纠错，时间冗余重试卷回，结构冗余TMR热备
**3. 事务ACID**：原一隔持——原子一致隔离持久
**4. 隔离四级**：脏读→不可重复读→幻读→串行化（READ UN→READ COM→REPEAT→SERIAL）
**5. 死锁四条件**：互占不环（互斥/占有等待/不可剥夺/环路），预防破一条件，银行家避免最优
**6. EVM四大指标**：SV=EV-PV，CV=EV-AC，SPI=EV/PV，CPI=EV/AC
**7. EVM预测三件套**：EAC=AC+(BAC-EV)/CPI典型，VAC=BAC-EAC，TCPI=(BAC-EV)/(BAC-AC)
**8. ATAM九步法**：方法→业务→架构→效用树→分析→风险术语→权衡→结论→最终讨论
**9. 风险术语四剑客**：敏感点单属性，权衡点多属性，风险点不利后果，非风险点无关决策
**10. ISO 25010八特性**：功性性能容可用靠安维移（功能/性能/兼容/可用/可靠/安全/维护/移植）
