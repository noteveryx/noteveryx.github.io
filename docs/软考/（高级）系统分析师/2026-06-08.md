## 今日核心概念

### 1. 行为型设计模式

定义：行为型模式关注对象之间的职责分配和通信机制，通过封装对象间的交互行为，使系统更具灵活性和可扩展性。与结构型模式关注"对象的组合"不同，行为型模式关注"对象怎么做"。

关键要点：
- 十一种经典行为型模式：责任链、命令、解释器、迭代器、中介者、备忘录、观察者、状态、策略、模板方法、访问者
- 策略（Strategy）：定义一系列算法，将每个算法封装起来使它们可互换。上下文持有一个策略引用，运行时切换算法
- 观察者（Observer）：定义对象间一对多的依赖关系，当一个对象状态改变时自动通知所有依赖者。典型应用：事件监听、数据绑定
- 状态（State）：允许对象在其内部状态改变时改变行为，不同状态下同一操作产生不同结果
- 命令（Command）：将请求封装为对象，支持请求排队、日志记录、撤销操作。核心三角色：调用者、接收者、命令对象
- 模板方法（Template Method）：在父类中定义算法骨架，将某些步骤延迟到子类实现。体现"开闭原则"
- 责任链（Chain of Responsibility）：将请求沿处理者链传递，每个处理者决定处理或转发。典型：审批流程、异常处理
- 中介者（Mediator）：用一个中介对象封装一系列对象交互，使各对象不需要显式引用彼此。典型：聊天室、MVC控制器

### 2. 数据库事务管理与并发控制

定义：事务是数据库操作的逻辑工作单元，具有ACID四个基本特性。并发控制解决多事务同时访问相同数据时的一致性问题。

关键要点：
- ACID四特性：原子性（Atomicity，要么全做要么全不做）、一致性（Consistency，事务前后数据库状态合法）、隔离性（Isolation，事务间互不干扰）、持久性（Durability，提交后永久生效）
- 并发问题四类：脏读（读到其他事务未提交的数据）、不可重复读（同一事务内两次读同一行结果不同）、幻读（同一事务内两次范围查询行数不同）、丢失更新（两事务同时改同一数据后一个覆盖另一个）
- 隔离级别四级（由低到高）：读未提交（Read Uncommitted）→ 读已提交（Read Committed，解决脏读）→ 可重复读（Repeatable Read，解决不可重复读）→ 串行化（Serializable，解决幻读，但并发性能最低）
- 锁机制：共享锁（S锁，读锁）、排他锁（X锁，写锁）、乐观锁（版本号机制，适用于读多写少）、悲观锁（先加锁再操作，适用于写多场景）
- 两阶段锁协议（2PL）：加锁阶段（只加锁不释放）→ 解锁阶段（只释放不加锁），保证可串行化
- 死锁四个必要条件：互斥、持有并等待、不可剥夺、循环等待。预防策略：一次性加锁、按序加锁、超时回滚

### 3. 系统需求工程与需求管理

定义：需求工程是系统分析的核心活动，涵盖需求的获取、分析、规约、验证和管理全过程。系统分析师必须确保需求完整、一致、可追踪、可验证。

关键要点：
- 需求分类三维度：功能需求（系统做什么）、非功能需求（性能/安全/可靠性等约束）、约束条件（法规/标准/技术限制）
- 非功能需求关键指标：性能（响应时间/吞吐量）、安全性（认证/授权/审计）、可用性（MTBF/MTTR）、可扩展性、可维护性
- 需求获取方法：访谈（一对一结构化提问）、问卷调查（广覆盖低成本）、观察（用户现场跟岗）、原型（快速验证理解）、头脑风暴（激发创新）、文档分析（研究已有系统资料）
- 需求变更管理流程：提出变更请求→ 变更影响分析→ CCB（变更控制委员会）评审→ 批准/拒绝→ 实施变更→ 验证→ 更新基线
- 需求追踪矩阵（RTM）：建立需求与设计/代码/测试用例之间的双向追踪链，确保每条需求可追溯到实现和验证
- 需求属性：优先级（MoSCoW：Must/Should/Could/Won't）、稳定性、来源、版本号、状态（已批准/已实现/已验证）

### 4. UML动态建模——交互图

定义：交互图是UML中描述对象之间动态协作行为的建模工具，主要包含时序图（Sequence Diagram）和通信图（Communication Diagram），二者表达相同信息但侧重点不同。

关键要点：
- 时序图（顺序图）：按时间顺序从上到下展示对象间的消息交互，强调消息的时间顺序
  - 五要素：参与者（Actor）、对象（Object）、生命线（Lifeline）、消息（Message）、激活框（Activation Bar）
  - 消息类型：同步消息（实心箭头，等待返回）、异步消息（开放箭头，不等待）、返回消息（虚线开放箭头）
  - 控制结构：循环（loop框）、条件（alt框）、可选（opt框）、并行（par框）、中断（break框）
- 通信图（协作图）：展示对象间的空间关系和链接，强调对象间的结构组织
  - 以对象为节点、消息为边，消息上标注序号表示顺序（如1.1、1.2表示嵌套）
  - 适合展示复杂的对象间静态连接关系
- 时序图 vs 通信图选择：需展示时间顺序→用时序图；需展示对象间结构关系→用通信图
- 高频考点：消息类型区分、循环和条件框的UML表示、从用例场景推导时序图、生命线与激活框的含义

### 5. 业务流程管理（BPM）与工作流技术

定义：BPM是对企业业务流程进行建模、执行、监控和优化的系统化管理方法。工作流引擎是BPM的技术实现核心，负责按照预定义的流程定义驱动业务过程的自动流转。

关键要点：
- BPM生命周期四阶段：流程设计（建模与定义）→ 流程部署（发布到引擎）→ 流程执行（实例化运行）→ 流程监控与优化（性能分析与改进）
- 工作流参考模型（WFMC）五部件：流程定义工具、工作流引擎、工作流客户端、被调用应用、管理监控工具
- 流程建模标准：BPMN（业务流程模型与符号，业界主流）、BPML（业务流程建模语言）、WS-BPEL（Web服务业务流程执行语言）
- BPMN核心元素：开始事件（薄圆圈）、结束事件（粗圆圈）、任务节点（圆角矩形）、网关（菱形：排他/并行/包含）、顺序流（实线箭头）、消息流（虚线箭头）
- 网关类型：排他网关（XOR，只走一条分支）、并行网关（AND，所有分支同时执行再汇聚）、包含网关（OR，满足条件的分支都执行）
- 业务流程与系统需求的关系：业务流程建模是需求分析的前置活动，通过流程梳理识别功能需求、数据需求、角色权限需求和集成需求

## 案例分析题

**题目**：

某省级医保结算系统正在进行核心模块重构。该系统面向全省5000万参保人员，日均交易量约800万笔，峰值时段集中在每日9:00-11:00和14:00-16:00。系统涉及医保局、医院、药店、银行等多个参与方。

现有系统存在以下问题：
1. 医院端上传处方和患者信息后，经常出现重复扣费（同一笔交易被计费两次）
2. 高峰期系统响应时间从平均200ms飙升至3秒以上，偶发超时
3. 新增险种或报销规则时需停机修改代码，每次上线需2周以上
4. 各参与方间的消息交互流程复杂，缺乏统一规范，维护困难

重构团队拟采用工作流引擎管理核心结算流程，并引入事务管理机制保障数据一致性。

**问题1**（5分）：请用BPMN建模医保结算主流程，至少包含以下节点：处方上传、合规审核、费用计算、医保报销、个人支付、银行扣款、结算完成。请说明至少三个网关的作用。

**问题2**（5分）：针对"重复扣费"问题，请从事务管理和并发控制角度分析可能的原因，并给出完整解决方案。

**问题3**（5分）：高峰期系统性能下降，请从数据库事务隔离级别角度分析原因，并给出优化方案（至少包含调整隔离级别和引入缓存两个方向）。

**问题4**（5分）：系统需要支持新增险种和报销规则的快速上线。请结合行为型设计模式，设计一个可扩展的费用计算模块架构，说明模式选型理由和关键类设计。

**参考答案**：

**问题1**：

BPMN流程设计：
开始事件 → 处方上传 → 排他网关（合规审核）→ [通过] 费用计算 → 并行网关（医保报销并行个人支付）→ 并行网关（汇聚）→ 银行扣款 → 包含网关（判断是否需补扣）→ [是] 补扣流程 / [否] → 结算完成 → 结束事件

三个网关作用：
1. 排他网关（合规审核后）：判断处方是否合规，合规则进入费用计算，不合规则返回医院端拒保通知（只走一条分支）
2. 并行网关（费用计算后）：医保报销和个人支付两支流可同时进行，提升效率（所有分支同时执行再汇聚）
3. 包含网关（银行扣款后）：根据结算结果判断是否需要补扣操作，可能走也可能不走，也可能两条都走（满足条件的分支执行）

**问题2**：

可能原因：
1. 医院端因网络超时重发请求，医保系统收到两笔相同请求但未做幂等校验，导致重复扣费
2. 系统采用"读未提交"或"读已提交"隔离级别，并发事务同时读到同一笔待结算记录导致重复处理
3. 分布式环境下，消息中间件至少一次投递（at-least-once）导致消息重复消费

解决方案：
1. 幂等设计：每笔结算生成全局唯一交易流水号（txnId = 医院编码 + 处方编号 + 时间戳哈希），医保端建立"结算记录表"，处理前先查重，已存在则直接返回结果
2. 事务控制：结算操作封装为数据库事务，结算记录表插入与费用扣减在同一事务中，利用唯一索引防止并发插入
3. 分布式锁：以txnId为key获取Redis分布式锁（SETNX），获取成功才执行结算，防止并发
4. 隔离级别：至少采用"读已提交"（Read Committed），推荐"可重复读"（Repeatable Read），配合乐观锁版本号机制

**问题3**：

原因分析：
- 高峰期大量并发事务争抢同一数据资源（如医保账户余额），若隔离级别过高（串行化）则锁等待严重，若隔离级别为可重复读则间隙锁范围大，导致锁冲突
- 长事务占用连接池资源，新请求排队等待

优化方案：
1. 调整隔离级别：将结算核心链路的隔离级别从可重复读降为读已提交（Read Committed）。医保结算场景下，"不可重复读"的可接受程度较高——同一事务内余额变化不影响结算正确性（结算以提交时刻的余额快照为准），但换来锁持有时间缩短、并发度提升
2. 引入Redis缓存：
   - 参保人基本信息和医保账户余额缓存Redis（TTL设为30秒），结算时先读缓存，减少数据库读取压力
   - 报销规则（险种政策、报销比例表）全量缓存，规则变更时主动刷新缓存，结算时从缓存读取避免频繁查库
3. 事务拆分：将结算流程拆分为"快速校验事务"（50ms内）+"异步扣费事务"，校验通过后立即返回，扣费异步执行
4. 读写分离：高峰期将报表查询和余额查询分流到只读从库，主库专注写操作

**问题4**：

模式选型：策略模式（Strategy Pattern）+ 简单工厂模式。

架构设计：
- 抽象策略接口：`CostCalculator`，定义统一方法 `calculate(Prescription prescription, Policy policy): BigDecimal`
- 具体策略类：`BasicMedicalCalculator`（基本医保）、`BigDiseaseCalculator`（大病保险）、`SupplementaryCalculator`（补充医疗）、`CommercialInsuranceCalculator`（商保对接）
- 策略工厂：`CalculatorFactory`，根据险种代码（policyType）创建对应策略实例，利用Map维护注册关系
- 上下文类：`CostCalculationContext`，持有一个 `CostCalculator` 引用，运行时由工厂注入具体策略

关键类设计：
```
<<interface>>
CostCalculator
  + calculate(prescription, policy): BigDecimal

BasicMedicalCalculator implements CostCalculator
BigDiseaseCalculator implements CostCalculator
SupplementaryCalculator implements CostCalculator

CalculatorFactory
  - calculatorMap: Map<String, CostCalculator>
  + register(type, calculator): void
  + getCalculator(type): CostCalculator

CostCalculationContext
  - calculator: CostCalculator
  + setCalculator(calculator): void
  + executeCalculation(prescription, policy): BigDecimal
```

选型理由：
1. 策略模式将每种险种的费用计算算法封装在独立类中，符合开闭原则——新增险种只需新建策略类并注册，无需修改已有代码
2. 与模板方法对比：各险种的计算流程差异大（基本医保按比例、大病保险有起付线和封顶线），不适合固定骨架，策略模式更灵活
3. 与状态模式对比：费用计算不存在"状态转换"语义，险种是静态属性而非动态状态
4. 简单工厂负责策略的创建和注册，使上下文类与具体策略解耦

**踩分要点**：

问题1：流程完整性2分（至少7个节点+起止事件）、网关作用分析3分（每个网关1分，需说明类型和判断逻辑，只画图不解释不给分）

问题2：原因分析2分（至少2个原因）、解决方案3分（幂等设计1.5分+隔离级别/锁1.5分，仅说"加锁"不给满分，需说明锁的类型和粒度）

问题3：原因分析2分（隔离级别与锁竞争的关系）、优化方案3分（隔离级别调整1分+缓存方案1分+至少一个补充方案如事务拆分或读写分离1分）

问题4：模式选型及理由2分（必须明确策略模式，且需说明与至少一个其他模式的对比理由）、类设计2分（接口+具体类+工厂+上下文四要素齐全）、扩展性说明1分

## 记忆口诀

**行为型十一模式**：
`责命解迭中，备观状策模访`
责任链（传递）、命令（封装）、解释器（语法）、迭代器（遍历）、中介者（协调）、备忘录（快照）、观察者（通知）、状态（行为随状态变）、策略（算法互换）、模板方法（骨架延迟）、访问者（操作分离）
速记联想：`责命解迭中，备观状，策模访` — "责命解决迭中，备观状，策略模仿访问"

**ACID四特性**：
`原一持久` — 原子性（全有全无）、一致性（合法状态）、隔离性（互不干扰）、持久性（永久生效）
谐音：`阿西的` — A(原子)C(一致)I(隔离)D(持久)

**隔离级别四等与并发问题**：
`读未提脏读，读已提脏读消，可重读不可重，串行化全消掉`
读未提交→脏读/不可重复读/幻读全有；读已提交→消脏读；可重复读→消不可重复读；串行化→全消

**BPMN网关三兄弟**：
`X排他一条路，+并行全走齐，O包含看条件`
排他网关（XOR，菱形加×）→一条分支；并行网关（AND，菱形加+）→全部执行；包含网关（OR，菱形加○）→满足条件就走

**需求管理MoSCoW**：
`M必做S应做C可做W不做`
Must（必须有）→ Should（应该有）→ Could（可以有）→ Won't（这次不做）
谐音："莫斯科"（MoSCoW）
