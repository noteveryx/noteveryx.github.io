## 今日核心概念

### 1. 结构型设计模式

定义：结构型模式关注如何将类或对象组合成更大的结构，通过继承或组合的方式实现功能扩展。与创建型模式（解决对象创建问题）不同，结构型模式解决的是对象之间的组织和协作关系。

关键要点：
- 七种经典结构型模式：适配器、装饰器、代理、外观、桥接、组合、享元
- 适配器（Adapter）：将一个接口转换成客户端期望的另一个接口，解决接口不兼容问题
- 装饰器（Decorator）：动态地给对象添加额外职责，比继承更灵活
- 代理（Proxy）：为另一个对象提供一个替身或占位符以控制访问
- 外观（Facade）：为子系统中的一组接口提供统一的高层接口
- 桥接（Bridge）：将抽象部分与实现部分分离，使它们可以独立变化
- 组合（Composite）：将对象组合成树形结构以表示"部分-整体"层次
- 享元（Flyweight）：运用共享技术有效地支持大量细粒度对象

### 2. 分布式数据库与CAP定理

定义：分布式数据库是将数据物理上分布存储在计算机网络的不同节点上，但逻辑上仍属于同一个系统。CAP定理（2000年由Eric Brewer提出）指出，分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）三个特性中的两个。

关键要点：
- 一致性C：所有节点在同一时间看到的数据相同，读操作总能读到最新写入
- 可用性A：每个请求都能收到非错误的响应，但不保证包含最新数据
- 分区容错性P：系统在任意网络分区故障情况下仍能继续运行
- AP系统（放弃C）：Cassandra、DynamoDB，优先可用，最终一致
- CP系统（放弃A）：HBase、ZooKeeper，优先一致，网络分区时可能拒绝请求
- CA系统（放弃P）：传统单机关系型数据库，无分区环境
- BASE理论：Basically Available（基本可用）、Soft-state（软状态）、Eventually Consistent（最终一致性），是CAP中AP的延伸

### 3. 软件复用与构件技术

定义：软件复用是利用已有软件组件来构建新系统的过程。构件是软件系统中具有相对独立功能、可复用、接口明确的可部署单元，是软件复用的核心要素。

关键要点：
- 复用三个层次：代码复用（最底层）、设计复用（架构/模式）、分析复用（需求模型）
- 构件三要素：接口（对外契约）、实现（内部逻辑）、部署（独立可替换）
- 构件模型标准：CORBA（OMG）、COM/DCOM（微软）、EJB（Java EE）
- 构件组装方式：黑盒组装（不动内部实现，通过接口拼装）、白盒组装（需修改或理解内部）
- 领域工程三阶段：领域分析（识别共性/可变性）、领域设计（构建参考架构DSSA）、领域实现（开发可复用构件）
- 复用的障碍：Not Invented Here综合征、构件信任问题、构件检索与分类困难、投资回报长期化

### 4. 面向服务的架构（SOA）

定义：SOA是一种将应用程序功能作为服务提供给其他应用的架构风格，服务之间通过定义良好的接口和契约进行通信。服务是自治、无状态、可复用的功能单元，独立于底层技术和平台。

关键要点：
- 核心角色三要素：服务提供者、服务消费者、服务注册中心（UDDI）
- 服务特征：松耦合、可发现、可组合、自描述、自治
- Web Service技术栈：SOAP（消息协议）、WSDL（服务描述）、UDDI（服务注册发现）
- 服务编排（Orchestration）vs 编排（Choreography）：前者有中央控制节点（如BPEL引擎），后者各服务平等协作无中心
- SOA与微服务区别：SOA侧重企业级服务重用和集成（通常带ESB），微服务侧重独立部署和快速迭代
- 企业服务总线（ESB）：SOA的核心基础设施，负责消息路由、协议转换、数据转换、安全等功能

### 5. 消息中间件与异步通信

定义：消息中间件是利用高效可靠的消息传递机制进行平台无关的数据交换，基于消息队列实现异步通信，使分布式系统的各组件之间解耦。核心模式包括点对点（队列模式）和发布/订阅（主题模式）。

关键要点：
- 消息队列两大模式：P2P（点对点，队列，消息只能被一个消费者消费一次）、Pub/Sub（发布订阅，主题，消息可被多个订阅者收到）
- 典型产品：RabbitMQ（AMQP协议）、Kafka（高吞吐日志型）、RocketMQ（阿里系事务消息）、ActiveMQ
- 消息可靠性：持久化（写入磁盘）、消息确认ACK机制、事务消息（Kafka的Exactly Once语义）
- 关键应用场景：削峰填谷（秒杀系统流量缓冲）、系统解耦（订单-库存-物流异步联动）、异步处理（邮件发送等非核心流程）、日志收集
- 消息有序性：全局有序（单分区）、局部有序（同一键值进同一分区）
- 死信队列（DLQ）：处理无法消费的消息，避免消息堆积阻塞

## 案例分析题

**题目**：

某大型电商平台正进行系统架构升级。当前系统为单体架构，核心模块包括：商品管理、订单处理、库存管理、支付对接、物流追踪和用户评价。每个模块都直接调用数据库，数据集中在同一套MySQL集群中。

升级目标如下：
1. 将核心业务拆分为独立服务，实现独立部署和弹性伸缩
2. 确保订单服务和库存服务之间实现最终一致性（不能超卖，不能重复扣减）
3. 系统需要支持双十一峰值流量（预估QPS达到日常的50倍），不能因瞬时流量导致系统崩溃
4. 商品详情页需要展示库存实时数量，但允许短时间（不超过5秒）的数据延迟

请回答以下问题：

**问题1**（4分）：该电商平台计划引入消息中间件进行异步解耦。请分别说明点对点模式和发布/订阅模式在该系统中各至少一个适用场景，并解释为什么。

**问题2**（6分）：订单服务扣减库存时，由于网络延迟可能导致消息重复投递。请设计一套方案保证"库存不能重复扣减"，并说明其实现机制。请画出关键交互流程。

**问题3**（5分）：针对双十一场景，消息中间件会积压大量消息。请分析可能产生的问题，并给出至少三种应对策略。

**问题4**（4分）：商品详情页的库存实时数量展示，你建议采用什么方案？请从CAP定理角度给出分析，并说明所选方案在CAP上的取舍。

**参考答案**：

**问题1**：

1. 点对点模式适用场景——订单创建后通知仓库发货。原因：一个订单只需被一个仓库系统处理一次，消息消费后从队列中删除，保证发货运单唯一。

2. 发布/订阅模式适用场景——商品价格变更后同步更新购物车、收藏夹、推荐系统等多个服务。原因：一个价格变更事件需要被多个下游服务同时感知和处理，各订阅者独立消费不影响彼此。

**问题2**：

方案：基于数据库唯一约束 + 消息幂等性设计。

实现机制：
1. 订单服务发送扣减库存消息时携带全局唯一的业务ID（如订单号+SKU编号组合）
2. 库存服务收到消息后，先查询"扣减记录表"判断该业务ID是否已处理过；若已处理则直接返回成功（幂等）
3. 若未处理，在数据库事务中同时执行两个操作：库存扣减（UPDATE stock SET quantity = quantity - N WHERE quantity >= N） + 插入扣减记录
4. 利用数据库唯一索引（业务ID）防止并发插入重复记录；若并发冲突则回滚并重试
5. 库存扣减SQL带条件 WHERE quantity >= N 防止超卖；若扣减失败则回滚事务

关键交互流程：
订单服务 → 消息队列（消息体含orderId+skuId+quantity）→ 库存服务 → 查询扣减记录表 →（未处理）→ 开启事务→ 扣减库存(带条件) + 插入扣减记录 → 提交事务 → 回复ACK

**问题3**：

可能产生的问题：
- 消息积压导致消费延迟增大，用户体验下降（如订单状态长时间不更新）
- 内存或磁盘耗尽，消息中间件自身崩溃
- 下游服务被积压消息冲刷，导致雪崩效应

应对策略：
1. 消费者横向扩容——临时增加库存服务、物流服务等消费者实例数量，加速消息消化
2. 消息优先级队列——区分核心流程消息（支付回调、库存扣减）与非核心消息（短信通知、日志），优先保障核心链路
3. 降级与限流——对非关键业务（如用户评价、积分累计）的消费者主动降低消费速率或暂停消费，释放资源给核心链路
4. （加分项）Kafka场景下可增加分区数来提升并行度；RocketMQ场景下利用定时消息将非紧急消息延后投递

**问题4**：

建议方案：采用Redis缓存 + 数据库最终一致，商品详情页读取Redis中的库存缓存值而非直接查库。

CAP分析：
- 放弃强一致性（C），选择AP方案（可用性+分区容错性）
- 库存服务更新数据库后异步刷新Redis缓存，允许5秒内缓存与数据库数据不一致
- 用户看到的是缓存中的库存值，可能存在5秒延迟，但页面响应快、系统可用性高
- 极端场景：若Redis缓存过期或不可用，降级为直接查库（此时牺牲部分可用性换取准确性）

**踩分要点**：

问题1：每个模式1个场景+1个理由，缺场景或缺理由各扣1分
问题2：幂等机制3分（业务ID+记录表查重+事务原子性）、并发控制2分（唯一索引+条件扣减）、交互流程1分
问题3：问题分析1分、每个应对策略1.5分（需具体方案而非泛泛而谈，如只说"扩容"不给满分，需说明扩消费者还是队列本身）
问题4：方案选型2分（Redis缓存必须明确提及）、CAP取舍分析2分（必须说明放弃C选择AP并解释）

## 记忆口诀

**结构型七模式**：
`适装外代，桥组享来`
适配器（转接口）、装饰器（加功能）、外观（统入口）、代理（控访问）、桥接（分两层）、组合（树结构）、享元（共对象）

**CAP定理三选二**：
`C强一致A总可用，P分区容错要取舍；AP最终一致高并发，CP强一致舍可用，CA单机无分区`
谐音联想："CAP"→"帽子扣头三选二"

**构件复用三层次**：
`代码抄、设计套、分析学`
代码复用（代码抄写复用）、设计复用（设计套路复用）、分析复用（分析方法复用）

**SOA三角色**：
`提工登` — 服务提供者（Provider）、服务消费者（Consumer）、服务注册中心（UDDI Registry）

**消息积压四招解**：
`扩容消、排优先、降限非核、加分拉` — 扩容消费者、优先级队列、降级限流非核心、增加分区数
