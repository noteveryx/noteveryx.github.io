## 今日核心概念

### 1. 模块化设计：内聚与耦合

定义：模块化设计是将系统分解为若干个功能相对独立、接口明确的模块的过程。其核心评价标准是内聚度（模块内部元素结合的紧密程度）与耦合度（模块之间相互依赖的程度），遵循"高内聚、低耦合"的设计原则。

关键要点：
- **内聚度七层（由低到高）**：巧合内聚（最差，元素间无关联）→ 逻辑内聚 → 时间内聚 → 过程内聚 → 通信内聚 → 顺序内聚 → 功能内聚（最优，模块只完成单一功能）
- **耦合度七层（由紧到松）**：内容耦合（最坏，直接修改对方数据）→ 公共耦合 → 外部耦合 → 控制耦合（传递控制参数）→ 标记/数据结构耦合 → 数据耦合（仅传数据参数）→ 无直接耦合
- **模块分解原则**：每个模块功能单一、接口简洁；模块大小适中（通常50~100行）；扇入要大（被多模块调用），扇出要小（调用的模块数不宜过大）
- **软考高频考点**：自动识别内聚/耦合类型（给场景判断属于哪一类）；区分功能内聚与顺序内聚；内容耦合与公共耦合的危害描述

### 2. 数据库设计：ER模型与关系规范化

定义：数据库设计是信息系统的核心环节，包括概念结构设计（ER模型）、逻辑结构设计（关系模式转换）和物理结构设计三个阶段。关系规范化理论通过范式等级确保数据结构的合理性与完整性。

关键要点：
- **ER模型三要素**：实体（用矩形表示）、属性（用椭圆表示）、联系（用菱形表示，标注1:1、1:N、M:N）
- **ER图转关系模式规则**：1:1联系→任选一端合并；1:N联系→在N端加入1端主键；M:N联系→建立独立关系表；弱实体→依赖关系必须用实线双边框标注
- **六大范式**：1NF（属性不可再分）→ 2NF（消除部分函数依赖）→ 3NF（消除传递依赖）→ BCNF → 4NF → 5NF。软考重点考到3NF
- **反规范化策略**：增加冗余列、派生列、重组表、水平/垂直分割——目的是以空间换时间
- **软考高频考点**：给定关系模式判断达到第几范式；范式分解即模式分解（保持无损连接与函数依赖）；ER图画法及转换关系模式

### 3. 软件测试策略与方法

定义：软件测试是确保软件质量的关键活动，按阶段分为单元测试→集成测试→确认测试→系统测试→验收测试，按方法分为静态测试（不执行程序）与动态测试（执行程序），按逻辑分为黑盒测试与白盒测试。

关键要点：
- **测试阶段模型**：单元测试（模块级，开发人员自测）→ 集成测试（模块组装，策略有非增量式的"大爆炸"与增量式的自顶向下/自底向上/三明治）→ 确认测试（验证需求符合性）→ 系统测试（在整个目标环境中验证）→ 验收测试（用户主导，含α测试与β测试）
- **白盒测试**：关注内部逻辑结构，语句覆盖 → 判定覆盖 → 条件覆盖 → 判定-条件覆盖 → 条件组合覆盖 → 路径覆盖（强度递增）
- **黑盒测试**：关注外部功能，等价类划分、边界值分析、因果图法、决策表法、正交实验法、错误推测法
- **回归测试**：每次修改后重新执行已通过的测试，确保不引入新Bug
- **软考高频考点**：自顶向下集成需编写桩模块（Stub），自底向上需编写驱动模块（Driver）；白盒覆盖标准强度排序；边界值测试是对等价类划分的补充

### 4. 软件维护类型与管理

定义：软件维护是软件生命周期中持续时间最长、成本最高的阶段，指软件交付后进行的修改活动，占总生命周期成本约60%~70%。科学的维护管理需要建立配置管理、问题跟踪和变更控制体系。

关键要点：
- **四大维护类型**：纠错性维护（修复Bug，约20%）→ 适应性维护（适应环境变化，约25%）→ 完善性维护（增加新功能或改进，约50%，占比最高）→ 预防性维护（预防潜在问题，约5%）
- **维护成本占比口诀**：完善性占一半，适应性四分之一，纠错性五分之一，预防性零星
- **逆向工程与再工程**：逆向工程是从代码恢复到设计甚至需求模型；再工程是在逆向基础上对系统进行重构和优化后正向生成
- **遗留系统处理策略**：高水平高质量→继续维护；低水平高质量→再工程；高水平低质量→改造/替换；低水平低质量→淘汰/重新开发
- **软考高频考点**：给定场景判别属于哪类维护；完善性vs适应性最易混淆（完善性=主动加功能，适应性=被动适应外部变化）；遗留系统四象限决策

### 5. 软件质量模型与度量

定义：软件质量模型（ISO/IEC 25010）提供了评估软件产品质量的标准化框架，将质量属性分解为八个基本特性，每个特性下包含子特性。软件度量是对软件质量、进度、规模等进行量化评估的方法体系。

关键要点：
- **ISO 25010八大质量特性**：功能适用性（正确性+完备性）、性能效率（时间特性+资源利用率）、兼容性（共存性+互操作性）、易用性（可识别性+易学性+可操作性）、可靠性（成熟性+可用性+容错性）、安全性（保密性+完整性+抗抵赖性）、可维护性（模块化+可复用性+易分析性）、可移植性（适应性+可安装性）
- **McCall模型的三种视角**：产品运行（正确性/可靠性/效率/完整性/可用性）、产品修正（可维护性/灵活性/可测试性）、产品转移（可移植性/可复用性/互操作性）
- **面向对象度量CK指标套件**：WMC（类加权方法数）、DIT（继承树深度）、NOC（子类数量）、CBO（类间耦合度）、RFC（类的响应集大小）、LCOM（方法内聚缺乏度）
- **软考高频考点**：ISO 25010八大特性名称及其子特性记忆；六大特性的英文与中文全称对应；CK指标套件中哪个衡量耦合（CBO）、哪个衡量内聚（LCOM）

## 案例分析题

**题目**：

某电商公司原有订单管理系统已运行8年，随着业务快速增长，系统出现以下问题：订单高峰期响应缓慢、数据库查询频繁超时、多个模块之间直接访问数据库表导致修改困难。公司决定对系统进行全面升级，组建了新项目团队。

（1）在对现有系统分析后发现："订单创建"模块同时负责库存检查、价格计算、优惠券核销和短信通知四个操作；而"用户管理"模块直接修改其他模块的数据库表来同步用户状态。

（2）原数据库中存在如下订单记录表：ORDER（订单编号、商品编号、商品名称、单价、数量、客户编号、客户名称、客户地址、下单日期），其中订单编号和商品编号共同作为主键。经分析，该表中存在数据冗余和异常问题。

（3）项目团队打算采用自底向上的集成测试策略。测试组长建议先在每个模块单元测试完成后，直接一次性组装所有模块进行测试。

**问题1**：从内聚度和耦合度角度，分析"订单创建"模块和"用户管理"模块分别存在什么问题，并提出改进方案。（8分）

**问题2**：分析ORDER表属于第几范式，简述存在的问题，并将其规范化到3NF，写出分解后的关系模式（主键加下划线）。（8分）

**问题3**：测试组长的集成策略有哪些问题？请给出正确的自底向上集成测试步骤，指出需要编写的辅助模块类型。（9分）

---

**参考答案**：

**问题1（8分）：**

**"订单创建"模块——内聚度问题（4分）：**
- 当前属于"逻辑内聚"（2分）：该模块同时完成库存检查、价格计算、优惠券核销、短信通知等多项关联不紧密的功能，通过调用逻辑混杂在一起，但每项任务的逻辑相互独立。
- 改进方案（2分）：按单一职责原则拆分为独立模块——库存模块（检查库存）、价格模块（计算价格）、优惠券模块（核销优惠）、通知模块（发送通知），每个模块实现功能内聚；通过服务编排或进程调用串联流程。

**"用户管理"模块——耦合度问题（4分）：**
- 当前属于"内容耦合"（2分）：直接修改其他模块的内部数据库表属于最严重的耦合类型，破坏了模块的封装性和数据安全，任何一方的表结构变更都会导致关联模块崩溃。
- 改进方案（2分）：引入统一的数据访问层或数据服务接口，各模块通过标准API/接口调用访问数据，降低为数据耦合；遵循"迪米特法则"——只与直接友元通信。

**问题2（8分）：**

**范式判断（2分）：**
该表的主键为（订单编号，商品编号）→（客户编号，客户名称，客户地址）仅依赖于订单编号，存在部分函数依赖，故该表属于1NF，未达到2NF。

**存在问题（2分）：**
插入异常（新增客户但未下单时无法插入客户信息）、删除异常（删除最后一条订单时客户信息丢失）、更新异常（客户地址变更需修改多处）、数据冗余（同一客户下每一条订单记录都重复存储客户信息）。

**规范化至3NF（4分）：**

```
（1）订单-商品表：ORDER_ITEM（订单编号，商品编号，数量，单价）
（2）商品表：PRODUCT（商品编号，商品名称）
（3）订单表：ORDER_MASTER（订单编号，客户编号，下单日期）
（4）客户表：CUSTOMER（客户编号，客户名称，客户地址）
```

评分要点：分解出4张表2分，主键标注正确2分。订单表中若将单价放入ORDER_ITEM亦可；客户地址如需进一步规范可拆地址表为城市代码表（无硬性要求）。

**问题3（9分）：**

**问题分析（3分）：**
- 测试组长描述的是"大爆炸集成"而非自底向上，自底向上不等同于一次性组装；（1分）
- 大爆炸集成风险极高：所有模块之间依赖关系一次性暴露，定位故障困难；（1分）
- 自底向上应逐层增量组装，每集成一个新模块就进行测试验证。（1分）

**正确步骤（4分，每步1分）：**
1. 底层模块（叶子模块，不调用其他模块）先做单元测试
2. 编写驱动模块（Driver），聚合底层模块逐一进行集成测试
3. 由底向上逐层引入新的模块，每次集成后进行回归测试
4. 最终集成顶层模块，移除所有驱动模块完成系统集成

**需编写的辅助模块（2分）：**
自底向上集成需要编写驱动模块（Driver Module）（1分），用于模拟上层调用场景并传递测试数据（1分）。自底向上不需要桩模块（Stub）。

---

**踩分要点**：

| 问题 | 得分点 | 易丢分陷阱 |
|------|--------|-----------|
| 问题1 | 内聚→类型判断（逻辑内聚）+改进；耦合→类型判断（内容耦合）+改进 | 把逻辑内聚误解为"过程内聚"或"通信内聚"；内容耦合与公共耦合区分不清 |
| 问题2 | 先判范式再列异常再分解，三步缺一步即扣分 | 只分表不写主键标注；ORDER_MASTER中忘保留客户编号做外键；主键忘记用下划线标注 |
| 问题3 | 指出策略名错误→分析风险→给出正确步骤→指明Driver | 把自底向上的辅助模块写成Stub（自顶向下才用Stub）；步骤中忘提回归测试；只说策略名称不说步骤 |

## 记忆口诀

**口诀一：内聚七级高低排**

> 巧逻时过通信序功，
> 最好只做一件事。
> 从低到高逐级升，
> 功能内聚是顶峰。
> （巧=巧合、逻=逻辑、时=时间、过=过程、通=通信、序=顺序、功=功能）

**口诀二：耦合按序由紧松**

> 内设法控标数无，
> 最危险的是内容耦合。
> （内=内容、设=公共(设谐音涉)、法=外部、控=控制、标=标记/数据结构、数=数据、无=无直接耦合）

**口诀三：三大范式一句话**

> 一消原子二消部，三消传递数据清。
> （1NF消原子=属性不可再分；2NF消部=消除部分函数依赖；3NF消传=消除传递依赖）

**口诀四：维护四种比一比**

> 五成完善两五适应，
> 两成纠错半成防病。
> 完善主动适应被动，
> 四字区分看动因。
> （完善性50%、适应性约25%、纠错性约20%、预防性5%）

**口诀五：ISO 25010八面玲珑**

> 功性兼易，可安维移。
> 八个大字记心中：
> （功=功能适用性、性=性能效率、兼=兼容性、易=易用性、可=可靠性、安=安全性、维=可维护性、移=可移植性）
