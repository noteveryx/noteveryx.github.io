## 今日核心概念
### 1. 领域驱动设计（DDD）
定义：以业务领域为核心，通过统一语言、限界上下文、领域模型等手段，将复杂业务知识转化为软件模型的一种设计方法。
关键要点：
- 战略设计：限界上下文、上下文映射、子域划分（核心域/支撑域/通用域）。
- 战术设计：实体、值对象、聚合、领域服务、仓储、工厂。
- 统一语言：业务人员与技术人员使用同一套术语，消除理解鸿沟。
- 适用场景：业务复杂、领域知识密集、需要长期演化的系统。

### 2. 事件驱动架构（EDA）
定义：以事件为核心通信机制，系统组件通过发布、订阅、处理事件进行解耦协作的架构风格。
关键要点：
- 事件三要素：事件生产者、事件通道、事件消费者。
- 事件类型：领域事件、集成事件、系统事件。
- 核心优势：组件高解耦、水平扩展能力强、实时响应能力好。
- 常用技术：消息队列、流处理平台、事件总线。
- 主要挑战：事件顺序保证、幂等性处理、最终一致性调试、死信队列管理。

### 3. 命令查询职责分离（CQRS）与事件溯源
定义：CQRS将写操作与读操作分离为不同模型；事件溯源以事件序列作为系统状态的单一真相来源，通过重演事件重建状态。
关键要点：
- CQRS适用场景：读写比例悬殊、查询模式多变、读性能要求高的系统。
- 事件溯源三要素：事件不可变、事件存储、版本快照与重演机制。
- 组合价值：读模型通过消费事件异步构建，写模型保持领域完整性。
- 注意事项：引入最终一致性、调试复杂度上升、需防范事件存储爆炸。

### 4. 微服务拆分与限界上下文
定义：在微服务架构中，以业务领域边界为依据划分服务单元，使每个服务围绕一个完整的业务能力自治运行。
关键要点：
- 拆分原则：按业务能力垂直拆分、高内聚低耦合、数据库按服务私有化、服务边界与限界上下文对齐。
- 康威定律：组织沟通结构会镜像到系统架构中，需匹配团队结构。
- 聚合根识别：以领域聚合根为最小数据边界，避免分布式事务。
- 常见反模式：分布式单体、共享数据库、贫血领域模型、纳米服务。

### 5. 架构腐化与架构治理
定义：系统在持续演化过程中因需求压力、技术债务、规范缺失导致架构质量下降的现象；架构治理通过制度、工具、流程保证架构健康演进。
关键要点：
- 腐化信号：循环依赖、上帝类、核心域扩散、重复代码、违反分层、接口膨胀。
- 治理手段：架构决策记录（ADR）、依赖检查、代码门禁、架构评审、技术雷达、平台化团队。
- 目标平衡：架构演进速度与可维护性、短期交付与长期健康。
- 量化指标：圈复杂度、耦合度、核心域代码覆盖率、API变更频率。

## 案例分析题
**题目**：某省级人社厅“智慧就业公共服务平台”已运行5年，采用单体架构，随着业务增长出现以下问题：
1. 核心模块耦合严重，一个小需求改动常影响多个模块；
2. 查询响应慢，首页统计报表加载经常超时；
3. 系统发布窗口长，每月只能发布一次。
现计划进行架构重构。请回答：
1. 从领域驱动设计角度，如何划分核心域、支撑域、通用域，并说明限界上下文？
2. 针对查询性能瓶颈，如何结合CQRS与事件溯源进行改造？
3. 在微服务拆分时，应遵循哪些原则，可能出现哪些反模式？
4. 如何建立架构治理机制，防止新架构再次腐化？

**参考答案**：
1. **领域划分**：
   - 核心域：就业智能匹配、补贴申领审核（体现差异化竞争力的业务）。
   - 支撑域：招聘企业管理、求职者画像、培训管理（重要但非核心）。
   - 通用域：用户认证、消息通知、文件存储、支付（可采购或复用）。
   - 限界上下文示例：以“就业补贴申领”为限界上下文，内部统一语言包括申领人、补贴类型、审批节点、发放状态；外部与招聘管理上下文通过“申领资格校验”集成事件交互。

2. **CQRS与事件溯源改造**：
   - 命令侧：保留申领、审批、发放的写模型，使用事件溯源记录“申领提交→审核通过→发放完成”事件流。
   - 查询侧：构建独立读取模型（如搜索引擎+物化视图），消费领域事件异步更新首页统计、个人进度等查询数据。
   - 效果：写操作即时响应，读操作从读取模型获取，报表加载从秒级降至毫秒级。

3. **微服务拆分原则与反模式**：
   - 原则：按业务能力垂直拆分；数据库按服务私有化；识别聚合根，服务边界与限界上下文对齐；团队结构遵循康威定律。
   - 反模式：分布式单体（拆分后仍频繁跨服务调用）、共享数据库、贫血领域模型、纳米服务（粒度过小）。

4. **架构治理机制**：
   - 建立架构决策记录（ADR）机制，重大技术选型留痕。
   - 引入依赖扫描工具，禁止跨服务循环依赖。
   - 设立架构评审委员会，对新服务接口、数据库变更进行门禁。
   - 定期进行架构演进回顾，使用技术雷达跟踪中间件版本。
   - 将核心域代码覆盖率、圈复杂度等指标纳入持续集成门禁。

**踩分要点**：
- 核心域/支撑域/通用域必须给出具体例子，不能仅列名称。
- CQRS必须体现“命令写模型”与“查询读模型”分离，并说明事件消费异步更新。
- 事件溯源必须强调事件不可变与状态重建。
- 微服务拆分必须提到按业务能力、数据库私有化、康威定律至少两点。
- 反模式至少答出2个。
- 架构治理至少答出3项具体机制。

## 记忆口诀
1. DDD子域三分法：核心支撑加通用，限界上下文要划清。
2. DDD战术五件宝：实体值对象聚合，领域服务加仓储。
3. EDA三件客：生产通道消费者，发布订阅解耦合。
4. CQRS读写分：命令一侧写模型，查询一侧读视图。
5. 事件溯源真相源：事件不可变，重演状态现。
6. 微服务拆分四原则：业务内聚、数据私有、康威对齐、聚合根定。
7. 微服务反模式：分布式单体、共享数据库、贫血模型、纳米服务。
8. 架构腐化四信号：循环依赖、上帝类、核心域扩散、重复码。
9. 架构治理五件套：ADR留痕、依赖扫描、评审门禁、技术雷达、演进回顾。
