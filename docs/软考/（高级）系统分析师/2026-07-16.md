## 今日核心概念

### 1. 结构化分析方法（SA）
定义：以数据流为核心的自顶向下分解方法，使用DFD（数据流图）和DD（数据字典）描述系统逻辑模型，适合数据处理类业务系统。
关键要点：
- DFD四要素：外部实体、加工、数据流、数据存储
- 分层模型：上下文图（0层）→ 一层图 → 二层图，逐层分解加工
- 父图与子图平衡：父图某加工的输入输出数据流必须与子图对应加工的输入输出保持一致
- 数据字典：定义数据流、数据存储、数据项的逻辑结构

### 2. 面向对象分析方法（OOA）
定义：以对象为核心，识别问题域中的类与对象，建立系统静态结构与动态行为模型的方法，强调封装、继承、多态。
关键要点：
- 五大活动：识别对象、识别类、识别结构、识别主题、定义属性与服务
- 三模型：用例模型（功能视图）+ 分析模型（结构视图）+ 交互模型（行为视图）
- 核心原则：抽象、封装、继承、多态、消息通信
- OOA与OOD边界：OOA关注"做什么"，OOD关注"怎么做"

### 3. 需求工程
定义：从用户业务需求出发，通过系统化方法获取、分析、规格说明、验证、管理需求的全生命周期活动。
关键要点：
- 需求层次：业务需求 → 用户需求 → 功能需求 → 非功能需求
- 需求获取技术：访谈、问卷、观察、原型、头脑风暴、焦点小组
- 需求验证：一致性、完整性、可行性、可验证性、可追溯性
- 需求管理：变更控制、版本管理、需求跟踪矩阵（RTM）

### 4. UML 2.5 13种图分类
定义：统一建模语言（Unified Modeling Language）提供的可视化建模标准，覆盖系统静态结构、动态行为和物理部署。
关键要点：
- 结构图（静态）：类图、对象图、构件图、部署图、包图
- 行为图（动态）：用例图、活动图、状态机图
- 交互图：顺序图、通信图、时序图、交互概览图
- 考试重点：用例图与参与者、类图关系（关联/聚合/组合/泛化/依赖/实现）、顺序图生命线

### 5. 软件测试方法体系
定义：通过人工或自动化手段执行程序，验证软件是否满足需求、发现缺陷、提高质量的技术集合。
关键要点：
- 按阶段：单元测试 → 集成测试 → 系统测试 → 验收测试
- 按方法：黑盒测试（等价类/边界值/判定表/因果图）、白盒测试（语句/分支/路径覆盖）、灰盒测试
- 覆盖标准：语句覆盖最弱 → 判定覆盖 → 条件覆盖 → 判定/条件覆盖 → 条件组合覆盖 → 路径覆盖最强
- 测试类型：功能测试、性能测试、安全测试、兼容性测试、回归测试

## 案例分析题

**题目**：
某大型电商平台计划重构订单系统，采用面向对象方法开发。系统分析师小王完成了以下工作：
① 通过用户访谈获取了买家、卖家、运营三类用户需求；
② 绘制了顶层DFD图描述系统与外部实体交互；
③ 使用UML用例图描述订单管理子系统，包含下单、支付、退款三个用例；
④ 在分析类时发现"订单"类既属于"交易域"主题又属于"履约域"主题；
⑤ 测试团队采用等价类划分法对"订单金额校验"功能设计测试用例。

请指出小王工作过程中存在哪些问题，并说明正确做法。

**参考答案**：
小王工作存在以下问题：
1. 第②步错误：采用面向对象方法开发，不应使用DFD数据流图描述系统，应改用UML用例图描述系统与外部参与者的交互。DFD是结构化分析方法的产物，与OOA方法论不匹配。
2. 第③步问题不完整：UML用例图除展示用例外，还应识别参与者（Actor），本案例中缺少对买家、卖家、运营三类参与者的显式建模，且应建立参与者与用例之间的关联关系。
3. 第④步处理不明确：单个类不应跨主题，应识别主题间的耦合点，将"订单"类根据职责拆分或确定其归属主题（通常归属核心业务域），避免多主题归属导致耦合过强。
4. 第⑤步覆盖不足：等价类划分法单独使用易遗漏边界场景，应结合边界值分析法共同设计测试用例，重点校验金额上界、下界、临界值（如0.01元、999999.99元）。

正确做法：
- OOA阶段使用UML建模，保留DFD仅作业务流程辅助分析；
- 用例图必须包含参与者、用例、关联关系三要素；
- 主题划分遵循高内聚低耦合，单类归属单主题；
- 测试用例设计采用多种方法组合，覆盖等价类划分与边界值分析。

**踩分要点**：
- 答出"DFD与OOA方法不匹配"（3分）
- 答出"用例图缺少参与者"（2分）
- 答出"等价类+边界值组合使用"（2分）
- 主题划分原则（1分）
- 表达清晰、逻辑严谨（2分）

## 记忆口诀

结构化分析四要素：**外加流存**（外部实体、加工、数据流、数据存储）

OOA五大活动：**对结主属**（识别对象、识别结构、识别主题、定义属性与服务，识别类）

UML图记忆：**静动交**（结构图、行为图、交互图三大类）

测试覆盖强弱：**语判条判条路**（语句→判定→条件→判定条件→条件组合→路径，覆盖度递增）

需求层次记忆：**业用功非**（业务需求→用户需求→功能需求→非功能需求，自顶向下逐层细化）
