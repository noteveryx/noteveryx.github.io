## 今日核心概念

### 1. 软件架构风格

定义：软件架构风格是描述特定系统组织方式的惯用模式，定义了构件和连接器的类型及组合约束。架构风格决定了系统的整体结构骨架，是架构师进行架构选型的核心知识体系。

关键要点：

**五大架构风格分类**：

（1）数据流风格
- 管道-过滤器（Pipe-Filter）：数据流经一系列处理组件（过滤器），通过管道连接。特点：各过滤器独立、可组合、低延迟但需数据格式统一。典型应用：编译器（词法分析→语法分析→语义分析→代码生成）、Unix管道命令、ETL数据流水线
- 批处理序列：整批数据一次性处理完毕再传递，非实时。与管道过滤器的区别：批处理是推模式（前一步完成才推），管道-过滤器是拉模式（数据主动流过）

（2）调用/返回风格
- 主程序/子程序：经典面向过程结构，自顶向下调用，层次清晰但扩展性差
- 面向对象风格：封装+继承+多态，对象间通过方法调用交互。优点：封装性好、可复用；缺点：对象间耦合（方法调用导致依赖）、副作用难追踪
- 层次结构：每层为上层提供服务、使用下层服务。严格分层（仅相邻层交互）vs 松散分层（可跨层调用）。典型：OSI七层、TCP/IP四层、三层/四层企业架构

（3）独立构件风格
- 进程通信：构件间通过消息传递交互，消息可以是异步的。解耦但增加系统复杂度
- 事件驱动风格（隐式调用）：构件不直接调用，而是声明事件或监听事件。发布者不知道谁在监听，订阅者不知道谁在发布。优点：高度解耦、可动态增删构件；缺点：难以理解系统行为、性能不可控、调试困难。典型：GUI事件循环、观察者模式、消息总线

（4）虚拟机风格
- 解释器：包含解释引擎+工作内存（被解释的程序+执行状态）。适合需要自定义规则或动态执行的场景。典型：规则引擎（Drools）、脚本语言解释器、工作流引擎
- 基于规则的系统：规则集+推理引擎+工作内存，通过模式匹配触发规则执行。适合业务规则频繁变化的场景

（5）仓库风格（数据中心风格）
- 数据库系统：中心数据存储+独立客户端构件，客户端通过查询/更新操作共享数据。黑板系统是特殊变体：多个知识源协同求解，中央黑板存储当前解空间
- 黑板系统三要素：知识源（独立计算模块）、黑板数据结构（层次化组织的问题求解状态）、控制器（根据黑板状态选择知识源激活）

**C2架构风格**：独立构件与仓库的结合。构件和连接器都有顶部和底部，构件顶部连接连接器底部，构件底部连接连接器顶部。通知沿连接器向下传播，请求向上发送。完全解耦、可替换

**架构风格选型决策表**：

| 场景特征 | 推荐风格 | 理由 |
|---------|---------|------|
| 数据顺序处理、低延迟 | 管道-过滤器 | 流式处理天然匹配 |
| 规则频繁变化 | 解释器/规则系统 | 规则热更新无需重编译 |
| 高度解耦、动态扩展 | 事件驱动 | 发布订阅解耦发布者和订阅者 |
| 系统分层次抽象 | 层次结构 | 关注点分离、可替换层 |
| 多知识源协同求解 | 黑板系统 | 并行推理、增量求解 |

### 2. 设计模式（GoF 23种）

定义：设计模式是对在特定上下文中反复出现的设计问题的经验性解决方案。GoF（Gang of Four）23种设计模式是面向对象设计的核心知识，按创建型、结构型、行为型三大类组织。

关键要点：

**创建型模式（5种）——关注对象创建过程**：

| 模式 | 核心意图 | 关键特征 |
|------|---------|---------|
| 工厂方法 | 定义创建对象的接口，让子类决定实例化哪个类 | 父类定义接口、子类实现创建，符合开闭原则 |
| 抽象工厂 | 创建一系列相关对象的家族，无需指定具体类 | 多个工厂方法组合，产品族一致性约束 |
| 单例 | 确保类只有一个实例并提供全局访问点 | 私有构造、静态实例、线程安全（双重检查锁/枚举实现） |
| 建造者 | 将复杂对象的构建与表示分离 | 分步骤构建、Director控制流程、可复用建造器 |
| 原型 | 通过复制已有实例创建新对象 | clone()方法、深拷贝vs浅拷贝、减少new的开销 |

**结构型模式（7种）——关注类和对象的组合**：

| 模式 | 核心意图 | 关键特征 |
|------|---------|---------|
| 适配器 | 将一个类的接口转换为客户期望的接口 | 类适配器（继承）、对象适配器（组合），解决接口不兼容 |
| 桥接 | 将抽象部分与实现部分分离，使两者独立变化 | 两个维度正交扩展，避免类爆炸（继承改组合） |
| 组合 | 将对象组合成树形结构表示"部分-整体"层次 | 统一对待叶子和容器，递归结构（文件系统、组织架构） |
| 装饰器 | 动态给对象添加额外职责 | 比继承更灵活、可叠加、透明增强（Java I/O流） |
| 外观 | 为子系统中的一组接口提供统一的高层接口 | 简化客户端调用、降低耦合、不限制子系统直接使用 |
| 享元 | 通过共享技术有效支持大量细粒度对象 | 内部状态（可共享）vs外部状态（不可共享），对象池（字符串常量池） |
| 代理 | 为其他对象提供代理以控制对这个对象的访问 | 远程代理、虚拟代理（延迟加载）、保护代理（权限控制）、智能引用 |

**行为型模式（11种）——关注对象之间的职责分配和通信**：

| 模式 | 核心意图 | 关键特征 |
|------|---------|---------|
| 责任链 | 将请求沿处理链传递，直到有对象处理 | 解耦发送者与接收者、动态组合处理链（审批流程、过滤器链） |
| 命令 | 将请求封装为对象，支持撤销、排队、日志 | 命令对象封装接收者+动作、调用者无需知道具体操作 |
| 解释器 | 给定一个语言，定义其文法表示和解释器 | 终结符/非终结符表达式、语法树构建、适合简单文法 |
| 迭代器 | 顺序访问聚合对象的元素，不暴露内部表示 | 统一遍历接口、多种遍历策略、支持并发修改检测 |
| 中介者 | 用中介对象封装一组对象的交互 | 消除对象间直接引用、集中控制交互逻辑（空管塔台） |
| 备忘录 | 在不破坏封装的前提下捕获对象内部状态 | Originator+Memento+Caretaker三角色、支持撤销恢复 |
| 观察者 | 定义对象间一对多的依赖，状态变化自动通知 | Subject通知所有Observer、推模型vs拉模型 |
| 状态 | 允许对象在内部状态改变时改变行为 | 状态类封装行为、消除大量条件判断、状态转换显式化 |
| 策略 | 定义一系列算法，封装使可互相替换 | 算法独立变化、客户端选择策略、消除条件语句 |
| 模板方法 | 定义算法骨架，将某些步骤延迟到子类 | 父类控制流程、子类实现细节、钩子方法提供扩展点 |
| 访问者 | 在不改变元素类的前提下定义新操作 | 双分派机制、数据结构稳定+操作多变时适用、破坏依赖倒置 |

**高频易混淆对比**：
- 工厂方法 vs 抽象工厂：工厂方法创建一种产品，抽象工厂创建产品族
- 桥接 vs 适配器：桥接是设计阶段预先分离维度，适配器是事后补救接口不兼容
- 装饰器 vs 代理：装饰器增强功能（多层叠加），代理控制访问（一层隔断）
- 策略 vs 状态：策略是客户端选算法，状态是对象内部自动切换行为
- 命令 vs 策略：命令关注请求封装（撤销/排队），策略关注算法替换

### 3. 系统安全架构设计

定义：系统安全架构是在系统架构设计中从安全维度出发，建立纵深防御体系，确保系统的机密性、完整性、可用性（CIA三要素），并在架构层面实现安全与业务的融合。

关键要点：

**安全架构五层模型**：
- 物理安全：机房环境（温湿度/消防/防雷/防静电）、设备安全（防盗/防破坏）、介质安全（备份/销毁）
- 网络安全：防火墙（包过滤/状态检测/应用层网关）、IDS入侵检测（异常检测/误用检测）、IPS入侵防御、VPN（IPSec/SSL VPN）、网络隔离（双机网闸/物理隔离）
- 主机安全：操作系统加固（最小权限/关闭不必要服务/补丁管理）、安全审计、恶意代码防护、数据库安全（访问控制/审计/加密）
- 应用安全：身份认证（多因素MFA）、访问控制（RBAC/ABAC）、输入验证、会话管理、日志审计
- 数据安全：数据分类分级、传输加密（TLS/IPSec）、存储加密（TDE/字段级加密）、脱敏（静态脱敏/动态脱敏）、数据生命周期管理

**访问控制模型**：
- 自主访问控制DAC：资源所有者自主决定访问权限，灵活性高但安全性低（ACL是实现方式之一）
- 强制访问控制MAC：系统强制执行安全策略，安全标签（密级）匹配才能访问（军事系统，不上读不下写原则——Bell-LaPadula模型保证机密性、Biba模型保证完整性）
- 基于角色的访问控制RBAC：用户→角色→权限三级映射，最小权限原则。RBAC0基础→RBAC1角色继承→RBAC2约束（互斥/基数）→RBAC3统一
- 基于属性的访问控制ABAC：根据主体属性+资源属性+环境属性+操作动态决策，灵活性最高，适合云环境细粒度控制

**安全开发生命周期（SDL）**：
- 微软SDL七阶段：培训→安全需求→安全设计→安全编码→安全测试→安全响应→安全退役
- 安全设计原则：最小权限、深度防御、默认拒绝、故障安全（Fail-Safe）、经济机制（Keep It Simple）
- 威胁建模（STRIDE模型）：欺骗Spoofing→篡改Tampering→抵赖Repudiation→信息泄露Information Disclosure→拒绝服务Denial of Service→权限提升Elevation of Privilege

**安全认证与加密**：
- 对称加密：AES（128/192/256位，当前主流）、SM4（国密对称算法）。加解密同一密钥，速度快，密钥分发是难题
- 非对称加密：RSA（2048位以上）、SM2（国密非对称）、ECC。公钥加密私钥解密，密钥分发简单但速度慢
- 数字签名：私钥签名、公钥验证，保证完整性和不可抵赖。结合消息摘要（SHA-256/SM3）提高效率
- 数字证书与PKI：CA签发证书绑定公钥与身份，证书链验证至根CA。CRL证书撤销列表/OCSP在线证书状态查询

### 4. 系统性能评估与优化

定义：系统性能评估是通过量化指标度量系统运行状态，性能优化是在满足功能需求的前提下通过架构调整和资源配置提升系统效率的过程。架构师需掌握性能建模、瓶颈定位、优化策略的完整方法论。

关键要点：

**性能指标体系**：
- 响应时间：用户发起请求到收到响应的时间。含平均响应时间、P95/P99分位响应时间（长尾分析必看P99）
- 吞吐量：单位时间处理的请求数或数据量。QPS（每秒查询数）、TPS（每秒事务数）、BPS（每秒字节数）
- 并发数：系统同时处理的请求数。并发用户数≠在线用户数，一般并发用户数≈在线用户数×10%~20%
- 资源利用率：CPU、内存、磁盘I/O、网络带宽的使用百分比。警戒线一般设80%

**性能评估方法**：
- 基准测试（Benchmark）：标准化负载下的性能测量。TPC-C（OLTP）、TPC-H（OLAP）、SPEC（CPU）
- 负载测试：逐步增加负载直至系统达到预期性能指标
- 压力测试：超过系统容量的负载，找到系统崩溃临界点
- 稳定性测试：长时间运行（7×24小时），检测内存泄漏、资源耗尽
- 尖峰测试：瞬时大量请求冲击，测试系统弹性恢复能力

**阿姆达尔定律**：系统加速比 S = 1 / [(1-F) + F/N]，F为可并行化比例，N为处理器数量。核心启示：优化占比最大的瓶颈环节效果最显著，无限增加处理器也无法突破 (1-F) 的串行瓶颈

**性能优化六层策略**：

| 层次 | 优化手段 | 典型效果 |
|------|---------|---------|
| 架构层 | 缓存/异步/分区/读写分离/CDN | 10倍~100倍提升 |
| 数据库层 | 索引优化/SQL调优/分库分表/连接池 | 5倍~10倍提升 |
| 代码层 | 算法优化/减少锁竞争/对象池/批处理 | 2倍~5倍提升 |
| JVM层 | 垃圾回收调优/堆内存配置/JIT编译 | 1.5倍~3倍提升 |
| 操作系统层 | 内核参数/文件系统/网络栈调优 | 1.2倍~2倍提升 |
| 硬件层 | SSD替代HDD/增加内存/网络升级 | 2倍~10倍提升 |

**数据库性能优化核心手段**：
- 索引优化：EXPLAIN执行计划分析、覆盖索引（避免回表）、最左前缀原则、避免索引失效（函数/隐式转换/OR）
- 查询优化：避免SELECT *、分页查询优化（游标分页代替OFFSET）、子查询改JOIN
- 分库分表：垂直分库（按业务拆）、水平分库（按ID哈希/范围拆）、分片键选择（均匀分布、查询覆盖）
- 连接池：最小连接数、最大连接数、空闲超时、连接验证

**缓存架构优化**：
- 多级缓存：本地缓存（Caffeine/Guava）→ 分布式缓存（Redis Cluster）→ 数据库，逐层降级
- 缓存预热：系统启动时主动加载热点数据
- 缓存策略：LRU/LFU/FIFO淘汰策略选择，热点key识别与本地缓存保护

### 5. 中间件技术

定义：中间件是位于操作系统与应用软件之间的基础软件层，提供分布式环境下的通信、数据管理、事务处理、安全等共性服务，使应用开发者无需关注底层分布式复杂性。

关键要点：

**中间件六大分类**：

（1）消息中间件
- 核心模型：点对点（Queue，一对一消费）vs 发布订阅（Topic，一对多消费）
- Kafka：高吞吐、持久化、分布式、支持回溯消费。适用：日志收集、事件流、大数据管道
- RabbitMQ：AMQP协议、路由灵活（直连/主题/扇出/头部）、支持消息确认和死信队列。适用：业务消息、任务分发
- RocketMQ：事务消息、顺序消息、延迟消息。适用：电商交易、金融支付
- 选型：吞吐优先选Kafka、路由灵活选RabbitMQ、事务消息选RocketMQ

（2）数据库中间件
- 读写分离：MyCat/ShardingSphere-Proxy，自动路由读请求到从库
- 分库分表：ShardingSphere-JDBC（客户端嵌入）、MyCat（代理服务），分片策略（哈希取模/范围/一致性哈希）
- 连接池：HikariCP（性能最优）、Druid（监控能力强）

（3）缓存中间件
- Redis六种数据结构：String（缓存/计数器）、Hash（对象存储）、List（消息队列/最新列表）、Set（标签/交集运算）、ZSet（排行榜/延时队列）、Stream（消息流）
- Redis集群方案：主从复制（读写分离）→ 哨兵模式（自动故障转移）→ Cluster模式（数据分片+高可用）
- 缓存问题三连：穿透（查不存在的数据→布隆过滤器）、击穿（热点key过期→互斥锁）、雪崩（大量key同时过期→随机TTL）

（4）Web服务器中间件
- Nginx：反向代理、负载均衡（轮询/加权/IP哈希/最少连接）、静态资源服务
- Tomcat：Servlet容器、线程模型（Acceptor+Poller+Worker）
- 对比：Nginx处理静态和代理能力强（C语言），Tomcat处理动态Java请求

（5）API网关
- 核心功能：路由转发、认证鉴权、限流熔断、协议转换、日志监控、灰度发布
- 主流方案：Spring Cloud Gateway（响应式）、Kong（OpenResty+Lua）、APISIX（Apache项目）
- 架构师关注：网关是无状态的、网关不可成为单点瓶颈、网关层限流用令牌桶或漏桶算法

（6）分布式协调中间件
- ZooKeeper：分布式协调服务，提供命名服务、配置管理、分布式锁、集群管理。ZAB协议保证数据一致性。临时节点实现服务注册与发现
- Nacos：服务发现+配置管理+健康检查，CP（Raft）和AP（Distro）模式可切换。相比ZK更适合微服务场景

**中间件选型四原则**：
- 场景匹配：不是越新越好，而是最适合业务场景
- 运维成熟度：社区活跃度、文档完善度、线上案例数量
- 团队熟悉度：学习曲线与团队技术栈匹配
- 性能弹性：是否支持水平扩展、是否有性能瓶颈上限

## 案例分析题

**题目**：

某省级电子政务平台采用微服务架构，包含用户中心、审批中心、证照中心、支付中心、消息中心5个核心服务，通过API网关统一对外提供政务服务。系统日均请求量800万次，高峰期QPS达3000。近期出现以下问题：

1. 审批中心在月底集中审批时段，响应时间从平均200ms飙升至3秒以上，数据库CPU利用率达到95%
2. 消息中心通过RabbitMQ异步推送审批结果通知，高峰期消息积压超过50万条，通知延迟达30分钟
3. 支付中心与银行对接，每笔支付需调用银行接口确认，超时率约2%，超时后系统状态不一致
4. 用户查询个人审批进度时需跨审批中心、证照中心、消息中心三个服务聚合数据，响应时间长且任一服务异常即导致查询失败

请从架构层面分析并解决以上问题。

**问题1**（8分）：针对审批中心性能问题，请设计完整的性能优化方案，要求覆盖架构层和数据库层两个层次，并给出关键技术的选型理由。

**问题2**（7分）：针对消息积压和支付一致性问题，请分别给出解决方案。消息积压方案需包含架构调整和参数优化；支付一致性方案需说明如何保证本地事务与银行接口调用的一致性。

**问题3**（5分）：针对跨服务聚合查询问题，请设计CQRS（命令查询职责分离）架构方案，说明读写模型如何分别设计，以及如何保证查询数据的最终一致性。

**问题4**（5分）：请为该平台设计安全架构方案，要求覆盖应用安全和数据安全两个层次，并说明身份认证和访问控制的技术选型。

**参考答案**：

**问题1**（审批中心性能优化）：

架构层优化（4分）：
- 引入Redis集群作为查询缓存：审批状态、审批流程定义等读多写少数据缓存至Redis，TTL=30分钟+随机偏移，预计缓存命中率85%，读请求降至原来的15%
- 读写分离：MySQL主库写入+2个从库读取，通过ShardingSphere实现读写路由，从库延迟<1秒
- 异步化改造：审批提交后先写数据库返回受理成功，后续审批流转通过事件驱动异步执行，降低同步等待时间

数据库层优化（4分）：
- 索引优化：通过EXPLAIN分析慢查询，在approval表增加(user_id, status, create_time)联合索引覆盖高频查询，确保走索引而非全表扫描
- 分库分表：按org_id（审批机关）水平分库、按create_time按月分表，单表数据量从5000万降至500万以内
- 连接池调优：HikariCP最大连接数从默认10调至50，空闲超时600秒，避免连接创建开销
- 归档策略：超过2年的已完结审批数据归档至历史库，保持在线库数据量可控

选型理由：Redis是缓存中间件的事实标准，与Spring生态无缝集成；ShardingSphere国产开源、无需额外代理进程、性能损耗<5%；HikariCP是性能最优的JDBC连接池

**问题2**（消息积压与支付一致性）：

消息积压解决方案（3分）：
- 架构调整：将RabbitMQ替换为Kafka作为高吞吐消息通道。理由：Kafka单分区吞吐量10万+/秒，远高于RabbitMQ的万级；Kafka支持批量发送和压缩，降低网络开销；RabbitMQ更适合路由复杂的业务消息，但本场景是简单的审批结果推送，不需要复杂路由
- 参数优化：Kafka生产者批量发送（batch.size=64KB、linger.ms=10ms）、消费者增加分区数和消费者组大小（分区数=消费者数×2，预留扩容空间）、消费者采用手动提交offset（处理完成再提交，避免消息丢失）
- 降级策略：积压超过阈值时启动降级——批量合并通知（如同一用户多条通知合并为一条摘要短信）、非紧急通知降级为站内信（跳过短信通道）

支付一致性解决方案（4分）：
- 采用本地消息表+定时补偿的最终一致性方案：
  1. 支付中心创建支付记录时，在同一数据库事务中写入本地消息表（message_id、payload、status=待发送、retry_count=0、next_retry_time）
  2. 事务提交后，异步线程发送消息到银行接口。成功则更新status=已发送
  3. 定时任务每30秒扫描status=待发送且next_retry_time≤当前时间的消息，重试发送
  4. 重试策略：指数退避（1秒→2秒→4秒→8秒→16秒→30秒），最多重试10次后标记为异常需人工处理
  5. 银行回调：支付结果通过银行回调接口接收，更新支付记录状态。若回调超时未到，主动调用银行查询接口对账
- 关键保证：本地消息表与支付记录在同一事务中，要么同时成功要么同时失败，避免了分布式事务的复杂度

**问题3**（CQRS架构方案）：

写模型设计（2分）：
- 审批中心、证照中心、消息中心保持现有微服务写模型不变，各自管理自己的数据库
- 写操作走现有服务API，命令侧保持事务一致性（每个服务内部保证ACID）
- 每次写操作完成后发布领域事件到Kafka（如ApprovalCompletedEvent、LicenseIssuedEvent）

读模型设计（2分）：
- 新建查询服务，维护一个专门用于查询的物化视图数据库（如Elasticsearch或MySQL宽表）
- 查询服务订阅Kafka中的领域事件，实时更新物化视图。将用户ID作为文档ID，聚合审批状态、证照信息、通知记录为一份完整视图
- 前端查询只访问查询服务，一次请求返回全部信息，响应时间<100ms

最终一致性保证（1分）：
- 写操作→事件发布→查询服务消费→视图更新，延迟通常在秒级
- 查询服务记录消费位点offset，重启后从上次位点继续消费
- 设置定时对账任务（每5分钟），对比写库和读库的数据差异，发现不一致触发事件重放修复

**问题4**（安全架构方案）：

应用安全（3分）：
- 身份认证：采用OAuth2.0+JWT方案。统一认证中心（SSO）颁发JWT令牌（access_token有效期2小时、refresh_token有效期7天），API网关验证JWT签名和有效期。政务场景还需对接省级统一身份认证平台（如CA证书认证+多因素MFA）
- 访问控制：采用RBAC模型。用户→角色→权限三级映射，角色按政务职能定义（申请人/审批人/管理员/审计员），支持角色继承（区县审批人继承申请人权限）和互斥约束（同一用户不可同时拥有审批人和申请人角色）
- API网关层安全：统一限流（令牌桶算法，单用户100次/分钟）、请求签名校验（防篡改）、IP白名单（政务服务专网IP段）

数据安全（2分）：
- 传输加密：全站HTTPS（TLS1.2+），政务专网内采用国密SSL（SM2/SM3/SM4）
- 存储加密：数据库启用TDE透明数据加密，敏感字段（身份证号、银行卡号）使用SM4字段级加密
- 数据脱敏：查询接口返回数据时，身份证号仅返回前6后4位、手机号仅返回前3后4位，完整信息需二次授权验证
- 审计日志：全量记录数据访问行为（谁、何时、访问了什么、结果如何），审计日志独立存储、不可篡改、保留5年

**踩分要点**：

| 分值 | 踩分点 |
|------|--------|
| 8分 | 架构层含缓存+读写分离+异步化(3分)、数据库层含索引+分库分表+连接池+归档(4分)、选型理由(1分) |
| 7分 | 消息积压含架构替换+参数优化+降级(3分)、支付一致性含本地消息表+定时补偿+重试策略+回调对账(4分) |
| 5分 | 写模型不变+事件发布(2分)、读模型物化视图+订阅更新(2分)、对账机制(1分) |
| 5分 | 认证OAuth2.0+JWT+SSO(2分)、访问控制RBAC含互斥约束(1分)、数据加密+脱敏+审计(2分) |
