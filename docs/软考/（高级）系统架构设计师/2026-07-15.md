## 今日核心概念
### 1. 微服务架构治理与Service Mesh
定义：将服务间通信、流量管理、安全、可观测性等治理能力从应用代码中剥离，下沉到独立的基础设施层（Sidecar代理），由控制平面统一管理，实现业务逻辑与技术关注点解耦的架构模式。
关键要点：
- Sidecar模式：与业务服务同生命周期部署的代理进程，拦截所有出入口流量
- 数据平面（Data Plane）：负责实际数据转发，由Envoy、Linkerd Proxy等Sidecar组成
- 控制平面（Control Plane）：负责配置下发、证书颁发、策略管理，如Istio Control Plane
- 核心能力：服务发现、负载均衡、熔断限流、灰度发布、链路追踪、mTLS加密、流量镜像
- 与传统ESB区别：去中心化、边车模式、协议无关、对应用零侵入

### 2. 云原生架构与Kubernetes编排
定义：基于容器、服务网格、微服务、不可变基础设施和声明式API构建的松耦合、可弹性伸缩、可观测的分布式架构体系，依托Kubernetes作为资源编排与调度核心。
关键要点：
- Kubernetes核心对象：Pod、Deployment、StatefulSet、Service、Ingress、ConfigMap、Secret
- 调度三要素：资源请求（requests）、资源限制（limits）、调度器（Scheduler）
- 弹性伸缩：HPA基于CPU/内存指标、VPA垂直扩缩容、CA集群级伸缩
- 自愈机制：Liveness Probe（存活）、Readiness Probe（就绪）、Startup Probe（启动）
- 声明式API：用户描述期望状态，控制器持续协调（Reconcile Loop）逼近目标状态

### 3. 架构评估方法（ATAM/CBAM/SAAM）
定义：通过结构化流程对软件架构进行系统化评估的方法体系，从质量属性、可修改性、风险、成本等多维度审视架构方案的合理性。
关键要点：
- SAAM（场景驱动）：最早方法，基于用例场景分析架构，关注可修改性
- ATAM（架构权衡分析法）：9步流程，核心是识别质量属性效用树（Utility Tree）
- CBAM（成本效益分析法）：在ATAM基础上引入经济性分析，量化质量属性收益
- 效用树结构：根（系统效用）→ 质量属性（性能可用性安全性等）→ 属性求精 → 场景
- 风险点（Risk）、敏感点（Sensitivity）、权衡点（Tradeoff）、非风险点是评估的四大输出

### 4. 分布式事务与最终一致性
定义：在跨服务、跨数据库的分布式系统中，保证多个数据源操作满足ACID特性的事务处理机制，因CAP定理限制，强一致性与可用性不可兼得，需根据业务场景选择合适的一致性模型。
关键要点：
- CAP定理：一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance），三者只能取其二
- BASE理论：基本可用（BA）、软状态（S）、最终一致（E），放弃强一致换取高可用
- 常用方案：2PC两阶段提交（同步阻塞）、3PC三阶段提交（引入超时）、TCC补偿事务、Saga长事务拆分、可靠消息最终一致、本地消息表、最大努力通知
- Saga模式：通过正向服务+补偿服务实现长事务，编排器（Orchestrator）或Choreography协同

### 5. 高可用架构与灾备设计
定义：通过冗余设计、故障检测、自动切换等手段，使系统在面对硬件故障、网络异常、流量洪峰时仍能持续提供服务的能力评估与设计方法。
关键要点：
- 可用性指标：可用性 = MTBF / (MTBF + MTTR)，常以"几个9"衡量（99.99%即年停机52分钟）
- 冗余模式：主备（Active-Standby）、主主（Active-Active）、集群（Cluster）、异地多活
- 灾备等级：数据级（RPO小时级）、应用级（RPO分钟级）、业务级（RPO秒级，RTO分钟级）
- 限流策略：计数器、滑动窗口、令牌桶、漏桶；降级策略：熔断降级、开关降级、读降级、写降级
- 故障转移机制：健康检查、心跳检测、脑裂防护（Lease机制、Quorum多数派）

## 案例分析题
**题目**：某电商平台日均订单量500万，峰值QPS达10万。系统采用微服务架构，包含订单、商品、库存、支付、用户等60+服务，使用Spring Cloud技术栈。近期频繁出现以下问题：
1. 大促期间服务雪崩，订单服务故障导致整个交易链路不可用
2. 服务间调用链路复杂，平均调用深度8层，故障定位耗时长
3. 配置变更需要逐个服务重启，无法快速响应业务调整
4. 不同团队服务接口协议不统一（REST、gRPC、消息并存），联调成本高
作为系统架构师，请从架构治理角度提出完整优化方案，并说明技术选型理由。

**参考答案**：
一、引入Service Mesh统一治理层
选用Istio+Envoy Sidecar模式，将服务发现、负载均衡、熔断限流、流量管理、安全认证下沉至数据平面，业务服务专注核心逻辑，跨语言支持解决异构协议问题。

二、构建可观测性体系
部署Prometheus+Grafana+Jaeger+ELK四件套：Prometheus采集Sidecar指标、Jaeger实现全链路追踪（解决调用链过深问题）、ELK聚合日志，通过TraceID串联日志链路，故障定位时间从小时级降至分钟级。

三、配置中心化与动态化
采用Nacos替代本地配置：支持动态推送、灰度发布、版本回滚，配置变更秒级生效，避免服务重启。

四、流量防护与弹性设计
- 服务入口：Sentinel实现QPS限流、熔断降级、热点参数限流
- 大促预案：核心服务设置独立线程池隔离，防止雪崩
- 流量调度：Istio VirtualService配置按权重灰度、按Header路由

五、容灾与高可用
- 多活架构：核心服务至少双机房部署，DB采用一主多从+读写分离
- 降级开关：通过Nacos动态开关，非核心功能（推荐、评论）大促期间自动降级
- 超时与重试：Feign集成Resilience4j，设置合理超时阶梯（1s/3s/5s），避免请求堆积

六、组织治理
建立Service Owner制，每个服务明确责任人；建立服务契约库（OpenAPI/YAML），通过契约测试保证兼容性。

**踩分要点**：
- 治理层下沉与Sidecar解耦思想（必答）
- CAP/BASE与一致性模型的取舍（必答）
- 雪崩防护三板斧：隔离/熔断/限流（必答）
- 可观测性三大支柱：Metrics/Logging/Tracing（必答）
- 配置中心化的动态推送能力（必答）
- 容量评估与限流阈值的量化方法（加分项）
