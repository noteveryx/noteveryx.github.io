## 今日核心概念

### 1. 行为型设计模式（上）：责任链、命令、观察者、策略
定义：行为型模式关注对象之间的责任分配和算法封装，解决"谁做什么、何时做"的问题，共11种。本日聚焦论文和案例中最常考的四种。

关键要点：
- **责任链模式**：多个处理者形成链条，请求沿链传递直到被处理。典型应用：审批流程、过滤器链、中间件。优点：解耦请求发送者和接收者，动态调整处理顺序；缺点：可能请求无人处理、性能开销。
- **命令模式**：将请求封装为对象，支持参数化、排队、日志、撤销。核心四角色：Command接口、ConcreteCommand具体命令、Invoker调用者、Receiver接收者。典型应用：操作回滚、任务队列、宏命令。
- **观察者模式**：一对多依赖关系，主题状态变化时自动通知所有观察者。Java内置Observer/Observable，Spring事件机制。关键要区分"推模型"（主题推送全部数据）和"拉模型"（通知观察者后由观察者自己拉取）。
- **策略模式**：定义一系列算法族，分别封装，使它们可相互替换。核心三角：Context上下文、Strategy策略接口、ConcreteStrategy具体策略。典型应用：支付方式选择、排序算法切换、折扣计算。与状态模式的区别——策略是客户端主动选择，状态是内部状态驱动切换。

### 2. 行为型设计模式（下）：状态、模板方法、备忘录、访问者
定义：余下七种行为型模式，各自解决特定的对象交互与行为分发问题。

关键要点：
- **状态模式**：对象内部状态改变时改变其行为，看起来像换了类。核心：Context持有State引用，State定义状态行为。典型应用：订单状态流转、TCP连接状态、电梯运行状态。与策略模式区别——状态是上下文自身状态驱动的自动切换，策略是外部传入的主动选择。
- **模板方法模式**：在父类定义算法骨架，将可变步骤延迟到子类实现。核心：模板方法（final，定义流程）、基本方法（抽象/具体/钩子）。典型应用：JdbcTemplate、Servlet生命周期、JUnit测试框架。体现了"好莱坞原则——别打电话给我们，我们会打给你"。
- **备忘录模式**：在不破坏封装的前提下捕获对象内部状态，支持状态恢复。三角色：Originator发起人、Memento备忘录、Caretaker负责人。典型应用：游戏存档、编辑器撤销、事务回滚点。宽度与深度的权衡——频繁保存开销大，保存太少回滚幅度受限。
- **访问者模式**：将数据结构与数据操作分离，在不修改数据结构的前提下添加新的操作。双分派机制：Client调用element.accept(visitor)，element调用visitor.visitConcreteElement(this)。典型应用：编译器AST遍历、报表生成。缺点：数据结构变更需修改所有访问者，违反开闭原则，适用"数据结构稳定、操作多变"的场景。

### 3. 数据库架构设计
定义：数据库架构设计解决海量数据场景下存储、查询、一致性的架构问题，是系统架构设计师考试的高频考点。

关键要点：
- **读写分离**：一主多从，主库写从库读。核心机制：binlog同步（异步/半同步）。延迟问题处理：强制读主库、延迟阈值兜底、缓存补偿。适用场景：读多写少，如门户网站、报表系统。
- **分库分表**：垂直拆分（按业务模块，如用户库、订单库）和水平拆分（按行拆分，如按用户ID哈希取模）。路由算法：取模（扩容迁移代价大）、范围路由（热点问题）、一致性哈希（平衡扩容代价与分布均匀性）。跨库问题：分布式事务（两阶段提交/Seata AT/TCC）、跨分片Join（禁止/应用层聚合/全局表）、全局唯一ID（雪花算法/号段模式）。
- **NoSQL选型矩阵**：KV存储（Redis，高速缓存/会话）、文档存储（MongoDB，灵活Schema/日志）、列族存储（HBase，海量写入/时序）、图数据库（Neo4j，社交关系/风控）、搜索引擎（ES，全文检索/日志分析）。选型口诀：多写少读用列族，多读少写用文档，高速读写用KV，关系复杂用图，全文搜索用ES。
- **数据一致性**：强一致性（同步复制，性能差）、最终一致性（异步复制，有延迟窗口）、因果一致性（有因果关系的操作保证顺序）。CAP取舍：银行转账选CP放弃A，社交动态选AP放弃C，库存扣减用最终一致性的补偿方案。

### 4. 分布式系统核心设计
定义：分布式系统面临网络不可靠、时钟不同步、节点故障三大挑战，需要特定的设计手段来保证系统正确运行。

关键要点：
- **分布式锁**：保证跨进程互斥访问共享资源。实现方式：数据库唯一索引（简单但性能差）、Redis SETNX+Lua脚本（高性能，需处理锁续期和红锁）、ZooKeeper临时顺序节点（强一致性，Watch机制自动释放，但性能低于Redis）。必须考虑：锁超时释放（业务未完成时锁已过期→看门狗续期机制）、锁误删（释放时校验持有者身份）。
- **分布式事务**：跨多个数据源的事务一致性。方案演进：XA两阶段提交（强一致、性能差、协调者单点）→ TCC补偿事务（Try预留资源、Confirm确认提交、Cancel回滚释放，业务侵入性强）→ 可靠消息最终一致（本地事务+消息表、RocketMQ事务消息，高可用但对消息中间件有要求）→ Seata AT模式（自动回滚，对业务无侵入，依赖全局锁）。
- **分布式ID**：全局唯一ID生成策略。雪花算法：64位 = 1位未用 + 41位时间戳 + 10位机器ID + 12位序列号。优点：高性能、趋势递增、不依赖DB；缺点：时钟回拨问题（等待/抛异常/扩展回拨位）。对比：号段模式（依赖DB但批量分配降低压力）、UUID（简单但无序不利于索引）、Redis自增（依赖Redis可用性）。
- **分布式限流**：保护系统不被突发流量冲垮。算法对比：固定窗口（临界突刺问题）、滑动窗口（精确但内存开销大）、漏桶（恒定速率，削峰填谷）、令牌桶（允许突发，Guava RateLimiter、Sentinel）。架构层面：网关层限流（整体入口）+ 服务层限流（核心接口）+ 业务层限流（按用户/租户维度）。

### 5. 缓存架构设计
定义：缓存是性能优化的核心手段，但引入缓存也带来一致性、穿透、击穿、雪崩等新的架构问题。

关键要点：
- **缓存策略模式**：Cache Aside（旁路缓存，先读缓存→未命中读DB→写缓存，更新时先写DB→删缓存；最常用）；Read/Write Through（缓存层代理读写DB，应用与DB彻底解耦）；Write Behind（异步批量写DB，高性能但数据丢失风险）。Cache Aside为什么先删缓存而不是先更新缓存——避免并发读写导致缓存与DB不一致。
- **三大经典问题**：缓存穿透（查不存在的数据→布隆过滤器/缓存空值）、缓存击穿（热点Key过期→互斥锁/永不过期+异步刷新）、缓存雪崩（大量Key同时过期→过期时间加随机值/多级缓存/限流降级）。三级防线：事前（过期加随机）、事中（互斥锁排队）、事后（限流熔断兜底）。
- **缓存一致性**：更新DB后删除缓存（而非更新缓存）——延迟双删策略（先删缓存→更新DB→延迟再删缓存），解决读写并发导致的不一致。Canal监听binlog异步更新缓存，最终一致性方案。强一致性场景不适合缓存或使用分布式锁保证。
- **多级缓存架构**：客户端缓存（浏览器LocalStorage/HTTP缓存头）→ CDN边缘缓存 → 反向代理缓存（Nginx proxy_cache）→ 本地缓存（Caffeine/Guava, JVM内）→ 分布式缓存（Redis Cluster）→ 数据库。每层命中率逐层降低、容量逐层增大、延迟逐层增加。

## 案例分析题

**题目**：

某大型电商平台在"双十一"大促期间频繁出现以下问题：
1. 热门商品详情页在高并发下响应超时，经排查是缓存中对应商品的Key过期瞬间，大量请求直接打到数据库导致数据库CPU飙升；
2. 订单系统在创建订单时需要同时扣减库存（库存服务）、扣减余额（账户服务）、生成积分（积分服务），其中积分服务偶尔调用失败，导致整个订单流程失败率高；
3. 运营反馈部分恶意用户使用脚本遍历不存在的商品ID，大量穿透缓存直接查询数据库，增加了数据库无效负载；
4. 平台计划将单体应用拆分为微服务，但各服务间存在大量跨表Join查询，直接拆分后性能大幅下降。

请以系统架构设计师的身份回答以下问题：

（1）分别指出问题1、2、3对应的缓存/分布式领域的典型问题名称，并给出每种问题的架构解决方案。

（2）针对问题4，设计两种不同的数据层拆分策略，并分析各自的适用场景和优缺点。

（3）综合上述四个问题，绘制一个包含缓存层、数据库层、微服务层的整体架构优化方案图（用文字描述即可），说明各组件如何协同工作。

**参考答案**：

（1）问题识别与方案：
- 问题1对应**缓存击穿**。解决方案：①对热点Key设置互斥锁，同一时刻只允许一个线程回源查DB重建缓存；②热点Key永不过期，由异步任务定期刷新；③逻辑过期+互斥锁，缓存值中携带逻辑过期时间，发现过期时加锁重建。
- 问题2对应**分布式事务一致性问题**。解决方案：采用TCC补偿事务模式，Try阶段分别预留库存（冻结扣减）、预留余额、预留积分额度；Confirm阶段全部确认下发；Cancel阶段任一失败则全部回滚。积分服务可降级为非关键链路，主流程扣减库存+余额采用TCC，积分采用可靠消息最终一致性异步补发。
- 问题3对应**缓存穿透**。解决方案：①布隆过滤器，将所有合法商品ID预先加载到布隆过滤器，请求先过布隆过滤器检查是否存在，不存在直接返回空；②对不存在的数据也缓存空值，设置较短过期时间（如5分钟）。

（2）数据层拆分策略：
- **策略一：垂直拆分+禁止跨库Join**。按业务域拆分为商品库、订单库、用户库等独立数据库，数据层禁止跨库Join，跨库关联在应用层通过多次查询+内存聚合完成。适用场景：业务边界清晰、跨域关联查询频率低的系统。优点：解耦彻底、独立扩展、故障隔离；缺点：应用层聚合增加复杂度、数据一致性需额外保障。
- **策略二：读写分离+CQRS**。命令端（写）按业务域垂直拆分，每个微服务独立数据库；查询端（读）通过CDC（如Canal）将各库数据实时同步到一个统一的查询库（Elasticsearch或宽表），所有跨域查询走查询库。适用场景：读多写少的复杂查询场景、报表与数据分析需求多的系统。优点：读写性能分离、跨域查询高效、各域独立演进；缺点：数据实时一致性问题、基础设施复杂度增加、查询库存储冗余。

（3）整体架构优化方案（文字描述）：

```
[客户端层]
  ↓
[CDN + WAF] → 布隆过滤器（防穿透）
  ↓
[API网关层] → 统一入口 + 限流（令牌桶算法）+ 鉴权
  ↓                    ↓                    ↓
[商品服务]      [订单服务]          [积分服务]
  ↓ 缓存层        ↓ 缓存层            ↓
Redis Cluster  Redis Cluster    Redis
(Cache Aside   (热点永不过期     ↓
+逻辑过期)     +互斥锁防击穿)    MQ异步消息
  ↓              ↓                ↓
[商品DB]       [订单DB]          [积分DB]
  ↓              ↓                ↓
  └──────────┬───┴───────────────┘
             ↓
          Canal实时同步
             ↓
[查询库 ES / 宽表] ← 跨域只读查询

组件协同：
- 布隆过滤器在网关层拦截无效商品ID请求（解决穿透）
- 商品服务对热点商品采用逻辑过期+互斥锁机制（解决击穿）
- 订单服务核心链路（库存+余额）采用TCC事务，积分非核心链路通过MQ异步最终一致（解决分布式事务）
- 各服务独立数据库垂直拆分，跨域只读查询走Elasticsearch查询库（解决跨库Join）
- 全局限流分为网关层（总流量）、服务层（核心接口）、业务层（按租户维度）
```

**踩分要点**：
- 问题1/2/3的正确术语命名各1分（共3分）：缓存击穿、分布式事务/数据一致性、缓存穿透
- 每个问题的解决方案至少写出2点，且技术方案具体可落地（每问2分，共6分）
- 问题4两种策略需明确命名（垂直拆分+CQRS），分别阐述优缺点和适用场景（4分）
- 整体架构图需覆盖缓存层+数据库层+微服务层的完整链路，体现组件协同关系（3分）
- 面试加分项：提及Canal CDC、最终一致性、降级策略、TCC补偿（2分）
- 总分18分，12分及格线

## 记忆口诀

**行为型设计模式分类记**：
"责命观策状，模备访解代"——责任链、命令、观察者、策略、状态、模板方法、备忘录、访问者、解释器、中介者、迭代器

**分布式三大缓存问题**：
"穿透布隆空值拦，击穿永不过期加锁看，雪崩过期随机加限流板"

**分库分表四难题**：
"ID要唯一（雪花算法），事务要一致（TCC/消息），Join要禁止（应用聚合），扩容要平滑（一致性哈希）"

**数据库选型一句话**：
"列多写（HBase），文多读（MongoDB），KV高速（Redis），图关系（Neo4j），ES全文搜"

**分布式事务选型口诀**：
"XA强一致但性能差，TCC业务入侵强保障，消息最终一致高可用，Seata自动化成本低"

**CAP取舍速记**：
"金融选CP（转账不容错），社交选AP（动态不怕等），库存选BASE（最终一致补偿）"
