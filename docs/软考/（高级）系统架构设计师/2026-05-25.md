## 今日核心概念

### 1. ABSD 基于架构的软件开发
定义：Architecture-Based Software Design，以软件架构为核心驱动整个软件开发过程的方法论。架构不再只是设计阶段的产出物，而是贯穿需求分析、设计、实现、测试、部署和维护全过程的核心资产。

关键要点：ABSD 强调架构是第一位的——先有架构决策再有详细设计。核心活动包括：架构需求获取（识别质量属性和业务约束）、架构设计（选择架构风格和模式）、架构文档化（用ADL或视图描述）、架构评估（用ATAM/SAAM等方法评审）、架构实现（将架构映射到代码）、架构演化（随需求变化调整架构）。与传统开发的区别：传统开发是"功能需求→设计架构"，ABSD 是"质量需求→驱动架构→约束功能实现"，架构师在ABSD中承担比在传统开发中更前移的决策角色。

### 2. DSSA 特定领域软件架构
定义：Domain-Specific Software Architecture，面向某一特定问题领域的、在该领域内可复用的参考架构。DSSA 不等同于一个具体系统的架构，而是该领域内一类系统的架构模板。

关键要点：DSSA 构建包含三个阶段——①领域分析：识别领域范围、共性需求和变化点，产出领域模型；②领域设计：基于领域模型设计DSSA，定义参考架构中的组件类型、连接件类型和约束规则；③领域实现：构建可复用组件库和架构框架，提供开发工具支持。DSSA 与软件产品线的区别：DSSA 是架构层面的领域复用，产品线是资产层面（含架构+组件+文档等）的系统化复用。DSSA 是产品线架构的基础。典型DSSA示例：航空电子领域的ARINC 653架构、电信领域的TMN架构。

### 3. SAAM 软件架构分析方法
定义：Software Architecture Analysis Method，是最早提出的基于场景的架构分析方法，通过评审会议让各利益相关方用场景走查架构，发现架构潜在问题和风险。

关键要点：SAAM 评估六步——①描述候选架构（形成架构描述文档）；②开发场景（各利益相关方从各自角度提出场景）；③场景分类（分类为直接场景，即架构支持的场景，和间接场景，即架构需要修改才能支持的场景）；④场景走查（逐个场景在架构上走查，识别架构中哪些组件需要修改）；⑤场景交互分析（分析不同场景对同一组件的修改是否存在冲突）；⑥整体评估（汇总问题、给出改进建议）。SAAM 与 ATAM 对比：SAAM 更早、更简单，侧重场景覆盖度与组件冲突分析；ATAM 在此基础上增加了质量属性效用树和敏感点/权衡点的分析，更系统化。两者可以结合使用。

### 4. 质量效用树
定义：Quality Attribute Utility Tree，是ATAM中用于对质量属性场景进行优先级排序的结构化工具。以"效用"为根节点，以各质量属性为二级节点，以具体场景为叶节点，并通过（重要性，实现难度）二维评级来决策评估优先级。

关键要点：构建效用树的步骤——①识别核心质量属性（性能、可用性、安全性、可修改性、可测试性、易用性等）作为二级节点；②为每个质量属性细化具体场景（如"当数据库节点宕机时，系统在30秒内自动切换到备用节点"）；③为每个场景标注（H/M/L, H/M/L），即（对系统成功的重要性，当前架构实现的难度）；④优先分析标注为（H,H）和（H,M）的场景，这些是高风险高价值的关注点。效用树的作用：①将抽象的质量属性具象化为可验证的场景；②让利益相关方就评估优先级达成共识；③将有限评估时间集中在关键问题上；④为ATAM后续的属性模型分析提供输入。

### 5. 企业应用集成模式
定义：Enterprise Application Integration，EAI，指将企业内部多个异构系统（遗留系统、ERP、CRM、SCM等）连接起来实现数据共享和业务协同的技术方法和架构模式。

关键要点：四种核心集成模式——①点对点集成（Point-to-Point）：系统间两两直接连接，实现简单但连接数随系统数O(n²)增长，适用于小规模集成；②总线式集成（ESB/Message Bus）：各系统通过总线连接，消息路由和数据转换由总线统一处理，连接数退化为O(n)，实现松耦合；③集成适配器模式（Adapter）：在总线与各系统之间部署适配器，完成协议转换、数据格式转换和接口适配；④数据集成模式（Data Integration）：通过ETL工具抽取、转换、加载数据到数据仓库，实现跨系统的报表和分析。集成面临的主要挑战：数据语义不一致（同名不同义、同义不同名）、协议异构（SOAP/REST/JMS/文件）、事务一致性（跨系统分布式事务）、安全边界穿越。

## 案例分析题

**题目**：
某省级智慧政务平台需要整合人社局、医保局、公安局、住建局四个业务部门的现有信息系统，实现"一网通办"。各系统现状如下：人社局系统（基于Java，Oracle数据库，暴露SOAP接口）、医保局系统（基于.NET，SQL Server，仅支持数据库直连，无API）、公安局系统（基于Java，PostgreSQL，暴露REST接口）、住建局系统（老旧C/S架构，Sybase数据库，无API）。业务要求整合后：公民在统一门户提交一次材料，后台自动流转给各部门处理，办理结果汇总返回；系统需保证跨部门办事流程的事务最终一致性。

（1）分析上述集成场景面临的三个主要架构挑战，并简要说明原因。（4分）
（2）为该政务平台设计集成架构方案，说明采用何种集成模式及理由，绘制架构层次图（文字描述即可）。（4分）
（3）针对住建局老旧系统的集成，你会采用什么具体策略？说明技术方案和潜在风险。（2分）

**参考答案**：
（1）①协议异构——四个系统分别使用SOAP、数据库直连、REST和无API四种不同的交互方式，无法直接互联；需统一集成层的协议转换能力。②数据语义不一致——四部门对相同业务概念的定义不同，如"公民基本信息"在人社局包含社保号、在公安局包含身份证号、在住建局包含房产登记号，需要建立统一数据模型和映射规则。③事务最终一致性——跨部门办事流程涉及四个独立系统的数据变更，无法使用传统ACID分布式事务（各系统DB异构且非XA兼容），必须采用补偿型最终一致性方案。

（2）采用基于ESB的消息总线集成架构。架构层次自下而上：数据源层（四部门原有系统+适配器）→集成层（ESB消息总线，负责消息路由、协议转换、数据转换）→服务编排层（BPM业务流程引擎，编排跨部门办事流程）→服务网关层（统一API网关，对外暴露RESTful接口给前端门户）。选择ESB的理由：四个系统接口各异，ESB天然支持多协议适配、消息路由和数据转换，将O(n²)的集成复杂度降为O(n)；BPM引擎支持长流程编排和人工任务节点，适配政务审批场景；API网关提供统一的鉴权、限流和监控能力。

（3）针对住建局Sybase+C/S老旧系统：在住建局侧部署数据同步代理（CDC，变更数据捕获），监听Sybase数据库表变更日志，将变更实时推送到集成层消息队列。集成层将其转换为统一模型后流入业务流程。技术方案：采用Debezium或自研CDC Agent捕获Sybase变更日志→投递到Kafka消息队列→ESB消费后转换。潜在风险：①CDC对Sybase版本的兼容性需验证；②直连数据库捕获存在性能影响和安全隐患；③如住建局表结构与统一模型差异过大，映射规则可能复杂且易出错。备选方案：如CDC不可行，可采用定时数据同步+手动补录兜底方案。

**踩分要点**：
- 三个架构挑战各1.33分，须结合案例场景具体分析（如"协议异构"须列出四个系统的具体协议差异），泛泛而谈不给满分（4分）
- 架构方案须包含层次描述（至少三层）+ESB模式选择理由（至少两个理由），缺层次结构扣2分（4分）
- 老旧系统集成须给出具体技术方案（CDC/适配器/重建至少选一且自圆其说）+至少一个潜在风险，仅说"需要改造"不给分（2分）

## 记忆口诀

**ABSD五点要记住**：需设文评实演——需求获取、架构设计、文档化、评估、实现与演化。谐音"徐设文评实演"。

**DSSA三阶段对应三问**：分什么（领域分析）→怎么设计（领域设计）→怎么用（领域实现）。"分设计用"倒过来就是"用计分"——用DSSA计分。

**EAI四模式速记**：点总线适数——点对点、总线式、适配器、数据集成。"点总线"连接网络，"适数"处理数据。另：集成复杂度口诀——点对点 n方不可取，总线式 线性才靠谱。
