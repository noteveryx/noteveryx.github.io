## 今日核心概念
### 1. 结构型设计模式
定义：关注类和对象的组合方式，通过继承或组合来形成更大的结构体。主要解决如何将类或对象结合成更复杂的结构，同时保持结构的灵活性和高效性。

关键要点：
- 适配器模式（Adapter）：将一个接口转换为客户期望的另一个接口，使不兼容的接口可以协同工作。分为类适配器（继承）和对象适配器（组合）两种。
- 桥接模式（Bridge）：将抽象部分与实现部分分离，使它们可以独立变化。核心是"用组合代替继承"。
- 组合模式（Composite）：将对象组合成树形结构以表示"部分-整体"的层次关系，使客户端对单个对象和组合对象的使用具有一致性。
- 装饰模式（Decorator）：动态地给一个对象添加额外的职责，比生成子类更灵活。典型应用：Java I/O流体系。
- 外观模式（Facade）：为子系统中的一组接口提供一个统一的高层接口，简化子系统的使用。
- 享元模式（Flyweight）：运用共享技术有效支持大量细粒度对象的复用。核心是区分内部状态（可共享）和外部状态（不可共享）。
- 代理模式（Proxy）：为其他对象提供一种代理以控制对这个对象的访问。包括远程代理、虚代理、保护代理、智能引用等。

### 2. CAP定理与BASE理论
定义：CAP定理由Eric Brewer提出，是分布式系统的核心理论基石。它指出一个分布式系统无法同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）三个需求，最多只能同时满足其中两个。BASE理论是对CAP中AP方案的延伸，是对传统ACID理论的补充。

关键要点：
- C（Consistency一致性）：所有节点在同一时间看到的数据完全一致。
- A（Availability可用性）：每个请求都能接收到一个（无论成功或失败的）响应，系统始终可用。
- P（Partition Tolerance分区容错性）：系统在遇到网络分区故障时仍能继续运行。
- BASE理论三要素：Basically Available（基本可用，允许响应延迟或功能降级）、Soft state（软状态，系统中的数据存在中间状态）、Eventually consistent（最终一致性，不要求实时一致但保证最终达到一致）。
- 常见落地：电商系统通常选择AP（保证可用性和分区容错性，接受最终一致性）；银行转账通常选择CP（保证一致性和分区容错性，允许短暂不可用）。
- 最终一致性五种变体：因果一致性、读己之写一致性、会话一致性、单调读一致性、单调写一致性。

### 3. 微服务架构核心治理
定义：微服务架构将单体应用拆分为一组小型、自治的服务，每个服务围绕业务能力构建、独立部署和扩展。微服务治理是确保众多微服务有序协作的一系列技术和管理实践。

关键要点：
- 服务注册与发现：服务实例启动时向注册中心注册，调用方从注册中心获取可用实例列表。主流组件：Eureka、Nacos、Consul、Zookeeper。
- API网关：所有外部请求的统一入口，负责路由转发、身份认证、限流熔断、协议转换、日志监控。主流组件：Spring Cloud Gateway、Zuul、Kong、Nginx+Lua。
- 熔断与降级：熔断器（Circuit Breaker）在检测到下游服务频繁失败时自动断开调用链路，快速失败；降级是在资源紧张时有策略地放弃非核心功能。Hystrix、Sentinel、Resilience4j。
- 配置中心：将配置从代码中抽离，实现配置的集中管理和动态刷新。主流组件：Nacos、Apollo、Spring Cloud Config。
- 服务网格（Service Mesh）：将服务间的通信层下沉到基础设施，Sidecar代理接管流量管理、安全通信、可观测性。代表：Istio、Linkerd。

### 4. 系统安全架构设计
定义：系统安全架构是在系统设计层面构建的信息安全保障体系，涵盖身份认证、访问控制、数据安全、网络安全和审计追踪等多个维度，确保系统在面临内外部威胁时仍能保护信息的机密性、完整性和可用性（CIA三要素）。

关键要点：
- 认证与授权体系：认证（Authentication）确认"你是谁"，授权（Authorization）决定"你能做什么"。常见模型：RBAC（基于角色的访问控制）、ABAC（基于属性的访问控制）、OAuth2.0（第三方授权协议）、JWT（无状态令牌）。
- 安全通信：传输层加密（TLS/SSL）、API签名验证、防重放攻击（时间戳+随机数）、SQL注入与XSS防御。
- 数据安全：敏感数据加密存储（AES）、数据脱敏、数据备份与灾备、最小权限原则。
- 安全审计：操作日志记录与追溯、异常的实时告警、定期安全评估与渗透测试。
- 纵深防御原则：不在单一层面依赖单一机制，而是构建多层防御体系——网络层（防火墙/WAF）→ 应用层（输入校验/权限控制）→ 数据层（加密/审计）。

## 案例分析题
**题目**：某电商平台原有系统采用单体架构，随着业务量增长，系统在"双十一"大促期间频繁出现响应超时和服务不可用问题。技术团队决定将系统重构为微服务架构，将订单、商品、用户、支付、库存拆分为独立服务。但在实施过程中遇到以下问题：

（1）订单服务依赖库存服务和支付服务，库存服务在大促期间压力巨大，调用延迟超过5秒导致订单服务线程池耗尽。分析此问题的根源，提出至少两种改进方案并说明理由。

（2）系统拆分后，前端需要调用多个微服务来展示"我的订单详情"页面（需要用户信息、订单详情、商品信息三个服务的数据）。请设计一种方案降低前端的调用复杂度和网络开销，并说明方案的架构原理。

（3）微服务架构下，某个服务版本升级后接口不兼容，导致调用方出现大量500错误。请提出服务版本管理和灰度发布的解决方案，确保服务平滑升级。

**参考答案**：
（1）问题根源：没有对跨服务调用设置保护机制，库存服务的慢响应通过调用链向订单服务级联扩散，形成"雪崩效应"。改进方案：

方案一：引入熔断机制（如Sentinel/Hystrix）。当库存服务错误率或响应时间超过阈值时，熔断器自动打开，订单服务直接返回降级结果（如"库存查询繁忙，请稍后重试"），避免线程池耗尽。理由：熔断器是防止级联故障最直接的手段，快速失败（fail-fast）保护了调用方资源。

方案二：实施线程池隔离。为库存服务和支付服务的调用分配独立的线程池，即使库存服务线程池耗尽，支付服务的调用不受影响。配合超时控制（如3秒超时）和请求排队，防止单个慢服务拖垮整个系统。理由：线程池隔离是舱壁模式（Bulkhead）的经典实现，将故障限制在局部范围内。

（2）采用BFF（Backend For Frontend）模式 + API网关方案：

为前端App/Web各自创建一个BFF层，BFF负责聚合多个微服务的数据。前端只向BFF发起一个请求，BFF内部并行调用用户服务、订单服务、商品服务，聚合结果后返回给前端。

架构原理：BFF是API网关的细粒度变体，面向特定前端定制接口，承担了协议适配、数据聚合和字段裁剪的职责。API网关（如Spring Cloud Gateway）在更外层负责统一鉴权、限流和路由，BFF与API网关协同工作，层次分工清晰。

（3）服务版本管理与灰度发布方案：

版本管理策略：采用URL路径版本号（/api/v1/order、/api/v2/order）或请求头版本号两种方式。推荐URL路径版本，语义清晰、兼容性强。服务端同时维护新旧两个版本的接口，旧版本标记为@Deprecated并在文档中声明废弃时间表。

灰度发布方案：基于注册中心的流量权重控制。新版本服务以低权重（如5%）注册到注册中心，观察一段时间（如30分钟），监控错误率和响应时间。若指标正常则逐步增大权重（20%→50%→100%），期间旧版本保持在线。若新版本出现异常，立即将权重归零实现秒级回滚，调用方无感知。

**踩分要点**：
（1）必须点出"雪崩效应"和"级联故障"两个关键词；两种方案须给出具体技术名称并说明原理；每条改进须有对应的理由阐述。
（2）BFF模式是核心得分词；须说明请求从多次合并为一次的设计思路；须区分BFF层和API网关的不同职责。
（3）须同时给出版本管理（URL路径）和灰度发布（权重控制）两个维度；须说明回滚机制和无损切换。

## 记忆口诀
- 结构型七模式速记：适配桥接，组装外享代（适配器/桥接/组合/装饰/外观/享元/代理）——"适合桥边看风景，组装外观享代理"
- CAP口诀：CAP三者不共存，要么CP要么AP——"分区容错必须选，一致可用二选一"
- BASE三要素：基本可用、软状态、最终一致——"基软最终"（基本→软→最终）
- 微服务治理五件套：注册发现、API网关、熔断降级、配置中心、服务网格——"注关口降配网"（注册→网关→熔断口→配置→网格）
- 安全CIA三要素：机密性（Confidentiality）、完整性（Integrity）、可用性（Availability）——"机完整可用"
