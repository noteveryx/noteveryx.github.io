## 今日核心概念

### 1. 信息系统战略规划方法论

定义：信息系统战略规划（ISSP）是组织制定长期信息化建设目标、策略和实施方案的系统性过程，确保IT投资与业务战略对齐。

关键要点：
- 诺兰模型（Nolan Model）六阶段：初始期→蔓延期→控制期→集成期→数据管理期→成熟期。前三个阶段是计算机时代特征，后三个阶段是信息时代特征，拐点在控制期到集成期之间
- BSP企业系统规划法：自上而下规划、自下而上实现，核心输出是U/C矩阵（功能/数据类矩阵），通过矩阵变换识别子系统边界，C表示数据类的创建者、U表示使用者
- CSF关键成功因素法：找出组织成功的关键因素，围绕CSF确定信息需求。CEO→部门经理→基层逐层分解，适合高层决策支持系统规划
- 价值链模型（Porter）：基本活动（进料后勤→生产→发货后勤→销售→服务）+ 支持活动（基础设施、人力资源、技术开发、采购），在价值链各环节嵌入IT增强竞争力
- 战略一致性模型（SAM）：业务战略↔IT战略、组织基础设施↔IT基础设施，四象限对齐，避免IT与业务脱节
- 企业信息化五层体系：基础网络层→数据资源层→应用系统层→信息服务层→战略决策层，自下而上构建

BSP与CSF对比：BSP是全面规划法，适合中长期整体规划，输出信息架构蓝图；CSF是聚焦法，从关键成功因素倒推信息需求，适合特定领域快速切入。实践中常结合使用：BSP定框架，CSF补重点。

### 2. 数据库规范化理论

定义：规范化是一套通过分解关系模式来消除数据冗余和操作异常的形式化方法，基于Armstrong公理系统和函数依赖理论。

关键要点：
- 函数依赖FD：X→Y表示X函数决定Y。平凡函数依赖（Y⊆X）、完全函数依赖、部分函数依赖、传递函数依赖（X→Y, Y→Z, Y不决定X，则X→Z为传递依赖）
- Armstrong公理系统八条规则：自反律（若Y⊆X则X→Y）、增广律（X→Y则XZ→YZ）、传递律（X→Y且Y→Z则X→Z）+ 三条推论（合并、分解、伪传递）+ 闭包计算（求属性集X关于F的闭包XF⁺）
- 范式递进关系：1NF ⊃ 2NF ⊃ 3NF ⊃ BCNF ⊃ 4NF ⊃ 5NF，后一级消除前一级的特定异常

各级范式定义与判别：
| 范式 | 消除的问题 | 判别条件 |
|------|-----------|---------|
| 1NF | 非原子值 | 所有属性不可再分 |
| 2NF | 部分函数依赖 | 1NF + 每个非主属性完全函数依赖于候选键 |
| 3NF | 传递函数依赖 | 2NF + 每个非主属性不传递依赖于候选键 |
| BCNF | 主属性对非主属性的部分依赖 | 所有函数依赖的决定因素都是超键 |
| 4NF | 多值依赖 | 每个非平凡多值依赖的决定因素都是超键 |

模式分解标准：
- 无损连接分解（Lossless Join）：分解后自然连接能还原原关系，充要条件是两个子模式的交集是其中一个的候选键。必保
- 保持函数依赖：分解后所有FD都能从分解的子模式FD集中推导出。力争保

反规范化设计：在OLAP场景中，为提升查询性能主动引入冗余。常用技术——增加冗余列、增加派生列、表合并、表分割（水平/垂直）。代价是降低写入性能、增加存储空间、可能引入不一致。必须通过触发器或应用层维护数据一致性。适用场景：数据仓库星型模型的事实表+维度表设计、高并发读场景。

### 3. 计算机体系结构与流水线

定义：计算机体系结构是软硬件交界面上的抽象规范，包括指令集架构ISA、微架构、系统架构三个层次。Flynn分类法根据指令流和数据流将计算机分为SISD、SIMD、MISD、MIMD四类。

关键要点：
- Flynn四分类：
  - SISD（单指令单数据）：传统冯·诺依曼单核处理器
  - SIMD（单指令多数据）：向量处理器、GPU核心、现代CPU的SSE/AVX指令扩展
  - MISD（多指令单数据）：理论上存在，实际极少应用（容错系统勉强对应）
  - MIMD（多指令多数据）：多核CPU、多处理器系统，是并行计算主流形态

- 指令流水线五级经典分段：取指IF→译码ID→执行EX→访存MEM→写回WB。流水线周期 = max{各段执行时间} + 寄存器延迟，非流水线执行时间 = n条指令 × k段 × 周期，流水线执行时间 = k周期 + (n-1)周期

- 流水线加速比：S = nk / (k+n-1)，n→∞时S→k。实际加速受限于流水段不平衡和冒险

- 流水线三大冒险（Hazard）：
  - 结构冒险（硬件资源冲突）：多条指令同时访问同一功能部件。解决：增加硬件资源、流水化功能部件
  - 数据冒险（数据依赖）：RAW（先写后读，最常发生）、WAR（先读后写）、WAW（写后写）。解决：转发旁路（Forwarding/Bypassing）、流水线暂停（Stall/气泡）
  - 控制冒险（分支指令）：分支未决时后续指令无法确定。解决：分支预测（静态/动态）、延迟分支槽、预测执行

- Cache三级映射与替换：
  - 直接映射：每个主存块映射到唯一Cache行，冲突高
  - 全相联：任意块可映射到任意Cache行，硬件代价大
  - 组相联：折中方案，n路组相联将Cache分为若干组，每组n行，主存块映射到固定组内任意行
  - 替换算法：LRU（最近最少使用，效果最好代价大）、FIFO（先进先出）、随机（简单但性能不可控）

- 并行架构：SMP（对称多处理器，共享内存总线）、MPP（大规模并行处理，分布式内存）、NUMA（非一致内存访问，本地内存快远地内存慢，架构师需关注数据局部性）

### 4. 操作系统进程同步与死锁

定义：进程同步是协调多个并发进程对共享资源访问的机制，确保执行顺序的正确性和数据一致性。死锁是多个进程因循环等待资源而永久阻塞的状态。

关键要点：

PV操作与信号量：
- P操作（wait/proberen）：信号量值减1，若结果<0则阻塞当前进程
- V操作（signal/verhogen）：信号量值加1，若结果≤0则唤醒一个等待进程
- 互斥信号量初值=1，同步信号量初值=0或N
- PV操作必须成对出现：互斥的PV在同一进程，同步的PV在不同进程

经典同步问题（架构师考试高频）：
- 生产者—消费者：empty计数空缓冲区、full计数满缓冲区、mutex互斥访问缓冲池。注意P(mutex)必须在P(empty)/P(full)之后，否则死锁
- 读者—写者：允许多读单写。读者优先（写者可能饥饿）vs写者优先（读者可能饥饿）vs公平策略。关键是第一个读者P(rw_mutex)、最后一个读者V(rw_mutex)
- 哲学家进餐：五哲学家五筷子，每人先拿左再拿右。死锁四解法：最多允许四人就餐、奇数先左偶数先右、同时拿两根（AND信号量）、加超时

死锁四个必要条件：互斥条件、保持并等待、不可抢占、循环等待。缺一不死锁

死锁处理策略：
- 预防：破坏四个必要条件之一——SPOOLing破坏互斥、一次性分配破坏保持等待（资源利用率低）、可抢占破坏不抢占、按序分配破坏循环等待
- 避免：银行家算法——系统安全状态判定。Available可用资源向量、Max最大需求、Allocation已分配、Need还需需求。每个进程试探性分配后检查是否存在安全序列，不存在则不分配
- 检测与恢复：检测用资源分配图化简（无环路则无死锁/每种资源只有一个实例时环路=死锁）；恢复可终止进程或资源抢占
- 鸵鸟策略：忽略死锁，由人工干预（实际系统中用得最多的策略）

银行家算法安全序列判定四步：
1. Work = Available，Finish[i] = false
2. 找满足Finish[i]=false且Need[i] ≤ Work的进程i
3. Work = Work + Allocation[i]，Finish[i]=true，返回第2步
4. 若所有Finish[i]=true则系统安全，否则不安全

### 5. 数学建模与经济管理

定义：架构设计师需掌握运筹学与工程经济学基础方法，用于系统可行性论证、资源配置优化和决策分析。

关键要点：

线性规划三要素：决策变量、目标函数（max或min）、约束条件（线性等式/不等式）。求解方法——图解法（两变量）、单纯形法（多变量通解）、对偶理论（原问题与对偶问题的最优值相等）。灵敏度分析：目标系数C变化范围、右端常数b变化范围，各变量在最优基不变的前提下允许的变动区间

最短路径问题：Dijkstra算法（非负权有向图，贪心策略，O(n²)）、Floyd算法（全源最短路径，动态规划，O(n³)）。架构选型中的网络延迟分析典型应用

盈亏平衡分析：BEP产量 = 固定成本 / (单价 - 单位变动成本)。计算保本点、判断经营安全率（安全边际率>30%安全、10-20%较危险）、多方案比较时选盈亏平衡产量低的方案

净现值NPV：∑(CI-CO)ₜ / (1+i₀)ᵗ，NPV>0项目可行。内部收益率IRR：使NPV=0的折现率，IRR>基准收益率则可行。动态投资回收期：累计折现净现金流量首次≥0的年份

决策树与风险决策：
- 期望货币价值EMV = Σ(概率 × 损益)
- 完全信息期望价值EVPI = 有完全信息最大收益 - 无完全信息最大EMV
- 不确定型决策五准则：乐观Maximax（好中求好）、悲观Wald（坏中求好）、等可能Laplace（等概率求期望）、后悔值Savage（最小化最大后悔）、折中Hurwicz（α×最好 + (1-α)×最坏）

## 案例分析题

**题目**（数据库与并发综合）：

某市税务局开发"智慧税务综合服务平台"，系统需要满足以下需求：
1. 纳税人基础信息管理：纳税人统一社会信用代码、名称、类型、注册地址、法定代表人等多属性，且法人可关联多家企业
2. 申报记录管理：每个纳税人每月产生多条申报记录（税种、计税金额、税率、应纳税额），需支持按月/按税种/按纳税人多维汇总查询
3. 系统日均查询请求5万次以上，月底申报高峰期并发写入压力大，查询响应时间需<200ms
4. 系统需对接工商局、社保局、银行三个外部系统，获取企业变更信息后更新本地数据

问题1（8分）：请为纳税人基础信息和申报记录设计关系模式，要求至少达到3NF，并说明候选键和函数依赖。

问题2（7分）：针对月底申报高峰期读写冲突问题，请设计数据库优化方案，包括索引策略、读写分离方案、缓存策略。需说明方案涉及的技术和预期效果。

问题3（5分）：对接工商局的数据同步过程可能因网络故障中断。请设计一个保证数据一致性的同步方案，考虑断点续传和冲突处理。

问题4（5分）：假设单次查询平均耗时50ms，其中数据库耗时40ms。若要实现查询响应时间<200ms，同时支持500并发查询，请计算所需的应用服务器最小线程数，并说明是否有必要引入缓存。

**参考答案**：

问题1（关系模式设计）：

纳税人表 Taxpayer(corp_id, credit_code, name, type, reg_address, legal_person, ...)
- 候选键：corp_id；函数依赖：corp_id→{credit_code, name, type, reg_address}，credit_code→corp_id
- 若legal_person也是属性，存在传递依赖：corp_id→legal_person→legal_person_phone（假设），但本例中legal_person非唯一标识。需拆分出法人表Person(person_id, name, id_card, phone)，纳税人法人关联表Taxpayer_Legal(corp_id, person_id, role)以支持一人多企

申报记录表 Declaration(decl_id, corp_id, tax_type, tax_base, tax_rate, tax_amount, period, create_time)
- 候选键：decl_id；函数依赖：decl_id→所有非主属性，{corp_id, tax_type, period}→tax_base以及tax_amount。tax_rate依赖tax_type而非具体记录，设计独立税率表
- 税种税率表 TaxRate(tax_type, tax_rate, effective_date)，候选键(tax_type, effective_date)

最终设计达到3NF：各非主属性都完全函数依赖于候选键，无传递依赖。

问题2（优化方案）：

索引策略：申报记录表按(corp_id, period)建立联合索引（覆盖高频查询），按period建立单列索引（支持月度汇总），按tax_type建立单列索引。需注意索引过多会降低写入性能，控制在5个以内

读写分离：主库负责申报写入，两个只读从库处理查询。使用MySQL半同步复制保证从库延迟<1秒。应用层通过中间件（如ShardingSphere）实现读写路由

缓存策略：采用Redis缓存热点数据，对纳税人基础信息（变更频率低）全量缓存TTL=1小时，对申报汇总数据（T+1更新）缓存到每日凌晨失效。Cache Aside模式：查询先读缓存→未命中读数据库→写入缓存；写入先更新数据库→删除缓存

效果预估：缓存命中率预计85%，查询响应时间降至10ms（缓存命中）+40ms（缓存未命中场景相应减小），从库分担读压力后主库写入吞吐量提升2倍

问题3（同步方案）：

方案设计：
1. 增量同步：工商局推送变更事件到消息队列（如Kafka），税务系统消费事件。每条消息携带序列号seq_id和变更时间戳
2. 断点续传：消费端定期记录消费位点（offset/seq_id）到本地持久化存储。重启时从上次位点继续消费，避免漏数据。同时保证消费者幂等——相同seq_id被重复消费不会产生重复更新
3. 冲突处理：采用"最后写入胜出"（LWW）策略，以变更时间戳为准。工商数据变更时间晚于本地则覆盖，否则忽略。重要变更写入变更审计日志表，支持人工核查
4. 全量对账：每日凌晨进行全量对账，对比工商数据和本地数据的差异，生成差异报告并自动修复（以工商为准）

问题4（计算）：

查询响应公式：单次查询时间 = 排队等待时间 + 处理时间。应用服务器处理时间（不含DB）= 50-40 = 10ms

500并发下，若500个请求同时到达，数据库需排队。数据库每秒处理能力 = 1000ms/40ms = 25 QPS（线程池大小假设1）。若数据库连接池=50，DB QPS = 50×25 = 1250 QPS > 500 QPS，数据库不是瓶颈

应用服务器：若单线程处理，500个请求需排队，最后请求等待时间 = 499×10ms = 4.99秒 >> 200ms。需多线程，最小线程数 = 500/(200ms/10ms向上取整) = 500/20 = 25线程。安全起见设线程池50，应对峰值

结论：数据库当前可支撑500并发查询无需引入缓存；但实际需考虑查询复杂度差异，建议缓存热点数据和查询结果，长期仍需引入缓存降低DB压力。

**踩分要点**：

| 分值 | 踩分点 |
|------|--------|
| 8分 | 关系模式含主键和函数依赖分析(3分)、候选键正确标注(1分)、至少达到3NF(1分)、法人独立拆分正确(2分)、税率表独立设计(1分) |
| 7分 | 索引策略包含联合索引和单列索引设计(2分)、读写分离含主从复制方案(2分)、缓存策略含Cache Aside模式和TTL设计(2分)、效果预估量化指标(1分) |

## 记忆口诀

**信息系统规划方法**：诺BS致价战
诺兰六段初蔓控集数成 → 诺（初→蔓→控→集→数→成）
BSP自上而下规划自下而上实现 → BSP
CSF关键成功因素倒推 → 关（CSF）
价值链基本五活动+支持四活动 → 价（进场→生→发→销→服：支基人技采）
战略一致性四象限对齐 → 战
信息化五层 → 网数应服策

**数据库范式递进**：一二三B四五，主部传多连
1NF：原子性、不可再分 → 一原
2NF：消除非主属性对候选键的部分函数依赖 → 二部
3NF：消除非主属性对候选键的传递函数依赖 → 三传
BCNF：消除主属性对非主属性的部分依赖 → B主
4NF：消除多值依赖 → 四多
5NF：消除连接依赖 → 五连

**流水线三大冒险**：结数控，转旁预
结构冒险（硬件冲突）→ 加硬件、流水化
数据冒险（RAW/WAR/WAW）→ 转发旁路 + 暂停
控制冒险（分支）→ 分支预测 + 延迟槽

**死锁四条件与对策**：互持不循，破预检脱
互斥条件 → SPOOLing破坏
保持并等待 → 一次性分配破坏
不可抢占 → 可抢占破坏
循环等待 → 按序分配破坏
预防→避免（银行家）→检测恢复→鸵鸟

**PV操作经典三问题**：生消读写哲，PV必成对
生产者消费者：empty+full+mutex，P(mutex)在P(empty)/P(full)之后
