## 今日核心概念

### 1. 软件架构风格
定义：描述某一特定应用领域中系统组织方式的惯用模式，反映众多系统共有的结构和语义特性，是架构层面的设计复用。
关键要点：数据流风格（管道-过滤器、批处理）、调用/返回风格（主程序/子程序、面向对象、分层）、独立构件风格（进程通信、事件系统）、虚拟机风格（解释器、规则系统）、数据仓库风格（黑板系统、数据库系统）。选择架构风格时须结合质量属性需求：如管道-过滤器适合高可修改性、分层适合高可移植性、黑板风格适合无确定性求解策略的问题。

### 2. ATAM 架构权衡分析法
定义：Architecture Tradeoff Analysis Method，是一种基于质量属性场景的架构评估方法，用于识别架构决策中的风险点（Risk）、非风险点（Non-Risk）、敏感点（Sensitivity Point）和权衡点（Tradeoff Point）。
关键要点：评估过程分四阶段——①场景和需求收集（构建质量属性效用树）；②架构视图和实现（分析架构方法）；③属性模型建立和分析（量化各质量属性响应值）；④折中识别与输出。效用树将质量属性场景按（重要性，实现难度）两维排序（H,H）、（H,M）、（M,H）优先分析。权衡点是多个质量属性相互制约的参数，如加密强度提高安全性但降低性能。

### 3. 质量属性场景
定义：用六部分结构化描述的质量属性需求模板，包括刺激源、刺激、环境、制品、响应和响应度量。
关键要点：可用性场景——刺激源（系统内部/外部）、刺激（故障/错误）、环境（正常/降级运行）、制品（系统/组件）、响应（检测并修复）、响应度量（恢复时间）；性能场景——刺激（事件到达）、响应度量（响应时间、吞吐量）；安全性场景——刺激（攻击/非法访问）、响应度量（检测延迟、恢复时间）；可修改性场景——刺激（变更请求）、响应度量（修改人天数、波及模块数）。

### 4. 架构描述语言（ADL）
定义：Architecture Description Language，以组件、连接件和配置为核心建模元素，形式化描述软件架构的语言。
关键要点：核心三元素——组件（Component，计算与状态单元）、连接件（Connector，组件间交互规则）、配置（Configuration，组件和连接件的拓扑结构）。RUP 4+1视图模型从五个视角描述架构：逻辑视图（功能需求，面向终端用户）、过程视图（并发与同步，面向系统集成者）、开发视图（模块组织，面向程序员）、物理视图（部署拓扑，面向系统工程师）、场景视图（用例驱动，串联验证所有视图）。ADL与UML的区别：ADL精于架构层面语义描述，UML覆盖面更广但架构语义不如ADL精确。

### 5. 软件产品线架构
定义：基于一组共享的软件资产（核心资产库），通过预定义的架构框架和可变性管理机制，系统化地构建一系列相似软件产品的架构方法。
关键要点：核心资产库包含架构、组件、文档、测试用例等可复用制品。可变性管理是产品线的核心——通过变化点（Variation Point）定义系统哪些部分可变，变体（Variant）定义可选的具体实现。领域工程阶段负责建立核心资产库（共性+可变性建模），应用工程阶段基于核心资产衍生产品。相比传统复用，产品线架构是有计划、有管理的系统性复用而非偶然复用。

## 案例分析题

**题目**：
某电商平台日均订单量200万单，要求系统保持高可用性（99.99%）和高可伸缩性（可水平扩展）。现有架构为单体应用部署于多台服务器，采用数据库主从复制。业务增长后面临瓶颈：秒杀活动期间数据库连接池耗尽导致大量超时；新功能上线需整体停机部署；促销推荐算法调整牵动整个系统。

（1）指出当前单块架构存在的三个主要架构问题，并分析其根本原因。（4分）
（2）从架构风格和质量属性的角度，给出重构方案，说明选择依据。（3分）
（3）针对推荐算法模块频繁变更的问题，你会采用哪种具体架构模式来隔离变更影响？请说明该模式的结构。（3分）

**参考答案**：
（1）①数据库瓶颈——单块架构所有模块共享同一数据库连接池，秒杀高并发写操作耗尽连接导致整个系统不可用（可用性受损）；②部署耦合——任何模块修改都需整体重新部署，无法独立上线新功能（可修改性差）；③紧耦合——推荐算法内嵌于单体中，算法调整影响订单、支付等核心模块的编译和集成测试（模块间耦合度过高）。

（2）建议采用微服务架构风格+事件驱动风格结合。拆分为订单服务、商品服务、用户服务、支付服务、推荐服务等独立微服务，各服务独立部署独立数据库。选择依据：微服务天然支持独立部署和独立数据库（解决部署耦合和数据库瓶颈），配合消息队列（如Kafka）实现异步解耦（解决紧耦合），各服务可独立水平扩展以应对秒杀流量。事件驱动用于服务间最终一致性场景（如订单创建后通知推荐更新偏好）。

（3）采用微内核架构模式（插件架构模式）。结构包括：核心系统（Core System）——提供最小化的推荐框架，定义插件接口规范和调用流程；插件模块（Plugin Modules）——每个推荐算法封装为独立插件，实现统一接口（如获取候选集→特征构建→排序→过滤）；注册表（Plugin Registry）——管理插件的加载、卸载和版本信息。新算法或策略调整只需开发新插件并注册，无需修改核心系统和其他插件。

**踩分要点**：
- 三个架构问题各1分，须指出具体问题+根本原因（不是泛泛说"性能差"）（4分）
- 给出重构方案须同时说明质量属性依据，仅提微服务不给质量属性分析扣分（3分）
- 微内核模式须完整描述三要素（核心、插件、注册表），缺一扣分（3分）

## 记忆口诀

**架构风格六分类速记**：数调用独虚仓——数（数据流）、调（调用/返回）、用（独立构件）、独（虚拟机）、虚（仓库=数据仓库）。全句：数据流调用返回分开写，独立构件和虚拟机各放一边，数据仓库黑板一起来。**ATAM四步骤**：收（场景需求收集）、架（架构视图分析）、模（属性模型建立）、折（折中识别）——"收架模折"谐音"收假模折"。
