## 今日核心概念

### 1. Lambda架构

定义：由Nathan Marz提出的通用大数据处理架构，以「批处理层+速度层+服务层」三层协作，同时满足大数据系统的低延迟和高精度需求。

关键要点：
- 批处理层（Batch Layer）：用MapReduce/Spark对全量历史数据做高延迟、高精度计算，产出Batch View，T+1或小时级更新
- 速度层（Speed Layer）：用Storm/Flink/Spark Streaming对增量实时数据做低延迟、近似精度计算，产出Realtime View，秒级/毫秒级更新
- 服务层（Serving Layer）：合并Batch View和Realtime View，对外提供统一查询接口，客户端读时合并两者结果
- 核心公式：Query = λ(Complete Data) = Batch View + Realtime View
- 本质取舍：用两套计算逻辑冗余换取延迟与准确性的兼得，代价是两套代码两套维护
- 不可变数据模型：原始数据只追加不修改，错误通过重新计算修正而非删除旧数据

### 2. Kappa架构

定义：由LinkedIn的Jay Kreps提出的「纯流处理」架构，取消批处理层，将所有数据视为事件流，用单一流处理引擎同时满足实时与历史计算需求。

关键要点：
- 核心思想：取消批处理层，速度层升级为唯一的计算层，流处理引擎足够快就不需要批处理
- 历史数据重算：需要修改计算逻辑时，启动新的流处理Job，从Kafka等消息队列中从头消费全部历史数据重新计算，完成后切换输出表
- 技术前提：消息队列必须长期保留全量数据（如Kafka设置retention=-1永久保留或按存储容量保留）
- Lambda vs Kappa选型：批流代码可统一→选Kappa；批流逻辑天然不同且难以统一→选Lambda；已有成熟批处理资产→选Lambda
- 优势：一套代码、一套引擎、运维简单、无批流结果合并延迟
- 局限：历史数据重算时间长（全量回放）、要求消息队列存储成本高、不适合需要真正批量大窗口分析的场景

### 3. 数据湖与数据仓库

定义：数据湖存储原始格式的全量数据（结构化+半结构化+非结构化），数据仓库存储经过ETL清洗建模的结构化数据，服务于不同的分析场景。

关键要点：
- 数据湖三特征：存储原始数据（Schema-on-Read读时定义结构）、存全量（不丢弃任何数据）、支持多种分析引擎（SQL/机器学习/全文检索）
- 数据仓库核心：Schema-on-Write（写时定义结构）、面向主题、集成、时变、非易失（Inmon四特征）
- 数据湖选型：HDFS+Spark/Hive（开源生态）、Delta Lake（ACID事务+时间旅行）、Apache Hudi（增量更新+记录级索引）、Apache Iceberg（表格式标准+隐藏分区）
- 典型反模式：数据沼泽（Data Swamp）——只入不出、无目录、无元数据管理、无生命周期治理，数据湖变垃圾场
- 数据仓库分层：ODS操作数据层→DWD明细数据层→DWS汇总数据层→ADS应用数据层（阿里数仓分层标准）

### 4. 流处理技术选型

定义：实时流处理框架的核心技术选型对比，涵盖Spark Streaming（微批）、Flink（真正流处理）、Kafka Streams（轻量库）三大主流方案。

关键要点：
- Spark Streaming：微批架构，将流数据切成小批次（默认1秒Batch Interval）用Spark引擎处理，Exactly-Once靠WAL+幂等输出，吞吐高但延迟秒级
- Flink：真正逐条流处理，事件时间+Watermark+Checkpoint三件套，Exactly-Once靠分布式快照（Chandy-Lamport算法改版），毫秒级延迟，支持CEP复杂事件处理
- Kafka Streams：嵌入应用的轻量流处理库（非平台），无独立集群，利用Kafka分区实现并行和容错，Exactly-Once靠Kafka事务，适合轻量ETL和实时聚合
- 选型决策：毫秒延迟+复杂事件→Flink；秒级延迟+已有Spark技术栈→Spark Streaming；Kafka生态+轻量转换聚合→Kafka Streams
- 一致性模型：At-Most-Once（最多一次，可能丢）< At-Least-Once（最少一次，可能重）< Exactly-Once（精确一次，端到端）
- 流表二元性：流是动态的变更序列，表是流在某一时刻的快照，流和表可以互相转化

### 5. 湖仓一体（Lakehouse）

定义：融合数据湖灵活性和数据仓库管理能力的新一代数据架构，在数据湖上增加ACID事务、Schema管理、数据治理，实现湖仓统一。

关键要点：
- 核心理念：一套存储（对象存储/HDFS）+ 一套元数据（Iceberg/Hudi/Delta Lake表格式）+ 多引擎访问（BI/ML/SQL多样化工作负载）
- 三大技术支柱：开放表格式（Delta Lake/Iceberg/Hudi）+ 高性能计算引擎（Spark/Trino/Presto/Flink）+ 统一元数据治理（数据目录/血缘/权限）
- 对比传统：数据湖只有存储无管理（数据沼泽风险），数据仓库管理好但贵且不支持非结构化数据，Lakehouse取两者之长
- 开放格式避免锁定：Iceberg/Dltake/Hudi均为Apache开源，计算引擎可插拔，不绑定特定厂商
- 时间旅行（Time Travel）：读取任意历史版本的数据快照，支持审计、回滚、"昨天数据长什么样"查询
- 现实落地路径：通常从数据湖升级，引入Delta Lake或Iceberg表格式，逐步增加数据治理和查询优化

## 案例分析题

**题目**：

某大型电商平台日均产生交易日志500GB、用户行为埋点2TB、推荐系统反馈1TB。当前架构问题：
（1）批处理T+1产出报表，运营团队抱怨对"大促实时转化"无法及时反应，要求分钟级看板
（2）历史数据离线仓库仅保留最近90天，管理层想分析去年商品季节性规律却发现数据已被清理
（3）数据团队在Hive上做用户画像，数据科学家在数据湖上跑ML训练，两个口径产出的"活跃用户"定义不同，导致运营决策混乱
（4）某次Spark离线任务因上游文件格式变更失败，修复后需要重跑3天数据，恢复慢

请回答：
1. 请设计Lambda架构来同时满足批处理和实时看板需求，画出层级并说明各层技术选型（6分）
2. 针对问题（2）和（4），如何用Kappa架构的「重算机制」解决历史数据问题和故障恢复？说明该方案的前提条件和局限性（6分）
3. 针对问题（3），提出湖仓一体方案统一数据口径，说明如何用元数据治理解决"同名不同义"问题（5分）
4. 实时看板场景对流处理引擎在一致性和延迟方面有何要求？分析Spark Streaming和Flink哪个更合适（3分）

**参考答案**：

1. Lambda架构设计（6分）：
- 批处理层：HDFS存储全量原始日志，Spark做T+1批计算产出Batch View（用户标签、多维聚合Cube），输出到HBase/ClickHouse
- 速度层：Kafka接入实时日志流，Flink做分钟级增量计算产出Realtime View（大促实时UV/GMV/转化率），输出到Redis
- 服务层：查询引擎接收看板请求，合并Batch View（T+1精确历史）和Realtime View（当日实时增量），前端用Grafana/Datart展示；查询时对同一指标取Batch View截止昨日值+Realtime View今日增量值合并返回
- 关键点：两套代码需保持语义一致（同一UV定义），合并逻辑在服务层统一处理

2. Kappa重算机制（6分）：
- 解决问题（2）：Kafka保留全量原始数据（设置retention基于存储容量如30TB而非时间），当管理层要分析去年数据时，启动新的Flink Job从头消费去年Topic数据重新计算，产出历史分析结果
- 解决问题（4）：上游格式变更导致错误——修复后启动新Flink Job从头回放全量数据即可恢复，无需等3天
- 前提条件：消息队列必须永久保留全量数据（Kafka存储成本高）；流处理引擎吞吐足够高（Flink千亿级/天可应对500GB×365≈178TB级别需评估集群规模）
- 局限性：全量回放时间可能很长（海量数据回放需数小时甚至数天）；批处理场景（如PageRank迭代计算、全局排序）不适合纯流处理；历史数据重算期间不能影响在线业务需资源隔离

3. 湖仓一体统一口径（5分）：
- 整体方案：在HDFS/S3数据湖之上引入Apache Iceberg表格式，建「统一数据层」，所有分析引擎（Hive/Spark/Presto/Flink）通过Iceberg读写同一套表
- 元数据治理解决"同名不同义"：
  - 建立统一指标字典（Metrics Dictionary）：明确「活跃用户」标准定义为「近30天有下单或加购≥3次的注册用户」
  - 在元数据层注册该指标，关联其计算SQL逻辑和依赖表
  - 所有引擎读取同一元数据目录，从目录中获取活跃用户口径→计算一致
  - 引入数据血缘（Lineage）追踪：任一指标可回溯到原始表和计算逻辑，变更时影响分析
  - 加上数据质量监控：定期校验Hive和ML两边产出是否一致，不一致时预警
- 技术选型：Iceberg或Hudi作表格式 + Amundsen/DataHub作数据目录和血缘 + Great Expectations作数据质量

4. 实时看板流处理选型（3分）：
- 一致性要求：Exactly-Once（不能重复计算GMV导致虚高，不能丢失订单导致漏报）
- 延迟要求：分钟级（大促看板可接受1-2分钟延迟，不需要秒级）
- 选型结论：选Flink。理由——Flink原生Exactly-Once（Checkpoint+Barrier对齐机制），Spark Streaming的Exactly-Once依赖WAL且微批模式下延迟偏大；Flink事件时间处理天然适合"大促转化漏斗"类分析（用户浏览→加购→下单存在时间跨度，需要Watermark处理乱序）

**踩分要点**：

- Lambda三层结构及每层技术选型（2分）+ 合并逻辑说明（2分）+ Kappa重算前提条件至少列出存储和吞吐两点（2分）
- Kappa局限必须写到「海量回放耗时」「批迭代场景不适用」「资源隔离」至少两点（2分）+ 湖仓一体方案必须有表格式+指标字典+血缘三个要素（3分）+ 实时选型必须对比Flink和Spark Streaming并给出理由（2分）
- 漏写"合并逻辑在服务层统一处理"扣1分 / 漏写Exactly-Once要求扣1分 / 漏写元数据血缘扣1分
- 数据沼泽概念作为反面案例写出来可以额外加分

## 记忆口诀

Lambda三层批速服，速度快批精度补，双份代码双维护；Kappa去批一肩挑，全量回放重计算，队列存永久是前提；数仓写前定结构主题集时非易变，数据湖读时才定义原始全量多引擎；湖仓一体表格式元数据血缘指标字典三剑客；Flink毫秒精确一次事件时间水印齐，Spark微批秒延迟吞吐高，Kafka Streams轻量嵌入无集群。选型三问：延迟要多少（毫秒Flink/秒Spark）、一致性要什么（精确一次Flink/Kafka Streams）、有没有Spark团队（有就Spark Streaming别折腾）。
