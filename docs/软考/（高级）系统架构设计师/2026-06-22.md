## 今日核心概念

### 1. 企业应用集成（EAI）与集成模式

定义：企业应用集成（Enterprise Application Integration）是将企业内部不同时期、不同平台、不同技术构建的异构应用系统连接起来，实现数据共享和业务流程协同的技术体系。EAI解决的是"信息孤岛"问题，是架构师进行系统整合时必须掌握的核心能力。

关键要点：

**集成层次五级模型**：
- 数据层集成：最底层，通过ETL/CDC技术在数据库层面同步数据。优点：对应用无侵入；缺点：数据延迟、无法处理业务逻辑、跨系统数据一致性难保证
- 应用接口层集成：通过API/SDK/WebService调用其他系统的功能。优点：复用业务逻辑、灵活性高；缺点：紧耦合、同步调用导致故障传播
- 方法层集成：共享业务逻辑，对象间通过RPC/RMI互操作。优点：代码级复用；缺点：平台绑定（Java RMI只能在JVM环境）
- 用户界面层集成：在展示层整合多个系统的界面，如门户集成、爬屏。优点：对系统改造最小；缺点：脆弱（UI变化即失效）、无法处理复杂交互
- 业务流程层集成：通过工作流引擎编排多个系统的服务调用，实现端到端业务流程。优点：业务逻辑可视化、灵活性最高；缺点：需要业务建模投入

**企业集成模式（EIP，Hohpe/Woolf）核心模式**：
- 消息通道（Message Channel）：发送者写入通道，接收者从通道读取，实现时间解耦和空间解耦
- 消息路由器（Message Router）：根据消息内容或规则分发到不同通道。六种路由：内容路由（Content-Based Router）、过滤器（Message Filter）、拆分器（Splitter）、聚合器（Aggregator）、重排器（Resequencer）、消息代理（Message Broker）
- 消息转换器（Message Translator）：在异构系统间转换消息格式（如XML→JSON→CSV），解耦发送者和接收者的数据格式
- 消息端点（Message Endpoint）：连接应用与消息系统的适配器，封装消息收发细节
- 死信通道（Dead Letter Channel）：存放无法处理的消息，供人工排查

**集成拓扑架构**：
- 点对点集成（星型）：每个系统两两直接对接，N个系统需N×(N-1)个连接。适用于系统数≤5的场景，但维护成本随系统数平方增长
- 中心辐射集成（Hub-and-Spoke）：所有系统连接到中心集成平台（Hub），由Hub负责协议转换、消息路由、格式转换。N个系统只需N个连接，是EAI的经典架构
- 企业服务总线（ESB）：在Hub-Spoke基础上标准化接入协议和消息格式，支持热插拔。比Hub-Spoke更灵活，但比微服务API网关更重。典型产品：IBM Integration Bus、Mule ESB
- 微服务集成：去中心化集成，每个服务自主选择通信方式（REST/gRPC/消息），避免中心集成平台的单点瓶颈。对比ESB：ESB是"智能管道+哑端点"，微服务是"哑管道+智能端点"

**ESB核心能力对比**：

| 能力 | ESB（传统集成） | API网关（微服务） |
|------|---------------|-----------------|
| 协议转换 | 强（SOAP↔REST↔JMS↔FTP多协议） | 弱（主要是HTTP/HTTPS） |
| 消息路由 | 丰富（内容路由/动态路由/规则路由） | 简单（路径/Header路由） |
| 数据转换 | 强（XSLT/XML↔JSON复杂转换） | 弱（基本Header改写） |
| 服务编排 | 支持（BPEL/工作流） | 不支持（由BFF或Service层处理） |
| 定位 | 企业内部异构系统整合 | 微服务外部流量治理 |

### 2. 面向服务架构（SOA）与企业架构框架

定义：SOA（Service-Oriented Architecture）是一种将业务功能封装为可复用服务的架构风格，强调服务之间的松耦合、标准化接口和平台无关性。SOA的服务是业务级别的抽象而非技术级别的，一个服务通常对应一个完整的业务功能。

关键要点：

**SOA核心原则**：
- 标准化服务契约：每个服务通过WSDL（Web Service Description Language）或OpenAPI规范定义接口，契约先行
- 松耦合：服务消费者无需知道服务提供者的实现细节（平台、语言、位置），只依赖契约
- 服务抽象：隐藏内部逻辑，仅暴露契约定义的操作
- 服务可复用：设计时以跨场景复用为目标，避免为单一消费方定制
- 服务自治：服务独立部署、独立版本、独立数据库，不与其他服务共享资源
- 服务无状态：服务不保存客户端会话状态，每次调用独立完成（状态外化到共享存储）
- 服务可发现：服务注册到UDDI/注册中心，消费者可动态发现并绑定
- 可组合：服务可编排组合为更复杂的复合服务

**SOA技术栈四层**：
- 传输层：HTTP/HTTPS、JMS、MQTT、TCP
- 消息协议层：SOAP（基于XML，WS-*规范族）、REST（基于HTTP方法+资源URI+JSON）
- 描述层：WSDL（SOAP服务契约）、OpenAPI/Swagger（REST服务契约）
- 发现层：UDDI（Universal Description Discovery and Integration，老）、注册中心（新）

**SOA vs 微服务（高频考点）**：

| 维度 | SOA | 微服务 |
|------|-----|--------|
| 服务粒度 | 粗粒度（业务功能级，如订单管理服务） | 细粒度（单一职责，如订单创建/订单查询分离） |
| 通信协议 | 侧重SOAP/WS-*（重量级XML协议族） | 侧重REST/gRPC（轻量级） |
| 数据管理 | 共享数据库或企业数据总线 | 每服务独立数据库 |
| ESB | 中心化ESB做集成 | 去中心化，无ESB，用轻量级消息或API网关 |
| 治理 | 全局规范+中心化治理 | 团队自治+轻量级治理 |
| 适用场景 | 大型企业异构系统整合 | 互联网高并发快速迭代 |

**企业架构框架TOGAF**：
- 定义：TOGAF（The Open Group Architecture Framework）是全球最广泛采用的企业架构框架，提供了一套系统化的方法、工具和参考模型来设计、规划、实施和治理企业IT架构
- 架构开发方法ADM（Architecture Development Method）——TOGAF核心：预备阶段→A.架构愿景→B.业务架构→C.信息系统架构（数据+应用）→D.技术架构→E.机会与解决方案→F.迁移规划→G.实施治理→H.架构变更管理→需求管理（位于中央，贯穿全过程）
- 四大架构域（BDAT）：业务架构（组织/流程/功能/业务服务）、数据架构（数据资产/数据管理/数据治理）、应用架构（应用组合/应用交互/应用与业务对齐）、技术架构（技术平台/基础设施/中间件/标准）
- 企业连续统一体（Enterprise Continuum）：基础架构→通用系统架构→行业架构→组织特定架构，从通用到特化
- 其他企业架构框架对比：
  - DoDAF（美国国防部架构框架）：强调能力视角、作战视角、系统视角、技术标准视角。军事色彩重，体系结构产品8类52个
  - FEAF（美国联邦企业架构框架）：按业务/数据/应用/基础架构/安全/绩效六条参考模型组织
  - Zachman框架：6行（范围/业务模型/系统模型/技术模型/组件表示/运行系统）×6列（数据/功能/网络/人员/时间/动机），36个格子的企业本体论

**TOGAF ADM各阶段关键产出**：

| 阶段 | 核心活动 | 关键交付物 |
|------|---------|-----------|
| 预备阶段 | 确定组织架构能力、建立架构框架 | 架构原则、治理框架 |
| A.架构愿景 | 定义范围、利益相关者、业务目标 | 架构愿景文档、利益相关者映射 |
| B.业务架构 | 建模当前和未来业务能力 | 业务能力模型、流程模型 |
| C.数据架构 | 企业数据模型设计 | 数据模型、数据流图 |
| C.应用架构 | 应用组合梳理和规划 | 应用目录、应用交互矩阵 |
| D.技术架构 | 技术平台选型和标准 | 技术标准目录、平台模型 |
| E/F/G | 差距分析→路线图→实施 | 迁移路线图、实施计划 |
| H.变更管理 | 监控架构变更请求 | 变更请求评估、架构更新 |

### 3. DevOps与持续交付体系

定义：DevOps是Development和Operations的组合词，指通过自动化工具链将软件开发和运维打通，实现持续集成、持续交付、持续部署、持续监控的完整闭环。DevOps不是某个具体工具，而是一套文化理念+实践方法+工具链的组合。

关键要点：

**DevOps文化三原则（CALMS模型）**：
- Culture（文化）：打破开发与运维的部门墙，共担责任建立互信
- Automation（自动化）：将重复性操作自动化，降低人为差错和等待时间
- Lean（精益）：消除浪费，小批量交付，加速价值流动
- Measurement（度量）：建立可量化的交付效能指标
- Sharing（分享）：知识透明化，故障复盘共享

**DevOps三大核心实践**：

（1）持续集成CI——频繁将代码合并到主干并自动构建测试：
- 核心原则：所有人至少每天向主干合并一次代码
- CI流水线：代码提交→自动检出→编译→单元测试→静态代码扫描（SonarQube）→构建产物（jar/docker镜像）→归档制品库
- 触发方式：代码合并触发（Webhook）、定时触发、手动触发
- 质量门禁：测试通过率<100%或代码覆盖率<80%则阻断流水线

（2）持续交付CD——确保代码始终处于可部署状态：
- 当CI完成后，自动将构建产物部署到测试环境并运行集成测试
- CI vs CD：CI保证代码可编译可集成→持续交付保证代码随时可以部署（但需手动触发部署到生产）
- 持续部署（Continuous Deployment）：比持续交付更进一步——测试通过后自动部署到生产，无需人工批准
- 关键实践：环境一致性（开发/测试/预发布/生产环境配置差异仅存在于配置管理）、不可变基础设施（部署=新建环境而非修改现有环境）

（3）基础设施即代码（IaC）：
- 定义：用代码描述和管理基础设施（服务器/网络/存储/负载均衡），而非通过手动操作控制台
- 声明式（Terraform/CloudFormation/Pulumi）vs 命令式（Ansible/SaltStack）：声明式描述期望状态由工具调谐，命令式描述操作步骤
- 配置管理（Ansible/Chef/Puppet）：在已创建的服务器上安装软件、配置系统参数
- 编排（Terraform/CloudFormation）：创建和管理基础设施资源本身（VM/网络/存储）
- 不可变基础设施+蓝绿部署/金丝雀发布：蓝绿部署=同时维护蓝绿两套环境，通过切换流量发布；金丝雀发布=逐步将一小部分流量路由到新版本，验证后逐步扩大

**DevOps工具链全景**：

| 阶段 | 工具 | 核心能力 |
|------|------|---------|
| 代码管理 | Git/GitLab/GitHub | 版本控制、分支策略 |
| 构建 | Maven/Gradle/npm | 依赖管理、编译打包 |
| CI/CD | Jenkins/GitLab CI/GitHub Actions | 流水线编排、自动构建 |
| 制品管理 | Nexus/Artifactory/Harbor | 包仓库、镜像仓库 |
| 代码质量 | SonarQube/Checkstyle | 静态扫描、代码规范、安全漏洞 |
| 测试自动化 | JUnit/Selenium/JMeter | 单元/集成/性能测试 |
| 配置管理 | Ansible/Chef/SaltStack | 自动化配置 |
| 编排 | Kubernetes/Terraform | 容器编排/基础设施编排 |
| 监控 | Prometheus+Grafana/ELK | 指标/日志/链路追踪 |
| 告警 | AlertManager/PagerDuty | 告警路由、值班管理 |

**DevOps与架构师的关联**：
- 架构决策需考虑可部署性：服务拆分粒度影响部署独立性、数据库变更策略影响部署安全性
- 架构决策需考虑可观测性：日志格式标准化、链路追踪集成、指标暴露方式
- 12-Factor App方法论：I.基准代码一份多部署、II.显式声明依赖、III.配置存于环境变量、IV.后端服务作为附加资源、V.构建发布运行严格分离、VI.无状态进程、VII.端口绑定导出服务、VIII.并发通过进程扩展、IX.快速启动优雅终止、X.开发预发生产环境一致、XI.日志作为事件流、XII.管理任务作为一次性进程

**DORA四大黄金指标（度量DevOps效能）**：
- 部署频率：代码部署到生产的频率。精英团队：每天多次；低效团队：每月一次
- 变更前置时间：从代码提交到成功运行在生产环境的耗时。精英团队：<1小时；低效团队：1-6个月
- 变更失败率：导致服务降级或需要回滚/修复的变更比例。精英团队：0%-15%
- 故障恢复时间：从生产故障中恢复所需的时间。精英团队：<1小时

### 4. 系统测试与质量保证

定义：系统测试是按测试层次和测试类型对软件系统进行全方位验证和确认的过程。架构师需要在架构设计阶段就考虑可测试性，包括测试策略制定、测试环境设计、自动化测试架构和性能测试方案的规划。

关键要点：

**测试层次金字塔**：
- UI测试（最顶层，少量）：端到端验证用户可见功能，数量少、运行慢、维护成本高。Selenium/Cypress/Playwright
- 服务/集成测试（中间层，适量）：验证服务间交互、API契约、数据库访问
- 单元测试（最底层，大量）：验证类/方法级别的独立逻辑。JUnit/Pytest/Jest
- 金字塔原理：越底层的测试维护成本越低、执行速度越快、越容易定位问题，因此单元测试占最大比例

**测试类型全面覆盖**：

| 测试类型 | 关注点 | 阶段 | 典型方法 |
|---------|-------|------|---------|
| 功能测试 | 功能是否符合需求规格 | 单元/集成/系统 | 黑盒测试、等价类、边界值、因果图 |
| 性能测试 | 响应时间、吞吐量、资源利用率 | 系统测试 | 负载测试、压力测试、稳定性测试 |
| 安全测试 | 机密性、完整性、可用性 | 系统测试 | 渗透测试、漏洞扫描、SAST/DAST |
| 兼容性测试 | 不同环境下的运行表现 | 系统测试 | 浏览器/OS/设备兼容矩阵 |
| 可靠性测试 | 系统持续正常运行能力 | 系统测试 | MTBF/MTTR衡量、故障注入 |
| 恢复测试 | 在故障后恢复的能力 | 系统测试 | 强制故障→验证恢复过程 |

**性能测试深入**：
- 负载测试：验证系统在预期负载下的性能。确定系统正常使用的并发用户数、QPS阈值
- 压力测试：确定系统的极限和崩溃点。逐步增加负载直到资源耗尽或响应超时，找到瓶颈
- 容量测试：确定系统在不同资源配置下可支撑的最大负载。用于容量规划（需要多少台服务器）
- 稳定性/浸泡测试：长时间（8h-72h）运行看系统是否出现内存泄漏、线程死锁、资源衰减
- 尖峰/突发测试：模拟瞬时突发流量（如秒杀），观察系统弹性
- 关键指标：响应时间（avg/P95/P99）、吞吐量（TPS/QPS）、并发用户数、资源利用率、错误率

**性能测试工具对比**：
- JMeter：Java生态，GUI脚本录制+CLI执行，插件丰富，适合HTTP/数据库/MQ协议
- Gatling：Scala DSL编写脚本，异步非阻塞IO，资源消耗更低，报表更精美（Gatling Reports）
- Locust：Python编写脚本，协程并发，适合自定义复杂场景
- wrk/wrk2：C语言，极致性能，适合HTTP基准测试

**自动化测试策略**：
- 测试金字塔 = 原则（单元70% + 集成20% + UI10%），不是教条。根据系统特征调整比例
- 测试数据管理：测试数据需独立于生产数据，使用DataBuilder模式和Fixture工厂
- Mock vs Stub vs Fake：Mock=验证行为（验证是否调用及参数），Stub=提供固定返回值，Fake=轻量级实现（内存数据库替代MySQL）
- 契约测试（Consumer-Driven Contract）：服务消费者定义期望的接口契约，服务提供者验证是否满足所有消费者的契约。Pact是主流方案，解决微服务集成测试爆炸问题

### 5. 大数据架构与数据治理

定义：大数据架构是面向海量、多样、高速增长的数据，在采集、存储、计算、分析、应用全链路中采用分布式技术体系构建的系统架构。数据治理则是制定数据标准、保障数据质量、确保数据合规的管理体系，是大数据价值的基石。

关键要点：

**Lambda架构（批流分离）**：
- 三层模型：批处理层（Batch Layer，Hadoop/Spark大规模离线计算）→ 实时处理层（Speed Layer，Storm/Flink流式计算）→ 服务层（Serving Layer，合并批+实时结果对外提供查询）
- 优点：架构成熟、容错性好（批处理层是真理源，出错可重跑全量数据修正）
- 缺点：两套代码逻辑需保持一致（批和流的计算逻辑双写），维护成本高；数据合并有复杂性

**Kappa架构（全流式）**：
- 单一技术栈：全部数据视为流数据，统一用流处理引擎处理，去除批处理层
- 关键设计：Kafka存储全量不可变事件日志（日志即数据源），Flink消费Kafka进行流式计算
- 重处理机制：需要重新计算历史数据时，重放Kafka历史消息（配合分层保留策略，如Topic保留7天→冷数据归档至HDFS/S3）
- Lambda vs Kappa选择：业务需要复杂离线批处理（机器学习训练/全量报表）→Lambda；业务以实时处理为主+历史重算需求可控→Kappa

**流处理关键概念**：
- 事件时间（Event Time）vs 处理时间（Processing Time）：事件时间是数据产生的时间戳，处理时间是数据处理的时间。流处理应基于事件时间以确保正确性
- 水印（Watermark）：表示事件时间进度，Watermark(t)意味着所有事件时间<t的数据已到达，窗口在Watermark超过窗口结束时间时触发计算
- 三种窗口：滚动窗口（固定长度不重叠）、滑动窗口（固定长度+固定间隔有重叠）、会话窗口（按活动间隔动态划分，如用户操作30分钟无活动断开）
- 精确一次语义（Exactly-Once）：Flink通过分布式快照（Checkpoint）+两阶段提交实现端到端精确一次。Kafka 0.11+支持幂等生产者和事务生产保证精确一次

**数据仓库与数据湖**：

| 维度 | 数据仓库 | 数据湖 |
|------|---------|--------|
| 数据 | 结构化，清洗加工后 | 原始数据，结构化+半结构化+非结构化 |
| 模式 | Schema-on-Write（写入前定义结构） | Schema-on-Read（读取时定义结构） |
| 存储 | 高成本（高端存储，数据压缩） | 低成本（HDFS/S3对象存储） |
| 用户 | 业务分析师、BI报表 | 数据科学家、数据工程师 |
| 代表 | Teradata/Oracle/ClickHouse | Hadoop HDFS/MinIO |
| 敏捷性 | 低（需求→建模→ETL→报表） | 高（原始数据直接可用，随时定义新分析） |

湖仓一体（Lakehouse）：融合数据湖灵活性和数据仓库管理能力。通过Delta Lake/Hudi/Iceberg在数据湖上实现ACID事务、时间旅行、Schema进化。代表：Databricks

**数据治理核心域（DAMA框架）**：
- 数据架构管理：企业数据模型、数据流、数据分布
- 数据建模与设计：概念模型→逻辑模型→物理模型
- 数据存储与操作：数据库运维、备份恢复、容量管理
- 数据安全：分类分级、访问控制、加密脱敏、审计
- 数据集成与互操作：ETL/ELT、数据同步、主数据管理
- 数据质量管理：六个质量维度——完整性（数据不缺项）、唯一性（不重复）、一致性（各系统数据含义相同）、准确性（数据正确反映事实）、及时性（数据在需要时可用）、规范性（数据符合标准格式）
- 主数据与参考数据管理：主数据MDM（跨系统共享的核心业务实体，如客户/产品/组织）、参考数据（代码表/枚举值/分类标准）
- 元数据管理：技术元数据（表结构/字段类型/ETL血缘）+ 业务元数据（数据字典/指标口径/数据所有者）

**数据架构中架构师的职责**：
- 选型决策：数据仓库vs数据湖vs湖仓一体、批处理vs流处理、OLAP引擎选择（ClickHouse/Doris/StarRocks/Kylin）
- 数据分层：ODS操作数据层→DWD明细数据层→DWS汇总数据层→ADS应用数据层，分层原则是数据解耦、口径统一、血缘清晰
- 数据血缘：追溯数据从源头到消费端的全链路，支撑影响分析和问题排查（如上游表变更影响哪些下游报表）
- 数据生命周期：热数据→温数据→冷数据→归档→销毁，按数据价值和访问频率分配存储成本和访问性能

## 案例分析题

**题目**：

某大型零售集团建设新一代数据中台，整合线上线下10余个业务系统（ERP/CRM/POS/电商/会员/供应链/财务），日均新增数据500GB，存储总量已达PB级。业务部门提出以下需求：

1. 会员运营部门要求实时查看各渠道会员新增、活跃、消费数据，支持按地域/年龄段/消费层级多维度下钻分析，查询响应时间<3秒
2. 供应链部门需要每日凌晨对前一天所有销售订单进行全量分析，生成补货建议和库存预警，涉及百亿级数据关联计算，需在早8点前完成
3. CIO要求建立企业级数据标准，统一各系统的会员ID、商品编码、门店编码，解决"同一会员在不同系统ID不一致导致画像不准"的问题
4. 合规要求：会员个人信息需按《个人信息保护法》进行分级管理，敏感数据脱敏后方可进入分析环境，且所有数据操作留痕可审计

请从架构层面分析并解决问题。

**问题1**（8分）：请为该数据中台设计整体技术架构，要求覆盖数据采集、存储、计算、服务四层，说明各层的技术选型及选型理由。

**问题2**（7分）：针对实时运营分析（需求1）和每日批量分析（需求2），请分别设计技术方案。说明为什么不能用同一套技术方案解决两个需求，并给出各自的架构选择和关键技术细节。

**问题3**（5分）：针对主数据统一需求（需求3），请设计主数据管理（MDM）方案，说明主数据模型设计、数据合并与治理策略、数据同步机制。

**问题4**（5分）：针对数据安全合规需求（需求4），请设计数据安全架构方案，覆盖数据分类分级、脱敏策略、访问控制和审计机制四个维度。

**参考答案**：

**问题1**（数据中台整体技术架构）：

采集层（2分）：
- 业务数据库（MySQL/Oracle）：Canal监听binlog实时采集增量变更数据到Kafka，延迟秒级
- 日志/埋点数据：Filebeat→Kafka，前端埋点通过Nginx日志或直接上报
- 文件数据（供应商Excel/CSV）：FTP定时拉取+Flink批处理解析
- 选型理由：Canal+Kafka解决异构数据源实时同步问题，CDC（Change Data Capture）对源系统零侵入；Kafka统一消息通道解耦采集和处理

存储层（2分）：
- 实时数据：Kafka（消息缓冲，保留7天）+ ClickHouse（OLAP实时查询引擎，列存高压缩比+向量化执行，亿级数据亚秒级响应）
- 离线数据：HDFS（Parquet列存格式+Snappy压缩，存PB级历史全量数据）+ Hive Metastore（元数据管理）
- 选型理由：ClickHouse单表查询亿级数据秒级返回，适合实时分析场景；HDFS低成本存储历史全量数据；Kafka作为流批统一数据管道

计算层（2分）：
- 实时计算：Flink流处理引擎（消费Kafka实时数据，写入ClickHouse/Doris/HBase）
- 离线计算：Spark（批处理跑在HDFS上，处理百亿级关联计算）
- 选型理由：Flink基于事件时间的精确一次语义（Checkpoint+两阶段提交）保证计算准确性；Spark成熟稳定适合大规模离线处理

服务层（2分）：
- 数据服务API：统一查询网关，支持SQL查询（ClickHouse→Presto/Trino联邦查询→HDFS）
- 指标服务：指标缓存（Redis预聚合指标+定时刷新），支撑高并发看板
- 选型理由：Trino（原PrestoSQL）支持跨数据源联邦查询（ClickHouse+Hive同时查），无需数据搬运

**问题2**（实时OLS vs 离线OLAP双轨方案）：

为什么不能用同一套方案（2分）：
- 实时分析（需求1）核心约束：低延迟（<3秒）、高并发（数百用户同时看板）、数据量相对小（热数据：当天+近7天）
- 离线分析（需求2）核心约束：大数据量（百亿级全量扫描关联）、长作业（小时级可接受）、单次/少量用户执行
- 二者技术需求矛盾和资源竞争：列存分析引擎同一时间做高并发点查（OLTP）和全表扫描（OLAP）会互相影响性能，前者CPU消耗在后者的1/20以下。若用同一引擎做两件事，实时查询在离线任务运行期间会严重变慢
- 架构决策：实时热数据走ClickHouse（稀疏索引+向量化执行，亿级聚合秒级）、离线全量走Spark on Hive（MapReduce/Spark分布式计算引擎，支持百亿级复杂关联）

实时运营分析方案（3分）：
- 数据链路：Canal→Kafka→Flink（实时ETL：清洗+关联维表+预聚合）→ClickHouse
- 数据模型：星型模型，会员宽表（user_id, channel, age_group, city, consumption_level, last_active_time, today_order_count, today_amount）
- 多维度下钻：ClickHouse物化视图（Materialized View）按维度组合预计算聚合值，查询时直接读取预计算结果
- 查询加速：Bitmap索引加速用户ID圈选，分区按日期（PARTITION BY toYYYYMMDD(event_time)），TTL自动清理90天前细粒度数据
- 查询层：Apache Superset/Metabase提供自助式可视化分析

每日离线分析方案（2分）：
- 数据链路：Kafka数据落地HDFS（分层：ODS原始层→DWD明细清洗层→DWS汇总层→ADS应用层）
- 计算引擎：Spark SQL执行百亿级多表关联，分区裁剪（仅扫描前一天分区）+ 列裁剪（仅读取需要的列）+ 动态分区插入
- 调度：DolphinScheduler/Airflow编排，凌晨2点启动、预计5-6小时完成、8点前产出结果
- 结果写入：ADS层结果写入MySQL/ClickHouse供业务系统查询

**问题3**（主数据管理MDM方案）：

主数据模型设计（2分）：
- 三个核心主数据域：会员主数据（全局唯一member_id + 基础属性姓名/手机/身份证 + 各系统映射关系）、商品主数据（全局唯一sku_id + 商品属性 + 分类体系）、门店主数据（全局唯一store_id + 门店属性 + 组织层级）
- 主数据标准化规则：会员ID生成规则（8位日期+4位随机+2位校验）、手机号格式验证（正则+查重）、商品编码（品类码+品牌码+流水号）
- 映射关系表：source_system|local_id|global_id|match_confidence|status

数据合并与治理（2分）：
- 会员数据合并策略：手机号+身份证为高置信度匹配依据→包含姓名+手机号+身份证的模糊匹配（编辑距离/相似度）→低置信度匹配的人工审核队列
- 合并优先级规则：线下会员系统（身份证实名）准确度高于线上系统→当属性冲突时优先取线下系统值→特殊字段（生日/性别）按来源权威性加权
- 主数据质量管理：唯一性校验（同一主数据属性值不能重复）、完整性校验（必填字段非空）、一致性校验（派生字段与源字段逻辑一致）
- 数据标准编码体系：发布企业级数据字典（指标口径/编码规则/枚举值），变更需经数据治理委员会审批

数据同步机制（1分）：
- 注册中心模式：各业务系统创建/更新主数据时调用MDM注册服务，MDM根据匹配规则链接或新建全局主数据，返回global_id
- 批量初始化：首次上线时通过ETL抽取各系统主数据，运行合并去重算法，人工审核低置信度匹配后建立映射表
- 增量同步：MDM全局主数据变更后通过Kafka发布变更事件（MemberChanged/SkuChanged），各业务系统订阅更新本地映射表
- 数据一致性保证：建立定时对账任务，每日对比MDM与各系统主数据差异，生成差异报告触发修复

**问题4**（数据安全合规架构）：

数据分类分级（1分）：
- L1-公开数据：商品信息、门店地址、促销活动
- L2-内部数据：销售汇总统计（脱敏后）
- L3-敏感数据：会员手机号、身份证号、银行卡号、精确位置
- L4-核心数据：支付密钥、数据库凭证（不出分析环境）
- 分类标准参照《数据安全法》和《个人信息保护法》的敏感个人信息定义

脱敏策略（2分）：
- 静态脱敏（数据进入分析环境前）：生产→分析环境的数据同步链路中嵌入脱敏引擎
  - 手机号：保留前3后4位（138****5678）
  - 身份证号：保留前6后4位（210102*******1234）
  - 姓名：姓氏保留+名用*替代（张**）
  - 银行卡号：保留后4位（****1234）
- 动态脱敏（查询层）：基于用户角色+数据等级实时脱敏。普通分析师看到脱敏后数据、高级数据分析师需审批后可看完整数据
- 脱敏算法选择：替换（手机号→哈希映射保留关联性但不可逆）、遮盖（姓名/身份证直接截断）、泛化（具体地址→区县级）

访问控制（1分）：
- 三层访问控制：数据平台层（HDFS Ranger/Kerberos用户认证+目录权限）→数据库层（ClickHouse RBAC角色权限，数据行级安全策略Row Policy）→API层（OAuth2.0令牌+API网关限流鉴权）
- 数据权限模型：按组织（只能看本部门数据）+ 按数据等级（L3以上需审批）+ 按时间（有效期审批，超时自动回收）+ 按用途（数据使用申请需说明用途）
- 原则：最小权限+职责分离+审批留痕

审计机制（1分）：
- 全链路审计日志记录：谁（user_id+role）→何时（timestamp）→从哪（source_ip）→做了什么（query_type）→访问了什么（table/column/row_count）→结果（success/denied）
- 日志格式标准化：JSON格式统一输出，包含trace_id关联全链路日志
- 日志存储：审计日志写入Kafka→Flink消费→ClickHouse（热数据30天）+ HDFS归档（冷数据永久）
- 日志完整性与防篡改：日志产生即追加（Append-only）、每日日志Hash摘要留存、关键审计日志同步到独立的安全审计系统
- 异常检测：基于审计日志建立UEBA（用户实体行为分析），识别异常数据访问（如非工作时间大量导出数据）

**踩分要点**：

| 分值 | 踩分点 |
|------|--------|
| 8分 | 采集层含Canal+Kafka+CDC选型理由(2分)、存储层含ClickHouse+HDFS+选型理由(2分)、计算层含Flink+Spark+选型理由(2分)、服务层含Trino+缓存+选型理由(2分) |
| 7分 | 不可同一方案理由含性能矛盾+技术特征冲突(2分)、实时方案含ClickHouse+物化视图+Bitmap索引(3分)、离线方案含Spark+分区裁剪+分层ETL(2分) |
| 5分 | 主数据模型含三域+标准规则+映射表(2分)、合并治理含匹配策略+优先级+质量校验(2分)、同步含注册中心+批量初始化+增量变更(1分) |
| 5分 | 分类分级含四级标准(1分)、脱敏含静态+动态+脱敏算法(2分)、访问控制含三层+权限模型(1分)、审计含全链路+防篡改+UEBA(1分) |
