## 今日核心概念

### 1. 嵌入式系统架构与RTOS

定义：嵌入式系统是面向特定应用、以计算机技术为基础、软硬件可裁剪的专用计算机系统。RTOS（Real-Time Operating System，实时操作系统）是嵌入式系统的核心软件基础，其特征是任务的执行具有确定性时间约束——硬实时系统必须在截止时间内完成，否则视为系统失败；软实时系统允许偶尔超时。

关键要点：

**嵌入式系统分层架构**（自底向上四层）：
- 硬件层：MCU/MPU/SoC（ARM Cortex-M/A、RISC-V）、传感器/执行器、电源管理。架构师关注：处理器选型（功耗/性能/实时性三角）、外设接口（I2C/SPI/UART/CAN/USB）
- 抽象层（HAL/BSP）：硬件抽象层将硬件差异封装为统一API，便于上层软件跨平台移植。板级支持包BSP包含Bootloader、驱动、配置
- 系统软件层：RTOS内核（μC/OS-II、FreeRTOS、RT-Thread、VxWorks、QNX）+ 中间件（TCP/IP协议栈lwIP、文件系统FatFS、图形库LVGL）
- 应用软件层：业务功能模块，按功能拆分任务。架构师关注：任务划分、优先级设计、共享资源保护

**RTOS核心机制**：
- 任务调度：抢占式优先级调度（最高优先级就绪任务立即获得CPU）+ 时间片轮转（同优先级任务轮流执行）。调度延迟μs级，确定性是RTOS与Linux/Windows的本质区别
- 任务间通信IPC：信号量（互斥/计数/二值）、消息队列（任务间异步消息）、事件标志组（位运算通知）、邮箱、共享内存+关中断保护
- 中断管理：中断服务程序ISR运行在最高优先级，可打断任何任务。中断上下文不能阻塞、不能调用可能等待的API，必须通过信号量/队列通知任务处理下半部
- 内存管理：静态内存分配（编译时确定，实时性最强）+ 动态内存（malloc/free，可能产生碎片，实时系统慎用）+ 内存池（MemPool，固定大小块预分配）
- 时间管理：心跳时钟Tick（典型1ms-10ms）、软件定时器、vTaskDelayUntil周期性任务

**嵌入式架构师核心关注点**：
- 资源约束：内存KB级、CPU MHz级、功耗mW级 → 不能使用重型中间件
- 实时性保证：最坏执行时间WCET分析、优先级反转问题（优先级继承协议解决）
- 可靠性设计：看门狗（硬件看门狗+任务喂狗双保险）、掉电安全（关键数据写入铁电/EEPROM）、电磁兼容EMC

**FreeRTOS vs Linux选型**：
- FreeRTOS：KB级内存、μs级响应、裸机调度、适合MCU小系统（STM32/ESP32）
- Linux：MB级内存、ms级响应、丰富驱动和网络协议栈、适合MPU大系统（i.MX/树莓派）
- 中间方案：RT-Thread（小型RTOS带丰富组件）+ Linux双系统AMP（非对称多处理）

### 2. 通信系统架构与网络协议

定义：通信系统架构是面向消息传输、协议处理、网络互联的体系结构。架构师需要根据业务场景（实时性、可靠性、网络规模、带宽）选择合适的通信协议、网络拓扑和消息处理模型，是系统间数据流动的"血管"。

关键要点：

**OSI七层模型与TCP/IP四层模型对比**：
- OSI（理论参考模型）：物理层→数据链路层→网络层→传输层→会话层→表示层→应用层
- TCP/IP（实际工业模型）：网络接口层→网络层（IP/ICMP/ARP）→传输层（TCP/UDP）→应用层（HTTP/DNS/SSH/MQTT）
- 对应关系：OSI会话层+表示层合并到TCP/IP应用层。架构师实战中主要用TCP/IP四层模型

**传输层协议选型（高频考点）**：
- TCP（传输控制协议）：面向连接、可靠传输、流量控制、拥塞控制、三次握手四次挥手。适用于HTTP/Web/文件传输/邮件等可靠性优先场景
- UDP（用户数据报协议）：无连接、无确认、不可靠但低延迟。适用于视频直播/语音通话/DNS查询/物联网传感数据
- QUIC（基于UDP的新协议）：HTTP/3底层协议，0-RTT建连、多路复用无队头阻塞、内置TLS1.3加密、连接迁移（切换网络不重连）
- 对比维度：TCP三次握手+慢启动+队头阻塞 → 弱网高延迟场景表现差；QUIC在UDP上实现可靠传输，解决TCP核心痛点

**应用层协议分类**：
- HTTP/HTTPS：万维网标准，短连接（HTTP/1.1长连接+管线化、HTTP/2多路复用、HTTP/3基于QUIC）。REST API基于HTTP
- WebSocket：浏览器-服务器全双工通信，适合实时聊天/推送/在线协作。握手基于HTTP升级（Upgrade: websocket）
- MQTT vs AMQP：物联网/消息中间件主流协议
  - MQTT：发布订阅模式、消息头最小2字节、适合低带宽不稳定网络（移动/卫星/IoT）
  - AMQP：金融级可靠性、支持事务和确认、RabbitMQ默认协议
- gRPC：基于HTTP/2+Protocol Buffers二进制序列化、四种通信模式（Unary/Server Streaming/Client Streaming/Bidirectional Streaming）、适合微服务间通信

**网络架构模型**：
- C/S（客户端/服务器）：传统两层架构，中心化服务器处理所有业务逻辑。简单但扩展性受限
- B/S（浏览器/服务器）：C/S的Web版，客户端统一为浏览器，免安装
- P2P（对等网络）：节点既是客户端也是服务器，去中心化。文件共享（BT）、区块链、即时通信（WebRTC）
- CDN（内容分发网络）：边缘节点缓存静态资源，就近访问降低延迟。动静分离、边缘计算节点
- SDN（软件定义网络）：控制平面与数据平面分离，网络可编程化。OpenFlow协议，架构师关注5G/云网络的SD-WAN

**架构师实战关注点**：
- 网络分层：核心层（高速转发）+ 汇聚层（策略控制ACL/路由汇总）+ 接入层（用户接入PoE交换机）
- 高可用：VRRP/HSRP网关冗余、OSPF/BGP动态路由、链路聚合LACP
- 安全防护：防火墙（ACL/状态检测）、IDS/IPS（入侵检测/防御）、WAF（Web应用防火墙）、DDoS防护（流量清洗+黑洞路由）

### 3. 安全架构设计与密码学基础

定义：安全架构是保护信息系统资产（数据/服务/基础设施）免受威胁的整体性设计，包括机密性（Confidentiality）、完整性（Integrity）、可用性（Availability）三大核心目标，简称CIA三要素。架构师必须在系统设计阶段就嵌入安全机制（Security by Design），而非事后打补丁。

关键要点：

**信息安全五大目标（CIA + 扩展）**：
- 机密性C：数据不被未授权访问。技术：加密、访问控制
- 完整性I：数据不被未授权篡改。技术：数字签名、Hash校验、MAC消息认证码
- 可用性A：授权用户需要时系统可访问。技术：冗余、容灾、限流
- 不可否认性：通信双方不能否认已发生的行为。技术：数字签名、审计日志
- 可控性/可审计性：系统行为可被监管。技术：审计日志、监控告警

**对称加密 vs 非对称加密**：
- 对称加密（同一密钥加解密）：AES（128/192/256位分组，业界标准）、DES（56位已不安全）、3DES、SM4（国密）。优点：速度快（GB/s级）；缺点：密钥分发困难、N方通信需N×(N-1)/2个密钥
- 非对称加密（公钥私钥对）：RSA（基于大数分解，2048位起）、ECC（椭圆曲线，256位等价RSA3072位）、SM2（国密）。优点：密钥分发简单（N方通信N对密钥）、天然支持数字签名；缺点：速度慢千倍
- 混合加密（实战方案）：用非对称加密安全传输对称密钥（解决密钥分发）+ 用对称加密加密实际数据（保证性能）。典型应用：HTTPS握手、PGP加密邮件

**数字签名与数字证书**：
- 数字签名流程：发送方用私钥签名（Hash原文→私钥加密Hash值）→接收方用公钥验签（公钥解密→对比Hash）。提供完整性+身份认证+不可否认
- 数字证书：CA（证书颁发机构）用CA私钥对用户公钥+身份信息签名，绑定公钥和身份。解决"公钥属于谁"的信任问题
- PKI体系（公钥基础设施）：CA认证中心（签发证书）+ RA注册中心（审核身份）+ 证书库（发布/存储证书）+ CRL/OCSP（证书吊销列表/在线证书状态协议）

**PKI/CA体系组成**：
- 根CA → 中间CA → 用户证书的证书链结构
- 信任锚：操作系统/浏览器预装的根CA证书（信任锚点）
- 国密体系：SM1/SM2/SM3/SM4（商密算法）+ SM9（标识密码）

**哈希函数与完整性**：
- 常见算法：MD5（已破解，128位）、SHA-1（已破解，160位）、SHA-256（256位，主流）、SHA-3（Keccak算法，新一代）、SM3（国密256位）
- 特性：单向不可逆、固定长度输出、雪崩效应（输入1bit变化输出50%bit变化）
- 应用：密码存储（加盐bcrypt/Argon2）、文件指纹、Git版本标识、区块链（Merkle树）

**SSL/TLS握手流程**（架构师必须掌握）：
1. ClientHello：客户端发送支持的TLS版本、加密套件、随机数
2. ServerHello + Certificate：服务端选择版本+套件、返回证书（含服务端公钥）
3. 客户端验证证书链 → 用证书中的服务端公钥加密pre-master secret发送
4. 服务端用私钥解密得到pre-master secret
5. 双方基于随机数+pre-master secret派生对称会话密钥
6. Finished消息验证握手完整性，后续通信用对称密钥加密

### 4. 系统可靠性与容错设计

定义：系统可靠性是指系统在规定条件下、规定时间内完成规定功能的能力。容错设计是通过冗余、检测、恢复机制使系统在部分组件失效时仍能提供规定服务，是高可用架构的核心技术。

关键要点：

**可靠性核心指标**：
- MTBF（Mean Time Between Failures）：平均故障间隔时间，越大越好
- MTTR（Mean Time To Repair）：平均修复时间，越小越好
- 可用性Availability = MTBF / (MTBF + MTTR)。99.9% = 8.76小时/年停机；99.99% = 52.6分钟/年停机；99.999% = 5.26分钟/年停机（"5个9"电信级）
- 失效率λ（单位时间失效次数）：MTBF = 1/λ
- 浴盆曲线：早期失效期（磨合期bug）→偶然失效期（稳定运行）→损耗失效期（老化故障）

**硬件冗余三模冗余TMR**：
- 三模冗余TMR：三套相同硬件同时运行，表决器按多数原则输出结果（2:1或3:0）。容忍任意单模块故障
- 双模冗余DMR + 故障检测：双机热备，主备切换
- N+1冗余：N个运行模块+1个热备模块（电源/风扇/网卡场景）
- 2N双活：两套完整系统同时运行，流量分担，单套故障不影响整体（如双活数据中心）

**软件容错技术**：
- 冗余编程：N版本程序设计（不同团队不同语言实现相同功能，投票器选择结果）
- 恢复块策略：主块执行+接收方测试，测试失败则执行备用块
- 检查点（Checkpoint）：周期性保存程序状态到稳定存储，故障后从最近检查点恢复
- 微重启（Micro-Reboot）：Erlang/JVM应用，崩溃后毫秒级重启进程（不是系统）
- 进程监控+心跳：监督者进程定期检查被监控进程心跳，超时则重启

**RAID磁盘阵列**（高频考点）：
- RAID 0：条带化，无冗余，性能最高，故障率倍增（n块盘故障率1/n）
- RAID 1：镜像，50%空间利用率，可容忍单盘故障
- RAID 5：条带+分布式奇偶校验（n-1块盘容量），单盘故障可重建。校验分散到所有盘
- RAID 6：双重奇偶校验（n-2块盘容量），容忍双盘同时故障
- RAID 10（1+0）：先镜像后条带，性能+可靠性兼顾，50%空间利用率
- 写性能差异：RAID 1/10/5写性能受校验计算影响；RAID 0/10读性能最优

**集群与高可用**：
- 主备（Active-Passive）：主机工作，备机待命，故障切换。资源浪费50%但简单可靠
- 双主（Active-Active）：两台同时工作互为备份。资源利用率高但需脑裂防护
- 集群脑裂（Split-Brain）：心跳断开时双方都认为对方故障 → 抢占资源 → 数据不一致。解决方案：仲裁机制（Quorum，少数服从多数）、心跳多路径+磁盘锁、fencing机制（STONITH）
- 负载均衡集群：LVS（四层）+ Nginx（七层）+ HAProxy（兼具四/七层）
- Kubernetes高可用：etcd集群（基于Raft协议）+ kube-apiserver多实例 + 负载均衡器

**CAP定理与架构权衡**（与分布式章节呼应）：
- C（一致性Consistency）：所有节点同一时刻看到相同数据
- A（可用性Availability）：每个请求都能收到响应
- P（分区容错性Partition tolerance）：网络分区时系统仍能运行
- CAP三者只能取其二：网络分区P必选 → CP（如ZooKeeper/etcd，牺牲可用性保一致性） vs AP（如Eureka/Cassandra，牺牲强一致性保可用性）
- BASE理论：基本可用BA + 软状态S + 最终一致E（CAP中AP的工程化落地）

### 5. 软件工程过程与CMMI

定义：软件工程过程是指为获得高质量软件所需要完成的一系列任务的框架，它规定了完成各项任务的工作步骤。CMMI（Capability Maturity Model Integration）是软件过程能力成熟度模型集成的国际标准，是架构师评估和提升组织软件工程能力的核心参考模型。

关键要点：

**软件生命周期模型对比**：
- 瀑布模型：阶段顺序执行（需求→设计→编码→测试→维护），每阶段产出文档作为下一阶段输入。适合需求明确、变更少的项目
- 增量模型：分批次交付，每次增量是一个完整子集。早期版本可获得用户反馈
- 演化模型（原型+迭代）：快速构建原型→用户反馈→迭代改进。适合需求不清晰场景
- 螺旋模型：瀑布+原型+风险分析，每个周期包含目标→风险→开发→评估。适合大型复杂高风险项目
- 敏捷模型（Scrum/XP/SAFe）：迭代+增量+拥抱变化+小批量快速交付。适合互联网产品/需求快速变化场景

**CMMI五个成熟度等级**（高频考点，必须背诵）：
- L1 初始级（Initial）：过程不可预测、缺乏控制、依赖个人能力。"英雄主义"靠牛人救火
- L2 已管理级（Managed）：项目级过程被管理、有基本项目管理、里程碑跟踪、需求跟踪。关键过程域：需求管理、项目计划、项目监控、配置管理、过程与产品质量保证、测量与分析、供应商协议管理
- L3 已定义级（Defined）：组织级过程标准化、技术与过程改进。关键过程域：组织过程焦点、组织过程定义、集成项目管理、风险管理、决策分析与解决方案、验证与确认、组织级集成构建、需求开发、技术解决方案、产品集成
- L4 量化管理级（Quantitatively Managed）：建立量化目标、用统计过程控制。关键过程域：组织级过程性能、定量项目管理
- L5 优化级（Optimizing）：持续过程改进、缺陷预防、技术创新。关键过程域：组织级创新与推广、原因分析与解决

**CMMI评估方法**：
- SCAMPI（标准CMMI评估方法）：正式评估，颁发等级证书
- 评估维度：过程域目标（SG）→ 实践目标（SP）→ 实践 → 工作产品（工件）
- 评估结果：5级→4级→3级→2级→1级（满足所有过程域才可达到对应等级）

**CMMI vs 其他过程改进框架**：
- CMMI（综合能力成熟度）：通用框架，涵盖软硬件系统工程+采购+服务
- ISO 9001（质量管理体系）：通用质量管理标准
- ITIL（IT基础设施库）：IT服务管理最佳实践
- PMBOK（项目管理知识体系）：项目管理五大过程组+十大知识领域
- OPM3（组织项目管理成熟度）：PMI的项目管理成熟度模型，三个维度（项目组合/项目集/项目）+四个等级（标准化/测量/控制/持续改进）

**敏捷开发框架对比**：
- Scrum：3个角色（PO/SM/团队）+5个事件（Sprint/计划/站会/评审/回顾）+3个工件（ProductBacklog/SprintBacklog/Increment）
- XP极限编程：12个实践（结对编程/TDD/持续集成/简单设计/重构/小型发布/规划游戏/集体代码所有权/系统隐喻/40小时工作周/客户在场/代码规范）
- SAFe规模化敏捷：4层结构（团队/项目群/解决方案/投资组合），适合大型企业数字化转型
- DevOps与敏捷关系：敏捷偏重开发文化，DevOps扩展到运维，"敏捷+DevOps"是端到端快速交付最佳实践

**架构师过程关注点**：
- 需求工程：需求获取（用户访谈/原型/观察）+需求分析（DFD数据流图/IDEF0功能模型/E-R图）+需求验证（评审/原型确认/形式化方法）
- 设计方法：结构化设计（SD，模块化+自顶向下）vs 面向对象设计（OOD，SOLID原则+设计模式）
- 测试过程：V模型（开发阶段对应测试阶段，左边是开发右边是测试，单元↔单元测试、需求↔验收测试）
- 维护过程：四类维护（纠错性/适应性/完善性/预防性）。完善性维护占比最大（约50%）

## 案例分析题

**题目**：

某城市轨道交通集团启动"智慧地铁"信息化建设项目，覆盖AFC自动售检票系统、ISCS综合监控系统、信号系统、乘客信息系统PIS、车载视频监控、应急指挥调度等多个核心业务子系统。系统需满足以下关键要求：

1. 列车控制信号系统要求毫秒级（<100ms）实时响应，任何延迟都可能导致安全事故，需采用专用实时操作系统。系统故障时必须fail-safe（导向安全侧），关键控制指令必须三模冗余表决
2. AFC系统日均处理千万级乘客交易，地铁站覆盖全城200+站点，要求7×24小时不间断服务。年度可用性需达99.99%（年停机<53分钟）
3. 乘客信息系统PIS需实时推送列车到站信息、紧急通知，要求弱网（隧道环境）下也能可靠传输。终端设备在车站/列车上部署，无人值守
4. 应急指挥系统在突发大客流/火灾/恐怖袭击时，需在3秒内启动预案调度，整合AFC/PIS/广播/视频多系统联动
5. 集团要求建立统一的软件工程过程体系，三年后通过CMMI L3评估，并支撑后续向L4/L5迈进

请从系统架构层面分析并解决问题。

**问题1**（8分）：针对信号系统（需求1），请设计嵌入式实时控制架构，说明RTOS选型、任务划分策略、冗余容错机制，并分析为何不能使用通用Linux/Windows系统。

**问题2**（7分）：针对AFC系统（需求2），请设计高可用集群架构，说明双活/主备模式选择、数据一致性保障（异地多活交易流水不丢不错）、故障切换机制（脑裂防护），并给出RAID配置和数据库选型。

**问题3**（5分）：针对乘客信息系统PIS（需求3），请设计通信协议栈和弱网传输方案，对比MQTT/HTTP/WebSocket三种协议在该场景的适用性，并说明消息可靠性保障机制。

**问题4**（5分）：针对应急指挥系统（需求4），请设计多系统联动架构，说明事件驱动+消息中间件+服务编排的整合方案，以及如何保证3秒内启动预案。

**参考答案**：

**问题1**（信号系统实时控制架构）：

RTOS选型（3分）：
- 选用VxWorks/QNX/嵌入式Linux+RTAI实时补丁。CBTC（基于通信的列车控制系统）行业标杆采用VxWorks或QNX
- 选型理由：硬实时性（μs级中断响应+任务调度延迟确定性）✓、高可靠性（已通过SIL4安全完整性等级认证）✓、长期稳定运行支持（>10年不重启）✓、轨道交通行业认证背书
- 任务划分：按功能模块拆分独立任务
  - 任务1：ATP/ATO列车自动防护/驾驶（最高优先级，硬实时，WCET<10ms）
  - 任务2：联锁计算（次高优先级，安全关键）
  - 任务3：通信管理（与ATS/相邻列车通信，周期性）
  - 任务4：诊断与日志（最低优先级，非关键）
  - 任务5：人机交互（驾驶员HMI）
- 任务优先级设计：基于速率单调调度RMS（周期越短优先级越高），保证关键任务调度可预测

冗余容错机制（3分）：
- 三模冗余TMR：三套相同控制硬件同时运行（CPU+IO+通信），表决器对输出三选二。任意单模块故障不影响系统功能
- 软件冗余：双版本软件（N版本程序设计），不同团队不同语言实现同一控制逻辑，结果投票
- 看门狗：硬件看门狗+软件看门狗双重保护，任务未在周期内喂狗则强制复位
- 安全失效导向：故障时自动切换到安全状态（如降速→停车），不输出可能导致危险的控制指令
- 通信冗余：双通道冗余（双网口/双总线），主备切换<50ms

为何不用通用Linux/Windows（2分）：
- Linux/Windows是分时操作系统，调度延迟不确定（最坏情况可能达数百ms），无法保证硬实时
- 进程可能因缺页中断、GC、调度策略导致毫秒级卡顿，对安全关键系统是致命缺陷
- 缺乏功能安全认证（IEC 61508 SIL4、EN 50128/SIL4），无法通过轨道交通行业准入
- Linux可用于非安全关键子系统（PIS/ISCS），但核心ATP/ATO必须用RTOS

**问题2**（AFC系统高可用集群架构）：

双活vs主备选择（2分）：
- 选用双活（Active-Active）架构。两中心同时对外提供读写服务
- 选型理由：
  - 资源利用率：主备模式50%资源浪费，双活100%利用
  - 切换时间：主备切换通常10-30秒（VIP漂移+应用启动），双活故障秒级自动流量切换
  - 业务影响：地铁刷卡不允许10秒中断，主备切换会被乘客感知
- 部署模式：两个数据中心各承担50%交易流量，数据库双向同步（主主复制+冲突检测），应用层无状态部署

数据一致性保障（3分）：
- 数据库选型：交易流水强一致需求 → MySQL Group Replication（MGR基于Paxos）或OceanBase（原生分布式强一致）
  - MGR单主模式：写入单节点，多节点只读，强一致+故障自动选主
  - OceanBase：原生Paxos协议，多副本强一致，金融级可靠性
- 交易不丢不错保证：
  - 三阶段提交：客户端请求→本地事务→同步复制到对等节点→两节点都确认→返回成功。任何环节失败则回滚
  - 唯一流水号：全局唯一transaction_id（雪花算法+数据中心标识），避免重复入账
  - 幂等设计：相同transaction_id重复请求返回相同结果（基于transaction_id+状态机去重）
  - 异步对账：每5分钟定时对账任务，扫描双中心交易流水，差异告警人工介入

脑裂防护（1分）：
- 仲裁机制Quorum：3节点集群（2主+1仲裁节点），少数服从多数，避免两主各自为政
- 磁盘锁Fencing：抢占资源前先获取共享磁盘锁，失败则放弃自降为备
- 链路检测：多路径心跳（光纤+IPSec VPN双链路），避免单链路抖动误判

RAID与存储配置（1分）：
- 数据库盘：RAID 10（先镜像后条带），性能+可靠性兼顾，50%空间利用率
- 备份盘：RAID 5/6，冷数据存储
- 双活存储：存储双活（HyperMetro/Metro Cluster）实现跨数据中心存储同步

**问题3**（PIS通信协议栈与弱网传输）：

协议对比（2分）：

| 维度 | MQTT | HTTP | WebSocket |
|------|------|------|-----------|
| 协议开销 | 最小（2字节头） | 较大（HTTP头） | 中等 |
| 连接模型 | 长连接发布订阅 | 短连接请求响应 | 长连接全双工 |
| 弱网适应性 | 强（QoS等级+心跳） | 弱（每次重连） | 中（需应用层重连） |
| 实时推送 | 原生支持 | 需轮询/长轮询 | 原生支持 |
| 适用场景 | IoT/弱网推送 | API查询 | 浏览器实时通信 |

- 选择MQTT作为PIS主协议
- 选型理由：隧道弱网下带宽小+丢包率高+连接易断，MQTT的QoS 1（至少一次）+会话保持（clean session=false保留离线消息）+心跳保活机制最适配；HTTP开销大且每次请求重建连接；WebSocket适合浏览器场景但隧道环境终端为嵌入式设备

可靠性保障（3分）：
- QoS等级策略：
  - 列车到站信息（高频非关键）：QoS 0（至多一次，允许偶尔丢失，节省带宽）
  - 紧急通知（关键低频）：QoS 1（至少一次，配合业务幂等去重）
  - 调度指令（强关键）：QoS 2（恰好一次，最可靠但开销最大）
- 心跳保活：Keep Alive 60秒，心跳超时2倍即重连
- 离线消息：clean session=false，客户端断线期间消息保留在Broker，重连后投递
- 主题设计：分层主题（如pcd/{line_id}/{station_id}/{train_id}/arrival）便于订阅过滤
- 消息压缩：载荷>1KB启用压缩（gzip/zstd），节省隧道带宽
- 重连机制：指数退避（1s→2s→4s→...→30s上限），避免雪崩

**问题4**（应急指挥联动架构）：

整体架构（3分）：
- 事件驱动+消息中间件+服务编排三层架构
  - 事件源：视频分析（人群密度/异常行为检测）、AFC刷卡异常（闸机集中报警）、ISCS环境监测（烟雾/温度）、人工报警
  - 消息中间件：Kafka作为事件总线（高吞吐+事件回溯），每类事件一个Topic
  - 事件处理引擎：Flink实时事件关联（CEP复杂事件处理）→识别大客流/火灾/恐怖袭击模式
  - 预案引擎：Drools规则引擎+服务编排（BPEL/Conductor），根据事件类型匹配预案模板
  - 多系统联动：通过API Gateway调用各子系统
    - AFC：限流/关闭部分闸机
    - PIS：推送紧急通知+疏散引导
    - 广播：触发应急广播
    - 视频：联动切换监控画面+录像
    - 调度：通知应急人员+车辆调度

3秒启动保证（2分）：
- 预案预加载：所有预案加载到内存（避免运行时加载延迟）
- 决策路径优化：事件→Flink CEP识别（<200ms）→预案匹配（<100ms内存查找）→并行调用多系统API（<2s异步并发）→状态机推进+重试
- 异步并行：多系统联动调用不串行等待，并发执行+超时控制（每系统API超时2s）
- 性能压测：定期演练验证端到端时延，确保SLA

其他要点：
- 演练机制：每月应急演练（桌面推演+实战演练），验证系统可用性
- 监控告警：联动执行全链路trace_id追踪，失败子任务自动重试+告警
- 降级策略：Kafka不可用时降级为直接HTTP调用，关键事件直连核心子系统

## 记忆口诀

**嵌入式四层架构**：硬抽系应——硬件层(MCU/SoC)→抽象层(HAL/BSP)→系统软件层(RTOS+中间件)→应用软件层(业务模块)

**RTOS核心机制**：调通中内时——调度(抢占+时间片)/通信(信号量队列事件)/中断(不能阻塞)/内存(静态+内存池)/时间(心跳+定时器)

**RTOS vs Linux选型**：RTOS KB级μs级裸机调度MCU（STM32），Linux MB级ms级丰富栈MPU（i.MX）

**OSI七层记忆**：物数网传会表应——物理→数据链路→网络→传输→会话→表示→应用

**TCP vs UDP核心差异**：TCP握手+确认+可靠慢，UDP无握手无确认快

**应用协议选型**：HTTP请求响应REST、WebSocket全双工实时、MQTT弱网IoT、gRPC微服务二进制

**CIA三要素机密完整可用**：密加访控、完签Hash校、可用冗余容灾

**对称vs非对称加密**：对称AES快密钥分发难（杂凑密钥管理N²），非对称RSA/ECC慢密钥管理简单（N对），实战混合加密

**数字签名流程**：私签公验——发送方私钥签Hash、接收方公钥验签

**PKI体系四件**：CA认证中心+RA注册审核+证书库+CRL/OCSP吊销

**可靠性指标**：MTBF越大好、MTTR越小好、可用性=MTBF/(MTBF+MTTR)、5个9电信级

**冗余方式**：TMR三模两票、DMR双模热备、N+1备一、2N双活

**RAID对比**：0条带无冗快、1镜像50%容一损、5分布式校验n-1容一、6双重校验n-2容二、10先镜后条带性能+可靠

**集群脑裂防护**：少数服从多Quorum、磁盘锁Fencing、多路径心跳

**CAP三选二**：网络分区P必选，CP强一致牺牲可用（ZK/etcd），AP可用牺牲一致（Eureka/Cassandra）

**CMMI五级**：初管定量优——L1初始(靠人)→L2已管理(项目级)→L3已定义(组织级)→L4量化管理(统计控制)→L5优化(持续改进创新)

**CMMI关键过程域分布**：L2需求/计划/监控/配置/质量/测量/供应、L3焦点/定义/集成项目/风险/验证/确认/构建/需求开发/技术解决/产品集成

**瀑布vs敏捷一句话**：瀑布阶段顺序变更难，敏捷迭代增量拥抱变
