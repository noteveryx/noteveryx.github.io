## 今日核心概念

### 1. CMMI能力成熟度模型集成

定义：由美国卡内基梅隆大学软件工程研究所（SEI）提出的过程改进框架，用于评估和改进组织在软件开发、服务交付和产品采购方面的过程能力。CMMI从最初的软件CMM发展而来，当前主流版本为CMMI V2.0，覆盖开发（DEV）、服务（SVC）、供应商管理（SPM）等三个领域。

关键要点：
- 五级成熟度递进：初始级（Initial，过程混乱无序，依赖个人英雄）→ 已管理级（Managed，项目层面有计划有跟踪）→ 已定义级（Defined，组织层面标准化过程，项目根据剪裁指南定制）→ 定量管理级（Quantitatively Managed，基于统计技术的过程度量与控制）→ 优化级（Optimizing，持续改进创新，缺陷预防）
- CMMI V2.0以实践域（Practice Area）为核心组织方式，不再是V1.3的阶段式/连续式双表示法，每个实践域划分为5个能力等级，整体组织仍可按成熟度评级
- 关键实践域举例：需求开发与管理（RDM）、技术解决方案（TS）、产品集成（PI）、验证（VV）、确认（VAL）、过程质量保证（PQA）、配置管理（CM）、决策分析与解决（DAR）、风险管理（RSKM）等
- 架构师与CMMI的关系：架构设计活动对应技术解决方案实践域的特定实践（SP 2.1设计产品架构、SP 2.2开发架构视图），架构评审对应验证实践域，架构决策记录对应决策分析实践域
- 易混淆点：CMMI是过程模型而非技术标准，不规定用什么工具和方法，只规定应具备什么能力；CMMI≠瀑布模型，敏捷开发团队同样可以通过CMMI评估（CMMI V2.0专门增加了敏捷实践场景）

### 2. 挣值管理（EVM）与项目绩效度量

定义：将项目范围、进度和成本整合为一体的绩效度量方法，通过计划价值（PV）、挣值（EV）和实际成本（AC）三个核心指标，以货币化形式量化项目进展与偏差，是软考上午题计算类的高频考点。

关键要点：
- 三个基本值：计划价值PV（Planned Value）——截至某时间点计划完成工作的预算（即BCWS）；挣值EV（Earned Value）——实际完成工作的预算价值（即BCWP）；实际成本AC（Actual Cost）——实际花费的成本（即ACWP）
- 四个偏差指标：进度偏差SV=EV-PV（正=提前、负=滞后）；成本偏差CV=EV-AC（正=节约、负=超支）；进度绩效指数SPI=EV/PV（>1提前、<1滞后）；成本绩效指数CPI=EV/AC（>1节约、<1超支）
- 完工估算三种算法：①假定剩余工作按计划——EAC=AC+(BAC-EV)；②假定剩余工作按当前CPI——EAC=BAC/CPI；③假定SPI和CPI同时影响——EAC=AC+(BAC-EV)/(CPI×SPI)。选择依据：绩效趋势稳定则用②，绩效异常且不可恢复则用③
- 完工尚需绩效指数TCPI：TCPI=(BAC-EV)/(BAC-AC)——表示剩余资金需要达到的效率水平，TCPI>1说明剩余资金紧张，TCPI<1说明资金充裕
- 论文应用场景：在项目管理类论文中，EVM可体现架构师对资源管理的量化把控能力，结合偏差分析给出纠正措施

### 3. 知识产权保护与标准化体系

定义：软考上午题中法律法规与标准化的核心内容，涵盖著作权（版权）、专利权、商标权、商业秘密四种知识产权保护形式，以及国家标准、行业标准、地方标准、企业标准的代号体系。

关键要点：
- 四种保护形式对比：著作权——自动产生（创作完成即享有），保护表达不保护思想，自然人死后50年、法人首次发表后50年；专利权——须申请审批，保护技术方案本身，发明20年/实用新型10年/外观设计15年（自申请日）；商标权——须注册，保护商业标识，有效期10年可无限续展；商业秘密——须采取保密措施，无期限（只要保密），反向工程合法获取不侵权
- 职务作品归属：利用单位物质技术条件完成的、主要利用单位条件或合同约定由单位享有的，著作权归单位，作者享有署名权；未利用单位条件的职务作品，著作权归作者，但单位有2年优先使用权；委托创作作品，著作权归属由合同约定，无约定归受托人（创作者）
- 标准代号体系：GB（强制性国标）、GB/T（推荐性国标）、GB/Z（指导性技术文件）；行业标准代号——GJB（军用）、GA（公安）、YD（通信）、JR（金融）、SF（司法）等；地方标准——DBXX/T（如DB11/T为北京）；企业标准——Q/XXX
- 标准化层次：国际标准（ISO/IEC/ITU）→ 区域标准（EN欧洲）→ 国家标准（GB）→ 行业标准 → 地方标准 → 团体标准 → 企业标准（技术要求依次升高）
- 易混淆点：开源许可证中，GPL为强传染性（衍生作品必须同样GPL开源），LGPL为弱传染性（动态链接可闭源），MIT/BSD/Apache为宽松型（可闭源商用但须保留版权声明）；MPL为文件级传染性（修改Mozilla代码须开源修改部分）

### 4. 决策树分析与不确定性决策

定义：用树形图表示决策问题中各方案的期望收益或期望损失，通过逆向归纳法（从叶节点向根节点逐层计算期望值）选择最优方案。是软考上午题中管理科学基础的必考题型，也是架构师在技术选型和风险决策中的常用工具。

关键要点：
- 决策树四要素：决策节点（□方框，表示需要做出选择）、状态节点（○圆圈，表示不确定的自然状态）、结果节点（△三角，表示最终结果及损益值）、概率标注（连线上的概率，所有分支概率之和必须=1）
- 计算规则：从右向左逆向计算，状态节点期望值=∑（概率×损益值）；决策节点取各方案期望值的最大值（收益型问题）或最小值（成本型问题）
- 完全信息期望价值EVPI=有完全信息时的期望值-无完全信息时的最优期望值。EVPI代表了决策者为获取完全信息愿意支付的最高价格
- 不确定性决策五准则（已知收益矩阵但未知概率）：①乐观准则（大中取大Maximax）——取各方案最好结果中最好的；②悲观准则（小中取大Maximin）——取各方案最差结果中最好的；③等可能性准则（Laplace）——假设各状态等概率计算期望值；④后悔值准则（Minimax Regret）——先算后悔值矩阵，再取各方案最大后悔值中最小的；⑤折中准则（Hurwicz）——乐观系数α加权
- 架构决策场景应用：技术选型（自研vs外包vs采购）、架构风格选择（单体vs微服务）、部署策略（自建机房vs云服务）均可构建决策树，将成本、性能、团队能力、风险概率量化为期望值辅助决策

### 5. 区块链架构与分布式账本技术

定义：一种去中心化的分布式数据库技术，通过密码学保证的链式数据结构、分布式共识机制和智能合约，实现不可篡改、可追溯的数据共享与协作。是系统架构设计师需要掌握的新技术架构方向之一。

关键要点：
- 区块链六层架构模型：数据层（区块+链式结构+Merkle树+时间戳+哈希函数+非对称加密）→ 网络层（P2P组网+数据传播+验证机制）→ 共识层（PoW工作量证明/PoS权益证明/DPoS股份授权证明/PBFT实用拜占庭容错/Raft）→ 激励层（发行机制+分配机制）→ 合约层（智能合约+虚拟机EVM+DApp）→ 应用层（可编程货币/可编程金融/可编程社会）
- 共识算法选型对比：PoW（算力密集、去中心化最高、吞吐量低~7TPS、比特币使用）；PoS（代币抵押、能耗低、吞吐量中等）；DPoS（投票选举出块节点、吞吐量高~数千TPS、EOS使用）；PBFT（状态机复制、无分叉确定性、节点≤100、联盟链主流选择）
- 三种链类型：公有链（完全开放，任何人参与，如比特币/以太坊）、联盟链（多机构共同维护，权限管理，如Hyperledger Fabric/FISCO BCOS）、私有链（单一组织内部控制，用于内部审计与追溯）
- 智能合约三特性：确定性（相同输入必得相同输出）、不可篡改（部署后代码不可修改，升级须通过代理合约模式）、可审计（代码开源、执行记录公共可见）
- 架构师关切的非技术因素：国密算法合规（联盟链必须使用SM2/SM3/SM4替代国际密码算法）、数据隐私保护（链上数据加密+零知识证明+可信执行环境TEE）、与现有企业系统集成（预言机Oracle桥接链下数据、API网关对接ERP/CRM）

## 案例分析题

**题目**：

某省级政务数据共享交换平台需进行架构升级。当前平台采用基于ESB的集中式数据交换架构，连接省直42个委办局的业务系统，日均交换数据120万条。主要问题：①各委办局对数据主权高度敏感，不信任集中式交换中心；②数据交换审计追溯能力不足，跨部门数据使用缺乏可信记录；③多部门协同审批场景下，审批状态同步依赖中心数据库，存在单点故障和篡改风险。

省大数据局决定引入区块链技术改造数据共享平台，要求在保留现有业务系统接口的前提下，将关键交换记录上链存证，并实现多部门联合审批的链上协同。作为项目总架构师，请回答：

**问题1（8分）**：分析该场景应选择公有链、联盟链还是私有链？说明理由，并绘制区块链在政务数据共享平台中的融入架构图（文字描述各层即可）。

**问题2（7分）**：设计多部门联合审批的智能合约方案。说明智能合约的核心数据结构和关键方法，分析如何确保审批流程不可篡改且可审计，提出审批状态在区块链与现有业务系统间的一致性保障方案。

**问题3（5分）**：该项目涉及42个委办局的数据共享，各局对数据授权粒度要求不同（字段级、记录级、时间窗口级）。设计基于区块链的数据授权与访问控制方案，说明授权信息的存储方式和权限验证机制。

**问题4（5分）**：从国密算法合规和现有系统集成两个维度，各指出一个关键风险及应对措施。

**参考答案**：

**问题1**：

应选择联盟链。理由：①政务场景有明确的参与方边界（42个委办局），联盟链的多机构共同维护天然契合；②公有链性能（PoW~7TPS、PoS~数十TPS）无法满足日均120万条交换需求，联盟链PBFT可支持数千TPS；③公有链数据全公开与政务数据安全保密要求矛盾，联盟链支持通道机制和隐私保护；④联盟链支持国密算法（SM2/SM3/SM4）合规要求，大部分公有链使用国际密码算法。

架构融入（自底向上六层）：
- 数据层：关键交换记录（哈希+时间戳+交换双方ID）打包为区块，Merkle树组织，每区块附国密SM3哈希
- 网络层：42个委办局+省大数据局共43个节点，采用许可准入P2P网络，TLS加密通信
- 共识层：采用PBFT算法（节点数可控、确定性共识无分叉、所需2f+1≤43，f≤14容错），满足政务场景低延迟要求
- 合约层：部署数据交换存证合约、联合审批合约、授权管理合约
- API接入层：各委办局业务系统通过SDK/API接入，交换操作触发智能合约自动生成存证
- 应用层：数据交换监控大屏、审计追溯平台、链上审批进度查看

**问题2**：

智能合约核心数据结构：
```
struct ApprovalTask {
    string taskId;          // 审批任务唯一标识
    string applicant;       // 申请人（委办局ID）
    string[] approvers;     // 审批人列表（多部门）
    mapping(string=>bool) approvals;  // 各部门审批结果
    uint8 requiredCount;    // 所需通过部门数
    uint8 approvedCount;    // 已通过部门数
    enum Status {PENDING, APPROVED, REJECTED}
    Status status;
    uint256 createTime;
    uint256 updateTime;
}
```

关键方法：createTask()创建审批任务 → approve()各部门依次审批（仅审批列表中的节点可调用）→ autoFinalize()满足条件自动终结状态。每个审批操作生成Event日志供链下监听。

防篡改与可审计：①审批记录以交易形式写入区块，一旦确认不可修改；②每次审批操作自动触发链上Event（含操作者、时间、操作内容）；③完整审批链路可追溯至每个部门的审批签名，形成不可抵赖的证据链。

一致性保障方案：①链上为主数据源（Single Source of Truth），链下业务系统为快照副本；②业务系统通过监听合约Event异步同步状态变更；③设置链下状态对账定时任务（每小时全量对账），发现不一致自动申请链上重放修复。

**问题3**：

授权信息存储方案：采用链上授权元数据+链下数据实体的混合模式。链上存储授权策略元数据（被授权方ID、授权方ID、数据资源标识、授权粒度级别、授权有效期），链下数据实体在交换网关中完成分级脱敏。

授权粒度设计：①字段级授权——数据资源标识精确到字段路径（如"自然人基本信息.身份证号"），在智能合约中维护字段级授权白名单；②记录级授权——通过查询条件模板（如"仅企业法人"）限定可见记录范围；③时间窗口级授权——在授权结构中增加起始时间/结束时间字段，合约在每次数据请求时校验当前时间是否在窗口内。

权限验证三层机制：①API网关层粗粒度校验——请求方是否在该数据资源的授权名单中；②智能合约层精确校验——合约方法require语句校验字段级、时间窗口约束；③数据交换层脱敏执行——根据授权级别对返回数据做字段过滤和脱敏（如身份证号只返回前6后4位）。

**问题4**：

风险一（国密合规）：现有联盟链底层（如Hyperledger Fabric默认使用ECDSA/SHA256/AES）需全面替换为国密算法。应对措施——选择FISCO BCOS等原生支持国密的国产联盟链平台（SM2签名、SM3哈希、SM4对称加密），在项目启动阶段完成国密算法的性能基准测试（重点关注SM2签名验证速度是否满足120万/日存证量）。

风险二（现有系统集成）：42个委办局系统技术栈异构（Java/.NET/PHP/Python等），SDK适配成本高。应对措施——设计统一的RESTful API中间层（区块链网关），封装合约调用为HTTP接口，各业务系统无需引入链SDK，HTTP携带国密签名即可完成上链操作；网关内部做异步批量提交优化，支撑日均120万条吞吐量。

**踩分要点**：
- 问题1：选型理由须包含性能（TPS对比）、安全（数据公开性）、合规（国密）三个维度，缺一则扣分；架构描述须体现链上链下协同思路
- 问题2：数据结构包含状态字段和审批人映射；提到Event日志机制是加分项；一致性方案不能只说"同步"，须有具体对账策略
- 问题3：三种粒度授权须逐项说明存储和验证方式；链上元数据+链下脱敏执行的分离设计是核心得分点
- 问题4：风险须具体可操作，不能泛泛说"注意安全"；应对措施须点名具体技术方案（FISCO BCOS/RESTful网关）

## 记忆口诀

**CMMI五级递进**：初管已定两优化——初（初始级）→管（已管理级）→定（已定义级）→量（定量管理级）→优（优化级）。助记：一个团队的成长——起初靠英雄，管住做计划，统一流程定标准，数据说话量成效，持续改进优化路。

**EVM挣值管理核心公式**：记忆基数——SV=EV-PV（进度）、CV=EV-AC（成本），正数为好、负数为差。衍生——SPI=EV÷PV、CPI=EV÷AC，大于1是好事。TCPI=（剩余工作÷剩余钱）=（BAC-EV）÷（BAC-AC），剩下的活看剩下的钱，大于1说明得加把劲。

**知识产权四保护**：著专商密各有期——著作死后50年、专利发明20实用10外观15（自申请日）、商标10年可续无限、商业秘密保密即永久。职务作品归属诀：用我钱用我料，权力归我你署名；自己创造自己用，单位两年先用权。
