## 今日核心概念

### 1. 软件架构评估方法体系
定义：软件架构评估是在架构设计完成后、编码实施之前，通过系统化的方法对架构质量属性进行定量和定性分析，识别架构风险与敏感点，为架构决策提供依据的工程活动。

关键要点：
- ATAM架构权衡分析法（Architecture Tradeoff Analysis Method）：最主流的评估方法，由SEI提出。核心流程——利益相关者识别→效用树构建（质量属性优先级排序）→架构方案生成→质量属性场景分析→敏感点与权衡点识别→风险与非风险归档。输出成果：效用树、架构风险报告、敏感点/权衡点清单
- SAAM架构可替代性分析法（Software Architecture Analysis Method）：最早的架构评估方法，侧重可修改性。流程——场景描述→构件归组→场景评估→评估报告。适用于评估架构的灵活性和可扩展性，但对非功能性质量属性覆盖不足
- CBAM成本效益分析法（Cost Benefit Analysis Method）：与ATAM配合使用，在架构方案比选时引入成本维度，计算每个质量属性改进的投资回报率，辅助决策者做成本效益平衡
- 三种方法对比：SAAM侧可修改性、ATAM侧多质量属性权衡、CBAM侧成本决策；实际项目中通常先ATAM定性分析，再CBAM定量评估
- 架构评估的六个质量属性场景要素：刺激源（谁触发）、刺激（触发什么）、制品（影响哪个构件）、环境（系统所处状态）、响应（系统如何反应）、响应度量（量化指标值）

### 2. 实时系统与嵌入式架构设计
定义：实时系统指必须在明确的时间约束内对外部事件做出响应的计算机系统，分为硬实时（违反时限导致灾难性后果）和软实时（违反时限导致服务质量下降但不致灾难）两类。

关键要点：
- 硬实时与软实时的区分标准：硬实时要求最坏情况下的响应时间也不能超过截止期限（如航空航天、医疗设备、核电站控制），软实时允许偶尔超时（如视频流、在线游戏）
- 实时调度算法三类：（1）速率单调调度（RMS）——静态优先级，周期越短优先级越高，利用率上限n(2^(1/n)-1)，n趋近无穷时约为69%；（2）最早截止期限优先（EDF）——动态优先级，截止期限最近的任务优先，理论利用率可达100%；（3）最小松弛时间优先（LLF）——动态优先级，松弛时间=截止期限-剩余执行时间-当前时间
- 实时操作系统内核特点：可抢占内核、中断延迟确定、优先级继承协议（防优先级反转）、内存保护、看门狗定时器
- 嵌入式系统架构模式：循环执行架构（轮询，最简单）、前后台架构（中断+主循环）、多任务架构（RTOS任务调度）、分层架构（硬件抽象层HAL+驱动层+OS层+中间件层+应用层）
- 嵌入式系统与通用系统的五大差异：资源受限（内存/处理器/存储）、实时性要求、专用性强、可靠性要求高（MTBF通常>10万小时）、软硬件协同设计

### 3. 信息安全架构设计深化
定义：信息安全架构是从系统架构层面规划和部署安全控制措施的整体框架，覆盖机密性、完整性、可用性三大基本属性，纵深防御、最小权限、纵深防护是其核心设计原则。

关键要点：
- 纵深防御模型（Defense in Depth）：将安全控制分层部署在物理层、网络层、主机层、应用层、数据层，任一层被突破时仍有下一层防护。类比城堡的护城河→城墙→城门→内廷→金库
- 零信任架构（Zero Trust Architecture）："永不信任，始终验证"。三大原则：默认不信任任何用户/设备/网络、最小权限动态授予、持续验证与审计。核心技术：身份认证（MFA多因素）、微隔离、软件定义边界（SDP）、持续信任评估引擎
- 等保2.0（网络安全等级保护2.0）五级体系：1级（自主保护）→2级（指导保护）→3级（监督保护）→4级（强制保护）→5级（专控保护）。政务系统通常定级为3级。等保2.0相比1.0新增了云计算、物联网、移动互联、工业控制、大数据安全扩展要求
- 数据安全三维度：数据分类分级（公开/内部/机密/绝密）、数据全生命周期保护（采集→传输→存储→处理→交换→销毁）、数据脱敏（静态脱敏+动态脱敏+匿名化）
- 安全审计四要素：审计策略（定义审计规则）、审计记录（操作日志、安全事件、访问日志）、审计分析（关联分析、异常检测）、审计报告（合规性报告、事件报告）

### 4. 面向方面的编程（AOP）与横切关注点
定义：面向方面的编程是一种通过将横切关注点（cross-cutting concerns）从业务逻辑中分离出来、以模块化方式处理的技术。横切关注点是指贯穿系统多个模块的通用功能需求，如日志、安全、事务、缓存等。

关键要点：
- 横切关注点的典型场景：日志记录、安全检查、事务管理、性能监控、异常处理、缓存、连接管理、并发控制。这些关注点散布在多个模块中，传统面向对象编程导致代码重复和耦合
- AOP核心概念：（1）切面（Aspect）——横切关注点的模块化封装，如TransactionAspect；（2）连接点（Join Point）——程序执行中的某个点，如方法调用、异常抛出；（3）切入点（Pointcut）——匹配一组连接点的表达式，如execution(* com.service.*.*(..))；（4）通知（Advice）——在切入点执行的动作，Before/After/Around/AfterReturning/AfterThrowing五种；（5）织入（Weaving）——将切面应用到目标对象的过程，编译期/类加载期/运行期三种时机
- AOP与装饰器模式的区别：AOP通过代理或字节码增强在运行时/编译期织入，对业务代码零侵入；装饰器模式需要在代码中显式包装，侵入性更强且只能装饰单个对象
- Spring AOP的实现机制：基于JDK动态代理（接口）或CGLIB字节码生成（类），默认自动选择。仅支持方法级别的连接点，不支持字段访问拦截
- AOP在架构中的应用价值：统一处理横切关注点、降低模块间耦合、提升代码可维护性、支持声明式编程（如@Transactional声明式事务）

### 5. 数据仓库与数据湖架构
定义：数据仓库是面向主题的、集成的、相对稳定的、反映历史变化的数据集合，用于支持管理决策。数据湖是存储企业全部原始数据（结构化/半结构化/非结构化）的大规模存储系统，支持多种分析范式。

关键要点：
- 数据仓库四特性（Bill Inmon定义）：面向主题（按业务域组织而非应用）、集成性（统一编码和命名规则）、相对稳定（历史数据不变）、反映历史变化（记录时间维度）
- 数据仓库架构分层：ODS操作数据层（原始数据贴源存储）→ DW数据仓库层（DWD明细层+DWS汇总层）→ DM数据集市层（面向特定业务主题）→ APP应用层（报表/接口/大屏）
- Inmon vs Kimball建模路线之争：（1）Inmon企业级数据仓库——自顶向下，先建立统一的企业数据模型（三范式），再拆分数据集市，优点是数据一致性强，缺点是建设周期长；（2）Kimball维度建模——自底向上，先建设面向业务过程的维度模型（星型/雪花），再整合为企业仓库，优点是见效快、易理解，缺点是跨主题一致性难保证
- 数据湖架构三层： Landing原始层（原始数据入湖，不加工）→ Curated治理层（数据清洗、元数据标注、质量校验）→ Consumed消费层（数据产品、分析模型、ML训练）
- 数据仓库vs数据湖核心区别：数据仓库存加工后的结构化数据、采用Schema on Write、适合BI报表和SQL分析；数据湖存原始全量数据、采用Schema on Read、适合数据科学和机器学习。现代趋势是数据湖仓（Lakehouse）融合两者优势

## 案例分析题

**题目**：某银行核心系统升级改造项目

某城市商业银行正在进行核心交易系统升级，新系统采用微服务架构，涉及账户管理、转账汇款、信贷审批、理财购买、客户信息管理等10个核心域。系统要求：①日均交易量500万笔，高峰期QPS不低于2000；②账户转账必须保证资金不丢不重（强一致性）；③核心域平均响应时间<200ms，99%请求<500ms；④满足等保三级安全要求。技术选型采用Spring Cloud微服务+MySQL分库分表+Redis集群+RocketMQ。

**【问题1】（7分）**
项目组在微服务拆分时面临服务粒度决策难题。请说明微服务拆分的三种粒度策略（粗粒度/细粒度/适中粒度）各自适用场景和风险，并给出该银行核心系统的推荐拆分方案及理由。

**【问题2】（7分）**
账户转账涉及两个微服务的数据库操作（扣款方+收款方），项目组需要保证跨库强一致性。请对比分析四种分布式事务方案（2PC、TCC、Saga、本地消息表+最终一致性），说明各自的原理、一致性级别、适用场景和性能代价，并给出该转账场景的推荐方案。

**【问题3】（6分）**
为满足等保三级安全要求，请从纵深防御的五个层次（物理/网络/主机/应用/数据）设计安全控制措施，每层至少给出两项具体措施。说明如何在该微服务架构中实施零信任安全架构。

**【问题4】（5分）**
系统采用数据仓库架构支撑经营分析和风控决策。请对比Inmon与Kimball两种建模路线，给出推荐方案及理由，画出四层数据仓库分层架构，说明各层职责。

**参考答案**：

**【问题1 参考答案】（7分）**
**三种粒度策略对比**：

（1）**粗粒度拆分**：每个微服务包含多个业务能力（如将账户管理+客户信息合并为一个服务）。适用场景：团队规模小（<10人）、业务边界尚不清晰、系统初期快速上线。风险：服务内部耦合高、单点故障影响面大、部署粒度粗回滚困难。

（2）**细粒度拆分**：每个微服务对应单一业务能力甚至单一数据表（如转账拆为"转账发起""转账确认""转账通知"三个服务）。适用场景：大型团队（>50人）、业务域边界非常清晰、需要独立弹性伸缩。风险：服务数量爆炸导致运维复杂度高、分布式通信开销大、调试排障困难。

（3）**适中粒度拆分**：按领域驱动设计（DDD）的限界上下文拆分，每个服务对应一个业务域（如账户域、转账域、信贷域）。适用场景：中等规模团队（10-50人）、业务复杂度中等以上。风险：需要较高的领域建模能力，限界上下文划分需要经验积累。

**推荐方案**：采用适中粒度，按DDD限界上下文拆分为10个核心域服务（账户域、转账域、信贷域、理财域、客户域、风控域、清算域、渠道域、参数域、通知域）。理由：银行核心系统业务复杂度高、合规要求严格、各域有明确的业务边界和数据所有权，适中粒度兼顾团队协作效率与系统弹性。

**【问题2 参考答案】（7分）**
**四种方案对比**：

| 方案 | 原理 | 一致性级别 | 性能代价 | 适用场景 |
|------|------|-----------|---------|---------|
| 2PC（两阶段提交） | 协调者发送prepare→参与者投票→协调者发送commit/rollback | 强一致性 | 高（同步阻塞、参与者需持锁等待、单点故障风险） | 数据库层面跨库事务，参与者数量少（2-3个），低并发 |
| TCC（Try-Confirm-Cancel） | 业务层面的两阶段：Try预留资源→Confirm确认提交→Cancel补偿回滚 | 强一致性 | 中（无数据库锁但需实现三个接口，网络开销2-3倍） | 资金类操作、订单扣库存、高并发下的资金转账 |
| Saga | 长事务拆分多个本地事务序列，每个本地事务有对应补偿操作，失败时反向补偿 | 最终一致性 | 低（无阻塞、各参与者独立提交、补偿可能失败需人工介入） | 长流程业务编排（机票+酒店+租车）、跨多系统的业务流程 |
| 本地消息表+最终一致性 | 业务操作与消息写入同一本地事务，消费者消费消息后执行远程操作 | 最终一致性 | 中低（异步解耦、消息可能有延迟） | 对实时性要求不严格的场景、通知类、数据同步类 |

**推荐方案**：转账场景推荐TCC方案。理由：银行转账是资金操作，必须保证强一致性且不能出现资金不一致的中间态；2PC在微服务跨网络场景下性能差且可靠性不足；Saga和最终一致性不适合资金场景。TCC的Try阶段冻结扣款方资金，Confirm阶段实际转账，Cancel阶段解冻资金，满足"不丢不重"要求。

**踩分要点**：四种方案必须每个都答出原理+一致性级别（各1.5分），漏答任一方案扣分。推荐方案必须说明推荐理由并关联业务场景（1分）。

**【问题3 参考答案】（6分）**
**纵深防御五层安全控制**：

（1）**物理层**：机房门禁+视频监控+UPS不间断电源+环境温湿度监控
（2）**网络层**：防火墙（DMZ区隔离内外网）+入侵检测系统（IDS）+VPN加密远程访问+网络分段隔离核心交易区
（3）**主机层**：操作系统加固（关闭不必要端口/服务）+防病毒软件+主机入侵检测（HIDS）+定期漏洞扫描与补丁管理
（4）**应用层**：OWASP Top10防护（SQL注入/XSS/CSRF）+API网关统一认证授权+请求签名防篡改+操作审计日志+输入校验白名单
（5）**数据层**：数据库访问控制（最小权限原则）+敏感数据加密存储（AES-256）+数据脱敏（手机号/身份证掩码）+数据库审计日志+数据备份与恢复策略

**微服务零信任安全架构实施方案**：
（1）服务间调用采用mTLS双向认证，每个微服务持有独立证书，通过服务网格（Istio/Linkerd）实现自动证书管理
（2）所有API调用必须携带JWT令牌，令牌包含调用者身份、权限范围、有效时间，由认证中心统一签发
（3）服务间调用实施细粒度RBAC控制，定义服务级别的访问策略（如"信贷服务仅允许调用账户服务的查询接口"）
（4）部署持续信任评估引擎，对异常调用频率、异常来源IP、异常时段访问进行实时告警和自动阻断

**踩分要点**：五层各至少两项措施，每层1分（共5分），漏层扣分。零信任方案需体现mTLS/JWT/细粒度RBAC/持续评估至少三个要点（1分）。

**【问题4 参考答案】（5分）**
**Inmon vs Kimball对比**：
Inmon路线自顶向下建立企业级三范式数据模型，数据一致性强但建设周期长、前期投入大；Kimball路线自底向上建设维度模型（星型/雪花），见效快、查询性能好但跨主题一致性难保证。

**推荐方案**：采用Kimball维度建模路线为主、Inmon思想为辅的混合策略。理由：银行项目通常有明确的上线时间要求，Kimball能快速交付核心分析场景（如贷款逾期分析、客户画像、交易流水分析）；同时在大屏级和全行级指标定义上引入Inmon的统一指标管理体系，确保跨部门数据口径一致。

**四层架构**：
```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  APP层   │ ←  │  DM层    │ ←  │  DW层    │ ←  │  ODS层   │
│ 应用接口 │    │ 数据集市 │    │ DWD+DWS  │    │ 贴源存储 │
│ 大屏报表 │    │ 按主题域 │    │ 明细+汇总│    │ 原始数据 │
│ API输出  │    │ 风控集市 │    │ 星型模型 │    │ 增量同步 │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```
- ODS层：从核心系统实时/准实时同步原始交易数据，保持与源系统表结构一致，用于数据回溯和审计
- DW层：DWD明细数据层（清洗、标准化、维度退化）+ DWS汇总数据层（按天/月汇总指标），采用Kimball星型模型
- DM层：面向特定业务域的数据集市，如风控集市（反欺诈评分模型输入）、营销集市（客户画像）、监管集市（报表报送）
- APP层：面向最终用户的数据产品，包括管理驾驶舱大屏、监管报表、自助分析平台、风控API

**踩分要点**：Inmon/Kimball对比说明理由得1分，四层架构每层职责说明各0.5分（共2分），架构图清晰得1分，混合策略说明得1分。
