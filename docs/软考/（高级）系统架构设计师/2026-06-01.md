# （高级）系统架构设计师 · 2026-06-01

## 今日核心概念

### 1. AI推理服务架构

定义：将训练好的机器学习模型部署为可在线提供预测服务的系统架构，核心目标是在满足延迟SLA的前提下最大化吞吐量与资源利用率。

关键要点：
- **两类部署模式**：同步推理（客户端阻塞等待，适合低延迟要求<100ms）；异步推理（消息队列解耦，适合批量离线场景）
- **模型服务三层拆分**：前置特征工程层（实时特征抽取）→ 模型推理层（GPU/CPU调度）→ 后处理层（业务规则融合）
- **关键优化技术**：模型量化（INT8/FP16降低显存，精度损失<1%）、动态批处理（将多个请求合并一次推理）、模型蒸馏（大模型→小模型，部署成本降低10倍）
- **高可用设计**：模型版本蓝绿发布（A/B切换零中断）、模型降级策略（GPU不足时自动降为规则引擎）
- **考试陷阱**：推理延迟瓶颈首先排查特征工程层（I/O密集），而非模型层（计算密集）

### 2. 边缘计算架构

定义：将计算能力下沉到靠近数据产生端（终端设备侧）的架构模式，与云计算形成"云-边-端"三层协同体系。

关键要点：
- **三层架构**：
  - 端层：IoT设备/摄像头，负责数据采集，计算能力极弱
  - 边缘层：边缘节点/MEC服务器，负责实时处理、本地决策，时延<10ms
  - 云层：中心云，负责模型训练、全局分析、跨区协同
- **核心优势**：低时延、减少回传带宽、数据本地化（符合隐私法规）、断网可用
- **与雾计算区别**：雾计算是边缘计算的一个子集，强调网络边缘的计算；边缘计算更强调靠近数据源侧
- **典型应用**：自动驾驶（毫秒级决策）、工业质检（本地实时图像识别）、智慧零售（门店本地人脸识别）
- **架构挑战**：边缘节点资源受限（内存<4GB）、大规模节点运维复杂、边云数据同步一致性

### 3. 知识图谱架构

定义：以图结构组织实体（Entity）、关系（Relation）、属性（Attribute）的语义知识库，核心存储结构为三元组（主体-谓词-客体）。

关键要点：
- **技术栈四层**：数据采集层（爬虫/结构化抽取）→ 知识融合层（实体对齐、消歧）→ 图存储层（Neo4j/JanusGraph/TigerGraph）→ 查询服务层（SPARQL/Cypher/Gremlin）
- **图数据库选型**：
  - Neo4j：OLTP场景，Cypher语法，社区版不支持集群
  - JanusGraph：分布式，支持HBase/Cassandra作为后端存储，适合超大规模图
  - TigerGraph：原生并行图计算，适合实时深度图分析
- **知识推理两类方法**：基于规则（Datalog，可解释性强）；基于嵌入（TransE/BERT，可发现隐式关系）
- **与关系数据库区别**：多跳关系查询（如"A的朋友的同事"）图数据库O(1)，关系数据库O(n³)
- **考试高频**：RDF三元组格式、SPARQL查询语言、实体链接vs实体对齐的区别

### 4. 事件驱动架构（EDA）深化

定义：以事件（状态变化的不可变记录）作为系统各组件通信媒介的架构模式，实现生产者与消费者的时间、空间、同步三维解耦。

关键要点：
- **事件三要素**：事件标识符（唯一ID）+ 时间戳 + 事件载荷（业务数据）
- **两种处理模式**：
  - Simple Event Processing（简单事件处理）：事件独立处理，如订单创建→发短信
  - Complex Event Processing（CEP复杂事件处理）：多事件关联分析，检测模式，如5分钟内同IP登录>10次→触发风控
- **事件溯源（Event Sourcing）**：不存储当前状态，而是存储所有历史事件，通过重放事件还原状态；优点：完整审计日志、时间旅行调试；缺点：查询复杂度高，需要配合CQRS
- **CQRS（命令查询职责分离）**：写模型（Command）和读模型（Query）分离，写库通常为事件存储，读库为物化视图
- **与消息队列区别**：消息队列是EDA的实现手段，EDA是架构模式；Kafka在EDA中既做消息通道又做事件存储（持久化）

### 5. 系统架构的可观测性设计

定义：通过系统产生的外部输出（Metrics/Logs/Traces）推断系统内部状态的能力，是分布式系统运维的核心支柱。

关键要点：
- **可观测性三支柱**：
  - **Metrics（指标）**：数值型时序数据，如CPU使用率、QPS、P99延迟；采集工具Prometheus，可视化Grafana
  - **Logs（日志）**：事件文本记录，结构化JSON日志优先；采集ELK（Elasticsearch+Logstash+Kibana）
  - **Traces（链路追踪）**：请求在分布式系统中的完整调用链，含Span树和耗时；工具Jaeger/Zipkin，遵循OpenTelemetry规范
- **黄金信号（Google SRE四大指标）**：延迟（Latency）、流量（Traffic）、错误率（Errors）、饱和度（Saturation）——**LTES口诀**
- **告警设计原则**：告警应基于SLO（服务水平目标）触发，而非单点指标阈值；避免"告警疲劳"
- **分布式追踪关键概念**：TraceID（全链路唯一标识）、SpanID（单次调用标识）、Context Propagation（跨服务传递TraceID）
- **考试高频**：三支柱各自解决什么问题的差异；OpenTelemetry vs OpenTracing vs OpenCensus的关系（前者是后两者的合并标准）

## 案例分析题

**题目**：

某省级政务数据共享平台承载200+政府部门的数据交换，日均数据交换量达5亿条。现有架构采用集中式ESB总线，所有数据均经ESB路由转发。近期出现以下问题：
1. ESB成为性能瓶颈，高峰期平均延迟超过3秒，单节点ESB崩溃导致全平台停服4小时
2. 各部门业务系统产生的数据存在实时感知需求（如公安需实时接收户籍变更事件）
3. 数据交换缺乏完整审计链路，无法追溯某条数据的完整流转过程
4. 平台需对接新建的AI智能审批系统，该系统需实时调用人口库进行身份核验，要求P99延迟<50ms

请回答：
1. 针对问题1，从架构层面提出ESB瓶颈解决方案，并说明改造思路（8分）
2. 针对问题2，应采用何种架构模式满足实时事件感知需求？说明核心组件与数据流向（6分）
3. 针对问题3，如何设计数据全链路追踪方案，实现完整审计？（6分）
4. 针对问题4，AI智能审批的身份核验场景有哪些架构设计挑战？给出解决方案（5分）

**参考答案**：

**第1题（8分）**

ESB瓶颈解决方案：**分层解耦 + 去中心化改造**

改造思路：
- **短期**：ESB水平扩展（集群部署）+ 异步化改造（同步调用改为MQ异步），减轻单点压力；ESB内部按部门/业务域做逻辑分区，避免全局热点
- **中期**：引入**事件驱动架构（EDA）**，将数据交换从"推模式（ESB主动路由）"改为"订阅模式（各部门订阅所需事件）"，消息中间件选用Kafka（高吞吐、持久化），ESB降级为仅处理同步RPC调用
- **长期**：采用**去中心化数据交换**架构：各部门保留本地数据服务，通过统一数据目录（API Gateway + 数据目录服务）完成服务发现；ESB变为轻量级API网关，只做身份认证和路由，不做业务处理
- **高可用设计**：消息中间件多副本（Kafka 3副本）+ 消费端幂等处理 + 死信队列兜底

**第2题（6分）**

采用**事件驱动架构（EDA）**：

核心组件：
- **事件生产者**：各部门业务系统（如户籍系统），在数据变更时发布领域事件到Kafka Topic
- **事件总线**：Kafka集群，按业务域分Topic（如`topic.huji.change`、`topic.shehui.change`），支持持久化回放
- **事件消费者**：公安等需实时感知的系统订阅对应Topic，实现毫秒级推送
- **事件路由规则**：通过Schema Registry统一事件格式（Avro/Protobuf），通过ACL控制跨部门订阅权限

数据流向：`户籍系统变更 → 发布事件到Kafka → 公安系统Consumer实时消费 → 触发本地业务处理`

**第3题（6分）**

全链路追踪方案（结合可观测性三支柱）：

- **链路追踪**：采用OpenTelemetry规范，数据流转的每个节点（生产→路由→消费）注入TraceID和SpanID，使用Jaeger存储和查询追踪链路
- **数据血缘**：在Kafka消息Header中携带TraceID，消费者处理时写入审计日志，包含：数据来源部门、时间戳、处理节点、目标部门
- **不可变审计日志**：基于**Event Sourcing**思想，所有数据流转事件追加写入审计专用存储（如HBase时序存储），禁止修改和删除
- **查询服务**：通过TraceID可还原数据完整流转路径；通过时间范围+数据标识可查询任意数据的生命周期

**第4题（5分）**

AI智能审批身份核验的架构挑战与方案：

挑战：
- 人口库数据量大（亿级），全量同步至AI系统不现实
- P99<50ms的延迟要求极高，跨部门网络调用存在不确定延迟
- 人口库属于敏感数据，需满足数据不出域的安全合规要求

解决方案：
- **数据不出域**：AI系统通过标准API调用人口库服务，人口库侧部署**身份核验微服务**，只返回核验结果（True/False），不返回原始数据
- **低延迟优化**：在AI系统侧部署**缓存层**（Redis），对频繁核验的身份信息缓存核验结果（TTL=5分钟）；同时人口库核验服务部署在同一数据中心内网，消除跨网延迟
- **降级方案**：核验服务超时（>30ms）时，触发降级策略（如使用本地黑名单库快速拦截高风险身份），避免全链路阻塞

**踩分要点**：

1. 第1题必须体现"架构层面"的演进思路（短期→中期→长期），仅答"加机器"只得1分；必须提到去中心化或EDA方向
2. 第2题必须画出或描述**数据流向**（生产→总线→消费），缺流向扣3分；必须提到Topic隔离和权限控制
3. 第3题关键词：**TraceID贯穿全链路**、**不可变追加写入**；如只答"写日志"不得分，须说明不可篡改机制
4. 第4题必须回答"数据不出域"合规点，否则该小题最多3分；P99延迟优化必须提到缓存，不能只说"优化网络"

## 记忆口诀

**可观测性三支柱**：
> 指标流量看Metrics，文本事件看Logs，请求链路看Traces
> **M-L-T，三驾马车**

**Google SRE黄金信号（LETS）**：
> **L**atency延迟 · **E**rrors错误 · **T**raffic流量 · **S**aturation饱和
> "LETS Go！"

**边缘计算三层**：
> 端（采集）→ 边（实时处理）→ 云（全局分析）
> "端边云，近快广"

**EDA三维解耦**：
> 时间解耦（异步）· 空间解耦（不知对方地址）· 同步解耦（不等待响应）
> "EDA解三耦，生产消费不相知"

**知识图谱三元组**：
> 主体-谓词-客体（Subject-Predicate-Object）
> "SPO一条边，万千知识织成网"

**AI推理五层优化**：
> 量化→蒸馏→批处理→缓存→降级
> "量蒸批缓降，推理五把刀"

**ESB改造演进三步**：
> 扩展（水平扩容）→ 异步（引入MQ）→ 去中心（EDA订阅）
> "扩异去，ESB三步走"
