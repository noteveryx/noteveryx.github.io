# 2026-06-24 信息系统管理工程师 学习内容

## 今日核心概念

### 1. 数字化转型与智慧城市
定义：数字化转型是利用数字技术（云计算、大数据、AI、物联网）对组织业务模式、运营流程、组织结构进行根本性重塑的过程；智慧城市是数字化转型在城市治理层面的集中体现，通过数据融合与智能决策实现城市精细化管理。

**数字化转型三阶段模型**：
- **信息化阶段**：业务流程电子化，建设信息系统替代手工操作（如OA、ERP）
- **数字化阶段**：数据驱动决策，打通系统孤岛实现数据融合，业务流程再造
- **智能化阶段**：AI赋能决策与执行，机器辅助甚至替代部分人工判断

**智慧城市四层架构**：
- 感知层：物联网传感器、摄像头、RFID采集城市运行数据
- 网络层：5G/光纤/政务外网实现数据传输
- 平台层：城市大数据中心、城市大脑、云平台进行数据汇聚与计算
- 应用层：智慧交通、智慧政务、智慧医疗、智慧安防等场景应用

关键要点：
- 数字化转型的核心不是"买系统"，而是"变模式"——技术只是手段
- 智慧城市评价指标体系（国标GB/T 36333-2018）：基础设施、治理、服务、产业、生态五大维度
- "数据中台"概念：统一数据标准、汇聚跨部门数据、对外提供数据服务接口

### 2. IT服务目录管理
定义：IT服务目录是IT部门向业务部门提供的全部服务的结构化清单，包含服务名称、服务描述、服务级别目标（SLA）、可用性要求、计费方式、服务流程等，是IT服务管理的"菜单"。

**服务目录两类结构**：
- **业务服务目录**：面向业务用户，用业务语言描述（如"邮件服务""OA审批""电子证照办理"）
- **技术服务目录**：面向IT运维，用技术语言描述（如"Exchange邮件集群""数据库主从复制""负载均衡器"）

**服务目录关键属性**：
| 属性 | 说明 | 示例 |
|------|------|------|
| 服务名称 | 业务可理解的服务标识 | 一网通办申报服务 |
| 服务分类 | 按业务域或技术域归类 | 政务服务/内部办公 |
| 服务级别（SLA） | 可用性、响应时间、解决时间 | 可用性99.9%，响应≤15分钟 |
| 计费方式 | 免费/按用户/按用量 | 按委办局用户数计费 |
| 服务状态 | 规划/试运行/正式/退役 | 正式运营 |
| 服务负责人 | 该服务的业务owner和技术owner | 业务:审批科 技术:运维组 |

关键要点：
- 服务目录是SLM（服务级别管理）的前提——没有目录就无法定义级别
- 服务目录必须经业务部门确认，不能仅由IT部门自编
- 新服务上线前须先写入目录并约定SLA，退役服务须走变更流程从目录中移除
- 服务目录与CMDB的关系：目录描述"做什么"，CMDB记录"用什么组件做的"

### 3. 项目范围管理与需求变更控制
定义：项目范围管理是确保项目包含且仅包含完成项目所需的工作；需求变更控制是通过规范流程管控项目需求的增删改，防止范围蔓延和镀金。

**范围管理六过程**：
- 规划范围管理：制定范围管理计划和需求管理计划
- 收集需求：使用访谈/问卷/焦点小组/原型法等工具获取干系人需求
- 定义范围：将需求转化为详细的项目范围说明书
- 创建WBS：将范围分解为可管理的工作包（最底层为工作包，上方为控制账户）
- 确认范围：由客户/发起人正式验收已完成的可交付成果
- 控制范围：监控范围状态，管理变更请求

**需求变更控制流程（六步法）**：
1. 提出变更请求（书面提交）
2. 变迁影响分析（评估对进度、成本、质量的影响）
3. CCB评审决策（变更控制委员会审批）
4. 更新项目文档（范围说明书、WBS、基线）
5. 执行变更（按新基线实施）
6. 验证与关闭（确认变更效果，关闭变更单）

关键要点：
- 范围蔓延（Scope Creep）：未经控制的范围渐进扩大——无声增长，最危险
- 镀金（Gold Plating）：项目团队自行添加超出需求的功能——不受控的增值
- WBS的100%规则：WBS必须覆盖100%的项目范围，不多不少
- 需求跟踪矩阵（RTM）：将每条需求关联到WBS工作包、测试用例、可交付成果

### 4. 操作系统安全加固
定义：操作系统安全加固是通过配置调整、补丁管理、权限精简、审计增强等手段，提升操作系统抵御攻击和减少漏洞风险的能力。

**安全加固四大维度**：

| 维度 | 核心措施 | 示例 |
|------|----------|------|
| 账号与权限 | 禁用默认账户、最小权限原则、密码复杂度策略 | 禁用Guest、设置8位以上密码含大小写数字 |
| 系统配置 | 关闭不必要服务/端口、文件系统权限收紧 | 关闭Telnet(23)/FTP(21)，仅保留SSH(22)/HTTPS(443) |
| 补丁与更新 | 定期安全补丁扫描与安装、内核版本管理 | 每月补丁窗口，高危漏洞72小时内修复 |
| 审计与监控 | 启用安全审计日志、设置关键操作告警 | 记录登录/权限变更/文件访问，日志异地备份 |

**Linux加固要点**：
- /etc/passwd与/etc/shadow分离存储，shadow仅root可读
- SELinux/AppArmor强制访问控制（MAC）补充自主访问控制（DAC）
- iptables/firewalld最小端口开放原则
- umask默认值设为027（组可读，其他无权限）

**Windows加固要点**：
- 组策略（GPO）统一管理安全策略：密码策略、账户锁定策略、审核策略
- 关闭SMBv1（永恒之蓝漏洞载体）、启用SMBv3加密
- BitLocker全盘加密防止物理窃取后数据泄露
- Windows Defender+防火墙高级安全规则

关键要点：
- 加固不是一次性的，是持续过程——每月评估、每季度审计
- CIS Benchmark（互联网安全中心基准）是行业通用的加固参考标准
- 加固后必须验证：功能测试确认业务未受影响
- 最小化原则贯穿所有维度：最小账户、最小服务、最小端口、最小权限

### 5. 数据生命周期管理
定义：数据生命周期管理是对数据从创建/采集、存储、使用/共享、归档到销毁的全过程进行规范化管控，确保数据在各阶段的安全性、可用性和合规性。

**数据生命周期五阶段**：
- **创建/采集**：数据录入、传感器采集、系统生成；此时须定义数据分类等级和归属责任
- **存储**：选择存储介质与架构（在线/近线/离线）、设置访问权限、加密存储敏感数据
- **使用/共享**：数据查询、分析、跨部门共享；须有授权机制、脱敏规则、审计日志
- **归档**：不再频繁访问但需保留的数据迁移至低成本存储；归档策略与法规保留期限对齐
- **销毁**：过期数据安全删除；物理介质须粉碎/消磁，逻辑删除须不可恢复

**数据分类分级（数据安全法要求）**：
| 分类 | 说明 | 保护要求 |
|------|------|----------|
| 一般数据 | 公开信息、内部通知 | 基本访问控制 |
| 重要数据 | 业务运营数据、统计报表 | 加密存储、权限管控 |
| 核心数据 | 涉密数据、个人敏感信息 | 最强加密、严格审计、离网存储 |

关键要点：
- 数据安全法第21条：国家建立数据分类分级保护制度
- 个人信息保护法：处理个人信息须遵循最小必要原则，保留期限到期必须删除
- 归档策略与法规保留期对齐：财务数据保留10年、医疗数据保留30年、政务审批数据保留15年
- 销毁须可举证：保留销毁记录（销毁时间、方式、执行人、监督人），应对审计核查

## 案例分析题

**题目**：

某市启动"智慧城市"建设，依托城市大数据中心汇聚各委办局数据，打造"一网通办"政务服务平台。项目建设半年后，出现以下问题：

**问题1**：城市大数据中心已完成基础设施建设和数据汇聚，但各委办局反映"数据用了没用处"。IT部门发布了业务服务目录和技术服务目录，但业务部门表示"看不懂技术目录"，IT部门表示"业务目录不够详细无法运维"。服务目录中仅标注"可用性99.9%"，未约定响应时间和解决时间。

**问题2**：项目建设过程中，业务部门先后提出12次需求变更，其中5次未经任何审批直接交给开发团队实施，3次由项目经理自行决定接受，其余4次提交了变更单但CCB从未召开评审会议。最终项目实际交付范围超出立项范围30%，预算超支40%，交付延期3个月。

**问题3**：安全审计发现：大数据中心8台Linux服务器中，3台仍开放Telnet端口，2台root密码为默认值"admin123"，1台未安装最近6个月的安全补丁。4台Windows服务器中，2台仍启用SMBv1协议，1台Guest账户处于启用状态。审计人员要求立即整改，运维组长称"改了怕业务停"。

**问题4**：大数据中心汇聚了全市600万居民的个人数据（身份证号、手机号、住址、社保信息）。数据共享时，某委办局要求获取全量居民数据做统计分析，但数据中心仅做了字段级脱敏（隐藏身份证中间4位），手机号和住址未脱敏直接提供。数据销毁环节，对过期的测试数据仅做了DELETE操作，未验证是否可恢复。

请分别回答以下问题：

（1）针对问题1，请指出服务目录管理存在的三个问题，并说明业务服务目录和技术服务目录的正确关系。

（2）针对问题2，请分析该项目在范围管理和变更控制方面的四个主要缺陷，并提出改进措施。

（3）针对问题3，请按照操作系统安全加固四大维度，逐项指出每台服务器存在的问题，并给出"不影响业务"的加固实施策略。

（4）针对问题4，请从数据生命周期管理的创建、使用、归档、销毁四个阶段分别指出数据安全管理缺陷，并提出改进方案。

**参考答案**：

（1）服务目录管理三个问题：
- 问题一：两套目录割裂——业务目录与技术目录缺乏映射关系，业务人员看不懂技术描述，运维人员觉得业务描述不够具体
- 问题二：SLA定义不完整——仅约定可用性99.9%，缺少响应时间（如≤15分钟）和解决时间（如≤4小时）等关键指标
- 问题三：目录未经业务确认——IT部门单方面编制，未与业务部门共同评审确认

业务服务目录与技术服务目录的正确关系：
- 业务目录面向业务用户，技术服务目录面向IT运维，两者必须建立映射对应关系
- 一个业务服务（如"一网通办申报"）对应多个技术服务（如"Web应用集群""数据库服务""负载均衡"）
- 业务目录的SLA驱动技术目录的SLA：业务要求99.9%可用性→分解为每个技术组件的可用性目标
- 目录必须由业务和IT共同确认，IT不能"自编自用"

（2）范围管理与变更控制四个缺陷：

| 缺陷 | 表现 | 改进措施 |
|------|------|----------|
| 未建立范围基线 | 变更随意，无参照标准 | 项目立项后立即冻结范围基线（范围说明书+WBS） |
| 变更流程缺失 | 5次变更未经任何审批 | 所有变更必须书面提交，口头变更一律拒绝 |
| CCB未运作 | 4次变更单提交但CCB从未开会 | 建立CCB运作机制：固定成员、定期会议、紧急变更24小时内评审 |
| 项目经理越权 | 3次变更由PM自行决定 | PM无权单方面接受变更，仅负责评估影响后提交CCB |
| 范围蔓延+镀金 | 范围超30%、预算超40% | 范围控制：用需求跟踪矩阵监控每条需求的实现状态 |

改进措施：
- 制定需求管理计划，明确变更分类（紧急/常规/重大）和审批权限
- CCB组成：项目经理+业务代表+技术代表+质量代表，重大变更需发起人参与
- 所有已批准变更更新范围基线、WBS、进度计划和成本预算
- 引入需求跟踪矩阵（RTM），关联需求→WBS→测试→交付

（3）操作系统安全加固逐项分析与实施策略：

账号与权限维度：
- 2台Linux root密码为默认值"admin123" → 立即修改为复杂密码（8位以上含大小写数字特殊字符），禁用root远程登录，创建普通管理账户+sudo授权
- 1台Windows Guest账户启用 → 通过组策略禁用Guest账户（不影响业务，Guest本就无人正常使用）

系统配置维度：
- 3台Linux开放Telnet端口 → 关闭Telnet服务，统一使用SSH（可设置SSH仅允许密钥认证，比密码更安全，不影响正常运维）
- 2台Windows启用SMBv1 → 禁用SMBv1，启用SMBv3（永恒之蓝漏洞修复，现代系统均支持SMBv3，不影响正常文件共享）

补丁与更新维度：
- 1台Linux 6个月未打补丁 → 在补丁窗口（非业务高峰时段）分批安装安全补丁，先在测试环境验证再上线

审计与监控维度：
- 所有8台服务器均未提到审计日志 → 启用安全审计日志（Linux: auditd，Windows: 安全日志策略），日志异地备份

"不影响业务"的加固策略：
- 分批实施：每周加固1-2台，先非核心后核心
- 变更窗口：所有加固操作在周末凌晨低峰时段执行
- 回退预案：每台服务器加固前备份配置，发现问题5分钟内回退
- 功能验证：加固后运行业务功能测试脚本确认无影响

（4）数据生命周期管理四阶段缺陷：

| 阶段 | 缺陷 | 改进方案 |
|------|------|----------|
| 创建/采集 | 未在数据创建阶段定义分类分级和归属责任 | 数据入库前须完成分类分级标注，指定数据责任部门和责任人 |
| 使用/共享 | 脱敏不完整——身份证号仅隐藏中间4位，手机号和住址未脱敏直接提供 | 建立脱敏规则库：身份证全号脱敏/手机号仅保留前3后4/住址模糊化到街道级；按需求方业务目的提供最小必要字段 |
| 归档 | 未提及归档策略和法规保留期 | 建立数据保留策略：政务审批数据保留15年/社保数据保留30年/统计分析数据到期后归档至离线存储 |
| 销毁 | DELETE操作不等于安全销毁，逻辑删除后数据可恢复 | 物理介质：粉碎/消磁；逻辑数据：加密后删除密钥（密码学销毁）；保留销毁举证记录 |

补充改进方案：
- 数据共享审批机制：申请方提交数据使用申请，数据责任部门审批同意后方可提供
- 脱敏分级原则：统计分析需求提供聚合数据（不提供个体明细），个体查询仅返回必要字段
- 销毁举证记录格式：销毁时间+销毁方式+执行人+监督人+数据类别+数量，留存备查至少5年

**踩分要点**：
- （1）必须明确"两套目录必须建立映射关系"这个核心观点，仅说"业务目录和技术目录不同"不给满分。提到"业务SLA驱动技术SLA"是加分项
- （2）必须区分"范围蔓延"和"镀金"两个概念，必须写出CCB的角色和PM不能越权。提到需求跟踪矩阵RTM加分
- （3）必须按四大维度逐项分析，笼统说"加固服务器"不给分。给出"不影响业务"的具体实施策略（分批、变更窗口、回退）是加分项
- （4）必须按数据生命周期四阶段逐一指出缺陷，笼统说"加强数据安全"不给分。提到"密码学销毁"和"销毁举证记录"两项是加分项

## 记忆口诀

**数字化转型三阶段**：
"信数智，三级跳"
信息化→数字化→智能化，对应：电子化→数据化→AI化。

**智慧城市四层架构**：
"感网平用，城脑四方"
感知层→网络层→平台层→应用层。

**服务目录两套表**：
"业务说做什么，技术说怎么做的"
业务目录对用户，技术目录对运维，一表映射两表联动。

**范围管理六过程**：
"规收定创确控"
规划→收集→定义→创建(WBS)→确认→控制。

**变更控制六步法**：
"提评审更执验"
提出→评估→评审(CCB)→更新文档→执行→验证关闭。

**OS加固四维度**：
"账配补审"
账号权限→系统配置→补丁更新→审计监控。

**数据生命周期五阶段**：
"创存用归销"
创建→存储→使用→归档→销毁，谐音：创存用归销（存了有用的归零）。

**范围蔓延vs镀金**：
"蔓延是外部偷加，镀金是内部自加"
蔓延：客户/干系人偷偷加需求；镀金：团队自己加功能讨好客户。两者都不受控。
