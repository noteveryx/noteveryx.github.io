## 今日核心概念
### 1. 信息系统项目管理十大知识领域（PMBOK）
定义：项目管理的核心框架，十大领域分为五大过程组（启动/规划/执行/监控/收尾）和十大知识领域（整合/范围/进度/成本/质量/资源/沟通/风险/采购/干系人），是项目管理知识体系的骨架。
关键要点：
1. 十大领域对应47个过程，每个过程都有输入、工具与技术、输出（ITTO）
2. 核心三角：范围、进度、成本三重约束，质量贯穿其中
3. 风险=威胁+机会，消极风险应对策略：规避/转移/减轻/接受/上报
4. 干系人参与级别：不了解/抵制/中立/支持/领导五级，需动态管理
5. 事业环境因素EEF与组织过程资产OPA是项目管理的重要输入

### 2. 系统分析与设计方法（结构化与面向对象）
定义：系统开发的两大方法论，结构化方法采用自顶向下分解（DFD数据流图+ER图），面向对象方法采用UML统一建模（用例图/类图/时序图/状态图），各有适用场景。
关键要点：
1. 结构化三要素：数据流图DFD（加工/数据流/数据存储/外部实体）+数据字典+结构化语言
2. 面向对象三大特性：封装、继承、多态，五大原则SOLID（单一/开闭/里氏/接口/依赖倒置）
3. UML九种图：用例图（功能视图）、类图（静态结构）、时序图/协作图（动态交互）、状态图（生命周期）、活动图（业务流程）
4. 结构化适合数据处理类系统，OO适合业务复杂、需求多变的系统
5. 现代开发多采用OO+敏捷+领域驱动设计DDD组合

### 3. 信息系统的可靠性与容错设计
定义：通过冗余、检错、纠错等技术手段，使系统在部分组件故障时仍能持续提供规定功能的设计方法，是信息系统高可用性的核心技术。
关键要点：
1. 三大策略：避错（设计阶段消除故障）、容错（运行时容忍故障）、排错（故障后快速恢复）
2. 冗余类型：硬件冗余（双机热备/集群）、软件冗余（N版本程序/恢复块）、信息冗余（校验码/CRC）
3. 容错技术：N+1/N+2冗余、双机互备（Active-Standby）、双机双工（Active-Active）
4. 可靠性指标：MTBF平均故障间隔↑、MTTR平均修复时间↓、可用性=MTBF/(MTBF+MTTR)
5. 金融/政务核心系统要求可用性≥99.99%（年停机≤52分钟）

### 4. 数据库设计与ER建模
定义：数据库设计是从需求到物理存储的完整过程，包括概念设计（ER图）、逻辑设计（关系模式）、物理设计（索引/分区）三个阶段，ER建模是概念设计的核心方法。
关键要点：
1. ER三要素：实体（矩形）、属性（椭圆）、联系（菱形），联系类型1:1/1:N/M:N
2. 设计六步骤：需求分析→概念设计（ER图）→逻辑设计（关系模式）→物理设计→实施→运行维护
3. 三大范式：1NF（原子性）、2NF（消除部分依赖）、3NF（消除传递依赖）
4. 反范式化：适当冗余换性能，读多写少场景常用
5. 主键设计原则：稳定不变、唯一非空、优先自增整型或UUID

### 5. 信息系统战略规划与信息化规划方法
定义：信息化规划是对组织未来3-5年信息化建设目标、架构、实施路径的系统性规划，方法包括BSP企业系统规划法、CSF关键成功因素法、SST战略目标转移法。
关键要点：
1. BSP四阶段：立项→定义业务过程→定义数据类→定义信息系统架构（U/C矩阵）
2. CSF法：识别企业关键成功因素（3-6个），围绕CSF规划信息系统
3. SST法：从企业战略目标→信息化目标→IT项目，自顶向下分解
4. 三种方法对比：BSP适合大型企业全盘规划，CSF适合聚焦关键业务，SST适合战略驱动型组织
5. 规划输出：信息化战略报告+项目清单+投资估算+实施路线图

## 案例分析题
**题目**：
某市政务服务局启动"一网通办"平台升级改造项目，投资1200万，工期12个月，由信息中心主任老张担任项目经理，聘请A公司承建、B公司负责监理。项目实施半年后出现严重问题：
1. 范围管理：开发过程中A公司不断提出"增值功能"建议，局长现场视察后口头要求新增电子证照、信用核查等5项功能，均未走变更流程直接开发
2. 进度管理：原计划12个月的项目，已延期4个月仍未完成核心模块开发，老张未做关键路径分析
3. 质量与可靠性：核心数据库采用单机部署未做主备切换，2026年5月因硬盘故障宕机8小时，丢失当日申报数据
4. 监理失职：B公司监理工程师从未参加需求评审和设计评审，月报仅写"项目进展正常"
5. 战略规划：该项目未纳入全局信息化规划，与已建成的电子证照库、信用信息系统数据标准不统一

请结合项目管理十大知识领域，分析该项目存在的主要问题，并给出整改方案。

**参考答案**：
一、范围管理与变更控制严重缺失
问题：局长口头要求直接开发，违反范围管理与变更控制流程，典型"镀金"和"范围蔓延"。
整改：
1. 立即启动变更控制流程，对5项新增功能补办变更申请单
2. 变更评估影响范围（成本/进度/质量），提交CCB变更控制委员会审批
3. 价值评估：仅保留与核心业务强相关的高价值功能，砍掉非必要需求
4. 建立变更管理信息系统，所有变更留痕可追溯

二、进度管理与关键路径失控
问题：延期4个月未做关键路径分析，缺乏挣值管理与进度压缩手段。
整改：
1. 立即采用关键路径法CPM重新梳理网络图，识别关键路径与浮动时间
2. 引入挣值管理EVM，计算PV/EV/AC、CPI（成本绩效指数）/SPI（进度绩效指数）
3. 采用进度压缩：赶工（增加资源）/快速跟进（并行活动）缩短工期
4. 每周输出项目状态报告，月度里程碑评审

三、可靠性与容错设计严重不足
问题：核心数据库单机部署无冗余，违反政务核心系统高可用要求，是典型的可靠性设计缺陷。
整改：
1. 立即搭建主备双机热备（Active-Standby）+共享存储或主主双活
2. 实施数据备份策略：每日全量+实时增量+WAL日志，RPO≤5分钟、RTO≤30分钟
3. 配置自动故障切换（Keepalived/Pacemaker）+定期切换演练
4. 故障丢失数据通过binlog从备份恢复，并对业务受损方致歉补偿

四、监理与质量保证形同虚设
问题：监理未参加关键评审，月报流于形式，违反监理工作三阶段（事前/事中/事后）控制要求。
整改：
1. 监理立即介入整改，重新审查需求规格说明书和设计文档
2. 强制要求监理参加需求评审、设计评审、测试评审"四审"环节
3. 实行旁站监理，对关键节点（如数据库切换）现场监督
4. 月报须含数据：进度偏差、质量问题、变更情况、风险预警，否则扣监理费

五、信息化战略规划与数据治理缺失
问题：项目未纳入统一规划，造成数据标准不统一、系统孤岛。
整改：
1. 立项必须先做信息化规划对标，确保与全局架构一致
2. 制定统一的数据标准（参照GB/T 21063/数据元规范）
3. 建立主数据管理MDM，电子证照、信用信息等基础数据统一编码
4. 项目验收须通过CIO或信息化领导小组审批，确保战略一致性

**踩分要点**：
1. 项目管理：十大知识领域对应问题（范围/进度/成本/质量/风险）+CCB变更控制+CPM关键路径+挣值EVM三参数
2. 变更控制：CCB+变更申请单+影响分析+范围蔓延vs镀金区分
3. 容错设计：双机热备/双活+RAID+备份策略+RPO/RTO指标
4. 监理工作：四审+三管+旁站监理+月报数据化
5. 规划方法：BSP/CSF/SST三方法对比+U/C矩阵+主数据管理MDM
6. 关键提醒：答题必须对照PMBOK十大领域框架逐项分析，不要遗漏

## 记忆口诀
**PMBOK十大领域口诀**：整范进成质、资沟风采干（整合、范围、进度、成本、质量、资源、沟通、风险、采购、干系人）

**结构化三要素**：DFD数据流图、数据字典DD、结构化语言，自顶向下分解

**OO三大特性封装继承多态，SOLID五原则单一开闭里氏接口依赖倒置**

**容错三策略**：避错设计阶段消、容错运行阶段忍、排错故障后快恢

**冗余三类型**：硬件冗余双机集群、软件冗余N版本、信息冗余校验码

**可靠性两指标**：MTBF越长越好、MTTR越短越好、可用性=MTBF/(MTBF+MTTR)

**ER三要素**：实体矩形属性椭圆联系菱形，联系三类1:1/1:N/M:N

**三范式递进**：1NF原子不可分、2NF消除部分依赖、3NF消除传递依赖

**信息化规划三方法**：BSP企业系统规划法、CSF关键成功因素法、SST战略目标转移法
