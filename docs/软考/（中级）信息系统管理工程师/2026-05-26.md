## 今日核心概念

### 1. 变更管理
定义：变更管理是 ITIL 服务运营中的核心流程，旨在以标准化、受控的方式管理所有对 IT 基础设施和服务的变更，最大限度降低变更带来的风险和对业务的影响。

关键要点：
- 变更类型：标准变更（低风险、已验证、流程化审批，如重置密码）、紧急变更（因业务紧急需要而快速实施的变更，事后补充完整流程）、普通变更（需经过完整评估审批流程的变更）。
- 变更管理流程：提出变更请求（RFC）→ 变更分类与优先级评估 → 变更影响和风险评估 → 变更顾问委员会（CAB）审批 → 变更实施 → 变更评审（回顾是否达到预期效果）→ 变更关闭。
- CAB（变更顾问委员会）：由 IT 经理、业务代表、技术专家组成，负责评审和审批变更请求。紧急变更由 ECAB（紧急变更顾问委员会）快速审批。
- 回退计划：每个变更实施前必须制定回退方案，当变更失败或产生未预期影响时能将系统恢复到变更前状态。没有回退计划的变更不得实施。
- 变更窗口：选择对业务影响最小的时间段实施变更，通常为非工作时间。紧急变更除外。

### 2. 问题管理
定义：问题管理是识别和分析事件根本原因、防止事件重复发生的流程。问题与事件的区别：事件是 IT 服务中断或质量下降的表象，问题是导致事件重复发生的根本原因。

关键要点：
- 问题管理目标：一是主动识别问题根源，消除引起事件的根本原因；二是防止已解决事件再次发生。
- 问题管理流程：问题识别与记录 → 问题分类 → 问题调查与诊断（找到根本原因）→ 提出规避措施或变更请求（RFC）→ 评估规避措施效果 → 关闭问题。
- 主动问题管理与被动问题管理：被动问题管理是在事件重复发生或影响严重时触发调查；主动问题管理是在事件发生前通过趋势分析、容量监控等手段主动识别潜在问题。
- 已知错误数据库（KEDB）：记录已识别的根本原因及其规避措施，当事件发生时可在 KEDB 中快速匹配解决方案，大幅缩短事件恢复时间。
- 问题管理与事件管理的关系：事件管理关注"快速恢复服务"，问题管理关注"找到并消除根因"，两者互补。问题管理通常由事件触发。

### 3. 网络管理与 SNMP 协议
定义：网络管理是对网络设备、链路和服务的运行状态进行监控、配置、故障诊断和安全管理的综合活动。SNMP（简单网络管理协议）是互联网最广泛使用的网络管理标准协议。

关键要点：
- 网络管理五大功能域（FCAPS）：故障管理（Fault，检测和定位故障）、配置管理（Configuration，管理设备配置）、计费管理（Accounting，统计资源使用量）、性能管理（Performance，监控网络性能指标）、安全管理（Security，控制网络访问）。
- SNMP 体系结构：管理工作站（NMS，运行管理软件的设备）、被管设备（运行 SNMP Agent 的网络设备如路由器/交换机）、管理信息库（MIB，以树形结构存储被管对象的参数）。
- SNMP 版本对比：SNMPv1（明文传输，安全性差）、SNMPv2c（增加了 GetBulk 操作，提升批量查询效率，但仍使用明文 community 认证）、SNMPv3（增加了 USM 安全模型，支持认证和加密，安全性最高）。
- SNMP 操作类型：Get（获取单个参数）、GetNext（获取下一个参数，用于遍历 MIB 树）、GetBulk（批量获取，v2c 新增）、Set（设置参数值）、Trap（代理主动向 NMS 发送告警）。
- MIB-Ⅱ 标准分组：system（系统信息）、interfaces（接口信息）、ip（IP 层信息）、icmp、tcp、udp、snmp 等组。每个被管对象用 OID（对象标识符）唯一标识。

### 4. 容量管理
定义：容量管理是确保 IT 资源（计算能力、存储空间、网络带宽等）在当前和未来都能满足业务需求的过程，目标是实现资源供需的动态平衡。

关键要点：
- 容量管理三个子流程：业务容量管理（从业务角度预测未来资源需求）、服务容量管理（监控和管理特定 IT 服务的资源使用）、资源容量管理（管理具体硬件和软件资源的使用情况）。
- 容量管理活动：监控当前资源使用率（CPU、内存、磁盘、带宽）、分析使用趋势、预测未来需求、制定容量计划、在资源不足前提前扩容。
- 关键指标：资源利用率（通常 CPU 持续超过 70% 需要扩容）、响应时间、吞吐量、并发用户数。容量基线是历史性能数据的基准线。
- 容量管理与性能管理的关系：性能管理关注实时监控和调优，容量管理关注中长期规划和预测。性能问题是容量管理的输入。
- 容量计划：容量管理的核心产出，通常覆盖 6-12 个月的周期，内容包括当前资源状况评估、业务增长预测、资源缺口分析、扩容方案和时间表、成本预算。

### 5. 业务连续性与灾难恢复
定义：业务连续性管理（BCM）是组织为应对可能中断业务运行的灾难事件而建立的完整管理体系；灾难恢复（DR）是 BCM 的技术子集，侧重于 IT 系统的恢复能力。

关键要点：
- 相关概念辨析：RTO（恢复时间目标，业务中断后可接受的最大恢复时间）、RPO（恢复点目标，可接受的最大数据丢失量，对应时间长度）、MTBF（平均故障间隔时间，衡量可靠性）、MTTR（平均修复时间，衡量可维护性）。
- 灾难恢复等级（国标 GB/T 20988）：第 1 级（基本支持，手动恢复、数据备份）、第 2 级（备用场地支持）、第 3 级（电子传输和部分设备支持）、第 4 级（电子传输和完整设备支持）、第 5 级（实时数据传输和完整设备支持）、第 6 级（数据零丢失和远程集群支持）。等级越高，RTO 和 RPO 越小。
- 灾难恢复策略：数据备份策略（完全备份、增量备份、差异备份）、备份介质管理、异地容灾（同城容灾、异地容灾、两地三中心）。
- 业务连续性计划（BCP）包含：业务影响分析（BIA，识别关键业务流程及其优先级）、风险评估、恢复策略制定、应急响应计划、灾难恢复计划、测试与演练。
- 演练类型：桌面演练（讨论推演）、模拟演练（模拟场景操作测试）、全面演练（实际切换到灾备环境运行）。定期演练是确保 BCP 有效性的必要手段。

## 案例分析题

**题目**：

某省级政务数据中心承载 120 多个省级部门的业务系统运行。数据中心采用 ITIL 框架进行 IT 服务管理，建立了事件管理、问题管理、变更管理、配置管理等核心流程。

近期该数据中心连续发生以下情况：

（1）运维团队收到多条关于"核心交换机端口流量异常"的告警，值班工程师仅将该告警标记为"已处理"并关闭了工单。三天后，核心交换机因过载导致整个政务网络瘫痪 2 小时，影响了 50 多个部门的正常业务。

（2）为解决网络拥堵问题，运维团队决定在周中工作时间直接更换核心交换机。更换过程中配置参数出现错误，导致多个关键业务系统中断 4 小时，且原配置未提前备份，恢复困难。

（3）事后复盘发现，近半年内数据中心的服务器 CPU 利用率持续从 45% 上升到 82%，磁盘 I/O 响应时间从 5ms 上升到 50ms，但运维团队未对此趋势进行过分析或预警。

**问题**：

1. 请分析情况（1）中运维工程师的处理方式存在哪些问题，应如何正确处理？
2. 请分析情况（2）中运维团队的变更管理流程存在哪些缺陷？
3. 请从容量管理的角度分析情况（3）中存在的问题，并给出改进措施。

**参考答案**：

1. 情况（1）中运维工程师的处理问题及正确做法：

问题分析：
- 仅将告警标记为"已处理"而未进行根本原因排查，属于治标不治本——这违反了事件管理与问题管理协同的原则；
- "流量异常"告警反映的是潜在故障征兆，但工程师未将重复性或异常事件升级为问题进行深入分析；
- 缺少主动问题管理意识，未通过趋势分析发现潜在风险。

正确做法：
- 收到异常告警后，首先执行事件处理（检查告警详情、初步排查原因），记录完整的排查过程和结果；
- 当发现异常告警无法通过简单操作消除时，应创建问题记录，提交问题管理流程进行深度调查；
- 将问题线索录入已知错误数据库（KEDB），形成知识积累；
- 对核心网络设备的异常告警应提高处理优先级，必要时上报管理层。

2. 情况（2）中变更管理的缺陷：

缺陷分析：
- 未经过变更管理流程：更换核心交换机属于高风险变更，必须提交变更请求（RFC），经过 CAB 评审审批后方可实施；
- 未选择合适的变更窗口：核心设备更换应在业务低谷期（周末或夜间）进行，周中工作时间实施对业务影响过大；
- 未制定回退计划：变更前必须制定详细的回退方案，但运维团队未准备任何回退措施；
- 未提前备份配置：更换前必须备份原设备配置，这是网络变更的基本操作规范，缺失导致恢复困难；
- 未进行风险评估：未评估变更对关联业务系统的影响范围和程度。

改进措施：
- 所有非标准变更必须提交 RFC，经过 CAB 评审后方可实施；
- 核心网络设备变更必须制定完整的实施方案和回退方案，并在 CMDB 中更新配置信息；
- 变更实施前在测试环境验证，确认配置正确后再在生产环境执行；
- 建立变更评审制度，对高风险变更进行技术评审和管理审批双重把关。

3. 情况（3）中容量管理的问题及改进措施：

问题分析：
- 缺少容量监控机制：CPU 和磁盘 I/O 指标持续恶化半年未被及时发现，说明容量监控缺失或未设置告警阈值；
- 缺少趋势分析：容量管理的核心活动之一是分析资源使用趋势并预测未来需求，运维团队未执行这一活动；
- 未制定容量计划：没有基于当前资源使用状况和业务增长预测进行容量规划；
- 缺少容量基线管理：未建立性能基线，无法判断当前指标是否在合理范围内。

改进措施：
- 建立全面的容量监控体系，对 CPU、内存、磁盘、网络带宽等关键资源设置分级告警阈值（如 CPU 超过 60% 预警、超过 80% 告警）；
- 定期（至少每月）生成容量分析报告，分析资源使用趋势，预测未来 3-6 个月的资源需求；
- 制定年度容量计划，根据业务增长预测提前规划扩容方案和预算；
- 建立容量基线，将历史正常运行的性能数据作为参考基准；
- 对已超阈值资源立即启动扩容流程（当前 CPU 82%、磁盘 I/O 50ms 已属严重不足），优先解决最紧迫的瓶颈。

**踩分要点**：
- 第 1 问关键得分点：指出"仅标记已处理未排查根因"的问题、提出事件应升级为问题、提及已知错误数据库（KEDB）、强调主动问题管理。
- 第 2 问关键得分点：未提交 RFC 和 CAB 审批（必答）、未选择合理变更窗口、未制定回退计划、未提前备份配置、未做风险评估（至少答出 3 点）。
- 第 3 问关键得分点：指出监控缺失/未设阈值、未做趋势分析、未制定容量计划（必答三点），提出具体的阈值设置建议和扩容措施。

## 记忆口诀

**变更管理三变类：**
标准变低走流程，紧急变更快审批，普通变更全评估，回退计划不能少。

**问题事件要分清：**
事件解决快恢复，问题管理找根因，已知错误建库中，被动主动两结合。

**SNMP 网管 FCAPS：**
故配计性安——故障配置和计费，性能安全五大块。v1 明文 v2 增，v3 加密最安全。

**容量管理三子流：**
业务服务加资源，监控趋势和预测，基线之上看变化，七八十线要预警。

**RTO 与 RPO 记忆法：**
RTO 是时间目标——多久能恢复；RPO 是数据目标——能丢多少数据。RTO 管恢复速度，RPO 管数据损失。
