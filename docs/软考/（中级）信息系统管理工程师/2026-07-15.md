## 今日核心概念
### 1. 变更管理（Change Management）
定义：从变更请求提出、评估、批准、实施到后评估的全生命周期闭环管理流程，目标是最大化变更收益、最小化变更风险，确保变更在受控环境下进行。
关键要点：
1. 变更类型：标准变更（低风险预授权）/紧急变更（紧急流程快批）/重大变更（CAB变更咨询委员会审批）
2. 三阶段：变更请求RFC→评估审批CAB→实施后评估PIR
3. 变更窗口：避开业务高峰期，重大变更须有回滚方案
4. 变更成功率=成功变更数/总变更数，事故率=变更引发事故数/总变更数

### 2. 软件质量保证SQA与质量控制QC
定义：SQA是过程性预防活动（建立标准、培训、审计），QC是产品性检查活动（评审、测试），两者共同保障软件质量。
关键要点：
1. SQA属于管理职能贯穿全过程，QC属于技术职能集中在测试阶段
2. 质量模型六特性：功能性、可靠性、易用性、效率、维护性、可移植性（ISO/IEC 9126）
3. 质量保证手段：技术评审、走查、审查、测试、过程检查
4. 缺陷分布遵循帕累托二八原则，80%缺陷集中在20%模块

### 3. 数据库连接池与性能优化
定义：连接池是预先创建并维护一定数量数据库连接的缓冲池技术，避免频繁创建/销毁连接带来的性能开销。
关键要点：
1. 核心参数：初始连接数、最小空闲、最大连接数、连接超时、获取超时
2. 主流连接池：Druid（阿里带监控）、HikariCP（高性能）、C3P0、DBCP
3. 常见问题：连接泄漏（未close）、连接风暴（突发并发）、慢SQL占用连接不释放
4. 性能调优三板斧：慢SQL定位（慢日志/explain）+索引优化+连接池参数调优

### 4. 信息系统运维指标体系与KPI设计
定义：通过建立量化指标体系衡量运维工作质量与服务水平，为运维改进和绩效考核提供数据支撑。
关键要点：
1. 五大维度：可用性（MTBF平均故障间隔/MTTR平均恢复时间）、性能（响应时间/吞吐量）、稳定性（事故数）、服务（满意度/SLA达成率）、成本（人均运维成本）
2. KPI设计SMART原则：具体、可衡量、可达成、相关、有时限
3. SLA达成率=（实际服务时间-SLA停机时间）/实际服务时间×100%
4. 指标需配套基线值、目标值、采集周期、责任人四要素

### 5. 信息系统退役与数据销毁
定义：信息系统到达生命周期终点或业务下线时，按照规范流程完成数据迁移、系统关停、资产处置和数据销毁的全过程管理。
关键要点：
1. 退役四阶段：评估决策→数据迁移归档→系统关停→资产与数据销毁
2. 数据销毁三级：软销毁（覆写重写）/硬销毁（消磁）/物理销毁（粉碎熔炼），涉密数据须物理销毁
3. 数据迁移校验：抽样比对+全量校验双保险，确保业务连续性
4. 退役后须保留完整档案：退役申请、审批、数据备份、设备去向、销毁证明

## 案例分析题
**题目**：
某市政务办"互联网+政务服务"平台上线运行三年，近期进行运维管理专项审计，发现以下问题：
1. 变更管理方面：系统升级和功能调整均未走变更申请流程，开发人员直接通过FTP上传部署包到生产环境，统计一年内发生47次变更，仅3次填写变更单
2. 数据库运维方面：核心业务数据库连接数设置为最大200，实际高峰期经常出现"获取连接超时"错误，开发人员反馈SQL慢查询较多
3. 运维指标方面：信息中心从未建立量化KPI，仅凭"系统运行正常"作为评价依据，SLA合同连续两年未做达成率统计
4. 系统退役方面：去年下线的老OA系统硬盘未做销毁处理直接当二手设备出售，硬盘中包含历年公文流转数据
5. 软件质量方面：新功能上线前仅做开发自测，无独立测试环节，上线后平均每月因程序缺陷引发2-3次事故

请结合ITIL运维管理知识，分析该政务平台运维存在的主要问题，并提出整改建议。

**参考答案**：
一、变更管理失控（最严重问题）
问题：47次变更仅3次走流程，属于严重变更管理失控，违反ITIL变更管理规范，重大故障风险极高。
整改：
1. 立即上线变更管理流程，所有变更必须提交RFC（变更请求单）
2. 区分标准变更、紧急变更、重大变更，建立CAB变更咨询委员会
3. 禁止FTP直连生产，开发-测试-生产三环境分离，通过CI/CD工具部署
4. 设立变更窗口期，重大变更须有回滚预案并做变更后评估PIR

二、数据库性能与连接池优化
问题：连接池最大200但高峰期仍超时，根因是慢SQL长期占用连接不释放，连接池成为系统瓶颈。
整改：
1. 开启慢SQL日志（slow_query_log），定位Top10慢查询
2. 通过explain分析执行计划，针对性加索引或改写SQL
3. 引入Druid连接池，开启连接泄漏检测（removeAbandoned=true）
4. 设置连接获取超时（maxWait）和合理超时时间，避免线程长时间阻塞

三、建立量化运维KPI与SLA达成率统计
问题：缺少量化评价体系，SLA名存实亡，运维工作无法度量也无法改进。
整改：
1. 建立五维KPI：可用性（99.9%）、MTTR（≤30分钟）、月均事故数（≤1次）、用户满意度（≥90分）、SLA达成率
2. 每月输出运维月报，包含指标完成情况、趋势分析、改进措施
3. 按SLA合同条款执行奖惩，建立服务台工单闭环跟踪

四、老旧系统数据销毁与退役规范
问题：未做数据销毁即出售硬盘，违反《数据安全法》《个人信息保护法》及等级保护要求，存在数据泄露重大风险。
整改：
1. 立即追回已出售硬盘，组织专业机构做数据恢复检测
2. 建立退役管理制度：评估→数据迁移→系统关停→数据销毁→资产处置五步流程
3. 涉密/敏感数据须做物理销毁（消磁或粉碎），保留销毁证明
4. 退役全过程形成档案保存至少3年备查

五、建立独立SQA与测试体系
问题：缺乏独立测试和SQA过程，质量完全依赖开发自测，缺陷逃逸率高。
整改：
1. 设立专职测试岗，新功能上线必须经过单元测试、集成测试、系统测试三轮
2. 建立SQA过程审计制度，技术评审纳入开发流程强制环节
3. 推行缺陷预防：用柏拉图分析缺陷分布，重点模块重点测试
4. 引入自动化测试框架（Selenium/JUnit），提升回归测试效率

**踩分要点**：
1. 变更管理：RFC/CAB/变更窗口/回滚方案/CI/CD是标准踩分点
2. 数据库调优：慢SQL日志+explain+索引+连接池参数+Druid/HikariCP必答
3. KPI设计：五维度（可用/性能/稳定/服务/成本）+SLA达成率+MTBF/MTTR公式
4. 数据销毁：三级销毁方式（软/硬/物理）+《数据安全法》依据+退役五步流程
5. 软件质量：SQA过程预防+QC产品检查+独立测试+ISO/IEC 9126六特性+柏拉图

## 记忆口诀
**变更管理三步走**：变更请求（RFC）→评估批准（CAB）→实施评估（PIR），重大变更走委员会，紧急变更走快批，标准变更走预授权

**数据库调优四板斧**：慢SQL日志开启、explain执行计划、索引优化重建、连接池参数调优

**KPI五维度口诀**：可用性能稳定服务成本（可性定务本），MTBF越长越好MTTR越短越好

**数据销毁三级**：软销毁（覆写）→硬销毁（消磁）→物理销毁（粉碎），涉密数据必走物理销毁

**退役管理五步曲**：评估决策→数据迁移→系统关停→数据销毁→档案留存

**质量保证SQA口诀**：SQA管过程QC管产品，过程预防优于事后检查

**缺陷帕累托**：80%缺陷集中在20%模块，重点模块重点测
