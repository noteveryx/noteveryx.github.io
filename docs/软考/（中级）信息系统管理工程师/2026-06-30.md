# 2026-06-30 信息系统管理工程师 学习内容

## 今日核心概念

### 1. 信息系统战略规划与IT治理

定义：信息系统战略规划是将组织业务战略与IT建设相结合，确定信息化发展方向、目标和重点项目的系统化过程；IT治理是确保IT投资与组织战略一致、风险可控、价值最大化的决策与监督框架。

**信息系统战略规划三层架构**：
- **战略层**：信息化战略（IS）与业务战略对齐，明确"为什么做IT"
- **战术层**：信息技术战略（ITS），明确"用什么技术、建什么系统"
- **执行层**：信息管理系统战略（ISS），明确"系统怎么建、怎么用"

**常用规划方法四类**：
| 方法 | 核心思想 | 适用场景 |
|------|----------|----------|
| 关键成功因素法（CSF） | 识别影响目标实现的关键因素 | 高层管理者驱动 |
| 战略目标集转化法（SST） | 业务目标→IT目标→信息系统 | 业务目标清晰 |
| 企业系统规划法（BSP） | 自上而下规划，自下而上实现 | 大型企业 |
| 价值链分析法（VCA） | 围绕核心价值链识别信息系统 | 流程驱动型组织 |

**IT治理三大框架**：
- **COBIT**：以"控制目标"为核心，连接业务与IT，五大域：计划-建设-运行-监控-评估
- **ITIL**：以"服务管理"为核心，强调流程化、标准化
- **ISO 38500**：以"治理原则"为核心，六原则：责任、战略、获得、性能、合规、行为

**关键要点**：
- 战略对齐是灵魂——IT规划必须服务于业务目标，避免"为技术而技术"
- 治理≠管理：治理是"定方向、定规则"，管理是"执行、运营"
- 信息化战略要写进组织正式文件：经董事会/办公会审议，具有约束力
- IT治理委员会是关键：通常由CIO牵头，业务+IT+财务+审计多方参与

### 2. 信息系统生命周期管理与立项决策

定义：信息系统生命周期管理是对信息系统从立项、规划、设计、实施、运行到退役的全过程统筹管理；立项决策是生命周期"入口"，决定项目的合法性与合理性。

**信息系统五阶段生命周期**：
1. **立项阶段**：项目识别、可行性研究、立项审批
2. **规划阶段**：需求分析、总体方案、详细设计
3. **实施阶段**：开发、测试、部署、培训
4. **运行阶段**：运维、监控、优化、变更
5. **退役阶段**：数据迁移、系统下线、资产处置

**立项决策三要素**：
- **必要性**：业务痛点、用户需求、政策驱动
- **可行性**：技术、经济、社会、运行、法律五维论证
- **可负担性**：预算落实、资金筹措方案

**立项审批权限分级**：
| 投资额 | 审批层级 | 关键文件 |
|--------|----------|----------|
| 50万以下 | 单位内部 | 项目申请书+可研报告 |
| 50-500万 | 主管部门 | 项目建议书+可研报告+评审意见 |
| 500万以上 | 财政+主管 | 完整立项文件+风险评估+后评价方案 |

**关键要点**：
- "立项"是法定程序——未经立项不得开工建设，违规立项是审计高风险点
- 立项后变更需"重大变更"流程：投资额变化≥10%或范围重大调整需重新审批
- 可研报告须独立编制：避免"既是运动员又是裁判员"——承建方不能既可研又建设
- 退役不是"删库跑路"：数据归档≥5年、资产处置合规、系统下线路径记录

### 3. 信息系统项目范围管理与WBS分解

定义：项目范围管理是"做且只做"项目所需的全部工作的管理过程；WBS（工作分解结构）是将项目交付物逐层分解为可管理的工作包的结构化工具。

**范围管理六过程**：
1. **规划范围管理**：制定范围管理计划
2. **收集需求**：识别并记录干系人需求
3. **定义范围**：编制范围说明书，明确项目边界
4. **创建WBS**：将交付物分解为工作包
5. **确认范围**：与客户共同验收交付物
6. **控制范围**：监控范围，管控范围变更

**WBS分解三层原则**：
- **第一层（项目）**：项目名称
- **第二层（阶段/可交付成果）**：需求、设计、开发、测试、部署等
- **第三层（工作包）**：8/80原则——工作包≤80小时、可由一人完成

**WBS三种表示方法**：
- **树形结构（组织结构图式）**：直观展示层级关系
- **列表式（缩进式）**：便于文档管理
- **WBS字典**：每个工作包配详细说明（责任人、预算、工期、验收标准）

**关键要点**：
- WBS是项目管理的"骨架"——所有计划（进度、成本、质量）都基于WBS展开
- 100%原则：WBS必须包含项目全部工作，不能遗漏也不能多出
- 范围蔓延（Scope Creep）是项目头号杀手——必须用WBS+范围基准严格管控
- 范围变更必经"CCB"审批：变更控制委员会（CCB）签字后方可执行

### 4. 信息系统项目进度管理与关键路径

定义：项目进度管理是制定项目活动计划、安排活动顺序、估算工期、编制进度计划并控制变更的过程；关键路径是项目网络图中工期最长路径，决定项目最短完工时间。

**进度管理六过程**：
1. **规划进度管理**：制定进度管理计划
2. **定义活动**：识别完成交付物所需的具体活动
3. **排列活动顺序**：确定活动之间的依赖关系
4. **估算活动持续时间**：三点估算（最乐观O+最可能M+最悲观P）
5. **制定进度计划**：编制甘特图/网络图，识别关键路径
6. **控制进度**：监控进度偏差，采取纠正措施

**关键路径计算（CPM）**：
- **正推法（最早开始ES/最早完成EF）**：ES + 工期 = EF
- **逆推法（最晚开始LS/最晚完成LF）**：LF - 工期 = LS
- **总浮动时间（TF）**：LS - ES = LF - EF
- **自由浮动时间（FF）**：紧后活动ES - 本活动EF
- **关键路径**：TF=0的活动组成的路径，决定项目最短工期

**三点估算公式**：
- 期望工期 = (O + 4M + P) ÷ 6
- 标准差 = (P - O) ÷ 6
- 完工概率：根据正态分布计算

**关键要点**：
- 关键路径可能"漂移"——项目执行中需动态更新关键路径
- 进度压缩两方法：赶工（增加资源，加成本）+快速跟进（并行活动，加风险）
- 资源平衡是平衡艺术：避免人员/设备过载
- 进度基准是"红线"：偏差超过±10%需启动变更控制

### 5. 信息系统项目成本管理与挣值分析

定义：项目成本管理是在批准的预算内完成项目所需的规划、估算、预算、控制过程；挣值分析（EVM）是综合范围、进度、成本的三维绩效测量方法。

**成本管理四过程**：
1. **规划成本管理**：制定成本管理计划
2. **估算成本**：自下而上估算法、参数估算法、三点估算法
3. **制定预算**：将总成本分摊到工作包，形成成本基准
4. **控制成本**：监控成本执行，纠偏

**挣值分析三参数四指标**：
- **三参数**：
  - 计划值（PV）：计划完成工作的预算成本
  - 挣值（EV）：实际完成工作的预算成本
  - 实际成本（AC）：实际完成工作的实际成本
- **四指标**：
  - 进度偏差（SV）= EV - PV（>0 提前，<0 滞后）
  - 成本偏差（CV）= EV - AC（>0 节约，<0 超支）
  - 进度绩效指数（SPI）= EV ÷ PV（>1 提前，<1 滞后）
  - 成本绩效指数（CPI）= EV ÷ AC（>1 节约，<1 超支）

**完工估算（EAC）公式**：
- **EAC = BAC ÷ CPI**（当前绩效持续到项目结束）
- **EAC = AC + (BAC - EV)**（剩余工作按原预算完成）
- **EAC = AC + ETC**（重新估算剩余工作成本）

**关键要点**：
- 成本基准≠总预算：成本基准是按时间分布的预算曲线，管理储备不在其中
- CPI<1连续2个月必须预警：超支已成趋势，需立即纠偏
- 挣值分析是项目经理"体检表"——周例会必须看CPI/SPI
- 成本失控的根因：范围蔓延+估算不准+变更失控+资源浪费

## 案例分析题

**题目**：

某市大数据管理局启动"智慧政务一网通办平台"二期项目，总投资5200万元，工期18个月，2025年6月开工。项目采用公开招标确定总集成商A公司，签订固定总价合同。市信息中心作为建设单位，委托B监理公司全程监理。

**问题1（规划）**：项目立项时，市大数据管理局委托A公司（拟中标方）编制可行性研究报告，报告重点阐述了技术方案和经济收益，对运行可行性、社会可行性、法律可行性三章用"基本可行"一笔带过。立项审批时，财政部门直接盖章通过，未组织专家评审。

**问题2（范围）**：项目WBS分解到第三层后，A公司项目经理认为"第三层工作包有的需要200小时，分配给1人难以按时完成"，决定将部分工作包再拆细到第四层。在第一次客户评审会上，业主代表提出增加"电子证照共享""区块链存证""AI智能问答"三大新功能模块，承诺"加功能不加钱"。A公司项目经理口头同意。

**问题3（进度）**：项目执行至第6个月，原计划完成需求分析+总体设计+部分详细设计。实际完成情况：需求分析完成，详细设计仅完成60%。项目经理采用快速跟进法，将详细设计剩余工作与开发并行，并申请了进度偏差报告。

**问题4（成本）**：项目执行至第10个月，A公司向业主方提交"成本预警报告"：PV=1800万，EV=1500万，AC=1700万。A公司项目经理告知业主方"项目已超支且进度滞后，必须追加预算800万，否则无法按期交付"。业主方认为"固定总价合同不应追加"。

**问题5（治理）**：项目执行至第12个月，市审计局开展专项审计，发现以下问题：①A公司项目经理同时担任承建方"质量主管"和"安全主管"；②监理B公司总监同时为A公司另一项目监理；③业主方信息科小李既负责需求审批又负责验收测试；④信息化战略由信息科自行编制，未报市政府常务会议审议。

请分别回答以下问题：

（1）针对问题1，请指出立项阶段存在的违规之处，并补充可行性研究"五维论证"的完整内容。

（2）针对问题2，请分析"工作包拆细到第四层"是否符合WBS原则，"加功能不加钱"口头承诺有何风险，并提出范围变更的规范流程。

（3）针对问题3，请计算该项目执行至第6个月的SV、SPI、CPI（假设BAC=5200万，AC=1100万，EV=1500万），并分析"快速跟进法"对该项目的影响。

（4）针对问题4，请计算SV、CPI、完工估算EAC，并判断项目成本/进度状态。A公司"追加预算800万"是否合理？业主方"不应追加"是否合规？

（5）针对问题5，请从IT治理角度分析"四个问题"各自违反了哪项治理原则，并提出整改建议。

**参考答案**：

（1）立项阶段违规分析：

| 违规项 | 表现 | 依据 |
|--------|------|------|
| 可研编制方违规 | 拟中标方A公司编制可研 | 违反"独立编制"原则，承建方不能既可研又建设 |
| 可研内容残缺 | 三维用"基本可行"带过 | 违反"五维论证"完整性要求 |
| 审批环节违规 | 财政部门直接盖章 | 违反"专家评审"前置程序 |
| 立项材料不全 | 缺风险评估+多方案比选 | 违反《政府投资条例》立项管理要求 |

**可行性研究五维论证完整内容**：
- **技术可行性**：现有技术能否实现？技术成熟度、技术风险、人员能力评估
- **经济可行性**：投资估算、NPV、IRR、投资回收期、TCO分析
- **社会可行性**：便民效益、用户接受度、社会影响、就业影响
- **运行可行性**：运维能力、用户培训、制度配套、应急能力
- **法律可行性**：等保合规、数据安全法、个人信息保护法、政府采购法、招投标法

**整改建议**：撤销原立项审批，委托第三方咨询机构重新编制可研报告，组织5-7人专家组评审通过后再行立项。

（2）WBS与范围变更分析：

**WBS拆细到第四层合规性判断**：
- 8/80原则：工作包应在8-80小时之间
- 若第三层工作包>80小时，拆细到第四层**符合**WBS原则（保证工作包可管理）
- 但拆细后需同步更新WBS字典、范围基准，并经CCB审批

**"加功能不加钱"口头承诺的风险**：
- 口头承诺**无法律效力**——一旦发生争议，空口无凭
- 三个新功能（电子证照+区块链+AI问答）工作量巨大，预计增加成本600-1000万
- 范围蔓延是项目失控主因——A公司项目经理无权单方面承诺范围变更
- 业主代表越权——业务方只能提需求，变更需走正式流程

**范围变更规范流程**：
```
业务方提出变更需求
   ↓
变更影响分析（范围/进度/成本/质量四维）
   ↓
承建方提交《变更申请单》
   ↓
监理方初审（合规性、合理性）
   ↓
CCB评审（业主+承建+监理+专家）
   ↓
审批通过？ ──否──→ 驳回或重新议价
   ↓ 是
签订《变更补充协议》→纳入范围基准→执行
```

**整改建议**：业主方与A公司召开正式评审会，对三大新增功能重新议价，签订《变更补充协议》，调整进度计划与预算。

（3）挣值分析与快速跟进影响：

**挣值计算**：
- 第6个月挣值数据：PV=1800万（计划完成1800万的工作），EV=1500万（实际完成1500万），AC=1100万
- 进度偏差 SV = EV - PV = 1500 - 1800 = **-300万**（进度滞后）
- 成本偏差 CV = EV - AC = 1500 - 1100 = **+400万**（成本节约）
- 进度绩效指数 SPI = EV ÷ PV = 1500 ÷ 1800 = **0.83**（进度滞后，进度绩效不佳）
- 成本绩效指数 CPI = EV ÷ AC = 1500 ÷ 1100 = **1.36**（成本节约，效率较高）

**挣值状态综合判断**：
- 进度状态：SPI=0.83<1，项目**进度滞后**约17%
- 成本状态：CPI=1.36>1，项目**成本节约**约26%（但需警惕"以质量换进度"）

**快速跟进法影响分析**：
- **优点**：通过并行缩短关键路径工期，能抢回部分进度
- **风险**：
  - 详细设计未完成就开始开发，需求变更概率增加
  - 设计缺陷会传递到开发阶段，导致返工率上升
  - 团队工作量增加，沟通成本上升
  - 后续测试和集成阶段风险剧增
- **本项目适用性判断**：在详细设计仅完成60%的情况下盲目快速跟进，**风险极高**——很可能引发"挣值从SPI<1演变为CPI<1"的二次危机

**整改建议**：暂停快速跟进，先完成剩余详细设计（即使耽误1个月），再启动开发；如确实需要抢工期，优先采用"赶工"（增加资源）而非"快速跟进"。

（4）成本预警分析与合同合规性：

**第10个月挣值计算**：
- 进度偏差 SV = EV - PV = 1500 - 1800 = **-300万**（进度滞后300万）
- 成本偏差 CV = EV - AC = 1500 - 1700 = **-200万**（成本超支200万）
- 进度绩效指数 SPI = EV ÷ PV = 1500 ÷ 1800 = **0.83**（进度落后17%）
- 成本绩效指数 CPI = EV ÷ AC = 1500 ÷ 1700 = **0.88**（成本超支约13%）

**完工估算（EAC）计算**：
- 假设当前绩效持续到项目结束：EAC = BAC ÷ CPI = 5200 ÷ 0.88 ≈ **5909万**
- 即项目最终可能超支约700万

**项目状态综合判断**：
- 进度滞后 + 成本超支 = **"双重失控"**，是项目管理最不利状态
- 项目从"成本节约、进度滞后"演变为"成本超支、进度滞后"，说明前期快速跟进产生副作用

**A公司"追加预算800万"是否合理**：
- 从项目管理角度：项目超支已成事实，追加预算有合理性
- 从合同角度：**固定总价合同下，A公司应承担主要风险**——非业主方原因的超支应由A公司自行消化
- 追加预算需经严格审批：需区分"业主方原因"还是"承建方原因"
- 800万数字需要审计验证：EAC显示约需追加700万，A公司报价基本合理但略偏高

**业主方"不应追加"是否合规**：
- **过于绝对，不合规**——若超支全部由业主原因导致，业主方应承担
- **正确的处理方式**：
  - 第一步：审计超支原因（业主原因 vs 承建方原因）
  - 第二步：依据合同约定分担超支责任
  - 第三步：若属承建方原因，业主方有权拒绝追加
  - 第四步：若属业主方原因，业主方应按变更流程追加预算

**整改建议**：双方启动合同争议调解程序，委托第三方审计超支原因，依据审计结论和合同约定合理分担超支责任。

（5）IT治理问题分析与整改：

**"四个问题"对应的治理原则**：

| 问题 | 现象 | 违反原则 | 依据 |
|------|------|----------|------|
| ①A公司项目经理兼任质安主管 | 职责未分离 | 职责分离原则 | 质量与安全需独立监督 |
| ②B监理总监兼A公司另一项目监理 | 利益冲突 | 独立性原则 | 监理与承建方应独立 |
| ③小李既审批需求又验收测试 | 职责未分离 | 职责分离原则 | 审批与验收须不同人 |
| ④信息化战略未报政府常务会审议 | 治理层级缺失 | 战略对齐原则 | IT战略须经决策层审议 |

**IT治理六原则（ISO 38500）**：
1. 责任原则：明确IT决策的责任主体
2. 战略原则：IT战略与业务战略一致
3. 获得原则：IT投资合理、风险可控
4. 性能原则：IT服务满足业务需求
5. 合规原则：IT活动符合法律法规
6. 行为原则：IT决策符合人文伦理

**本案例四个问题对应原则**：
- ①②③：违反**责任原则**和**合规原则**——职责未分离、缺乏独立性
- ④：违反**战略原则**——IT战略未上升至决策层

**整改建议**：

针对①A公司：项目经理与质安主管必须分设，质安主管直接向公司质量部汇报，独立性增强

针对②B监理：更换本项目监理总监，确保监理与承建方无任何利益关联

针对③业主方：调整小李职责，需求审批由信息科负责人承担，验收测试由独立测试组承担

针对④市政府：信息化战略经市大数据管理局编制后，报市政府常务会议审议，纳入市级专项规划

**深层次整改**：
- 建立IT治理委员会：CIO+业务+财务+审计四方联动
- 制定"职责分离矩阵"：明确各岗位的审批、监督、执行权限
- 建立"利益申报制度"：项目相关方定期申报利益关联
- 信息化战略与业务战略同步规划：纳入市"十四五"信息化规划

**踩分要点**：
- （1）必须明确"承建方不能既可研又建设"+"五维论证完整内容"+"专家评审前置"三个关键点
- （2）必须区分"WBS拆细符合原则"和"口头承诺无效"两个判断。"CCB变更控制"是踩分核心
- （3）必须准确计算SV、SPI、CPI三个指标。明确"快速跟进风险大于收益"是得分点
- （4）必须准确计算EAC。强调"业主方不能简单拒绝追加"+"需区分责任原因"是踩分关键
- （5）必须将"四个问题"对应到IT治理六原则。提到"职责分离矩阵"+"IT治理委员会"是加分项

## 记忆口诀

**战略规划三层架构**：
"战ITS战ISS"
战略层+信息技术战略+信息管理系统战略，对应：做什么+用什么+怎么做。

**规划方法四类**：
"成战企价"
关键成功因素+战略目标集转化+企业系统规划+价值链分析。

**IT治理六原则**：
"责战获性合行"
责任+战略+获得+性能+合规+行为，ISO 38500核心六原则。

**生命周期五阶段**：
"立规实运退"
立项+规划+实施+运行+退役，对应：入口+蓝图+建设+运维+终结。

**WBS拆分8/80原则**：
"8到80，一人能搞"
工作包小于80小时，可由1人完成，是WBS拆分标准。

**WBS三种表示法**：
"树列字典"
树形结构+列表式+WBS字典，对应：展示+管理+说明。

**范围管理六过程**：
"规收定WBS确控"
规划范围管理+收集需求+定义范围+创建WBS+确认范围+控制范围。

**进度管理六过程**：
"规定排估制控"
规划+定义活动+排列顺序+估算工期+制定计划+控制进度。

**挣值三参数**：
"PV/AC/EV"
计划值/实际成本/挣值，对应：预算+花了+干了。

**挣值四指标**：
"SV/CV/SPI/CPI"
进度偏差+成本偏差+进度绩效+成本绩效，对应：偏差+比率。

**挣值结果判断**：
"减落减超，率一落超"
EV-PV为负=进度落后；EV-AC为负=成本超支；SPI<1落后；CPI<1超支。

**完工估算EAC**：
"BAC除CPI"
假设当前绩效持续到项目结束：EAC = BAC ÷ CPI。

**三种成本估算方法**：
"参三自"
参数估算+三点估算+自下而上估算，对应：模型+概率+累加。

**进度压缩两方法**：
"赶快"
赶工（加资源加成本）+快速跟进（并行加风险）。
