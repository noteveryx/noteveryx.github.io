## 今日核心概念

### 1. 风险管理七过程与ITTO体系

定义：项目风险管理包括七个过程，覆盖规划→识别→分析→应对→实施→监督全生命周期。

关键要点：

| 过程 | 核心工具与技术 | 关键输出 |
|------|--------------|---------|
| 规划风险管理 | 会议、专家判断 | 风险管理计划 |
| 识别风险 | SWOT、德尔菲、头脑风暴、核对单、假设分析、文档审查 | 风险登记册（初版） |
| 定性风险分析 | 概率影响矩阵、风险数据质量评估、风险紧迫性评估 | 风险优先级排序、观察清单 |
| 定量风险分析 | EMV决策树、蒙特卡洛模拟、敏感性分析（龙卷风图） | 量化风险报告 |
| 规划风险应对 | 消极四策略+积极四策略、应急应对策略 | 风险应对计划、应急计划 |
| 实施风险应对 | 项目管理信息系统 | 变更请求 |
| 监督风险 | 风险审计、偏差与趋势分析、储备分析、技术绩效测量 | 风险登记册更新 |

- 风险分解结构RBS：按类别划分风险（技术风险/外部风险/组织风险/项目管理风险）
- 风险偏好（愿意承担的风险程度）→风险容忍度（可接受偏差范围）→风险临界值（触发应对的阈值）
- 风险登记册随项目推进持续迭代更新，是风险管理的核心文档

### 2. 项目管理者的权力与领导力风格

定义：项目经理权力分为职位权力（来自组织赋予）和个人权力（来自自身能力与魅力），需合理组合使用。

关键要点：
- French & Raven权力五类型：
  - 法定权力（正式职权）→ 来自职位任命
  - 奖励权力（给予奖励）→ 绩效奖金/晋升推荐
  - 惩罚/强制权力（施加惩罚）→ 慎用，易引发抵触
  - 专家权力（专业技能）→ 最持久的权力来源
  - 参照/感召权力（人格魅力/人际关系）→ 团队自愿追随
- 最佳组合：专家权力 + 参照权力（靠能力赢得信任）
- 矩阵型组织中，PM职位权力弱，更需要依赖专家权力和参照权力

领导力风格与阶段匹配：
- 独裁型：危机时刻、新手团队（形成期）
- 民主型：成熟团队（规范期/成熟期）、需要群策群力时
- 放任型：高绩效自组织团队、专家团队
- 管理（管事）vs 领导（带人）：管理是把事做对，领导是做对的事

### 3. 软件维护四类型与质量模型

定义：软件维护是软件生命周期中占比最大、成本最高的阶段，占总成本60%-80%。

关键要点：
- 四种维护类型及占比：
  - 纠正性维护（21%）：修复已发现的Bug和缺陷
  - 适应性维护（25%）：适配外部环境变化（新OS/新数据库/新法规）
  - 完善性维护（50%）：新增功能、优化性能，**占比最高**
  - 预防性维护（4%）：主动改进，防止未来问题发生
- ISO 25010质量模型（替代ISO 9126）八大特性：
  - 功能适用性、性能效率、兼容性、易用性
  - 可靠性、安全性、可维护性、可移植性
- 软件测试V模型：需求分析↔验收测试、概要设计↔系统测试、详细设计↔集成测试、编码↔单元测试
- Alpha测试：开发方内部模拟用户环境测试；Beta测试：真实用户在真实环境下测试

### 4. 对称加密与非对称加密体系

定义：加密技术是信息安全的核心，对称加密用同一密钥加解密，非对称加密用公钥-私钥对实现。

关键要点：
- 对称加密（速度快、适合大量数据）：
  - DES（56位密钥，已不安全）
  - 3DES（三重DES，过渡方案）
  - AES（128/192/256位，当前主流）
  - SM4（国密算法）
- 非对称加密（速度慢、安全性高）：
  - RSA（基于大素数分解难题，最常用）
  - ECC（椭圆曲线，同等安全下密钥更短）
  - SM2（国密算法）
- 数字签名原理：发送方用私钥签名→接收方用公钥验证，实现不可否认性
- 数字证书：CA（认证中心）签发，绑定公钥与持有者身份
- 实用组合：用非对称加密传输对称密钥（密钥交换），用对称加密传输大量数据

## 案例分析题

**题目**：

某省卫健委启动"智慧医疗数据共享平台"项目，预算1200万元，工期12个月。项目涉及40余家医院HIS系统对接、居民健康档案共享、医保数据交换等功能。由某软件公司承建，采用固定总价合同。

第8个月末，项目经理汇报如下情况：
- 计划价值PV=800万元，实际成本AC=850万元，挣值EV=700万元
- 甲方临时要求增加"疫苗接种数据对接"模块，项目经理口头答应并安排团队加班实施
- 系统上线后部分医院频繁出现数据传输错误，经排查是各医院HIS系统接口标准不一致导致
- 项目团队对数据加密方案不熟悉，居民健康档案数据传输采用了DES算法
- 项目经理职位权力不足，无法调动公司其他部门资源配合多医院接口联调
- 项目启动至今未建立风险登记册，未进行过任何正式的风险识别和分析活动
- 项目质量标准未按GB/T 39725-2020《信息安全技术 健康医疗数据安全指南》执行
- 多个医院接口开发人员已离职，新接手的开发人员不了解原有的接口实现逻辑

**问题1**（10分）：计算当前CV、SV、CPI、SPI，并对项目状态进行综合分析。

**问题2**（8分）：从风险管理角度，分析该项目存在哪些问题？应采用哪些工具和技术进行改进？

**问题3**（8分）：从软件维护角度，各医院HIS接口不一致导致的问题属于什么维护类型？结合本项目，提出运维阶段的改进建议。

**问题4**（6分）：数据加密方案存在哪些问题？应如何改进加密体系？

**问题5**（8分）：分析项目经理在权力运用方面存在的问题，并给出提升建议。

**参考答案**：

**问题1**：
- CV = EV - AC = 700 - 850 = -150万元（成本超支150万）
- SV = EV - PV = 700 - 800 = -100万元（进度滞后100万）
- CPI = EV/AC = 700/850 = 0.824（每花1元仅完成0.824元价值）
- SPI = EV/PV = 700/800 = 0.875（进度效率87.5%）
- 项目状态：成本超支且进度滞后。CPI<1、SPI<1，属于典型的"双亏"状态。若不采取纠正措施，按典型偏差预测 EAC = BAC/CPI = 1200/0.824 = 1456.3万元，完工时需追加约256万元。

**问题2**：
风险管理问题：
① 未开展风险规划——无风险管理计划，无法指导后续活动
② 未进行风险识别——未建立风险登记册，对潜在风险无知
③ 未实施定性和定量风险分析——无法区分风险优先级
④ 未制定风险应对计划——风险发生后只能被动应对

改进措施：
① 制定风险管理计划，明确角色职责、预算和周期
② 使用头脑风暴+德尔菲技术+核对单，全面识别风险，建立风险登记册
③ 使用概率影响矩阵对风险排序，对高概率高影响风险进行定量分析（EMV/蒙特卡洛）
④ 针对关键风险制定应对策略（人员离职→交叉培训+文档化；接口不一致→制定统一接口规范+适配层）

**问题3**：
属于适应性维护——各医院HIS系统接口标准不一致，软件开发方需要适配不同的外部环境（各医院的异构系统）。

运维改进建议：
① 制定统一的接口标准和数据交换规范，强制各医院对接时遵守
② 开发统一的数据交换中间件/适配层，屏蔽各医院HIS系统差异
③ 建立接口版本管理机制，接口变更走CCB审批
④ 完善接口文档（完善性维护），确保新接手人员能快速理解
⑤ 预计运维阶段适应性维护将占较大比例，在合同中提前约定维保条款和费用

**问题4**：
存在的问题：
① DES算法已过时（56位密钥可被暴力破解），不应再用于敏感数据传输
② 团队缺乏加密技术能力，方案制定草率
③ 未遵循国家医疗数据安全标准GB/T 39725-2020

改进方案：
① 传输加密：使用TLS 1.3协议（握手用非对称加密RSA/ECC交换密钥，数据传输用AES-256对称加密）
② 存储加密：数据库层使用AES-256加密存储敏感字段
③ 如涉及商用密码应用，采用国密算法SM2（非对称）+SM4（对称）
④ 部署数字证书体系（CA签发），实现身份认证和不可否认性
⑤ 进行等保三级测评，确保系统整体安全合规

**问题5**：
存在的问题：
① 过度依赖职位权力——职位权力在矩阵型组织中本身有限
② 缺乏专家权力——接口联调时无法有效协调跨部门资源
③ 未建立参照权力——与各医院IT部门缺乏信任关系

提升建议：
① 增强专家权力：深入学习医疗行业标准和接口技术，成为技术权威
② 建立参照权力：主动与各医院IT负责人建立关系，用沟通和真诚赢得信任
③ 善用奖励权力：为配合联调的部门争取绩效认可和资源支持
④ 慎用惩罚权力：避免命令式沟通，以合作共赢为主导
⑤ 在不同阶段切换领导风格：启动期用民主型（共识方案）、执行期用民主型+适当独裁（危机时）、收尾期用放任型（尊重专业）

## 记忆口诀

**风险管理七过程**：
"规矩定计，识定定应，施监"
规（规划风险管理）→识（识别风险）→定性→定量→规（规划应对）→施（实施应对）→监（监督风险）

**权力五型速记**：
"法官励惩专参"
法（法定权力）→官（即职位）→励（奖励权力）→惩（惩罚权力）→专（专家权力）→参（参照权力）
专家+参照=王道，惩罚慎用少用。

**软件维护四型**：
"纠适完预，完占一半"
纠（纠正性21%）→适（适应性25%）→完（完善性50%）→预（预防性4%）
完善性占比最高，考选择判断必出。

**加密算法速记**：
"对称快：DAESM（大爱SM），非对称慢：RESM（热SM）"
对称：DES→3DES→AES→SM4（D-A-E-S）
非对称：RSA→ECC→SM2（R-E-S）
国密：SM2（非对称签）+SM3（哈希）+SM4（对称加）="2签3哈4加"

**质量模型ISO 25010**：
"功效兼易，可安可维可移"
功能适用性、性能效率、兼容性、易用性、可靠性、安全性、可维护性、可移植性 —— 八个字头凑一句："功能效率兼容易，可靠安全维护移"
