## 今日核心概念

### 1. 范围蔓延与镀金

定义：范围蔓延指未经正式变更控制流程的范围扩大；镀金指项目团队主动增加客户未要求的功能或特性。

关键要点：
- 范围蔓延分两类：**客户蔓延**（外部压力导致需求膨胀）和**团队蔓延**（内部自行添加功能）
- 镀金属于团队蔓延的一种，是PMI明确反对的行为——"多做不等于多得分"
- 正确做法：任何范围变更必须走**整体变更控制流程**，经CCB审批后纳入基准
- 范围确认≠质量控制：范围确认关注"有没有做"，质控关注"做得对不对"
- 防范手段：详细的需求跟踪矩阵（RTM）+ 严格的变更控制流程 + 定期范围确认

### 2. 关键链法（CCPM）

定义：关键链法是在关键路径法基础上，考虑资源约束和不确定性，通过缓冲管理来保护项目进度的方法。

关键要点：
- 关键链≠关键路径：关键链是考虑**资源约束**后的最长路径
- 三类缓冲：**项目缓冲**（放在关键链末端）、**接驳缓冲**（放在非关键链与关键链汇合处）、**资源缓冲**（资源就位提醒）
- 缓冲大小确定方法：**根方差法**——将各活动的安全时间按√(∑σ²)聚合，而非简单相加
- 缓冲消耗监控：消耗<1/3为绿灯、1/3~2/3为黄灯、>2/3为红灯需采取行动
- 与CPM核心区别：CPM只看逻辑依赖，CCPM同时看逻辑依赖+资源约束；CPM各活动留安全余量，CCPM把余量集中到缓冲区

### 3. 项目沟通管理

定义：确保项目信息及时、恰当地生成、收集、分发、存储和最终处置的过程。

关键要点：
- 沟通模型六要素：**编码→信息→媒介→解码→确认→反馈**，噪声贯穿全程
- 三种沟通方法：**交互式**（会议/电话，最有效）、**推式**（邮件/报告，确保送达但不确保理解）、**拉式**（门户/知识库，受众自主获取）
- 沟通渠道公式：N(N-1)/2，如6人团队=15条渠道
- 沟通管理计划核心内容：干系人沟通需求、信息格式与载体、汇报频率、升级机制、术语表
- 沟通障碍根源：信息过滤（层级越多失真越大）、信息过载、文化差异、情绪干扰
- PM沟通时间占比：项目经理约**90%**的时间用于沟通

### 4. 需求工程四阶段

定义：需求工程是系统工程的核心环节，包含需求获取、需求分析、需求规格说明和需求验证四个阶段。

关键要点：
- **需求获取**：技术包括访谈、问卷调查、观察、原型法、用例法、头脑风暴；关键在于识别显性需求+隐性需求
- **需求分析**：构建系统逻辑模型，工具包括数据流图（DFD）、实体关系图（ERD）、状态转换图、用例图；核心输出为需求基线
- **需求规格说明（SRS）**：是需求工程的里程碑文档，是用户与开发方的契约；应具备正确性、无歧义性、完整性、一致性、可验证性
- **需求验证**：通过需求评审（正式/非正式）确认SRS正确反映用户需求；验证通过后形成**需求基线**，后续变更走变更控制
- 需求跟踪矩阵（RTM）：从需求→设计→代码→测试用例的全程追溯工具，是范围管理的核心制品

### 5. 数字化转型与数字政府

定义：数字化转型是利用数字技术重塑业务模式、组织结构和价值交付方式；数字政府是政府在数字化转型中的具体实践。

关键要点：
- 数字政府三阶段：**信息发布→交互服务→一体化智能服务**
- "一网通办"核心要求：事项标准化、数据共享、流程再造、统一入口
- 数字化转型四大要素：**技术驱动**（云大物移智链）、**业务重塑**（流程再造非简单信息化）、**组织变革**（扁平化/敏捷化）、**文化转变**（数据决策）
- 数据要素市场化：数据资源→数据资产→数据资本的转化路径；数据确权三权（持有权/使用权/经营权）
- 政务信息化常见问题：信息孤岛、重复建设、数据不通、安全合规缺失、运维保障不足

## 案例分析题

**题目**：

某市"一网通办"政务服务平台项目，预算1200万元，工期12个月。项目采用瀑布模型开发，已执行到第8个月末。项目团队18人，其中5人为甲方业务部门人员。

当前状况：
- 计划价值PV=800万元，实际成本AC=960万元，挣值EV=720万元
- 项目经理在开发过程中，多次应甲方业务人员口头要求增加"智能推荐""人脸识别"等新功能，均直接安排开发
- 甲方5名业务人员仅在项目启动时参加过一次需求评审会，此后再未参与
- 项目未建立正式的变更控制流程，也无需求跟踪矩阵
- 近期安全审计发现系统存在SQL注入漏洞，且未按等保二级要求进行安全设计

**问题**：

1. 计算CV、SV、CPI、SPI，并判断项目成本和进度状况。
2. 若偏差为典型偏差，计算EAC和TCPI。
3. 指出项目管理中存在的问题，并给出改进建议。

**参考答案**：

**问题1：**

- CV = EV - AC = 720 - 960 = **-240万元**（成本超支）
- SV = EV - PV = 720 - 800 = **-80万元**（进度滞后）
- CPI = EV / AC = 720 / 960 = **0.75**（每元只产生0.75元价值）
- SPI = EV / PV = 720 / 800 = **0.90**（进度效率仅90%）

项目成本严重超支，进度滞后，需立即采取纠偏措施。

**问题2：**

- EAC = BAC / CPI = 1200 / 0.75 = **1600万元**
- TCPI = (BAC - EV) / (BAC - AC) = (1200 - 720) / (1200 - 960) = 480 / 240 = **2.0**

TCPI=2.0远超1.0，在剩余预算内完成几乎不可能，需申请追加预算或大幅缩减范围。

**问题3：**

| 问题 | 改进建议 |
|------|----------|
| 口头变更直接开发，无变更控制流程 | 建立整体变更控制流程，所有变更经CCB审批后方可实施 |
| 增加新功能属于范围蔓延/镀金 | 严格按需求基线执行，新增需求走变更流程并评估影响 |
| 无需求跟踪矩阵 | 建立RTM，实现需求→设计→代码→测试全链路追溯 |
| 甲方人员参与不足 | 制定干系人参与计划，明确甲方参与节点和职责 |
| 存在SQL注入漏洞，未做等保安全设计 | 引入安全开发生命周期（SDL），按等保要求进行安全设计与测试 |
| 瀑布模型不适合需求频繁变更场景 | 评估采用混合生命周期，需求稳定部分用瀑布，变化部分用迭代/敏捷 |

**踩分要点**：

1. CV/SV/CPI/SPI计算必须列出公式和结果，**不带公式扣分**
2. 典型偏差EAC必须用BAC/CPI公式，非典型用AC+(BAC-EV)
3. TCPI>1.0要说明"剩余工作成本密度超过1，需追加预算"
4. 范围蔓延要指出具体行为（口头增加功能）和正确做法（走变更流程）
5. 每个问题必须**问题+建议成对出现**，只列问题不提建议扣一半分

## 记忆口诀

**范围蔓延三防线**：变更流程第一关，需求基线第二关，范围确认第三关

**CCPM缓冲口诀**：关键链看资源，项目缓冲保全局，接驳缓冲保汇合，根方差算余量，三区灯号控风险

**沟通管理口诀**：编码信息媒介解码，确认反馈闭环查；交互推拉三方法，九成时间在沟通

**需求工程口诀**：获取访观察原用例，分析建模型定基线，规格写SRS无歧义，验证过评审锁需求，RTM全程追到底

**数字化转型口诀**：技术驱动是手段，业务重塑是核心，组织变革是保障，文化转变是根基；数字政府三阶段，发布交互到智能
