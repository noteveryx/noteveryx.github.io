# 信息系统项目管理师 第20次学习

## 今日核心概念

### 1. PMBOK第七版八大绩效域

定义：项目绩效域是项目运行过程中需要重点关注和管理的八个相互关联的关键能力领域，是PMBOK第七版（2021年）的核心变革之一，取代了传统的十大知识领域结构，强调"价值交付"和"系统性思考"。

关键要点：
- **八个绩效域**：①团队（Team）②开发方法与生命周期（Development Approach & LifeCycle）③规划（Planning）④项目工作（Project Work）⑤交付（Delivery）⑥度量（Measuring）⑦不确定性（Uncertainty）⑧沟通（Stakeholders），简称"团开规工交度不沟"
- **关系本质**：八个绩效域是项目执行中需要同时关注的能力集合，不是孤立的过程，相互交织、共同支撑价值交付
- **绩效域与过程的区别**：绩效域强调"能力/结果"，过程强调"动作/输入输出"
- **核心转变**：从"基于过程"（Process-based）转向"基于原则"（Principle-based）+ 绩效域
- **十二项项目管理原则**（PMBOK第七版新增）：①成为勤勉、尊重、关心他人的管家②营造协作团队③让干系人有效地参与④聚焦价值⑤识别、评估、应对系统交互⑥展现领导力⑦根据环境进行裁剪⑧将质量融入过程和可交付成果⑨驾驭复杂性⑩优化风险应对⑪拥抱适应性⑫实现变革管理以获得成功

### 2. 团队绩效域

定义：团队绩效域涵盖与团队组建、建设、管理、领导力相关的活动与能力，目标是建立高绩效项目团队以实现项目目标。

关键要点：
- **关键内容**：团队组建、领导力风格、资源分配、团队发展（塔克曼五阶段：形成-震荡-规范-成熟-解散）
- **领导力风格**：服务型领导、仆人式领导、SBI反馈模型（情境Situation-行为Behavior-影响Impact）
- **冲突管理五种方法**：回避/包容/妥协/竞争/合作（合作/共赢为最佳）。注意：第七版绩效域中"妥协"改为"调停/调解"
- **决策风格**：命令、咨询、共识、投票。紧急时命令，长期效益时共识
- **成熟度模型**：D-A-C-I（Driver-Approver-Contributor-Informed，决策驱动者-审批者-贡献者-知情人）
- **远程/混合团队管理**：虚拟协作、跨文化沟通、心理安全、异步沟通规范

### 3. 规划绩效域

定义：规划绩效域围绕项目的组织与协调开展，包含规划活动的交付与行动，与其他绩效域交互最频繁（最核心的横向绩效域）。

关键要点：
- **规划内容**：交付规划、范围规划、进度规划、成本规划、团队规划、沟通规划、风险规划、采购规划
- **交付规划核心**：交付节奏（节奏可预测、节奏紧迫、节奏灵活）、开发方法选择（预测/敏捷/混合）
- **估算方法**：类比估算、参数估算、三点估算、自上而下/自下而上
- **敏捷规划层级**：产品愿景→产品路线图→版本计划（PI/迭代）→迭代计划→每日站会
- **规划关键原则**：渐进明细、滚动式规划、规划应服务于交付价值、规划是持续活动而非一次性活动
- **与其他绩效域的关系**：规划绩效域是枢纽，所有其他绩效域的输出都汇入规划，规划的输出又指导其他域的执行

### 4. 不确定性绩效域

定义：不确定性绩效域关注项目在运行过程中面临的风险、模糊性和复杂性的识别、评估与应对，是PMBOK第七版相比第六版"风险管理"知识领域的最大扩展。

关键要点：
- **三类不确定性**：①风险（Risk，可识别概率/影响，已知未知）②模糊性（Ambiguity，信息不完整/因果不清，未知未知）③复杂性（Complexity，系统交互难以预测，因果非线性）
- **风险应对策略**：规避/转移/减轻/接受（消极风险）+ 开拓/分享/提高（积极风险）
- **风险登记册全流程**：识别→定性分析→定量分析→响应→监控
- **模糊性应对**：迭代开发、原型法、渐进交付、假设分析与实验、设置假设条件与制约因素日志
- **复杂性应对**：系统思维、自组织团队、涌现设计、适应性规划、减化交互界面
- **与项目风险管理的区别**：传统风险管理聚焦"风险"，绩效域覆盖"风险+模糊+复杂"三类——这是考试新热点

### 5. 度量绩效域

定义：度量绩效域围绕项目绩效的评估、度量指标体系建立和数据驱动的决策支持，是与"价值交付"直接关联的领域。强调"用数据说话"。

关键要点：
- **关键度量类别**：①交付度量（进度偏差SV、成本偏差CV）②质量度量（缺陷密度、测试覆盖率）③价值度量（ROI、NPV、投资回收期）④团队度量（士气、效率、流动速率）
- **挣值管理（EVM）三大基本值**：PV（计划值）、EV（挣值）、AC（实际成本）——顺序"PEA"
- **核心绩效指标**：CPI（成本绩效指数）=EV/AC，SPI（进度绩效指数）=EV/PV
- **预测指标**：EAC（完工估算）、ETC（完工尚需）、VAC（完工偏差）、TCPI（完工绩效指数）
- **敏捷度量**：燃尽图、燃起图、累积流量图（CFD）、前置时间（Lead Time）、周期时间（Cycle Time）、吞吐量（Throughput）
- **度量陷阱**：度量过剩（Goodhart's Law——"当一个指标成为目标时，它就不再是好指标"）、虚荣指标（只升不降的指标往往是虚假的）
- **度量四步法**：识别度量对象→定义度量指标→收集度量数据→分析度量结果用于决策

## 案例分析题

**题目**：

某省级政府数字化项目"智慧政务服务平台"，投资1500万元，工期18个月，由信息处王处长担任项目经理，采用预测型（瀑布）开发方式。项目执行到第9个月时发生以下情况：

（1）项目已完成工作量40%，实际成本已花费850万元（PV=750万，AC=850万，EV=600万）

（2）项目团队与业务部门出现严重分歧——业务部门要求增加"领导驾驶舱"和"AI智能问答"两个新模块，但原合同中未包含

（3）核心开发人员小李被公司调往其他项目，技术骨干流失2人

（4）系统集成商提出因硬件涨价，要求追加采购成本80万

（5）王处长发现原需求文档中部分功能描述模糊，团队理解不一致

问题：

1. 计算当前项目的CV、SV、CPI、SPI并分析项目状态
2. 针对业务部门的新增需求，应该如何走变更控制流程
3. 小李调走属于什么类型风险？应如何应对
4. 集成商追加费用的要求应如何处理
5. 需求文档模糊带来的影响属于哪类不确定性（风险/模糊/复杂）

**参考答案**：

1. **绩效度量计算**：
   - CV = EV - AC = 600 - 850 = **-250万元**（成本超支）
   - SV = EV - PV = 600 - 750 = **-150万元**（进度滞后）
   - CPI = EV/AC = 600/850 ≈ **0.71**（<1，成本绩效差，每投入1元仅产生0.71元价值）
   - SPI = EV/PV = 600/750 = **0.80**（<1，进度绩效差，只完成80%的计划工作量）
   - **结论**：项目处于"成本超支+进度滞后"双重不利状态，需立即启动纠偏行动（如赶工、快速跟进、范围优化、采购协商、人员补强）

2. **变更控制流程**：
   - 业务部门提出新增需求（"领导驾驶舱"、"AI智能问答"）→ 提交**变更请求**（书面形式）
   - 项目经理王处长初步评估变更影响（范围、进度、成本、质量、风险五维度）
   - 提交**变更控制委员会（CCB）**审批（本项目政府背景，CCB应由信息处领导+业务部门+承建方+监理方代表组成）
   - CCB审批通过后，更新**项目管理计划**、**范围基准**、**成本基准**、**进度基准**和**需求跟踪矩阵**
   - 通知相关方执行变更，并记录到**变更日志**中
   - 若涉及合同变更，还需走**合同变更控制程序**，可能产生索赔，签订合同变更令

3. **人员流失风险分析**：
   - 性质：**项目风险**（具体风险中的资源风险/人员风险类，PMBOK第七版归为"团队绩效域"的不确定性）
   - 应对策略：执行**风险应对计划**中的"减轻"策略——启动资源日历调整、知识转移（让小李完成知识交接文档、录制关键模块操作视频）、紧急招聘或内部调配、考虑将该部分工作**外包**或**转移**
   - 同时启动**风险登记册**更新，进行二次风险评估（人员流失是否还可能再次发生）
   - 防范二次风险：建立**AB角**制度，关键岗位设置备份人员；建立**知识库**避免单点依赖

4. **集成商追加费用处理**：
   - 这是合同**变更请求**或**索赔**事项，需按合同条款处理
   - 王处长应审核合同条款：合同是固定总价（FFP）还是成本补偿（CPFF/CPIF）？是否含价格调整条款？
   - 若是**固定总价合同（FFP）**：供应商应承担物价波动风险，原则上不予追加，但可通过友好协商签订补充协议
   - 若是**成本补偿合同**：需提供涨价证据（厂商通知、合同外购凭证等），按合同约定的成本计算方式追加
   - 处理结果须经**CCB**审批，涉及金额较大（>5%成本基准）应报**上级主管部门**批准
   - 签订**变更令**（Change Order）后方可执行。**重要原则**：未签变更令不能付款，否则失控

5. **需求模糊性分析**：
   - 这属于**模糊性（Ambiguity）**不确定性，而非传统"风险"
   - 应对方法：①组织需求研讨会（联合应用开发JAD会议）②建立**需求原型**让业务方确认③采用**用户故事+验收标准**形式细化需求④使用**MoSCoW**优先级法（Must/Should/Could/Won't）⑤无法澄清的部分纳入**假设条件与制约因素日志**进行跟踪⑥使用**原型法/迭代法**让用户在使用中明确需求
   - **核心理念**：PMBOK第七版认为，许多项目失败并非因为"风险"未识别，而是因为"模糊性"未被管理——这是绩效域升级的重要意义

**踩分要点**：
- 第1题：必须写出CPI/SPI公式，结论必须含"超支""滞后"两词，并给出EAC预测
- 第2题：变更流程要点必须包含"变更请求→CCB→更新基准→变更日志"四步
- 第3题：风险类别和应对策略（减轻）要同时答出，并体现"防二次风险"意识
- 第4题：要分合同类型讨论，体现合同管理意识，强调"未签变更令不付款"
- 第5题：必须明确指出"模糊性"——这是PMBOK第七版绩效域的新考点，必须区别于"风险"

## 记忆口诀

- **八大绩效域口诀**："**团开规工交度不沟**"——团队、开发、规划、项目工作、交付、度量、不确定性、沟通（干系人）
- **十二项原则口诀**："**管家团队干系人价值系统领导环境质量风险适应变革**"（前4为人员相关，中间4为系统相关，后4为不确定性相关）
- **挣值三值口诀**："**P-E-A**"——Plan(计划值PV)、Earned(挣值EV)、Actual(实际成本AC)
- **CPI/SPI判断**："**>1绿，=1黄，<1红**"——绩效指数小于1亮红灯
- **风险类型口诀**："**风险可见，模糊看不清，复杂理还乱**"
- **冲突管理口诀**："**回避包容妥协，竞争合作最优**"（合作/共赢为上策）
- **塔克曼团队五阶段**："**形震规成解**"——形成、震荡、规范、成熟、解散
- **风险应对七策略**："**避转减接受，开分提**"（消极4+积极3）
- **度量四步法**："**对象→指标→数据→决策**"——识别对象、定义指标、收集数据、分析决策
- **十二项原则三组分类**："**人系不**"——人员（管家/团队/干系人/领导）+ 系统（价值/系统/质量/环境）+ 不确定性（风险/复杂/适应/变革）
